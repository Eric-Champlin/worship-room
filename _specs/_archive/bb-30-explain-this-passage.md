## Overview

Add an "Explain this passage" feature to the verse action sheet from BB-6. A user who taps a verse and picks "Explain this passage" gets a sub-view with a thoughtful, denominationally neutral scholarly explanation generated by Google Gemini 2.5 Flash-Lite using a carefully engineered system prompt that establishes a scholar-not-pastor voice with explicit rules against proselytizing, prescription, and pastoral ventriloquism. Every passage in scripture is handled by the same prompt and the same model. If future usage reveals specific passages where the LLM consistently produces unacceptable output, a future spec can add targeted curated overrides.

This is the first AI-powered feature in the wave. It establishes the Gemini client pattern and the prompt engineering pattern that BB-31, BB-32, and the eventual Ask AI migration will build on.

### Why this matters more than it sounds

Most Bible apps with AI features use the LLM as a shortcut around real curation. They prompt the model with something like "explain this verse in a Christian way" and ship whatever comes back. The results are predictable: theological platitudes, culturally Protestant interpretations presented as universal truth, aggressive evangelism dressed up as explanation, and a complete absence of the scholarly or historical context that would actually help a user understand the text.

Worship Room has been building trust with vulnerable users through BB-22 through BB-25 by writing devotional content with honesty, restraint, and respect for the user's emotional state. BB-30 is where that trust gets tested against the efficiency temptation of "just let the LLM handle it." If the AI-generated explanations feel like a different app speaking — louder, more confident, more preachy — users will feel the whiplash and trust will erode.

The spec's job is to make sure the LLM output sounds like Worship Room's voice: restrained, grounded, honest about what scripture does and doesn't say, never rushing past uncertainty to a neat resolution. This is prompt engineering as content work.

There's also a deeper argument: "explain this passage" is the feature that lets users without biblical training actually engage with scripture they find confusing. A user who opens Leviticus 19 and wonders why there are rules about not wearing mixed-fabric clothing alongside "love your neighbor as yourself" deserves an explanation that takes the question seriously — not a lecture about dietary laws and certainly not a deflection. Scholars have real answers about genre, audience, historical context, and redaction history; most Bible apps hide those answers behind paywalls or inside denominational framings. BB-30 makes scholarly honesty available to everyone, for free, in the same reader where they're already reading.

## User Story

As a **logged-out visitor or logged-in user** reading a passage in the Bible reader, I want to **tap a verse and request a thoughtful scholarly explanation** so that **I can understand what the text is actually doing in its own context without being preached at, proselytized to, or handed denominational platitudes**.

---

## Architecture: LLM-only

Every passage in scripture is explained at runtime by Gemini 2.5 Flash-Lite. There is no two-tier lookup, no override check, no fallback path. The `generateExplanation` function takes a reference and verse text, constructs the prompt, sends it to Gemini, and returns the result.

Future specs MAY add targeted overrides for specific passages if real-world usage reveals the LLM consistently fails on certain cases. The pattern for doing so is documented in the "Future work" section near the end of this spec but is **not built in BB-30**. BB-30 ships as a pure LLM feature.

## LLM provider and model

**Provider:** Google Gemini, via the Google Gen AI SDK for JavaScript/TypeScript.

**Package:** Verify the current canonical package name against Google's documentation before installing. As of recent documentation this should be `@google/genai`, but CC must confirm against <https://ai.google.dev/gemini-api/docs/sdks> during the plan update. Do not guess; verify.

**Model:** `gemini-2.5-flash-lite`

**Rationale:** cost. Gemini 2.5 Flash-Lite is currently priced at $0.10 per million input tokens and $0.40 per million output tokens — roughly an order of magnitude cheaper than Claude Sonnet or GPT-4 class models. A typical BB-30 request (~500 input tokens, ~400 output tokens) costs approximately $0.0002. A thousand requests costs about $0.20. Production scaling is not a meaningful cost concern at this price point, which is important for a free non-commercial app with no monetization plan.

The tradeoff: Flash-Lite has weaker prompt adherence than larger models. It's more likely to drift into preachy language, violate length constraints, or soften its handling of difficult passages despite explicit prompt instructions. The spec mitigates this with two strategies:

1. More defensive prompt engineering with specific counter-examples to common failure modes
2. More rigorous prompt testing during execution (iterate 2-3 times if needed, not one-shot success)

BB-30 accepts the residual risk on morally difficult passages and ships without a curated-override gating mechanism. See the "Future work" section for how targeted overrides could be added later if real-world usage reveals consistent LLM failures.

**Why not Gemini 2.0 Flash-Lite:** Deprecated. Google's official documentation warns it will be shut down on June 1, 2026. Building against it would break the feature within weeks.

**Why not Gemini 3.1 Flash-Lite:** Currently in preview status. API contracts may change with minimal notice. Stable models are safer for production code. BB-30 can upgrade later once 3.1 Flash-Lite reaches general availability.

**Why not Gemini 2.5 Flash (non-Lite) or 2.5 Pro:** Cost. For BB-30's scholarly explanation task, Flash-Lite is adequate. A future spec could upgrade to Flash if prompt iteration reveals quality gaps — the cost delta is about 3x input and 6x output, still cheap in absolute terms. Document this as a future decision in the deep review backlog.

**Why not Claude, GPT-4, or any other provider:** The user specifically chose Gemini for cost reasons. Not a technical argument — a product decision already made. Don't revisit this in the plan phase.

The exact model string `gemini-2.5-flash-lite` should be hardcoded as a constant in the client file with a comment explaining why this specific model was chosen and linking to the deprecation notice for 2.0 Flash-Lite. CC must verify the string against <https://ai.google.dev/gemini-api/docs/models> during the plan update to make sure Google hasn't changed the identifier since April 2026.

## The Gemini client

A new module at `frontend/src/lib/ai/geminiClient.ts`.

### Responsibilities

- Read the API key via `requireGeminiApiKey()` from `src/lib/env.ts`. Do NOT read `import.meta.env.VITE_GEMINI_API_KEY` directly. The indirection through `env.ts` is intentional — it gives clear error messages when the key is missing, makes key rotation touch one file, and matches the centralized env module pattern the foundation task established.

- Initialize the Google Gen AI SDK client lazily, on first use, not at module load time. Module-load initialization would throw for users who don't have a key configured, which defeats the whole point of the require-on-use pattern in `env.ts`. The client is constructed inside the `generateExplanation` function (or memoized on first call) so that users who never tap "Explain this" never trigger key validation.

- Export a single public function with this signature: `generateExplanation(reference: string, verseText: string): Promise<ExplainResult>`, where `ExplainResult` is a single shape: `{ content: string, model: string }`. There is no discriminated union and no `source` field — every result comes from the LLM, so there is nothing to discriminate. The `model` field identifies which Gemini model was used and is useful for tests and future analytics.

- The function internally:
    1. Lazily initializes the Gemini SDK client if not already initialized
    2. Constructs the request with the system prompt (from `explainPassagePrompt.ts`) and a user prompt template containing the reference and verse text
    3. Sends the request via the SDK with temperature 0.7, maxOutputTokens 600, and default safety settings
    4. Parses the response, distinguishing successful generation from safety-block refusals
    5. Returns the LLM result on success, or throws a typed error on failure

- **Critical Gemini-specific detail:** the system prompt must be passed via the SDK's system instruction parameter, NOT prepended to the user message. Gemini has noticeably worse prompt adherence when system content is in the user turn, and this matters more for Flash-Lite than for larger models. CC must verify during the plan update which parameter in the current SDK handles system instructions, and route the system prompt through that parameter.

### Error handling

The client throws typed error classes that map to user-facing error states:

- `GeminiNetworkError` — the SDK or underlying fetch failed before reaching the API (offline, DNS failure, etc.). User message: "Couldn't load an explanation right now. Check your connection and try again."

- `GeminiApiError` — the API returned a non-success response (invalid key, quota exceeded, service down, 5xx error). User message: "This feature is temporarily unavailable. Try again in a few minutes."

- `GeminiSafetyBlockError` — the API returned successfully but the response was blocked by safety filters. This can happen on morally difficult passages like Judges 19, Psalm 137:9, 1 Samuel 15:3, Joshua 6, or any other passage where Gemini's safety layer refuses to generate output. User message: "This passage is too difficult for our AI helper to explain well. Consider reading a scholarly commentary or asking a trusted teacher."

- **Critical Gemini-specific detail:** Gemini returns safety blocks as part of the response payload, not as an error status code. The client must parse the response correctly to distinguish "LLM responded successfully with content" from "LLM refused via safety block." Look for the `promptFeedback.blockReason` field (or whatever the current SDK exposes for this) and check whether the response's candidates array is empty or contains only blocked results.

- `GeminiTimeoutError` — the request took longer than 30 seconds. User message: "The request took too long. Try again in a moment." with a retry button. 30 seconds is the timeout budget; do not shorten it, because Flash-Lite can occasionally be slow on longer prompts.

- `GeminiKeyMissingError` — `requireGeminiApiKey()` threw because the key is not in the environment. Logged to the developer console with a clear message pointing at `frontend/.env.local`. Shown to the user as the generic "temporarily unavailable" message (do not expose the specific "key missing" detail — it's a developer concern, not a user concern).

### What the client does NOT do

- No caching. BB-32 will add caching. Every request hits Gemini fresh. This is acceptable because Flash-Lite is cheap enough that re-running the same explanation costs fractions of a cent.
- No client-side rate limiting. BB-32 will add this. BB-30 trusts Gemini's server-side rate limits as a last-resort backstop.
- No retry logic. A failed request surfaces the error to the user, who can tap retry. Automatic retries mask bugs and produce ambiguous error states.
- No streaming. The full response is awaited before being rendered. Streaming would be nicer UX but adds complexity. Defer to a future polish spec.
- No multi-turn conversation. Single request, single response. Users cannot follow up with "tell me more." The sub-view is a one-shot reading aid, not a chatbot.
- No telemetry. The client does not log prompts, responses, or user behavior to any backend. Requests go directly from the browser to Gemini with no intermediate server.
- No development-mode mock. The client always calls Gemini. There is no "mock mode" toggle. Developers testing without a key see the `GeminiKeyMissingError` path and can add a key to their `.env.local` to test real behavior.

## The prompt engineering

This is the load-bearing section of the feature.

### The system prompt

Stored in `frontend/src/lib/ai/prompts/explainPassagePrompt.ts` as an exported constant named `EXPLAIN_PASSAGE_SYSTEM_PROMPT`. The exact text of the prompt (do not "improve" this during planning):

> You are a thoughtful biblical scholar helping a user understand a scripture passage they're reading in the World English Bible. Your explanations are grounded in scholarship — historical context, literary genre, linguistic observations, and honest acknowledgment of interpretive difficulty.
>
> You are not a pastor. You are not a preacher. You do not proselytize. You do not assume the user is a Christian or tell them what they should believe. You serve users across the full spectrum of religious backgrounds, including those deconstructing their faith and those who have been hurt by religious communities.
>
> Your explanations follow these rules:
>
> 1. Lead with context. What kind of text is this (narrative, poetry, letter, law, prophecy)? Who wrote it, when, and to whom? What was happening in the world of the text?
>
> 2. Then explain what the passage is doing. What is the author's argument, story, or concern? What literary or rhetorical moves are being made? What does the passage mean in its own context, before we ask what it means for us?
>
> 3. Acknowledge uncertainty honestly. If scholars disagree about a passage, say so. If the Hebrew or Greek is ambiguous, say so. If the passage has been read in multiple ways across Christian traditions, say so. Do not paper over difficulty with confident platitudes.
>
> 4. Do not prescribe application. Do not tell the user what to do, feel, or believe. Explain what the text says and let the user decide what to do with it. Do not end with "so what does this mean for you" or any variant.
>
> 5. Avoid triumphalism. Do not say "this proves," "this shows us," "this means we must," or similar. Use tentative, scholarly language: "scholars suggest," "this passage likely," "one reading of this is."
>
> 6. Stay in the text. Do not bring in external theological doctrines the passage doesn't directly address. Do not invoke systematic theology categories unless the passage explicitly engages them.
>
> 7. Acknowledge hard passages honestly. If the passage depicts violence, slavery, patriarchy, or other troubling material, acknowledge it plainly. Do not defend, explain away, or spiritualize. Say what the text says, note the scholarly consensus or disagreement on how to read it, and leave the moral assessment to the user.
>
> 8. Be restrained in length. Explanations should be 200-400 words, not 800. Users are reading on a phone screen. Every sentence earns its place.
>
> 9. Never use the phrases "God wants you to," "God is telling you," "the Lord is saying to your heart," "God is calling you to," or any variant that speaks for God to the user. These phrases are presumptuous and violate the user's agency.
>
> 10. Never recommend prayer, church attendance, further study, or spiritual practices unless the user specifically asks about these topics. The user asked for explanation, not instruction.
>
> Respond with the explanation text only. Do not include a greeting, a summary header, a closing question, or any framing text. Just the explanation.

### The user prompt template

Stored in the same file as a function `buildExplainPassageUserPrompt(reference, verseText)` that returns a string in this shape:

> Explain this passage from the World English Bible:
>
> [REFERENCE]
> [VERSE TEXT]
>
> Give me the context, what the passage is doing in its own setting, and any interpretive difficulties an honest reader should know about.

Where `[REFERENCE]` is the formatted reference (e.g., "1 Corinthians 13:4-7") and `[VERSE TEXT]` is the actual WEB translation text for that range.

### Why the prompt works

The system prompt establishes three things the off-the-shelf LLM behavior fails on:

1. **Scholar, not pastor.** Framing the LLM as a biblical scholar rather than a spiritual advisor immediately changes the default register. Scholars lead with context and uncertainty; pastors lead with application and confidence. We want the first voice.

2. **Explicit rules against common failure modes.** Rules 4, 5, 9, and 10 are direct counters to the specific behaviors that make AI Bible commentary bad. "Do not prescribe application" prevents the LLM from ending every explanation with "so what does this mean for you today?" "Never use the phrases 'God wants you to'" prevents pastoral ventriloquism, which is the worst failure mode. Rules that name failure modes prevent them.

3. **Permission to acknowledge difficulty.** Rule 7 explicitly gives the LLM permission to say "this passage is hard" without needing to resolve it. Most LLM prompts implicitly pressure the model to produce a satisfying, complete answer. BB-30's prompt explicitly removes that pressure and the output quality goes up as a result.

### Prompt testing methodology (expanded for Flash-Lite)

Because Flash-Lite has weaker prompt adherence than larger models, prompt testing needs to be more thorough than a single-pass test.

**Test passages (8 total):**

Note: the four morally hardest passages in scripture (Joshua 6, Psalm 137:9, Judges 19, 1 Samuel 15:3) are deliberately **not** in the prompt testing list. They will be handled by the LLM at runtime with the same prompt and are not individually verified before ship. Rationale: if they were in the testing list, a failing test would force a decision about writing curated overrides for the failing cases, which is the thing this spec explicitly defers. Not testing them is a deliberate choice to ship without a gating mechanism on those specific passages and trust the prompt + disclaimer to handle the risk.

Easy cases:

- John 3:16 (most-quoted verse, tests whether the LLM can say something non-obvious)
- Psalm 23:1 (famous metaphor, tests whether the LLM explains pastoral imagery without being preachy)
- 1 Corinthians 13:4-7 (love chapter, tests whether the LLM avoids wedding-speech clichés)
- Philippians 4:6-7 (the anxiety verse from BB-24, tests whether the LLM acknowledges historical context rather than weaponizing it)

Medium cases:

- Leviticus 19:19 (mixed fabrics law, tests whether the LLM explains ancient purity categories)
- Genesis 22:1-2 (Abraham and Isaac, tests whether the LLM handles the binding narrative honestly)
- 1 Timothy 2:11-12 (women silent in church, tests whether the LLM honestly addresses patriarchy without defending it)
- Romans 1:26-27 (same-sex passages, tests whether the LLM handles contested interpretation without taking a side)

**The testing process:**

1. Write the prompt (starting version is the one above)
2. For each of the 8 test passages, manually run a request through the Gemini client (via a test script, a dev-mode REPL, or the actual feature in dev)
3. Capture each output verbatim in `_plans/recon/bb30-prompt-tests.md`
4. Evaluate each output against the 10 rules in the system prompt
5. Flag any violation
6. If more than 2 of the 8 outputs violate any rule, iterate on the prompt — add a more specific counter-example or a stricter rule — and re-test all 8 from scratch
7. Continue until no more than 1 of the 8 outputs has any rule violation
8. Document the final prompt version and the passing test outputs in `_plans/recon/bb30-prompt-tests.md`
9. If after 3 iterations the prompt still fails on more than 1 of 8, document the failures and escalate to the user — this may indicate Flash-Lite is insufficient and a model upgrade is needed

Specific Flash-Lite failure modes to watch for during testing:

- **Application drift:** drifting into application/prescription despite rule 4, often in the last paragraph. Counter-rule: "Do not end with application. End with the scholarly observation, not with 'what this means for you.'"
- **Pastoral ventriloquism:** using "God is telling you" or similar despite rule 9. Counter-rule: add more specific examples of forbidden phrases.
- **Length drift:** producing output longer than 400 words despite the soft cap. Counter-rule: "The maximum length is 400 words. Count your words. If you exceed 400, cut ruthlessly."
- **Softening on medium passages:** softening the acknowledgment of difficulty on passages like Leviticus 19:19 or Romans 1:26-27. Counter-rule: "For passages that modern readers find morally difficult, name the difficulty in the first or second sentence, not at the end."

Each iteration adds a more specific rule or counter-example until the output is clean.

## The UI surface

### Entry point

The verse action sheet from BB-6 currently has actions like Highlight, Bookmark, Note, Cross-references, Pray, Journal, Meditate, Share. BB-30 adds a new action: **Explain this passage**.

Placement: near the top of the action sheet, above the Daily Hub bridges (Pray/Journal/Meditate) but below the direct text actions (Highlight/Bookmark/Note/Cross-references). The order signals hierarchy: text-level actions first, explanation next, application later.

Icon: a small question mark or lightbulb glyph in the same visual language as the other action icons.

### ExplainSubView pattern

Critical: ExplainSubView follows the `CrossRefsSubView` pattern, NOT the `NoteEditorSubView` pattern. Here's why it matters.

`NoteEditorSubView` renders its own back button, title, and close X chrome on top of what `VerseActionSheet` already provides, which creates a double-header bug (two stacked headers, two back buttons, visually broken). `CrossRefsSubView` does NOT render its own chrome — it starts with a div containing subtitle, context strip, divider, content, footer. All the sheet chrome (back button, title, close X) is provided by `VerseActionSheet`.

ExplainSubView MUST follow `CrossRefsSubView`, not `NoteEditorSubView`. The component code should start with a div wrapping the content, with no header JSX inside the component itself. The sheet chrome comes from `VerseActionSheet`'s existing rendering.

### Sub-view contents

The ExplainSubView renders (top to bottom):

- **Context strip:** the passage reference (e.g., "1 Corinthians 13:4-7") and the verse text, displayed in a distinct visual treatment — slightly larger text, indented or bordered, so the user can see what they're reading about
- **Divider**
- **Loading state:** a skeleton or small spinner with the text "Thinking..." or similar while the Gemini request is in flight
- **Content:** the LLM-generated explanation text rendered as plain text with paragraph breaks. No markdown parsing, no HTML rendering, no embedded links. Just text with double-newline paragraph breaks.
- **Disclaimer:** a small muted disclaimer at the bottom of the explanation

### The disclaimer (exact text — do NOT "improve")

"This explanation was generated by an AI trained on biblical scholarship. It's one reading of the passage, not the only one. Use it as a starting point, not a final answer."

This disclaimer appears below every explanation. It is **not dismissible** and **not hidden behind settings**. There is only one disclaimer variant because there is only one content source (the LLM).

### Error states

- **Network failure:** "Couldn't load an explanation right now. Check your connection and try again." with a retry button.
- **Gemini API error:** "This feature is temporarily unavailable. Try again in a few minutes."
- **Gemini safety block:** "This passage is too difficult for our AI helper to explain well. Consider reading a scholarly commentary or asking a trusted teacher."
- **Timeout:** "The request took too long. Try again in a moment." with a retry button.
- **Key missing:** shown as the generic "temporarily unavailable" message (developers see the specific error in the console).

### Accessibility

The sub-view is fully keyboard navigable. The explanation text is read by screen readers in full. The disclaimer is read as part of the explanation flow, not skipped. Focus returns to the verse action sheet main view when the sub-view closes.

## Request flow

When the user taps "Explain this passage":

1. The action sheet transitions from its main view to the ExplainSubView (using `VerseActionSheet`'s existing sub-view navigation pattern)
2. The ExplainSubView opens in loading state
3. The `useExplainPassage` hook is called with the selected verse reference and text
4. The hook calls the Gemini client's `generateExplanation` function
5. The client lazily initializes the Gemini SDK (if not already initialized) and sends the request
6. On success, the response text is rendered in the sub-view with the disclaimer
7. On error, the appropriate error state is shown
8. The user reads the explanation and taps back to return to the action sheet main view, or closes the sheet entirely

## AI Safety Considerations

**Crisis detection needed?** No. BB-30 does not take free-text user input. The input to the Gemini call is a Bible reference and the corresponding WEB translation text — both are app-controlled data, not user-generated content. Crisis detection is not applicable because there is no user text to scan.

**User input involved?** No — the only "input" is the user's choice of which verse to explain, which comes from their existing selection in the reader.

**AI-generated content?** Yes. Gemini output is rendered as **plain text only**, with paragraph breaks preserved via `white-space: pre-wrap` or similar. **No markdown parsing, no HTML rendering, no `dangerouslySetInnerHTML`.** This matches the project's standing rule in `.claude/rules/02-security.md` that all AI output is treated as untrusted input and escaped on render.

**Content boundaries** (from `.claude/rules/01-ai-safety.md`):

- The system prompt's rules 4, 5, 7, 9, and 10 directly implement the project's theological boundaries: no "God is telling you" language, no pastoral prescription, no denominational bias, no definitive life advice.
- The LLM is instructed to be tentative ("scholars suggest," "one reading of this is") rather than authoritative.
- Rule 7 requires honest acknowledgment of hard passages rather than defense or spiritualization.

**Disclaimer** is mandatory on every explanation. It is not dismissible. This satisfies the project's standing requirement for disclaimers on AI-generated content. See the "Future work" section for how curated overrides could be added later if the prompt proves insufficient for specific passages.

**Safety block handling:** If Gemini's safety filter blocks a response, the user sees a graceful fallback ("This passage is too difficult for our AI helper to explain well. Consider reading a scholarly commentary or asking a trusted teacher."), not a raw error.

## Auth & Persistence

- **Logged-out users:** Full access. No database writes, no localStorage writes, no cookies, no anonymous IDs. Session-only. Every request hits Gemini directly; no persistence.
- **Logged-in users:** Identical to logged-out users. No personal layer, no history, no "passages I've asked about." Ephemeral by design.
- **localStorage usage:** **None.** BB-30 introduces zero new localStorage keys.
- **Database:** **None.** No new tables, no new columns, no backend persistence of any kind.
- **Telemetry:** **None.** No prompt logging, no response logging, no analytics on which passages get explained, no cost tracking at the client level (BB-32 will add server-side cost tracking if needed).

## Design Notes

- Follow the existing Bible reader visual language for the sub-view. Match `CrossRefsSubView`'s layout structure exactly — context strip, divider, content, footer — since that is the established pattern the sheet chrome was designed around.
- **Do not** replicate `NoteEditorSubView`'s self-rendered chrome. It causes a double-header bug. This is a specific architectural lock-in for BB-30.
- Typography for the explanation body should match the reader's body text defaults (serif or sans per user preference, theme-aware, respects type-size preference from `wr_bible_reader_type_size`).
- Context strip typography should match existing scripture callout patterns in the reader (slightly larger, indented or bordered).
- Disclaimer typography is muted — small, lower-contrast than the body — but still legible (not failing contrast checks). No raw hex values; use design tokens.
- Loading skeleton should feel in keeping with the calm reading atmosphere: subtle, not flashy. Respects `prefers-reduced-motion`.
- Zero raw hex values anywhere in BB-30 code. Use tokens from `.claude/rules/09-design-system.md` or the design system recon in `_plans/recon/design-system.md`.
- The "Explain this passage" action item in `VerseActionSheet` uses the same icon sizing, padding, and label typography as existing action items (Highlight, Bookmark, Note, Cross-references, etc.).

**New visual patterns introduced by BB-30:** none. Everything reuses existing reader and sheet patterns.

## Critical edge cases

### The user taps "Explain this" on a range spanning multiple verses

If the user has selected a multi-verse range (e.g., 1 Corinthians 13:4-7), the prompt includes all selected verses in the user prompt template. The LLM explains the range as a unit. If the range is very long (more than 20 verses), cap it at 20 verses or show an error "Please select 20 or fewer verses to explain." In practice, the action sheet from BB-6 probably already caps selection, so this edge case may be unreachable.

### The LLM returns content that violates the prompt rules

This will happen sometimes with Flash-Lite. BB-30 does NOT try to filter or post-process the LLM output — that's a losing game. Instead, the prompt testing methodology catches systematic violations and iterates on the prompt. Occasional violations are acceptable for v1. A future spec could add output filtering if needed.

### The user asks for an explanation of a passage they've already asked about

BB-30 doesn't cache, so every request hits the LLM. BB-32 will add caching. Until then, the user might notice slightly different explanations if they ask twice — LLM output is non-deterministic by default. This is acceptable.

### The Gemini API is rate-limited or down

The error state handles this. The user sees "temporarily unavailable" and can retry later.

### The user has no internet connection

The sub-view shows the network error state immediately. No offline fallback in BB-30 — every passage requires the LLM and a live connection.

### The user is logged out

The explain feature works for logged-out users. It's a read-only enhancement to the reading experience, not a personal data feature. Zero auth gates. No `useAuth()` calls, no auth modal, no localStorage writes.

### The LLM returns an extremely long response despite the word limit

The prompt asks for 200-400 words but LLMs don't always obey length constraints. If a response exceeds, say, 800 words, the sub-view still shows the full response — don't truncate. Users who get a long answer can scroll. The word limit is a soft guide, not a hard cap enforced by the client.

### The API key is missing from the environment

`requireGeminiApiKey()` throws, the client catches the error as `GeminiKeyMissingError`, logs the specific error to the developer console, and shows the user the generic "temporarily unavailable" message. Developers see the fix in the console; users see a friendly error.

### The user taps "Explain this" on one of the morally hardest passages

Joshua 6, Psalm 137:9, Judges 19, and 1 Samuel 15:3 are handled by the LLM with the same prompt as every other passage. There is no special-case handling in BB-30. If Gemini's safety layer refuses to generate output for one of these, the user sees the `GeminiSafetyBlockError` fallback ("This passage is too difficult for our AI helper to explain well. Consider reading a scholarly commentary or asking a trusted teacher."). If Gemini generates output, the user sees it with the standard disclaimer. This is a deliberate risk the spec accepts — see the Future work section below.

## Out of Scope

- **"What does this mean for me"** — that's BB-31, which uses a deliberately different prompt with a more personal framing. BB-30 is scholarly; BB-31 is applied.
- **Caching** — BB-32.
- **Rate limiting** — BB-32.
- **Ask AI page migration** — future spec.
- **User feedback on explanations** (thumbs up/down, "this was helpful") — out of scope.
- **Personalization based on the user's religious background** — every user gets the same explanation.
- **Denominational tagging** ("Catholic reading," "Reformed reading," etc.) — the explanation is ecumenical.
- **Multi-turn conversation** — single request, single response.
- **Citations or references** — the LLM is not asked to cite sources.
- **Image generation** — text only.
- **Audio narration of explanations** — text only.
- **Translation of explanations into other languages** — English only.
- **Scholarly commentary integration** (Bible Gateway's commentaries, Logos, etc.) — LLM only.
- **Sharing of explanations** — the sub-view is a private reading aid.
- **Saving of explanations to the user's personal layer** — ephemeral. Users who want to save a takeaway write a note via BB-8.
- **Integration with reading plans** — works on any passage inside or outside a plan.
- **Request history** — users cannot see "passages I've asked about before."
- **AB testing of prompt variants** — one prompt, shipped.
- **User-editable prompt** — the prompt lives in source code.
- **Streaming responses** — full response, then render.
- **Logging of prompts or responses to any backend** — zero telemetry.
- **Analytics on which passages get explained most** — not tracking.
- **Cost tracking or budget alerts** — BB-32 adds this.
- **Fallback to a smaller/cheaper LLM** — one model, used for all requests.
- **Admin UI to view or edit the prompt** — prompt lives in source code.
- **New localStorage keys** — zero client-side storage for this feature.
- **Auth gates** — feature is available to logged-out users identically to logged-in users.
- **Backend proxy for the API key** — the key is exposed in the client bundle, protected by HTTP referrer restrictions configured in Google Cloud Console.
- **Fixing pre-existing failing tests** — the 46 failing tests the foundation task identified are outside scope. Any changes to the test suite should only be additions for BB-30's new code.

## Future work

BB-30 ships as a pure LLM feature with no curated content overrides. This is a deliberate choice to minimize authoring time and ship faster, accepting the risk that Gemini 2.5 Flash-Lite may occasionally produce suboptimal output on morally difficult passages (e.g., commanded violence in Joshua 6, 1 Samuel 15:3; vengeful lament in Psalm 137:9; sexual violence in Judges 19). If real-world usage reveals consistent LLM failures on specific passages, a future spec can add targeted curated overrides by creating `frontend/src/lib/ai/overrides/explainPassageOverrides.ts` with a matcher-based lookup that runs before the LLM call in `generateExplanation`. The Gemini client's architecture leaves room for this addition without requiring a rewrite — the override check would be a single function call at the top of `generateExplanation` before the LLM request is constructed. The disclaimer on every LLM-generated explanation signals AI provenance to users, providing a baseline level of transparency even for difficult passages.

## Notes for execution

- **The prompt is the product.** Do not let the plan phase shortcut on prompt testing. A prompt that works 70% of the time on Flash-Lite is a broken feature. Insist on the 8-case test methodology and document the results in `_plans/recon/bb30-prompt-tests.md`.
- **Flash-Lite needs more iteration than a larger model would.** Budget 2-3 prompt iteration rounds, not one-shot success. If after 3 rounds the prompt still fails on more than 1 of 8 cases, escalate to the user — this may mean Flash-Lite is insufficient and a model upgrade is needed.
- **The LLM-only architecture is the load-bearing decision.** No curated overrides, no two-tier lookup. Every passage goes to Gemini. Future specs MAY add targeted overrides if real-world usage reveals consistent failures, but BB-30 does not build that infrastructure.
- **The `src/lib/env.ts` indirection is non-negotiable.** The Gemini client must read the key via `requireGeminiApiKey()`, not via `import.meta.env` directly. This is the pattern established by the foundation task.
- **The Gemini SDK initialization must be lazy.** If the client initializes at module load time, users without a key get a crash instead of a clean error state. Initialize inside the function on first call.
- **The system prompt must go through the SDK's system instruction parameter.** Prepending it to the user message produces measurably worse adherence on Flash-Lite. Verify the correct SDK parameter during the plan update.
- **The disclaimer text is exact.** One variant. Non-dismissible. Do not let the plan phase "improve" it.
- **ExplainSubView follows `CrossRefsSubView`, not `NoteEditorSubView`.** This is a specific architectural decision to avoid the double-header bug. The plan phase should lock this in explicitly and the component code should have no header JSX.
- **Zero auth gates.** The feature is available to logged-out users identically to logged-in users. No `useAuth`, no `authModal`, no localStorage.
- **The BB-20 gating concern from BB-26 is still in the backlog.** If the plan phase encounters any auth-gating machinery during implementation, flag it but do not fix it. That's a BB-37b concern.
- **Pre-existing failing tests are out of scope.** Do not touch them. The 46 failing tests the foundation task identified are a separate concern.
- **The integration test:** After this spec ships, a user reading 1 Corinthians 13 can tap any verse, tap "Explain this passage," and get a thoughtful 300-word explanation of Paul's rhetorical context, the literary structure of the love chapter, and a brief honest note that the "love is patient, love is kind" passage has been flattened by wedding-speech quotation and is actually Paul's rebuke to a factional Corinthian church. Every passage uses the same LLM path. That's the test.
- **BB-31 is next and builds directly on BB-30's infrastructure.** Same Gemini client, different prompt, same sub-view pattern. BB-32 comes after that with caching and rate limiting.
