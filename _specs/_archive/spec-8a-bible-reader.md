## Overview

The Bible cluster's outer surfaces ŌĆö BibleLanding, BibleBrowse, MyBiblePage, PlanBrowserPage, BiblePlanDetail, BiblePlanDay, the TodaysPlanCard URL bug, and the `bg-primary` ŌåÆ `<Button variant="subtle">` rollout across enumerated Bible CTAs ŌĆö were all addressed in Specs 8B and 8C. Three `bg-primary` CTAs remained at the end of 8C, and all three live on BibleReader's validation-error pages: book-not-found ("Browse books"), chapter-not-found ("Go to Chapter N"), and load-error ("Try Again"). Spec 8A migrates exactly those three CTAs.

These three CTAs are the outliers because they sit inside BibleReader.tsx, which is a documented Layout Exception per `09-design-system.md` and direction Decision 13. The reader's chrome ŌĆö `ReaderChrome` instead of `Navbar`, no `SiteFooter`, root-level skip link, three reader-only theme variants, four type sizes, three line heights, focus mode, audio coordination, swipe gestures, keyboard shortcuts, AI Explain/Reflect side panels, BB-41 contextual notification prompt, and the chapter-mount tracking effect from Spec 8B ŌĆö is intentional immersive design that does not migrate to canonical patterns. Decision 13 is explicit about that boundary: BibleReader's chrome stays as it is. But validation-error surfaces (rendered when the user lands on `/bible/<invalid-book>/N`, `/bible/genesis/<invalid-chapter>`, or hits a transient load failure) are NOT reader chrome ŌĆö they're full-screen FrostedCard recovery states that happen to live inside BibleReader.tsx because the validation guards run after the route mounts. They sit on the same `bg-hero-bg` background as the rest of the app, render inside a `<FrostedCard>`, and have CTAs that should match the rest of the cluster. They were missed in 8C only because they're co-located with the reader chrome that Decision 13 protects.

After 8A ships, every enumerated `bg-primary` CTA across the entire Bible cluster has been migrated to `<Button variant="subtle">`. The Round 4 visual-consistency rollout's Bible-cluster phase is complete. The reader chrome itself stays per Decision 13. The chapter-mount effect from Spec 8B (`recordChapterVisit` + `markChapterRead` ŌĆö the BB-43 reading heatmap and progress-map data feed) is preserved exactly; this spec does not touch the effect or its dependencies. Decoration-tier `bg-primary` tints (e.g., `bg-primary/[0.15]` selection background in ReaderBody, `bg-primary/[0.6]` left-border accents in scripture callouts, the `bg-primary/N` decorative tints flagged in 8C as Decision 11 categorical signals) stay per Decision 11 ŌĆö those are not CTAs. The skip-link's `focus:bg-primary` at line 730 stays as the canonical keyboard-focus treatment. `duration-200` drift stays per Decision 12. Reader theme CSS variables (`var(--reader-bg)`) are unchanged.

The change is small enough that the entire diff fits in roughly 30-50 LOC of edits to `frontend/src/pages/BibleReader.tsx` plus class-string updates in tests. There are no new imports beyond `Button` (which is likely already imported ŌĆö a fast re-check during pre-execution recon confirms that), no new components, no new copy, no new auth gates, no new tests of new behavior, and no new accessibility patterns. The migration's value is consistency across the cluster: a logged-out user typing `/bible/notabook/1` into the URL bar sees the same recovery UI chrome as they'd see on every other Bible cluster surface. Today they see a stranded purple solid-fill button; after 8A, they see the same translucent frosted button they see everywhere else.

The pre-execution recon flagged that the three CTAs vary in semantic element type: CTA #1 is a `<button>` (the action is "open the BibleDrawer"), CTA #2 is a `<Link>` (the action is "navigate to the last valid chapter of this book"), and CTA #3 is a `<button>` (the action is "retry the load"). The `<Button variant="subtle">` migration handles button-typed CTAs directly via `<Button variant="subtle" type="button" onClick={...}>`. For the `<Link>`-typed CTA #2, the canonical pattern is `<Button variant="subtle" asChild>` wrapping the existing `<Link>` element ŌĆö the `Button` primitive supports `asChild` for exactly this composition (verified during pre-execution recon by reading the Button primitive's source at `frontend/src/components/ui/Button.tsx`). The `asChild` form preserves the Link's routing semantics (semantic `<a>` element rendered, `to` prop honored by React Router) while applying the Button chrome's class string and 44├Ś44 minimum tap target.

After this spec ships, the Bible cluster's Round 4 chrome migration is mechanically complete: every documented `bg-primary` CTA across `/bible`, `/bible/browse`, `/bible/my`, `/bible/plans`, `/bible/plans/:planId`, `/bible/plans/:planId/day/:dayNumber`, and BibleReader's validation-error pages renders with the canonical `<Button variant="subtle">` chrome. The reader chrome remains the documented Layout Exception. The bg-primary decorative tints remain per Decision 11. Future Bible-cluster work can build on a fully migrated foundation.

## User Story

As a **logged-out visitor who hand-typed `/bible/notabook/1`** into the URL bar (or who clicked an outdated external link), I want to recover gracefully into the Bible browser. The page renders a FrostedCard with "That book doesn't exist." copy and a "Browse books" button. Today the button is a saturated purple solid-fill rectangle that doesn't match the rest of the Bible cluster I just came from. After this spec ships, the button is the same translucent frosted button I see everywhere else on the site ŌĆö visual continuity across the recovery, no jarring chrome shift.

As a **logged-in user who navigated to `/bible/genesis/51`** by typo or stale bookmark (Genesis only has 50 chapters), I want a friendly "go to the last valid chapter" CTA that matches the rest of the cluster. After 8A, it's the canonical subtle Button ŌĆö same routing behavior, same dynamic chapter-number interpolation, same 44├Ś44 tap target, but the chrome matches the rest of the Bible cluster.

After 8A, it's the same subtle Button as every other recovery CTA on the site ŌĆö visually consistent with the user's expectation of how recovery surfaces look.

As a **`/code-review` or `code-quality-reviewer` operator on a future Bible-cluster spec**, I want the Bible cluster's `bg-primary` CTA migration to be mechanically complete. After 8A, those three CTAs are migrated, and the only remaining `bg-primary` matches inside BibleReader.tsx are the skip-link's `focus:bg-primary` (intentional, canonical) and any decorative tints (intentional, Decision 11). The audit surface is clean.

As a **future spec author working on BibleReader chrome** (e.g., a hypothetical reader virtualization spec, a future read-along animation spec, or a Phase 4 light-mode reader spec), I want Decision 13's Layout Exception boundary to be intact. Logged-in users continue to benefit from existing BibleReader persistence (chapter-mount effect from Spec 8B writes to `wr_chapters_visited` and `bible:streak`; reader preferences read from `wr_bible_reader_*` keys; etc.) on the happy path, but the validation-error pages do not read or write those keys ŌĆö the chapter-mount effect runs only AFTER `book` and `chapter` validation succeed.
- **localStorage usage:** No new keys. No changes to existing keys. The validation-error CTAs trigger navigation (CTA #2's Link, "ŌåÉ Back to Bible" Link) or component-local actions (CTA #1's `bibleDrawer.open()`, CTA #3's `loadChapterWeb` retry) ŌĆö none of which write to localStorage.
