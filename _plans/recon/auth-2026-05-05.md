# Worship Room — Auth Surfaces Recon Report

**Generated:** 2026-05-05
**Branch:** forums-wave-continued
**Scope:** Pre-spec reconnaissance for the Auth surface redesign (Spec 7). Covers `/register`, `/login`, `AuthModal` (the canonical auth shell), `TermsUpdateModal` (sibling overlay that shares the auth idiom), and the auth context/service plumbing that the redesign will inherit. **No code changes made.**

**Surprise finding:** The `/login` route is a stub (`ComingSoon`) and there is no standalone `/forgot-password` or `/reset-password` route. ALL real auth — login, register, forgot-password (toast-stub) — flows through `AuthModal` (671 LOC), which is opened from 60+ call sites across the app. The redesign work is therefore concentrated in **two files** (`AuthModal.tsx` + `RegisterPage.tsx`) with one sibling pattern (`TermsUpdateModal.tsx`) that should move in lockstep, plus deprecated-pattern cleanup on the navbar/mobile-drawer auth CTAs and on the `ComingSoon` stub itself.

## Part 4 — Form Patterns

### Input field chrome (canonical, used for every input on AuthModal + TermsUpdateModal)

```
w-full rounded-xl bg-white/[0.06] border border-white/[0.12] px-3 py-2.5 text-sm text-white
placeholder:text-white/40
focus-visible:outline-none focus-visible:border-purple-400/50 focus-visible:shadow-[0_0_15px_rgba(139,92,246,0.15)]
```

This is the **utility input idiom** — flatter chrome than the Daily Hub textarea glow (which uses `shadow-[0_0_20px_rgba(167,139,250,0.18),0_0_40px_rgba(167,139,250,0.10)] border border-violet-400/30 bg-white/[0.04]`). Both are violet-themed but the textarea is significantly more decorative. **Spec 7 must decide whether auth inputs adopt the textarea glow idiom (more "trust moment" emphasis) or stay utility.** See Open Questions.

### Label pattern

```tsx
<label htmlFor="auth-email" className="mb-1 block text-sm font-medium text-white">
  Email<span className="text-purple-400 ml-0.5" aria-hidden="true">*</span><span className="sr-only"> required</span>
</label>
```

- Visible asterisk in `text-purple-400`, `aria-hidden="true"`
- `<span class="sr-only"> required</span>` for screen readers — canonical accessibility pattern
- Every `<label>` uses `htmlFor` paired with `id` on the input — verified across all 6 input fields (firstName, lastName, email, password, confirmPassword, resetEmail)

### Validation timing

| Field | When error appears |
|---|---|
| firstName, lastName, confirmPassword | After blur OR after first submit attempt — gated by `(touched.<field> || submitted) && error` |
| email, password | Set on submit only (no blur validation); cleared on input change |
| resetEmail | Set on submit only |

**Inconsistency to flag:** firstName/lastName/confirmPassword use the touched+submitted gating, but email/password do not — they only show errors after submit. This makes blur-validation behavior asymmetric across fields. Spec 7 may want to unify (either all blur+submit, or all submit-only).

### Per-field error rendering

```tsx
{(touched.firstName || submitted) && firstNameError && (
  <p id="firstname-error" role="alert" className="mt-1 flex items-center gap-1.5 text-sm text-red-400">
    <AlertCircle className="h-4 w-4 shrink-0" aria-hidden="true" />
    {firstNameError}
  </p>
)}
```

- `role="alert"` on every per-field error
- `aria-invalid` and `aria-describedby` wired on every input
- `text-red-400` is the canonical error tone for inputs (NOT `text-red-100` which is reserved for FormError severity wrappers). Tested at `AuthModal.test.tsx:106`.

### Form-level error rendering (FormError)

Uses the canonical `<FormError>` primitive (severity="error") wrapped in a focusable container:

```tsx
<div ref={formErrorRef} tabIndex={-1} className="mb-3 focus:outline-none">
  <FormError severity="error" onDismiss={() => setFormError(null)}>
    {formError.message}
  </FormError>
</div>
```

After API failure, `requestAnimationFrame(() => formErrorRef.current?.focus())` moves focus to the banner — canonical focus management per `09-design-system.md` § "Accessibility Checklist".

### Required-field indicator pattern

`<span className="text-purple-400 ml-0.5" aria-hidden="true">*</span>` followed by `<span className="sr-only"> required</span>`. Used on all 6 fields including the reset-email field.

### Password visibility toggle

**Not implemented.** Both password and confirmPassword inputs are `type="password"` with no eye-icon toggle. Common modern pattern; absence is a likely Spec 7 ask.

### Server-side field error mapping

`applyServerFieldErrors(fieldErrors)` (lines 146-151) maps backend `fieldErrors` keys onto the local error state — `email`, `password`, `firstName`, `lastName` only. Backend `VALIDATION_FAILED` errors with field-specific messages flow into the per-field UI; non-field errors land in the FormError banner via `AUTH_ERROR_COPY[err.code]`.

### AuthError code → copy map (`types/auth.ts`)

```ts
INVALIDCREDENTIALS  → "That email and password don't match our records. Try again, or use Forgot Password."
VALIDATION_FAILED   → "Please check the fields below."
RATE_LIMITED        → "Too many attempts. Please wait a moment and try again."
ACCOUNT_LOCKED      → "This account is temporarily locked. Please try again in a few minutes."
AUTO_LOGIN_FAILED   → "Your account is ready. Please log in to continue."
NETWORK_ERROR       → "We couldn't reach the server. Check your connection and try again."
VERSION_MISMATCH    → "The terms updated again while you were reading. Please review and accept the latest versions."
UNKNOWN             → "Something didn't go through. Please try again."
```

All copy passes the Anti-Pressure Checklist (no exclamation points near vulnerability, sentence case, period-terminated, blameless framing). Spec 7 should preserve this.

### Auth state pattern

`AuthProvider` wraps the app at the highest level (above `LegalVersionGate`, above `AuthModalProvider`). It reads initial state via `readInitialState()` (AuthContext.tsx:81-101):

1. If `wr_jwt_token` is present → real-mode boot, `isAuthResolving: true` until `getCurrentUser()` settles
2. Else if `wr_auth_simulated === 'true'` → legacy mock mode (synthetic AuthUser from `wr_user_name` + `wr_user_id`)
3. Else unauthenticated

Cross-tab sync via `storage` event listener. 401 invalidation via `wr:auth-invalidated` window event listener. Boot hydration via `useRef` guard to fire `getCurrentUser()` exactly once on mount.

**Transitional `mirrorToLegacyKeys(user)` helper** (lines 150-158, called from login/register/refreshUser/storage-sync) writes `wr_auth_simulated`, `wr_user_name`, `wr_user_id` for any JWT-authed user. Each call site is tagged with `// Transitional — removed in Phase 2 cutover`. **Spec 7 should preserve these comments** if the redesign touches AuthContext.

**"Sign in to X" action-gated prompts** — 60+ call sites across pages, components, and hooks (see Part 3 for the full list).

### Post-auth redirect logic

**There is no post-auth redirect.** AuthModal calls `handleClose()` on successful login/register (line 258). The user stays on the same page they were on when they opened the modal. The auth context state changes (`isAuthenticated: true`), and consuming components re-render to reflect the new state.

**Implication for Spec 7:** if the redesign moves login to a real `/login` route, the spec must define a `?redirectTo=` query-param contract OR a session-stored "last visited page" mechanism. The current modal pattern avoids this complexity entirely. **Recommendation in Open Questions: keep the modal pattern unless there's a strong reason to switch — the modal is the canonical Worship Room auth entry point and 60+ sites depend on it.**

For the auto-login-fail edge case (RegisterPage path → email already existed → wrong password), AuthContext throws `AUTO_LOGIN_FAILED` (lines 211-218) which surfaces in FormError as "Your account is ready. Please log in to continue." — the user must then manually log in. This is an anti-enumeration safety, NOT a redirect.

## Part 15 — Open Questions for the User

### Visual decisions

1. **Input chrome — utility or decorative?**
   Current: `rounded-xl bg-white/[0.06] border border-white/[0.12]` with `focus-visible:border-purple-400/50 focus-visible:shadow-[0_0_15px_rgba(139,92,246,0.15)]` — the **utility input idiom**.
   Alternative: adopt the Daily Hub textarea-glow pattern (`shadow-[0_0_20px_rgba(167,139,250,0.18),0_0_40px_rgba(167,139,250,0.10)] border border-violet-400/30 bg-white/[0.04]`) for trust-moment emphasis.
   **Recommendation in this recon:** lean utility — auth is task-completion, not emotional content. The textarea glow earned its decorative weight on the Pray/Journal textareas because those ARE emotional surfaces; AuthModal is a gate. But this is a judgment call, and a hybrid (focus-only glow on auth inputs, ambient violet glow on Pray/Journal) is also defensible.

2. **Submit buttons — gradient or subtle?**
   Current: white pill (`bg-white text-hero-bg shadow-[0_0_20px_rgba(255,255,255,0.15)]`) — the canonical Pattern 2 homepage primary CTA.
   Alternatives:
   - Keep white pill (recommended — matches "Get Started — It's Free" on FinalCTA, "Help Me Pray", "Save Entry")
   - Bump to `<Button variant="gradient">` (the violet-to-violet showstopper used on Pray/Journal) — risks drift away from homepage primary CTA pattern
   - Down-size to `<Button variant="subtle">` (frosted) — reads as utility, loses the "click here this is the way forward" visual weight

3. **Atmospheric layer for `/register` and a possible new `/login`?**
   Current `/register`: `<GlowBackground variant="fullPage">` — 5 distributed orbs across scroll length.
   Alternatives:
   - Keep `fullPage` (canonical for long-scroll marketing pages)
   - Switch to `HorizonGlow` (currently scoped to Daily Hub — would require widening the scoping rule in `09-design-system.md` § "Daily Hub Visual Architecture")
   - Solid `bg-hero-bg` with no atmospheric layer (utility — but RegisterPage is marketing, not utility, so this would feel flat)
   **Recommendation:** keep `fullPage` for `/register`; if a real `/login` route lands, it should also use a single-section `GlowBackground variant="center"` rather than `fullPage` (login is short, not scrollable).

4. **AuthModal chrome — match dialog patterns or adopt FrostedCard accent?**
   Current panel chrome IS the FrostedCard tier-1 visual signature applied to a modal — `rounded-3xl bg-hero-bg/95 backdrop-blur-md border border-white/[0.12]` with the dual box-shadow. This is already the trust-moment treatment. **Recommendation: preserve.** Spec 6D's two cyan-ring violations don't apply here (AuthModal is already canonical).

5. **`/login` route — leave as ComingSoon stub or replace?**
   Three options:
   - Keep ComingSoon (do nothing) — but clean up its `font-script`, `text-text-dark`, `text-primary` deprecated patterns
   - Make `/login` redirect to home with the AuthModal opened in login view (e.g., `/login` → `/?auth=login`) — hardest because route-driven modal opening requires App.tsx wiring
   - Build a real `/login` page with the form (parallel to RegisterPage, smaller) — creates a third surface to maintain; risks drift from AuthModal
   **Recommendation:** option 2 if a clean URL contract is wanted (deep-linkable, shareable login link), otherwise clean up the stub. Real `/login` page (option 3) is excessive for what AuthModal already does well.

### Behavior / a11y decisions

6. **Password visibility toggle?**
   Common modern UX. Currently absent on both password and confirmPassword fields. Adding it is small (~20 LOC + an aria-label toggle button + a 2-state icon). **Recommendation:** add it — low effort, expected by users, especially on mobile.

7. **Validation timing — symmetric blur+submit, or all submit-only?**
   Current asymmetry: firstName/lastName/confirmPassword validate on blur+submit; email/password validate on submit only.
   **Recommendation:** unify to blur+submit on all fields. Catches typos earlier, no UX downside.

8. **Backdrop click on mobile — gated or open?**
   AuthModal: open on all viewports (one tap dismisses). TermsUpdateModal: gated off on mobile (only X / Later / Esc dismiss).
   **Recommendation:** keep AuthModal open (it's user-initiated; no concern about accidental dismissal of a pending action) BUT document the difference in the spec rationale so future contributors don't unify them by reflex.

9. **Skeleton fallback for `/register`?**
   Currently `<Suspense fallback={null}>`. Adding `RegisterSkeleton` parallels the 13 page-level skeletons documented in `09-design-system.md` § "Skeleton Components". Cost: ~50-80 LOC of skeleton + a slightly larger initial chunk. **Recommendation:** defer unless the bundle measurement shows the chunk is large enough that the blank flash is user-visible. The bundle measurement script lives at `frontend/scripts/measure-bundle.mjs`.

### Drift cleanups (do these regardless of design direction)

10. **RegisterPage white-pill drift** — the two CTAs at lines 166 and 382 use `text-primary` instead of `text-hero-bg` (the canonical Pattern 2 white pill text color). Either bump them to Pattern 2 fully (`shadow-[0_0_30px_rgba(255,255,255,0.20)]` + `text-hero-bg`) OR document them as an intentional softer treatment.
11. **AuthModal hardcoded 150ms** — replace `setTimeout(..., 150)` at line 87 with `ANIMATION_DURATIONS.fast` per BB-33 discipline.
12. **Navbar deprecated `bg-primary`** — Navbar.tsx:130 non-transparent "Get Started" branch uses `bg-primary hover:bg-primary-lt`. Replace with the canonical white-pill or transparent-pill pattern.
13. **ComingSoon stub** — remove `font-script`, `text-text-dark`, `text-primary` from App.tsx:119-136. Either keep as a clean dark-theme stub or remove the route entirely (if `/login` becomes a redirect).

15. **OAuth placeholder** — the disabled "Continue with Spotify" button at AuthModal lines 627-638. Either keep as a deliberate "coming soon" affordance, remove, or commit to wiring real OAuth. Leaving a `disabled` button visible without explanation is a small but real UX cost.

17. **TermsUpdateModal is the SAME idiom as AuthModal** — they share the panel chrome, the backdrop gradient, the H2 gradient, the FormError pattern, the `useFocusTrap`, the dual button stack pattern. Treat them as a coupled pair. Drift between them is a real bug in the making.

18. **Legacy auth keys are still being mirrored** — AuthContext writes `wr_auth_simulated`, `wr_user_name`, `wr_user_id` for JWT-authed users via `mirrorToLegacyKeys()` until Phase 2 cutover. The visual redesign must NOT touch this code path; the comments tagged `// Transitional — removed in Phase 2 cutover` are load-bearing for the future Phase 2 grep-and-remove.

19. **No FormField adoption** — `09-design-system.md` documents `FormField.tsx` as "built but not yet adopted by all production forms." AuthModal rolls its own field markup. Spec 7 could either migrate to FormField (would simplify the 6 inputs ~30%) or stay with the hand-rolled markup. **Recommendation:** if the spec touches input chrome at all, migrate to FormField — it's strictly better and the migration will catch a11y regressions automatically.
