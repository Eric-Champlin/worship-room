**Critical observations:**

- **Single page component, single route.** No nested routes, no providers above `<Layout>` (the `AuthModalProvider` and `ToastProvider` are mounted higher in `App.tsx`).
- **Conversation lives in `useState<ConversationPair[]>`.** Lost on refresh. No localStorage shadow, no backend persistence. (Eric's scope decision #1 defers persistence to Spec 9b.)
- **`showInput = conversation.length === 0 && !isLoading`** — the entire input block (textarea + chips + submit + popular topics) is **unmounted** when loading or when the first response arrives. After the first submission, the user can only continue via Dig Deeper chips OR by clicking "Ask another question" (which clears state and remounts the input).
- **Two live regions:** the `aria-live="polite"` parent at line 333 wraps both the conversation list and the loading state. The `CrisisBanner` has its own `role="alert"` + `aria-live="assertive"`.
- **Response auto-scroll, not focus-move:** `setTimeout(100ms) → document.getElementById('latest-response')?.scrollIntoView(...)` (lines 90-98 + 162-170). Screen reader users get the live-region announcement but visual focus stays on the submit button (which is unmounted).

## Part 3 — Reactive Store Consumer Audit (BB-45 anti-pattern check)

**Stores Ask consumes:** none directly. Ask is a stateless feature from a reactive-store perspective.

**Indirect dependencies:**

- `VerseCardActions` reads from `bible:notes` reactive store via `getNoteForVerse(...)` (`store.ts`). It does NOT subscribe — it calls `getNoteForVerse` once inside a `useEffect` keyed on `[showNoteInput, parsedRef]` to pre-fill the note text. This is not a BB-45 violation because the use case is "load once when expanding to edit" rather than "stay in sync with cross-surface mutations." The user can't change a note from another surface during the few seconds the inline composer is open.
- After `upsertNote(...)` succeeds, the user is shown a toast and the composer collapses. The Ask page does not re-render based on note state changes from elsewhere.

**Spec 9 cross-feature additions:**

- **Memorize button on Ask verse cards (Eric scope #2):** uses the `addCard` and `removeCard` API from `lib/memorize/store.ts`, plus `isCardForVerse` for the toggle state. The button needs to subscribe to `useMemorizationStore()` (Pattern A — standalone hook) so the button's "Memorized" indicator stays current if the user adds the same verse from BibleReader and then returns to Ask. **This is the only new reactive-store consumer Spec 9 introduces.** Since memorize is Pattern A, the wiring is `const cards = useMemorizationStore(); const isMemorized = cards.some(...)`. No subscribe boilerplate needed.

**BB-45-shape risks:** none. Ask is a "fetch and render" surface — its state lives in `useState<ConversationPair[]>` and is bounded to the page. Cross-surface state is not a real concern for the existing scope.

## Part 4 — Conversation State System

This part is short because there is almost nothing to document — that itself is the finding.

### Persistence: NONE in production code

- Conversation thread: `useState<ConversationPair[]>` in `AskPage.tsx:34`. **Lost on every page reload.**
- localStorage shadow: NONE. No `wr_ask_*` key. No `bible:ask*` key. The only Ask-related localStorage is `wr_chat_feedback` (per-question thumbs up/down).
- Backend storage: NONE. The backend `AskController` is stateless — every request is independent. Conversation history is sent in the request body (`conversationHistory: ConversationTurn[]`) and reconstructed server-side for each Gemini call. There is no `ask_threads` or `ask_messages` table.
- Service worker / IndexedDB: NONE.

### What persists today

| Key | Type | Purpose | Where written |
|-----|------|---------|---------------|
| `wr_chat_feedback` | `ChatFeedback[]` | Thumbs up/down per response. Each entry: `{ questionId: string; helpful: boolean; timestamp: string }` | `AskPage.tsx:228` |

**`wr_chat_feedback` is documented in `11-local-storage-keys.md` line 125:** "Thumbs up/down feedback on AI answers." No further metadata. There is no surface that reads this back — the feedback is write-only from the user's perspective. (Future telemetry could mine it, but no surface today.)

### Conversation history limits

- React state: unbounded. A user could submit 50 follow-ups and the array would grow to 50 pairs in memory.
- Network request body: each follow-up sends the full prior history flattened to `[{role: 'user'|'assistant', content: string}]`. **No client-side trimming.** A long conversation will eventually exceed Gemini's input context window (1M tokens for 2.5 Flash Lite, so practically not a concern unless user is intentionally pushing it). No backend-side trimming either.
- TTL / rotation: N/A — no persistence.
- Browser refresh: kills the conversation. User starts fresh on next visit.

### AI request/response caching: NONE

- Frontend cache: BB-30 Explain and BB-31 Reflect both use the BB-32 `bb32-v1:` localStorage cache (7-day TTL, 2 MB cap). **Ask does NOT use any cache.** Every question is a fresh Gemini call.
- Backend cache: also none. `AskService` calls Gemini directly on every request.

**Why no cache?** A reasonable design choice, not an oversight: caching on `(question, conversationHistory)` keys would have a near-zero hit rate because conversation context shifts every turn AND because users phrase questions differently. The BB-32 cache works because Explain/Reflect are keyed on stable `(reference, verseTextHash)` tuples — there's no equivalent stable key for Ask.

### FINDING (Eric ask #6 — document but do not change)

The 30-second timeout in `services/ask-service.ts:12` (`REQUEST_TIMEOUT_MS = 30_000`) and the silent-fallback-to-mock behavior in the same file's `catch` block are an **intentional UX choice**, not an oversight:

```ts
// services/ask-service.ts:73-75
} catch {
  return getAskResponse(question)
}
```

The doc comment at the top of the file (lines 1-6) explicitly states: *"User never sees a raw error state — the mock's 16 canned responses are the graceful-degradation floor."* This is the canonical Worship Room error posture for emotionally vulnerable surfaces — a user asking "why does God allow suffering?" should never receive a 502 page. **The 16-topic mock at `mocks/ask-mock-data.ts` is the floor, not a fallback that should be removed.** Behavior is identical for: HTTP 4xx, HTTP 5xx, network failure, timeout, malformed JSON, missing required fields. All routes converge on `getAskResponse(question)`.

**Spec 9 does not change this.** Mark as a Spec 9b candidate for revisit when conversation persistence ships — at that point we may want explicit "we couldn't reach the AI right now, but here's a thoughtful response that might still help" framing in UI rather than indistinguishable degradation.

### FINDING (Eric ask #8 — confirm cancel/abort affordance)

**There is no user-visible cancel/abort button during the 30-second AI request window.** The `AbortController` exists (`fetchWithTimeout` in `services/ask-service.ts:24-36`) and its only consumer is the timeout's `setTimeout(() => controller.abort(), timeoutMs)` — there is no UI affordance to call `.abort()` from a user click.

The user is locked into the loading state for up to 30 seconds. They can navigate away (route change unmounts AskPage), but there's no "Cancel" or "Stop generating" button.

**This is a Spec 9b candidate. Spec 9 does not add one.** A cancel button would interact with conversation persistence (do we save a partial state? do we keep the pending question for retry?), so coupling it to the persistence spec is the correct sequencing.

## Part 5 — AI Integration

### Model + endpoint

- **Model:** `gemini-2.5-flash-lite` (`AskService.java:51`)
- **Endpoint:** `POST /api/v1/proxy/ai/ask` (`AskController.java:36`)
- **Frontend service:** `services/ask-service.ts:47` — `fetchAskResponse(question, conversationHistory?)`
- **Frontend timeout:** 30 seconds (`REQUEST_TIMEOUT_MS = 30_000`)
- **Backend timeout:** 30 seconds (`AskService.java:53` — `REQUEST_TIMEOUT = Duration.ofSeconds(30)`)
- **Gemini SDK:** `@google.genai.Client` Java SDK (server-side; frontend never sees keys per Key Protection Wave)

### Backend layered defenses (in order)

1. **Crisis-keyword short-circuit** (`AskCrisisDetector.detectsCrisis(question)`) — bypasses Gemini entirely on match, returns canned crisis response with 988 / Crisis Text Line / SAMHSA. PII rule: only `questionLength` is logged.
2. **Gemini structured-output call** with `responseMimeType="application/json"` + `responseSchema` (built in `AskService.buildResponseSchema()`). The schema enforces: exactly 3 verses (each with `reference`/`text`/`explanation`), exactly 3 follow-up questions, `id` from a 16-value enum (`suffering, forgiveness, anxiety, purpose, doubt, prayer, grief, loneliness, anger, marriage, parenting, money, identity, temptation, afterlife, fallback`), `answer` minLength 50, `prayer` minLength 30.
3. **Post-parse validation** (`AskService.validateResponse(dto)`) — re-checks the same rules in code. The schema is first-line defense; this is authoritative. Returns null on failure → triggers retry.
4. **Retry with corrective suffix** (`AskPrompts.RETRY_CORRECTIVE_SUFFIX`) — appends "Your previous response had validation issues..." to the system prompt. Max 1 retry (`MAX_RETRIES_ON_VALIDATION_FAILURE = 1`).
5. **Server-side canned `FALLBACK_RESPONSE`** (`AskService.java:341-373`) — used when retries exhausted. Topic: "Seeking Wisdom". Provides 3 verses (Psalm 46:10, Proverbs 3:5-6, Matthew 7:7) and 3 follow-ups. Returned with `id="fallback"`.
6. **Frontend graceful degradation** (`services/ask-service.ts:73-75`) — on ANY exception (HTTP, network, parse, malformed envelope), falls through to `mocks/ask-mock-data.ts` `getAskResponse(question)` which keyword-matches against 16 topic buckets.

### Rate limiting

- Per-IP via the proxy `RateLimitFilter` (60/min prod, 120/min dev — same as all `/api/v1/proxy/**` endpoints).
- No per-user rate limiting yet (Forums Wave Phase 1 will introduce JWT auth, then per-user takes over).
- No Ask-specific limits (Master plan target is 20 AI requests/hour per user once auth lands; not yet implemented).

### Error handling on backend

- `SafetyBlockException` → 422 `SAFETY_BLOCK` (when Gemini's safety filters reject the prompt). Three-path detection: prompt-level `blockReason`, candidate-level `FinishReason.SAFETY`, candidate-level `FinishReason.PROHIBITED_CONTENT`.
- `UpstreamException` → 502 `UPSTREAM_ERROR` (Gemini SDK throws non-timeout exception).
- `UpstreamTimeoutException` → 504 `UPSTREAM_TIMEOUT` (Gemini exceeds 30s).
- All caught by `ProxyExceptionHandler` (package-scoped `@RestControllerAdvice` for `com.worshiproom.proxy`).

**The frontend never displays these.** All three error codes hit the same `services/ask-service.ts` catch block and degrade silently to the mock.

### Prompt templates (visible client-side?)

**No.** The system prompt and user prompt builders live entirely in the backend:

- `AskPrompts.ASK_SYSTEM_PROMPT` (8 rules, ~290 words) — backend-only.
- `AskPrompts.RETRY_CORRECTIVE_SUFFIX` — backend-only.
- `AskPrompts.buildUserPrompt(question, history)` — backend-only.

The frontend sends only the raw user question + conversation history. This is correct architecture per Worship Room standards — prompts are server-side IP and shouldn't ship in the browser bundle.

### Crisis detection wiring

- **Server-side (authoritative):** `AskCrisisDetector` runs FIRST in `AskService.ask()`. If `detectsCrisis(question)` returns true, the canned crisis response is returned without calling Gemini.
- **Client-side (fast-path UX):** `CrisisBanner` (mounted at `AskPage.tsx:291`) reads from `text` (the textarea value) on every keystroke and displays the 988 / Crisis Text Line / SAMHSA resources inline before the user even submits. Same pattern as Pray and Journal tabs.

The two-layer design matches `01-ai-safety.md` § Crisis Intervention Protocol — the server-side check is the authoritative one; the client-side check is courtesy UX so the user sees resources immediately on typing.

### Streaming vs synchronous

- **Synchronous.** No streaming response. The backend awaits the full Gemini response (with retry-on-validation-failure), then returns the complete `AskResponseDto` envelope. The frontend sees a single `await response.json()` — no chunked transfer, no Server-Sent Events.
- The "loading" UX is the bouncing dots + Psalm 119:105 verse. There's no progressive reveal.
- **Spec 9 does not change this.** Streaming is a Spec 9b/future candidate.

### What's documented but not yet implemented

The frontend documents (in `services/ask-service.ts:1-6`) that the service falls back to mock on ANY error. This is intentional, mature behavior. The "user never sees a raw error state" is an architectural commitment.

### Focus management on async state changes

⚠ **Findings (Eric scope #7 + my prior #8):**

**FINDING #1 (Eric ask #7 — confirmed): Loading region lacks `aria-busy`.**

Lines 360-391 of `AskPage.tsx` render the loading UX (bouncing dots, "Searching Scripture for wisdom...", Psalm 119:105 verse). The wrapper `<div className="flex flex-col items-center justify-center py-16">` (line 370) has no `aria-busy="true"` and no `role="status"`. The parent `aria-live="polite"` at line 333 announces text changes, but assistive tech is not told that work is in progress.

Spec 9 fix: add `aria-busy="true"` and `role="status"` to the loading wrapper at line 370. Add an sr-only "Searching Scripture for wisdom" announcement that lives outside the visible loading text (so the announcement happens even if the visible text is later visually replaced).

**FINDING #2 (my prior — focus management): Response arrival does not move focus.**

Lines 90-98 + 162-170 do `document.getElementById('latest-response')?.scrollIntoView({...})` after a 100ms timeout. **Visual scroll moves; focus does not.** The submit button is unmounted (so focus visually goes to wherever the document moved focus next — typically `<body>`), and screen reader users get the live-region announcement but no semantic focus on the new response.

Spec 9 fix: when a response arrives, move focus to a `tabIndex={-1}` element on the response container (e.g., the `<h2>` "What Scripture Says" or the FrostedCard wrapping the answer). This is the same pattern AuthModal uses for focus management on submission failure (Spec 1.9 canonical pattern). Per `09-design-system.md` § "Accessibility Checklist" — moving focus to the FormError or first invalid field is the canonical fix.

**FINDING #3 (Eric ask #7 — submit button + isLoading): Button is unmounted, so the disabled state is structural rather than communicated.**

The submit button at line 317-325 has only `disabled={!text.trim()}` — it does NOT include `disabled={!text.trim() || isLoading}`. **In practice, the button is unmounted when `isLoading=true`** because `showInput = conversation.length === 0 && !isLoading` (line 233). So the user cannot click the button during loading.

However:
- A screen reader user navigating the page during the loading state will not find the submit button at all (it's gone from the DOM). They may not realize a request is in flight.
- If `showInput` ever changes (e.g., a future spec keeps the input visible during loading for follow-up flow), the button could be visible-but-not-disabled, allowing rapid double-submit by a screen reader user moving past the live-region announcement.

Spec 9 fix:
- Add `disabled={!text.trim() || isLoading}` to the button as a defensive guard even though `showInput` currently unmounts it. This makes the disabled-during-loading semantics explicit in the markup.
- Add an `aria-label` that communicates loading state when the button is in the loading-but-visible scenario. Example: `aria-label={isLoading ? 'Searching Scripture' : 'Find Answers'}`.
- Better still: add an `aria-label` that doesn't depend on the button being visible — combine with FINDING #1's sr-only announcement so the loading state is communicated regardless of the button's mount status.

### Screen reader announcements for new responses

⚠ The parent `aria-live="polite"` at line 333 announces ALL changes to its children — that's both the user question bubble AND the response. A screen reader user hears:
1. The new `UserQuestionBubble` text (their own question repeated back).
2. The loading state text ("Searching Scripture for wisdom...").
3. The full `AskResponseDisplay` content when it arrives — direct answer, "What Scripture Says", 3 verse cards, encouragement, prayer, AI disclaimer, Dig Deeper, action buttons, feedback row. **All in one announcement burst.**

## Part 12 — Performance and Rendering

### Long-conversation rendering

- Conversation thread lives in `useState<ConversationPair[]>`. No virtualization.
- Each `AskResponseDisplay` is heavy — 3 FrostedCards (1 answer + 3 verses), ~10 buttons, a `LinkedAnswerText` parsing pass on the answer for verse references, scroll-reveal hooks, animated `motion-safe:animate-fade-in-up`.
- Per response: ~217 LOC of React rendering. With 3 useScrollReveal hooks each (1 verses, 1 actions per first response).
- A 10-pair conversation = 10 instances of `AskResponseDisplay`, each with full content. Not virtualized.

**Practical impact:** in observed UX, conversations rarely exceed 3-4 pairs before users hit "Ask another question" (which clears state). The page would handle 20+ pairs technically, but the layout becomes very long.

**Spec 9 does not change this.** Virtualization would be a Spec 9b consideration tied to history persistence.

### Streaming response performance

- N/A — no streaming. The response arrives in a single `await response.json()` after up to 30s.
- Visual loading state (bouncing dots + Psalm 119:105) carries the wait. The Psalm 119:105 verse is a thoughtful UX choice — gives the user something meaningful to read while waiting.

### Bundle size considerations

- AskPage is lazy-loaded (`App.tsx:93`). The bundle splits Ask code from the main bundle.
- Bundle contents: AskPage + 8 components + service + constants + types + mock data (16 topics × ~4-line response each). Mock data is the largest individual piece — ~641 LOC of static content.
- The mock data is bundled even when the user has a working backend (because `services/ask-service.ts` imports it for the fallback path). Could be optimized to dynamic-import the mock only when needed, but that's a perf micro-optimization not in Spec 9 scope.

**Spec 9 changes that might affect bundle:**
- BackgroundCanvas import replaces GlowBackground import. Both are small wrappers. Net zero.
- New violet-glow textarea is class-string-only. No bundle impact.
- New `useMemorizationStore()` import for the memorize button. Already imported by other Bible-wave surfaces; no new code path.
- New BibleReader bridge button + Daily Hub Devotional bridge button. Small additions in those host components.

### Re-render cascades

- AskPage re-renders on every keystroke into the textarea (because `text` is local state).
- This causes:
  - `CharacterCount` to re-render (it's tied to the `text.length`)
  - `CrisisBanner` to re-render (it reads `text`)
  - The submit button to re-render (`disabled` changes when text crosses empty/non-empty)
  - The topic chips do NOT re-render — they're rendered from a constant array via `.map()`, and React reuses the elements because keys are stable
- `useScrollReveal()` for chips only fires once (`triggerOnce: true` default). No animation thrash on keystroke.

**No performance issue today.** Acceptable behavior for a textarea with a 500-char cap.

### React Strict Mode + auto-submit race

The `?q=` auto-submit + `pendingAutoSubmitRef` patterns include explicit guards against React Strict Mode double-mount (commented in `pages/__tests__/AskPage.test.tsx:13-14` of the older recon). Verified: the `useEffect` that handles `?q=` (line 105-113) does not re-trigger on remount because the dependency array is `[]` and the inner code uses a `setTimeout(0)` — the timeout's cleanup (line 112) prevents a duplicate submit if the effect cleanup runs before the timeout fires.

## Part 13b — BibleReader Bridge Architecture (Eric ask #9)

This part is dedicated to the BibleReader → Ask bridge because Eric specifically called out that it crosses Decision 13's reader-chrome boundary. The bridge is **a NEW entry-point that respects existing chrome**, NOT a chrome migration.

### Where the CTA should mount

Three candidate locations in BibleReader UI, with tradeoffs:

| Location | File | Pros | Cons | Recommendation |
|----------|------|------|------|----------------|
| `VerseActionSheet` (per-verse action menu) | `components/bible/reader/VerseActionSheet.tsx` (496 LOC) | Range-aware automatically (selection contains startVerse + endVerse). Mounts naturally next to existing AI actions (Explain, Reflect). Discoverable on tap/long-press. Symmetric with how BB-30 Explain and BB-31 Reflect are presented (via `getActionByType(action)` registry). | Requires user to select a verse first; can't ask a chapter-level question without selecting. | **PRIMARY mount** — register `'ask'` as a new action in `lib/bible/verseActionRegistry.ts` next to `explain` and `reflect`. |
| `ReaderChrome` top toolbar | `components/bible/reader/ReaderChrome.tsx` (186 LOC) | Always visible (when chrome not focus-mode-dimmed). Chapter-aware via `bookName` + `chapter` props. Symmetric placement next to existing AI toggles. | Chapter-level granularity only — can't ask about a specific verse without losing context. Reader-chrome icon density is already high (theme/typography/back/chapter selector/audio/focus/books/audio-play). Adding another would crowd the bar. | Defer — not in Spec 9 scope. |
| New entry-point card below ReaderBody | inline in `BibleReader.tsx` | Could be ambient ("Ask anything about this passage"). | Adds vertical content density, breaks the immersive reader flow. Conflicts with focus mode UX. | Defer — runs counter to BibleReader's documented "immersive" intent (Decision 13). |

**Recommendation: mount in `VerseActionSheet` only.** The verse action sheet is the canonical "do something with this verse range" surface. `Explain` and `Reflect` already live there. Adding `Ask` as a sibling action is the lowest-friction architectural fit.

### URL contract for passing passage context

Two reasonable patterns:

**Option A: Single `?q=` parameter with embedded passage context**

The existing `?q=<text>` parameter is auto-submitted on AskPage mount. Pre-format the question with the passage:

```
/ask?q=Help+me+understand+John+3%3A16-18%3A+%22For+God+so+loved+the+world...%22
```

Pros: Reuses existing infrastructure (no new code path on AskPage). The system prompt's `buildUserPrompt` already handles arbitrary user text. Frontend-only change.

Cons: Loses structured passage metadata. Gemini sees the verse text as inline-quoted in the user's question, not as separate "context" the system prompt can frame differently. The auto-prefilled text appears in the textarea visibly.

**Option B: Separate `?passage=` parameter alongside `?q=`**

Introduce a new URL contract that AskPage understands:

```
/ask?passage=John+3%3A16-18&passageText=For+God+so+loved+the+world...
```

AskPage reads both, displays a `DevotionalPreviewPanel`-style sticky inline panel showing the passage context, and either:
- Auto-prefills the textarea with a context-shaped placeholder ("Help me understand John 3:16-18..."), OR
- Leaves the textarea empty and adds passage as a separate field in the request body that the backend can incorporate into the system prompt.

Pros: Structured. The passage is visually distinct from the user's question. Matches the existing Daily Hub `DevotionalPreviewPanel` pattern (Spec X).

Cons: Requires new frontend state, new visual pattern, new request body field, new backend system prompt incorporation. Significantly more work than Option A.

**Recommendation for Spec 9: Option A.** It ships the bridge with minimal new architecture, gets us the "Ask about this passage" UX, and defers the structured-passage-context refinement to Spec 9b/9c when conversation persistence and richer context-passing land together. The system prompt (`AskPrompts.ASK_SYSTEM_PROMPT`) already handles inline-quoted scripture gracefully — its rule 1 ("WEB translation for all verse quotations") and rule 5 ("acknowledge pain when present") don't preclude quoted passages in the user input.

The pre-filled question template should be anti-pressure and exploratory. Suggested template:

```
"Help me understand <reference>: \"<verse text>\""
```

Where `<reference>` is `formatReference(selection)` (e.g., "John 3:16" or "John 3:16-18") and `<verse text>` is the joined verse texts (`getSelectionText(selection)`). Both helpers already exist in `lib/bible/verseActionRegistry.ts`.

### How the system prompt should incorporate passage context

Per Option A, the passage is incorporated **as part of the user's question**, NOT as separate context. Rule 1 of `AskPrompts.ASK_SYSTEM_PROMPT` ("Use the World English Bible (WEB) translation for all verse quotations. Never invent or paraphrase verse text") will guide Gemini to anchor the answer on the quoted passage. Rule 5 ("Acknowledge pain when it's present") still applies. No system prompt changes needed for Spec 9.

**Future Spec 9b/9c may revisit** if user testing shows Gemini ignores the passage context — at which point we'd switch to Option B with a `passage` field that the backend can pass as separate `Content` parts to Gemini's API, distinct from the user's question.

### What does NOT change in BibleReader chrome

Per Decision 13:
- `ReaderChrome` top toolbar stays as-is. NO new icon there.
- `BibleReader.tsx` page structure stays as-is. NO new ambient panel.
- `TypographySheet`, `AmbientAudioPicker`, `CrossRefsSubView`, `ExplainSubView`, `ReflectSubView`, `ShareSubView`, `NoteEditorSubView`, `HighlightColorPicker`, `BookmarkLabelEditor` — no changes.
- Focus mode behavior is preserved (the new Ask action is inside the verse action sheet, not in the chrome).

The change is scoped tightly to:
1. `verseActionRegistry.ts` — register a new `'ask'` action (next to `'explain'`, `'reflect'`).
2. `types/verse-actions.ts` — add `'ask'` to the `VerseAction` union if needed (verify against the current type).
3. `VerseActionSheet.tsx` — handle the new action's `onClick`. Likely a `handleAsk` that calls `routerNavigate(buildAskUrl(selection))` and closes the sheet.
4. A small URL builder util (potentially `lib/bible/verseActions/buildAskUrl.ts` mirroring the existing `buildDailyHubVerseUrl.ts` pattern at `lib/bible/verseActions/buildDailyHubVerseUrl.ts`).

### Does this need to be a deep-linkable action (BB-38)?

**No.** Looking at `lib/url/validateAction.ts`, `DEEP_LINKABLE_ACTIONS` are actions that have a sub-view inside the BibleReader's verse action sheet (Explain, Reflect, Highlight, Note, Share). Ask navigates AWAY from BibleReader to AskPage, so it doesn't have a sub-view to deep-link to. Don't add `'ask'` to `DEEP_LINKABLE_ACTIONS`.

The action's `hasSubView` would be `false`. It behaves like the existing `'pray'`, `'journal'`, `'meditate'`, `'copy'` actions — fire-and-forget navigations with no sub-view.
