**Atmospheric layer fragmentation (FINDING for Spec 8):**

Five different atmospheric treatments are live on Bible surfaces today:

1. **`<BackgroundCanvas>`** — BibleLanding, PlanBrowserPage. Homepage Round 3 pattern.
2. **`<HorizonGlow>`** — MyBiblePage **only**. Daily Hub Spec Y pattern. The MyBibleAuthenticatedInner inline comment (`pages/MyBiblePage.tsx:218`) acknowledges this: "Hero section — Daily Hub pt-36 pattern, no ATMOSPHERIC_HERO_BG".
3. **`ATMOSPHERIC_HERO_BG`** inline style + `bg-hero-bg` body — BibleBrowse, BiblePlanDetail, BiblePlanDay. Hero radial only, no atmosphere below.
4. **Reader theme variables (`var(--reader-bg)`)** — BibleReader. Documented exception (per `09-design-system.md` § "Layout Exception: BibleReader").
5. **`bg-dashboard-dark`** flat — `BiblePlanDetail.tsx:107` sets `min-h-screen bg-dashboard-dark` and `BiblePlanDay.tsx:113` does the same. The hero has the radial; everything below is flat.

The mismatch is the headline visual finding for Spec 8: a user who scrolls from BibleLanding (canvas) → tap a plan (inline radial only) → tap a day (inline radial only) experiences an inconsistent atmospheric drop, then jumps back to MyBiblePage (HorizonGlow) and sees yet a different layer. See Open Question 1.

**Critical observations:**

- **No Navbar, no SiteFooter on BibleReader.** Documented intentional exception. ReaderChrome is a `<nav aria-label="Reader controls">` fixed at `top-0 z-30`.
- **Root-level skip link.** Mounted at the top of the JSX outside any provider.
- **Reader theme is scoped via inline `style={{ background: 'var(--reader-bg)' }}` + `data-reader-theme={settings.theme}` attribute.** CSS custom properties switch midnight/parchment/sepia.
- **Single `<main>` element.** Both validation error states (book not found, chapter not found) render different content but consistently within the page chrome. Reader body and validation errors all share the `bg-hero-bg`/reader-bg surface.
- **974 LOC in BibleReader.tsx — heavily orchestrated.** Manages 16+ pieces of state, 11 useEffects, 2 store subscriptions for chapter highlights/bookmarks/notes, 3 keyboard handlers, swipe gestures, scroll-to-verse arrival highlighting, focus mode, ambient audio coordination, notification prompts.

### Authentication behavior

- **BibleLanding** (`/bible`) — public. `useAuth().isAuthenticated` is read only to gate the StreakChip (which is hidden when logged out).
- **BibleBrowse** (`/bible/browse`) — public. No auth check.
- **BibleSearchMode** (inline on `/bible?mode=search`) — public.
- **PlanBrowserPage** (`/bible/plans`) — public.
- **BiblePlanDetail** (`/bible/plans/:slug`) — public read. Auth-gated start/pause/restart and tap-on-day-row (opens auth modal: `'Sign in to start a reading plan'`).
- **BiblePlanDay** (`/bible/plans/:slug/day/:dayNumber`) — public read. Auth-gated `Mark day complete` and reflection save.
- **BibleReader** (`/bible/:book/:chapter`) — **fully public.** Highlights, notes, bookmarks, journal entries, memorization adds, AI Explain, AI Reflect, ambient audio control, BB-41 push subscription — all unauthenticated. Per the BB-0–BB-46 wave's documented design.
- **MyBiblePage** (`/bible/my`) — **auth-gated entirely.** When `!isAuthenticated`, renders a sign-in CTA shell with the homepage primary white-pill CTA (`'Get Started — It\'s Free'`). When authenticated, renders the full personal layer feed. **THIS IS THE NOTABLE DEVIATION** from BB-0–BB-46's "no auth gates" promise. The page reads from localStorage stores that the unauthenticated user has been writing to — but they can't see them. See Open Question 2.

## Part 3 — Reactive Store Consumer Audit (BB-45 anti-pattern check)

This is the highest-priority audit per the brief. Two patterns coexist (per `11b-local-storage-keys-bible.md` § "Reactive Store Consumption"):

- **Pattern A** — `useSyncExternalStore`-based hook: `useMemorizationStore`, `useStreakStore`, `usePrayerReactions` are the canonical Pattern A hooks. **There is no `useReactiveStorage` hook in the codebase.**
- **Pattern B** — inline `subscribe()` in `useEffect`: every other Bible store.

Below is the consumer-by-consumer audit for every Bible-relevant reactive store.

### `wr_bible_highlights` → `lib/bible/highlightStore.ts` (Pattern B)

| Consumer | Pattern | Lines | Status |
|----------|---------|-------|--------|

### Card creation flow

1. **From BibleReader verse menu (primary entry):** `<VerseActionSheet>` opens on tap or long-press. The "Add to memorize" action calls `addCard(...)` from `lib/memorize/store.ts`. Multi-verse range selection creates a single card with concatenated text (per BB-45 spec). Action becomes "Remove from memorize" if already in deck.
2. **From My Bible activity feed:** `<HighlightCard>` (108 LOC) renders an "Add to memorize" / "In deck" toggle button when `hasVerseInfo && verseText` are present. Calls `addCard(...)` or `removeCard(...)`.

### Card review flow

`MemorizationFlipCard.tsx` is the entire review surface. **No quiz, no scoring, no spaced repetition** — confirmed.

- Tap (or Enter/Space) flips the card 180deg around Y-axis (3D transform with `transformStyle: 'preserve-3d'`).
- On front-to-back flip, `recordReview(card.id)` fires via `queueMicrotask` to defer the store write outside React render. Updates `lastReviewedAt` and increments `reviewCount`.
- Review count and lastReviewedAt are **only used for sorting and display in the deck header** ("Last reviewed John 3:16 2 hours ago") — they never gate UI or trigger pressure.
- Remove flow: X-icon button enters a confirmation pill state ("Remove this card? Yes / Cancel"). No modal.

**Anti-pressure compliance:** PASS. The deck renders no progress score, no streak, no "you mastered this" celebration. Sorting the "last reviewed" line is the only acknowledgment of activity.

### Card detail views

There is **no separate card detail view.** A card has only the front (reference) and the back (verse text). Tapping flips between them. No drill-in surface for editing/practicing/spelling/etc.

### Integration with BibleReader

- The "Add to memorize" action lives in `<VerseActionSheet>` — which mounts when the user taps a verse in the reader.
- The reader does NOT display "this verse is in your memorization deck" inline. The status only surfaces inside the action sheet itself.
- Memorization cards are NOT highlighted differently in `<ReaderBody>` — only highlighted verses (highlights store) get visual treatment.

### Reactive store consumers + BB-45 risk

- `MemorizationDeck.tsx` — Pattern A via `useMemorizationStore()`. **Correct.**
- `HighlightCard.tsx` — Pattern A via `useMemorizationStore()` (side-effect only, then synchronous reads). **Correct but fragile** — the unused return value is the only thing keeping the subscription alive.
- VerseActionSheet — needs verification during Spec 8 (the memorize action's "in deck" badge needs to subscribe).

### Sound effects, animations, completion celebrations

- **No completion celebration on add.** Adding a card to the deck is silent.
- **No flip sound effect.** No `useSoundEffects` hook is imported into `MemorizationFlipCard.tsx` or `MemorizationDeck.tsx`. Compare to Grow's challenge completion: `useSoundEffects().playSoundEffect('ascending')`. **Memorization is intentionally quieter** per anti-pressure design — verify that's the intent.
- Flip animation uses `transition-transform duration-base ease-decelerate motion-reduce:duration-0` (BB-33 tokens). **Correct.**
- No confetti, no shimmer, no toast on add.
- Empty state uses `<FeatureEmptyState>` (BB-34 canonical). **Correct.**

**Open Question 8** addresses whether to add a celebration parallel to Grow's challenge `themeColor` — likely "no, anti-pressure design supersedes" but worth confirming.

## Part 12 — Performance and Rendering

### Long-scroll BibleReader

`BibleReader.tsx` body renders all verses in a chapter via `<ReaderBody>` mapping over the verses array. Psalms 119 has 176 verses — the longest. No virtualization. Each verse is a `<sup>` (verse number) + `<span>` text + verse-level event handlers attached via `useVerseTap`. Highlights/notes/bookmarks are rendered inline as overlays.

**Performance characteristics:**
- Chapter load time: lazy JSON import, ~50-200ms per chapter on warm browser.
- First render: ~5-50ms for typical 30-verse chapter.
- Scroll: smooth on modern hardware. Long chapters (Psalms 119, 1 Chronicles 9, etc.) may have noticeable scroll jank.

**Recommendation for Spec 8:** Defer virtualization unless Lighthouse Performance shows < 90 on long chapters. Today's scroll feel is acceptable on the chapters tested.

### Reactive store subscription cascades

Each chapter mount in BibleReader subscribes to highlight/bookmark/note stores AND has a separate effect that re-fetches on `bookSlug, chapterNumber` change. The diff-based "fresh highlight pulse" effect is well-bounded (only re-runs on store mutation).

A streak update from `recordReadToday()` in BibleReader on chapter mount cascades to:
- `useStreakStore` consumers in BibleLanding (different page, no impact)
- `useActivityFeed` consumer in MyBiblePage (different page)
- Any sibling component subscribing — none on the reader page

Within BibleReader itself, store subscriptions are scoped tightly. **Performance is fine.**

### Bundle size

- Lazy chapter JSON: ~5-100KB per book. ✅
- Lazy plan JSON: ~5-50KB per plan. ✅

### BB-32 cache hit rate considerations

Different verse ranges within the same chapter are separate cache entries. 7-day TTL. Cache is courtesy reduction in upstream calls — backend rate limiter is the actual enforcement.

### BB-42 search index size
