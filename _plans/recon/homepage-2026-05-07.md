  right: [...],

function GlowOrbs({ variant, glowOpacity }) { ... }

export function GlowBackground({ children, variant = 'center', className, glowOpacity }) {
  return (
    <div className={cn('relative overflow-clip bg-hero-bg', className)}>

- Children get `<div className="relative z-10">` wrapper to sit above orbs.

## Section E — `/register` GlowBackground migration assessment

### E.1 What does GlowBackground currently render?

Per Section B.2, `<GlowBackground variant="fullPage">` on `/register` renders:

- 5 inline orbs distributed at top-5%, top-30%, top-55%, top-75%, top-92% of the page scroll length

- Animation: `animate-glow-float motion-reduce:animate-none`

The `<div className="relative z-10">` wrapper holds the children (the `<main>` content) above the orbs.

### E.2 Canonical replacement / harmonization options (NOT prescribing)

Direction phase decides between these options:

#### Option E.2.a: Migrate to HeroSection-style atmospheric (homepage cinematic match)

Replace `<GlowBackground variant="fullPage">` with the HeroSection background pattern: video + dual gradient masks. Keeps the cinematic feel that links the homepage and register page. Adds a cinematic background video to `/register`, increasing emotional weight at the moment of conversion.

**Pros:** Visual continuity from homepage to register. Cinematic conversion moment.
**Cons:** Adds bandwidth cost to a route that's loaded by motivated users (registration is a conversion surface, fewer drive-by visits than homepage). The homepage video has a specific "How're You Feeling Today?" association that may not translate to "Your room is ready." conversion context.

#### Option E.2.b: Migrate to ATMOSPHERIC_HERO_BG (canonical inner-page atmospheric)

Replace the full-page wrapper with a single hero band using ATMOSPHERIC_HERO_BG (the canonical inner-page atmospheric layer used by Settings, Insights, RoutinesPage). The rest of the page would use plain `bg-hero-bg` between scroll sections.

**Pros:** Aligns with canonical inner-page pattern (Music Direction Decision 1). Consistent with Settings/Insights/RoutinesPage.
**Cons:** Loses the long-scroll seam-free atmospheric distribution that GlowBackground fullPage provides. ATMOSPHERIC_HERO_BG is a single section radial — a long-scroll page would have visible "atmospheric gaps" between scroll sections.

#### Option E.2.c: Migrate to HorizonGlow (Daily Hub atmospheric)

Replace with `<HorizonGlow />` — the 5 strategic-position glow blobs at 5/15/35/60/85% with opacities 0.32/0.28/0.35/0.30/0.28.

**Pros:** Higher opacity range than GlowBackground fullPage (0.28-0.35 vs 0.16-0.25) = more atmospheric depth. 5-point distribution matches register's long-scroll. Daily Hub-proven for sustained scroll experiences.
**Cons:** Per `09-design-system.md` § "Daily Hub Visual Architecture": "**Do not use on other pages without explicit reconsideration** — the layer is scoped to the Daily Hub controlled experience." Adopting HorizonGlow on register would expand its scope. Direction call.

#### Option E.2.d: New "register" variant on GlowBackground

Add a new variant to GlowBackground (e.g., `variant="register"`) with tuned opacities and orb positions specific to register's content rhythm. Or tune the `glowOpacity` prop to lift opacities to the 0.28-0.35 range (matching HorizonGlow without taking on the HorizonGlow component itself).

**Pros:** Bespoke control. Stays within the GlowBackground component contract (no new primitive). Could simply pass `glowOpacity={0.32}` to lift current variant.
**Cons:** Net-new variant adds component complexity. May not be reusable elsewhere.

#### Option E.2.e: Polish-only — keep GlowBackground variant="fullPage" but align chrome values

Keep the current architecture. Polish: tune the 5-orb opacities to better match the homepage section-glow opacity language (0.25-0.40 per section), align orb colors to a canonical palette, ensure the `glowOpacity` prop is used if a per-instance multiplier is needed.

**Pros:** Minimal disruption. Preserves current visual identity. Aligns drift without re-architecting.
**Cons:** Doesn't resolve the question of "is fullPage the right primitive for register, or should register match a peer surface's atmospheric pattern?"

**Recon does not pick.** Direction phase decides which option to lock based on:
- Visual continuity priority (cinematic vs canonical inner-page)
- Performance budget (video adds bandwidth)
- Component scope (broaden HorizonGlow vs keep GlowBackground)
- Feature work scope (polish-only vs new variant vs migration)
