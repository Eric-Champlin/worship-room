# Direction — Spec 9: AskPage Round 3 Visual Migration + Light UX

**Date:** 2026-05-05
**Recon:** `_plans/recon/ask-2026-05-05.md`
**Scope:** `/ask` route — single page, 8 child components — plus two
new cross-feature bridge entry points (BibleReader VerseActionSheet,
Daily Hub Devotional CTA) and a memorize button on Ask response
verse cards.

## Premise

AskPage is already mostly on Round 3 patterns. The page-level visual
delta is small: BackgroundCanvas atmospheric layer swap and textarea
violet-glow parity with Daily Hub 1B. The bulk of Spec 9 is light UX:
two cross-feature bridges (mirroring established patterns), three
a11y fixes (loading region semantics, focus management on response
arrival, defensive disabled guard on submit), and a memorize button
that wires Ask verse cards to the existing BB-45 store.

Conversation persistence, history surfaces, threading, shareable
Canvas cards, and additional cross-feature wiring are explicitly
deferred to Spec 9b/9c. Spec 9 is polish-only.

## Locked decisions

### 1. Atmospheric layer migration

**`<GlowBackground variant="fullPage">` → `<BackgroundCanvas>`**

`/ask` is currently the only logged-out-facing page using
`<GlowBackground variant="fullPage">`. Migrate to `<BackgroundCanvas>`
matching Dashboard, Local Support, BibleLanding, PlanBrowserPage,
CreatePlanFlow, PlanNotFound, ChallengeNotFound. The wrap pattern is
verbatim — children unchanged.

After Spec 9 the only fullPage holdout is `/register`, which migrates
with Spec 13.

### 2. Textarea idiom

**White-glow → violet-glow (Daily Hub 1B parity).**

AskPage's question textarea is the primary user-facing affordance.
Migrate to canonical violet-glow per `09-design-system.md` § "Textarea
Glow Pattern":

```
border-violet-400/30 bg-white/[0.04]
shadow-[0_0_20px_rgba(167,139,250,0.18),0_0_40px_rgba(167,139,250,0.10)]
focus:border-violet-400/60 focus:ring-violet-400/30
placeholder:text-white/40
```

Aligns Ask's emotional register with Daily Hub Pray/Journal and
CreatePlanFlow Step 1.

The inline note composer in `VerseCardActions.tsx:134` (also white-
glow) is OUT OF SCOPE — touching it would cross into Bible reader
audit territory. Defer.

### 3. Submit button — keep canonical white pill

**No change.** Already canonical Round 3 white pill primary CTA per
recon. Page-primary CTA reservation pattern preserved.

### 4. Hero h1 and subtitle

**No change.** Already canonical GRADIENT_TEXT_STYLE + animate-
gradient-shift on h1; canonical white sans subtitle.

### 5. Conversation persistence — DEFERRED

**No change.** Conversation lives in `useState`. Lost on refresh.
SaveConversationButton remains clipboard-copy-only (no real save).

Defer to Spec 9b alongside potential Forums Wave Phase 3 backend
`ask_threads` table.

### 6. Cross-feature bridges — INCLUDE 2

**6a. BibleReader → Ask.** Mount as new `'ask'` action in
`VerseActionSheet`. Register in `lib/bible/verseActionRegistry.ts`
next to `'explain'` and `'reflect'`. URL contract: `?q=<prefilled>`
(Option A — reuses existing infrastructure). NOT a deep-linkable
action (no sub-view inside BibleReader).

NO touches to `ReaderChrome` top toolbar. NO ambient panel below
ReaderBody. NO chapter-level entry. Decision 13 boundary respected.

**6b. Daily Hub Devotional → Ask.** Add inline CTA paralleling the
existing "Journal about this question" + "Pray about this" CTAs in
`DevotionalTabContent.tsx`. Same `<Button variant="subtle" size="sm">`
chrome.

Other potential bridges (BibleReader chapter-level, MyBible → Ask
history, Prayer Wall ↔ Ask, Ask → Bible note one-tap save, Ask →
Bible journal one-tap save, Ask → Daily Hub journal entry better-
than-state-passing) all deferred to Spec 9c.

### 7. Pre-filled question templates

**BibleReader bridge:** `"Help me understand <reference>: \"<verse text>\""`

Where `<reference>` comes from `formatReference(selection)` and
`<verse text>` from `getSelectionText(selection)`. Both helpers exist
in `lib/bible/verseActionRegistry.ts`.

**Daily Hub Devotional bridge:** the reflection question text directly
(same text the existing "Journal about this question" CTA passes).
No "Help me understand" wrapping — the reflection question already
reads as a question.

### 8. Auto-submit on bridge arrival

**Auto-submit enabled.** Both bridges set `?q=<prefilled>`; AskPage's
existing `?q=` handler auto-submits on mount. Consistent with homepage
typewriter pattern.

User can refine via Dig Deeper chips or "Ask another question" after
the first response lands.

### 9. Memorize button on Ask response verse cards

**Add as 4th action in `VerseCardActions`.** Order:
**Highlight, Memorize, Save note, Share.**

Memorize sits next to Highlight as the two "scripture engagement"
primitives. Save note follows as "your reflection." Share is the
social/external action.

Auth posture: **unauthenticated.** Mirrors BibleReader's BB-45
memorization auth posture. Logged-out users can memorize from Ask
verse cards just as they can from BibleReader verse action sheet.

Reactive store: `useMemorizationStore()` Pattern A hook. The
component subscribes for cross-mount sync (if the user adds the same
verse from BibleReader and returns to Ask, the indicator updates).

Mobile wrap: 4 actions in a row will wrap to 2 lines on small
viewports. Acceptable.

### 10. A11y fixes (3 small wins)

**10a. Loading region semantics.** Add `aria-busy="true"` and
`role="status"` to the loading wrapper at `AskPage.tsx:370`. Add an
sr-only "Searching Scripture for wisdom" announcement that lives
outside the visible loading text (so the announcement happens even
if the visible text is later visually replaced).

**10b. Focus management on response arrival.** Move focus to the
response container's `<h2>` "What Scripture Says" with `tabIndex={-1}`.
Pattern matches AuthModal's focus-on-error canonical reference. Keep
the existing `scrollIntoView` for visual continuity; add focus move
alongside.

**10c. Submit button defensive disabled guard.** Change
`disabled={!text.trim()}` to `disabled={!text.trim() || isLoading}`.
Add `aria-label={isLoading ? 'Searching Scripture' : 'Find Answers'}`.
The button is currently unmounted during loading via `showInput` —
the `disabled` is defense-in-depth for any future spec keeping the
button visible.

### 11. The 30-second timeout + silent fallback to mock — PRESERVED

Per `services/ask-service.ts:1-6` doc comment, the silent-mock-
fallback on any error is the canonical Worship Room error posture
for emotionally vulnerable surfaces. Spec 9 does NOT change this.

Mark as Spec 9b candidate for revisit when conversation persistence
ships (at which point we may want explicit "we couldn't reach the AI
right now" framing distinct from the mock degradation).

### 12. Cancel/abort affordance during loading — DEFERRED

No user-visible cancel button during the 30-second AI request
window. Adding one couples to conversation persistence (do we save
partial state? do we keep the pending question?). Defer to Spec 9b.

### 13. Crisis detection wiring — PRESERVED

Two-layer design (server-side authoritative `AskCrisisDetector` +
client-side `CrisisBanner` for fast-path UX) preserved exactly. No
changes.

### 14. AI integration — CLOSED SCOPE

No prompt template changes. No backend changes. No model swap. No
streaming. No BB-32-style cache. Backend Gemini 2.5 Flash Lite
orchestration is closed scope for Spec 9.

### 15. AskPage `wr_chat_feedback` schema — UNCHANGED

Don't add entry-source telemetry in Spec 9. Defer to Spec 9b
alongside conversation persistence schema work.

### 16. SaveConversationButton — UNCHANGED

Remains clipboard-copy-only. Despite the name, does not save anywhere.
Auth-gated render is preserved (component returns null for logged-
out users).

Real save flow (localStorage shadow, /ask/history, shareable URL)
defers to Spec 9b.

### 17. ConversionPrompt — UNCHANGED

"This is just the beginning" upsell card for logged-out users after
their first response. Already canonical FrostedCard chrome with
`animate-shine` on the white-pill register CTA. No changes.

### 18. PopularTopicsSection — UNCHANGED

5 topic FrostedCards (intentional 5, not 6). Click prefills + auto-
submits via `pendingAutoSubmitRef`. Already canonical Round 3 chrome.

Confirmed intentional 5 (not 6) per recon — older `ask-v2.md` flagged
the discrepancy with prompt expectations. Resolved as 5.

### 19. Tests update strategy

- Replace `data-testid="glow-orb"` count assertion with BackgroundCanvas
  presence check (precedent: `LocalSupportHero.test.tsx:99`)
- Add a11y tests for the 3 new wins (aria-busy, focus management,
  disabled guard)

  don't depend on shadow class strings)

## Sub-spec ordering and execution model

**Single spec, single execution.** Per Eric scope #1, no sub-spec
split. Total LOC delta ~200-300 production + ~80-150 test.

| Phase                  | Model               |
| ---------------------- | ------------------- |
| Plan from spec         | Opus 4.7 xHigh      |
| Execute plan           | **Sonnet 4.6 High** |
| Verify with Playwright | **Sonnet 4.6 High** |
| Code review            | Opus 4.7 xHigh      |

Execution model: Sonnet sweet spot. The bulk is pattern application
(BackgroundCanvas swap, textarea class string swap, parallel CTA
addition, memorize button wire-up). The architectural decisions are
in this direction doc, not deferred to execution.
