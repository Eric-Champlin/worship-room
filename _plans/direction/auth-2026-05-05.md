# Direction — Spec 7: Auth Surfaces

**Date:** 2026-05-05
**Recon:** `_plans/recon/auth-2026-05-05.md`
**Scope:** AuthModal, RegisterPage, /login stub, TermsUpdateModal (lockstep), Navbar/MobileDrawer auth CTA polish

## Premise

Auth surfaces are largely already canonical. The recon found AuthModal's chrome matches Round 3 / dark theme exactly (`rounded-3xl bg-hero-bg/95 backdrop-blur-md` with the dual box-shadow), no cyan drift, no font-script on headings, canonical FormError + Button + useFocusTrap + GRADIENT_TEXT_STYLE patterns throughout. RegisterPage uses the canonical GlowBackground fullPage atmospheric layer.

This is a drift-cleanup + small-adds spec. Not a major migration. The shape of Spec 7 is closer to Spec 6D (patches) than to Spec 6A/B/C (full migrations).

## Decisions (locked)

### Visual

**Decision 1: Input chrome stays utility.**

Auth inputs use the existing utility input idiom: `rounded-xl bg-white/[0.06] border border-white/[0.12] px-3 py-2.5 text-sm text-white placeholder:text-white/40 focus-visible:outline-none focus-visible:border-purple-400/50 focus-visible:shadow-[0_0_15px_rgba(139,92,246,0.15)]`.

The textarea-glow idiom (Daily Hub Pray/Journal pattern) does NOT apply here. Reasoning: textarea glow signals emotional authoring; auth is task completion. Don't decorate a gate.

**Decision 2: Submit buttons stay white pill (Pattern 2).**

AuthModal's submit button keeps the canonical Pattern 2 white pill: `w-full rounded-full bg-white py-3 text-sm font-semibold text-hero-bg shadow-[0_0_20px_rgba(255,255,255,0.15)] hover:shadow-[0_0_30px_rgba(255,255,255,0.25)]`.

Gradient variant is reserved for emotional climax (CreatePlanFlow Step 2). Auth submit is a different category. White pill matches FinalCTA, Help Me Pray, Save Entry, and the rest of the marketing/auth visual language family.

**Decision 3: AuthModal panel chrome preserved.**

Current chrome IS canonical: `mx-4 w-full max-w-md rounded-3xl bg-hero-bg/95 backdrop-blur-md border border-white/[0.12] p-6 shadow-[0_0_40px_rgba(139,92,246,0.15),0_8px_30px_rgba(0,0,0,0.4)]`. The dual box-shadow (violet glow + drop shadow) is the trust-moment treatment we want here. Do not migrate.

**Decision 4: RegisterPage atmospheric layer preserved.**

Keep `<GlowBackground variant="fullPage">`. Canonical for long-scroll marketing pages. Do not switch to BackgroundCanvas (that's the dashboard/feature visual language).

**Decision 5: TermsUpdateModal moves in lockstep with AuthModal.**

They share the auth idiom — same panel chrome, same backdrop, same gradient H2, same FormError, same button stack. Any visual change to AuthModal applies to TermsUpdateModal in the same spec. In practice this is a verification pass since neither is changing chrome — but if drift surfaces, fix it here.

### Behavior

**Decision 6: `/login` route becomes query-param-driven redirect.**

Replace the ComingSoon stub with a redirect: `/login` → `/?auth=login` (and similarly `/login?view=register` → `/?auth=register` if needed for parity). Add wiring in App.tsx (or a small wrapper component) that reads `?auth=login|register` from useSearchParams on mount, calls `openAuthModal(undefined, view)`, and strips the query param via `navigate(pathname, { replace: true })` to keep the URL clean post-open.

This gives us deep-linkable auth URLs (shareable login link for marketing emails, password reset email links once Spec 1.5b lands) without maintaining a parallel `/login` page.

The ComingSoon component is deleted entirely. LOGIN_METADATA stays for the redirect route's SEO `noIndex: true` posture.

**Decision 7: Password visibility toggle added.**

Both password and confirmPassword inputs get an eye-icon toggle button. Standard modern UX. Use Lucide Eye / EyeOff icons. Toggle button positioned absolute right inside the input wrapper, `min-h-[44px] min-w-[44px]` tap target, aria-label switches between "Show password" and "Hide password" based on state.

Implementation: each password field gets its own visibility state (independent — toggling password visibility shouldn't toggle confirmPassword visibility, and vice versa). State managed via two `useState<boolean>` hooks adjacent to the existing form state.

**Decision 8: Validation timing unified to blur+submit.**

Email and password fields gain blur validation alongside their existing submit validation, matching firstName/lastName/confirmPassword. resetEmail keeps its current submit-only timing (it's a one-field form; no value in blur there).

Implementation: extend the existing `touched` state object to include `email` and `password` keys. Update the per-field error rendering gates from `submitted && error` to `(touched.<field> || submitted) && error` for symmetry.

**Decision 9: Backdrop click on mobile stays open for AuthModal.**

Document the difference from TermsUpdateModal in spec rationale. AuthModal is user-initiated; accidental dismissal is low-stakes. TermsUpdateModal is gated content; accidental dismissal would lose the user's place in a required acceptance flow. Different consequence models = different defaults are correct.

### Drift cleanups

**Decision 10: RegisterPage white-pill drift fixed.**

Both white pills (RegisterPage.tsx:166, 382) migrate from `text-primary shadow-[0_0_20px_rgba(255,255,255,0.15)]` to canonical Pattern 2: `text-hero-bg shadow-[0_0_30px_rgba(255,255,255,0.20)]`. Match the rest of the marketing CTA family.

The `animate-shine` and `animate-gradient-shift` decorations stay (per RegisterPage.test.tsx assertions).

**Decision 11: AuthModal hardcoded 150ms replaced with token.**

Line 87's `setTimeout(() => closeFullyRef.current?.(), 150)` becomes `setTimeout(() => closeFullyRef.current?.(), ANIMATION_DURATIONS.fast)`. BB-33 token discipline.

Verify `ANIMATION_DURATIONS.fast` exists at `frontend/src/constants/animation.ts` and equals 150ms (or accept the small timing drift if the canonical value is 200ms; the close animation tolerates either).

**Decision 12: Navbar bg-primary "Get Started" branch migrated.**

Navbar.tsx:127-131 currently branches between transparent state (`bg-white/20 hover:bg-white/30 border border-white/30`) and non-transparent (`bg-primary hover:bg-primary-lt`). Migrate the non-transparent branch to canonical Pattern 2 white pill: `bg-white text-hero-bg shadow-[0_0_20px_rgba(255,255,255,0.15)] hover:shadow-[0_0_30px_rgba(255,255,255,0.25)]`.

Transparent branch stays as-is (canonical for over-hero placement).

MobileDrawer's "Get Started" link uses a `<Link to="/register">` without distinct chrome — verify its current treatment is canonical; if not, align to Pattern 2 white pill.

**Decision 13: ComingSoon stub deleted entirely.**

App.tsx lines 119-136 (the `function ComingSoon`) and the route at line 291 (`<Route path="/login" element={<ComingSoon ... />}`) get replaced. The route element changes from rendering ComingSoon to redirecting via the new query-param wiring.

`font-script`, `text-text-dark`, and `text-primary` deprecated patterns disappear with the component.

**Decision 14: Subtitle trailing-period unification.**

The 2 call sites with `'Sign in to save prayers to your list.'` (PrayerResponse.tsx:140, InteractionBar.tsx:218) drop the trailing period to match the other 31 unterminated subtitle strings. The third PrayerResponse call site at line 132 already omits the period; bring those two outliers in line.

Final unified subtitle: `'Sign in to save prayers to your list'` (no terminating period).

**Decision 15: OAuth Spotify placeholder removed.**

Delete the disabled Spotify button (AuthModal.tsx:627-638) and the "or" divider above it (lines 621-625). The button signals a feature that isn't in the immediate roadmap; better to remove than imply availability.

When real OAuth wires later (Phase 1 Spec 1.5g, likely Google + Apple), provider buttons can be re-introduced cleanly with a fresh divider. Don't preserve the divider for a hypothetical future.

Test impact: AuthModal.test.tsx:112-113 asserts on the Spotify button's `border-white/[0.12]` and `text-white`. These assertions get deleted with the button.

### Out of scope (deferred)

**Decision 16: FormField primitive adoption deferred.**

Migrating AuthModal's 6 hand-rolled fields to FormField would substantially expand Spec 7's scope (12 migration sites across both modals + extending FormField to support all current behaviors + significant test rewrites). FormField adoption is a refactor concern, not a visual concern. Belongs in its own dedicated spec (likely part of a future "adopt canonical primitives across forms" cleanup).

The current hand-rolled markup passes a11y audit. No regressions to fix via migration.

**Decision 17: RegisterSkeleton deferred.**

Bundle measurement first. Most users hit /register via AuthModal trigger, not as cold first-page load. Don't ship a skeleton until measurement justifies it.

**Decision 18: AuthContext / mirrorToLegacyKeys / legacy auth keys not touched.**

Spec 7 is visual migration only. AuthContext is out of scope. The `// Transitional — removed in Phase 2 cutover` comments are load-bearing for the future grep-and-remove.

**Decision 19: Real OAuth wiring deferred.**

Phase 1 Spec 1.5g territory. Spec 7 only removes the disabled placeholder.

**Decision 20: Real forgot-password flow deferred.**

Phase 1 Spec 1.5b territory (waits for SMTP). Spec 7 keeps the toast-stub behavior; only the visual chrome of the forgot-password view is in scope (and it's already canonical — verification pass only).
