## Design System Reminder

**Project-specific quirks that `/execute-plan` displays before every UI step:**

- **Active-state class string distinction matters.** For active-foreground variants (a card or row that's part of a parent surface, e.g., DrawerNowPlaying scripture toggle), use `bg-white/15 text-white` with NO extra border. For isolated-pill variants (a chip selector floating in space, e.g., TimerTabContent timer presets), use `bg-white/15 text-white border border-white/30`. Spec 10A's canonical pattern; both variants are correct in their respective contexts. Pre-execution recon Step 7 calls out which variant applies to the RoutineStepper current step (default = active-foreground; alternate = white-pill if pre-execution classifies as CTA).
- **White-pill primary CTA Pattern 2 is verbatim.** The class string in the Design System Values table is the canonical Pattern 2. Apply verbatim to all six 11A sites — do NOT improvise or omit any class. Cross-reference the homepage Get Started button, the RegisterPage hero CTA, and Spec 10A's AvatarPickerModal Save / ChangePasswordModal Update / NotificationsSection Send test notification CTAs for visual matching.
- **`text-primary` fails WCAG AA on `bg-dashboard-dark`.** Migrate to `text-violet-300 hover:text-violet-200` for text-buttons. Decorative `text-primary` adjacent to `bg-primary/X` tinted backgrounds (e.g., RoutineStepper line 41 inside a `bg-primary/30` circle) is a different context — pre-execution recon classifies whether the `text-primary` is decorative-only (preserve per Decision 20) or text-button (migrate per Decision 18). RoutineStepper.tsx:41 is on a small circular icon container — clearly decorative-only.
- **Decorative `bg-primary/X` tints preserved per Decision 20.** Categorical signals (book icon containers, completed-step tints, notification dots, animated waveform bars, active-verse highlights) are NOT CTA chrome. Do NOT migrate them. Decision 20 list is canonical: `BibleSleepSection.tsx:57`, `RoutineStepper.tsx:41` (background tint stays), `DrawerTabs.tsx:73`, `WaveformBars.tsx:12,20,28`, `ScriptureTextPanel.tsx:45`. A 6th site `RoutineCard.tsx:108` is touched by Spec 11B (not 11A) but follows the same precedent.
- **`AudioDrawer rgba(15, 10, 30, 0.95)` background is documented intentional drift.** Per Direction Decision 15, the dark frosted-glass purple background is genuinely different from canonical content cards. Border opacity migrates separately (`border-white/10` → `border-white/[0.12]`); the background does NOT change.
- **`SharedMixHero` purple gradient panel is documented intentional drift.** Per Direction Decision 16, the gradient celebrates receiving a mix from a friend. Sound chips and gradient stay; ONLY the "Play This Mix" CTA migrates.
- **Scene gradient backgrounds (FeaturedSceneCard / SceneCard) preserved.** Per Direction Decision 4, the gradient art IS the chrome — the cards intentionally have no border. Only the active-scene play overlay's inner circle migrates from `bg-primary/80 text-white` to `bg-white text-primary`.
- **Sound button transparent chrome (SoundCard) preserved.** Sound buttons are buttons, not cards. No border opacity migration applies; the transparency is intentional.
- **BB-33 token discipline.** Hardcoded `100ms` and `300ms` migrate to `ANIMATION_DURATIONS.fast` (150ms) and `ANIMATION_DURATIONS.base` (250ms). The 50ms drift is acceptable per `09-design-system.md` § "Animation Tokens" — cluster-wide consistency over millisecond precision. The `setTimeout(..., 3000)` in `SaveMixButton.tsx:92` is a SEMANTIC ARIA-live announcement clear, NOT animation; do NOT migrate.
- **Atmospheric layer preserved per Direction Decision 1.** `bg-dashboard-dark` body root + `<PageHero>` + `ATMOSPHERIC_HERO_BG` stay exactly as they are. Do NOT introduce per-tab atmospheric variants. Do NOT introduce `BackgroundCanvas`, `HorizonGlow`, or `GlowBackground` (those are scoped to Daily Hub or homepage).
- **Audio engine integrity per Direction Decision 24.** `AudioProvider`, `audioReducer`, `lib/audio-engine.ts`, and all 10 audio-engine hooks are OUT OF SCOPE. If apparent engine bugs surface during execution, REPORT them — do NOT silently patch. `/code-review` flags any diff to engine files.
- **`wr_listening_history` schema stability per Direction Decision 25.** Read-only on the visual side. No field additions, removals, or renames. Cross-cutting consumers (`useGardenElements`, `badge-engine`) depend on schema stability.

This block is displayed verbatim by `/execute-plan` Step 4d before each UI step to prevent mid-implementation drift back to default assumptions.
