## Spec-Category-Specific Guidance

This is a **constants-and-types foundation spec** — neither a dual-write spec nor a cutover spec nor a backend-only spec nor a frontend-only spec in the traditional sense. It has both backend and frontend touch points, but they are minimal:

- **Backend:** add ONE static method (`isValid(String)`) to `PostType.java` and create a new test file (`PostTypeTest.java`).
- **Frontend:** create the new canonical constants file (`post-types.ts`) plus its test file; modify `prayer-wall.ts` to remove the local type and add a re-export shim.

Step ordering follows the safety convention: backend first (one method + test), frontend second (new file + test + re-export shim). The drift test runs as part of the frontend test file. Validation is a single typecheck pass + both test suites.

No category-specific guidance from the four canonical patterns (DUAL-WRITE, CUTOVER, BACKEND-ONLY, FRONTEND-ONLY) applies cleanly. Guidance for this micro-spec:

1. Keep the Java enum lean — `isValid(String)` is the ONLY new method (D4).
2. Keep the constants tree-shakable — `icon` is a string, NOT a component (D8).
3. Keep the type alias unchanged in semantics — `PostType` remains the union of all 5 ids regardless of `enabled` flag (D2, W3).
4. Keep `PrayerRequest.postType` optional — the field-required flip is 4.2's work (D7, W4).
5. The drift test's `EXPECTED_BACKEND_VALUES` array MUST be hard-coded, NOT derived from `POST_TYPES` (D5, W11).

### Step 1: Add `isValid(String)` static method to backend `PostType.java`

**Objective:** Provide a non-throwing validation helper that callers can use to check a string value before parsing, complementing the existing throwing `fromValue(String)`.

**Files to create/modify:**
- `backend/src/main/java/com/worshiproom/post/PostType.java` — add ONE static method (~6 lines).

**Details:**

Append this method to the `PostType` enum (after the existing `fromValue`):

```java
public static boolean isValid(String value) {
    if (value == null) return false;
    for (PostType t : values()) {
        if (t.value.equals(value)) return true;
    }
    return false;
}
```

The method:
- Returns `false` for null (does NOT throw)
- Returns `false` for any string not matching one of the 5 enum values
- Returns `true` for exactly the 5 valid values: `prayer_request`, `testimony`, `question`, `discussion`, `encouragement`
- Is `null`-safe by deliberate design (W9)

- DO NOT modify the existing `fromValue(String)` method's behavior — it must continue to throw on null and unknown values (used by the JPA converter path where invalid values indicate data corruption and SHOULD throw).
- DO NOT add `displayLabel()`, `pluralLabel()`, or any other display-related method to the Java enum — those are frontend brand-voice concerns (D4, W10).
- DO NOT change the order of the existing 5 enum constants.
- DO NOT add new enum constants in this spec.

```typescript
/**
 * Canonical post-type definitions for the unified posts table (Phase 4.1).
 *
 * The `id` field MUST match the backend Java PostType enum's `value()` strings
 * character-for-character. The drift test in __tests__/post-types.test.ts is
 * the contract — see that file's top comment for the sync rule.
 *
 * Brand voice (D3): every label, pluralLabel, and description has passed the
 * pastor's-wife test. No exclamation points, no urgency, no obligation
 * language. Descriptions are sentence-case + period; labels are noun-phrase
 * title case without period.
 */
export const POST_TYPES = [
  {
    id: 'prayer_request',
    label: 'Prayer request',
    pluralLabel: 'Prayer requests',
    icon: 'HandHelping',
    description: 'Lift up a need to the Lord with the community.',
    enabled: true,
  },
  {
    id: 'testimony',
    label: 'Testimony',
    pluralLabel: 'Testimonies',
    icon: 'Sparkles',
    description: 'Share what God has done in your life.',
    enabled: false,
  },
  {
    id: 'question',
    label: 'Question',
    pluralLabel: 'Questions',
    icon: 'HelpCircle',
    description: 'Ask the community for wisdom or perspective.',
    enabled: false,
  },
  {
    id: 'discussion',
    label: 'Discussion',
    pluralLabel: 'Discussions',
    icon: 'MessagesSquare',
    description: 'Open a thread on a passage or topic worth exploring.',
    enabled: false,
  },
  {
    id: 'encouragement',
    label: 'Encouragement',
    pluralLabel: 'Encouragements',
    icon: 'Heart',
    description: 'Speak a word of life over the community.',
    enabled: false,
  },
] as const

export type PostType = (typeof POST_TYPES)[number]['id']

export type PostTypeEntry = (typeof POST_TYPES)[number]

export function isValidPostType(value: string | null): value is PostType {
  if (value === null) return false
  return POST_TYPES.some((t) => t.id === value)
}

export function getPostType(id: PostType): PostTypeEntry {
  const entry = POST_TYPES.find((t) => t.id === id)
  if (!entry) {
    throw new Error(
      `Unknown post type id: ${id}. Did you mean one of: ${POST_TYPES.map((t) => t.id).join(', ')}`,
    )
  }
  return entry
}
```

Key shape decisions:
- `as const` makes the array readonly and gives precise type inference for each `id` string literal (D1).
- `PostType` is derived from `(typeof POST_TYPES)[number]['id']` — automatically the union of all 5 ids (D1).
- `PostTypeEntry` is the per-entry type, exported for consumers that pass entries around (e.g., Composer Chooser cards).
- `isValidPostType` is a TypeScript type predicate (`value is PostType`) so consumers narrow correctly.
- `getPostType` throws with a descriptive error on unknown id — never returns undefined silently (W12).
- Single quotes for string literals per project convention (W14).

- DO NOT make `PostType` the union of only `enabled === true` ids — the type must include all 5 because the BACKEND accepts all 5. The `enabled` flag is for UI gating, not type narrowing (D2, W3).
- DO NOT add exclamation points to any description string (W6, Rule 12).
- DO NOT mix sentence case + period for labels (labels are title-case noun phrases without period; descriptions are sentence-case with period — W8).
- DO NOT add additional helper functions in this spec. Anything beyond `isValidPostType` and `getPostType` is YAGNI for 4.1; future specs can add as needed.
