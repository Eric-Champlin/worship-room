## Architecture Context

### Recon findings (R7–R16 from brief, OPENAPI-1, MODAL-1)

**R7 — Friend schema:** ✅ Path A (real classification). `friend_relationships` table shipped Phase 2.5 at `backend/src/main/resources/db/changelog/2026-04-27-009-create-friend-relationships-table.xml`. Composite PK on `(user_id, friend_user_id)`. Mutual friendships are stored as TWO rows (A→B and B→A inserted by `FriendsService.acceptRequest`). Status `'active'` or `'blocked'`. **To find friends of post.author_id (call them author A):** `SELECT friend_user_id FROM friend_relationships WHERE user_id = A AND status = 'active'`. Repository at `com.worshiproom.friends.FriendRelationshipRepository` exposes `findAllByUserIdAndStatus(userId, status)` returning `List<FriendRelationship>`.

**R8 — PrayerCard.tsx:** 229 lines. Has NO viewer plumbing today — `authorLink` derives from `prayer.userId` alone. Existing author-vs-viewer branching is anonymity-driven (`prayer.isAnonymous`). Insertion point for `<PrayerReceipt>`: directly above `{children}` at line 217, ABOVE `<InteractionBar>` which renders via the `children` prop from parent (`PrayerCardWithBar` pattern — `<PrayerCard><InteractionBar/></PrayerCard>`). **Decision:** add `useAuth()` inside PrayerCard to derive viewer; wrap `<PrayerReceipt>` with `viewer?.user_id === prayer.userId && !prayer.isAnonymous ? null` gate (anonymous posts: author still sees receipt via `prayer.userId` set server-side — see R13).

**R9 — PrayerWallDashboard.tsx (corrected path: `frontend/src/pages/PrayerWallDashboard.tsx`):** 761 lines. Renders the author's own posts list. Plan recon flag: the `/execute-plan-forums` step that touches this file MUST read it end-to-end first and identify the per-post card JSX block, then insert the mini-receipt component (`<PrayerReceiptMini postId={post.id} />`) as the last child of each own-post card's body. The mini-receipt component handles its own fetch + null-at-zero rendering.

**R10 — Settings page:** `frontend/src/pages/Settings.tsx` (230 lines). Section structure: `['profile', 'dashboard', 'notifications', 'privacy', 'account', 'app']` via `SECTIONS` constant + `?tab=` query param. Section components live in `frontend/src/components/settings/` (e.g., `PrivacySection.tsx`). **Decision:** add the toggle inside `PrivacySection.tsx` as a new subsection labeled "Prayer Receipts" — single click from `/settings` to reach the Privacy tab, then visible without scrolling on most viewports. This satisfies Eric's "single toggle directly on the settings page" requirement (out-of-band note in spec).

**R11 — `wr_settings` localStorage shape:** Backed by `services/settings-storage.ts`, hooked via `frontend/src/hooks/useSettings.ts`. `UserSettings` type at `frontend/src/types/settings.ts` currently has `{ profile, notifications, privacy }`. **Plan extension:** add `prayerWall: { prayerReceiptsVisible: boolean }` namespace to `UserSettings`; default `true`; `useSettings` gains `updatePrayerWall(updates: Partial<UserSettingsPrayerWall>)`. Backward-compat in `settings-storage.ts`: when reading an older blob missing the `prayerWall` key, merge in defaults.

**R12 — Existing fetch hook pattern:** Canonical client at `frontend/src/lib/api-client.ts` exposes `apiFetch<T>(path, options)`. Auto-prepends `VITE_API_BASE_URL`, attaches `Authorization: Bearer <token>` from `getStoredToken()`, parses `{ data, meta }` envelope and returns `data` typed, dispatches `wr:auth-invalidated` on 401, 30s timeout via AbortController, throws typed `ApiError` for non-2xx. **`usePrayerReceipt` uses `apiFetch<PrayerReceiptResponse>` — no custom fetch boilerplate.**

**R13 — Anonymous-post implementation:** `posts.author_id` is always populated server-side regardless of `is_anonymous`. The flag affects ONLY what non-authors see (the frontend hides the author chip + uses generic avatar when `is_anonymous=true`). The backend's 403 check uses `posts.author_id` directly — anonymous-post authors fetching their own receipt always succeed. **Frontend gate must use `prayer.userId === viewer.user_id` (server-supplied `userId` is the real author_id; the `isAnonymous` flag is orthogonal to receipt visibility).**

**R14 — Crisis-flag handling:** Crisis posts still have `posts.author_id` populated and `posts.praying_count` increments normally. 6.1 receipts behave identically for crisis-flagged and non-flagged posts. No special handling needed.

**R15 — Existing reaction-toggle service:** `backend/src/main/java/com/worshiproom/post/engagement/ReactionWriteService.java`. Methods `toggle(postId, userId, reactionType, requestId)` and `remove(postId, userId, reactionType, requestId)` are both `@Transactional`. Increment/decrement of `posts.praying_count` happens via `postRepository.incrementPrayingCount(postId)` / `decrementPrayingCount(postId)` (both `@Modifying(clearAutomatically=true, flushAutomatically=true)`). **`@CacheEvict("prayer-receipt", key = "#postId")` placement:** on both `toggle()` and `remove()` methods, alongside the existing `@Transactional`. Eviction fires AFTER method body returns (default `beforeInvocation=false`) but BEFORE transaction commit; a rollback scenario produces a cache miss (not stale data) since the next reader queries fresh DB. Acceptable per Gate-33.

**R16 — Existing PostController structure:** `backend/src/main/java/com/worshiproom/post/PostController.java`. Lives in `com.worshiproom.post` package so the existing `PostExceptionHandler` picks up its exceptions.

**PLAN-RECON-MODAL-1:** Canonical modal pattern: `AuthModal.tsx` consumes `useFocusTrap()` hook (`hooks/useFocusTrap.ts`). ### Cache architecture (consumer of 5.6)

`backend/src/main/java/com/worshiproom/cache/CacheConfig.java` is **profile-aware**:
- **dev/test profile** → `ConcurrentMapCacheManager` (in-process, no TTL enforcement)
- **prod profile** → `RedisCacheManager` (auto-prefixed `cache:` namespace per `CACHE_KEY_PREFIX`) wrapped in `CircuitBreakingCacheManager` (3 failures → 30s open → recovery)

Per-cache TTL declared via `spring.cache.redis.time-to-live.<cacheName>=<duration>` in `application.properties`. `RepoWideTtlEnforcementTest` (Spec 5.6) asserts every `@Cacheable` cacheName has a TTL property — **adding `@Cacheable("prayer-receipt")` without `spring.cache.redis.time-to-live.prayer-receipt=30s` will fail this enforcement test.**

**Cache key shape:** With `@Cacheable("prayer-receipt", key = "#postId")`, the Redis key in prod becomes `cache:prayer-receipt:<postId>` (per `computePrefixWith` rule). Matches brief W32/W44 verbatim.

**Testing strategy:** `PrayerReceiptIntegrationTest` registers Redis via `TestContainers.registerRedisProperties()` AND uses an inline `@TestConfiguration` to override the default in-memory `inMemoryCacheManager` bean with a real `RedisCacheManager` for this test class. This activates the production cache path (TTL, `cache:` prefix) under Testcontainers Redis. Test 1 #8 (`redisTemplate.getExpire("cache:prayer-receipt:<postId>")`) becomes feasible.

### Existing rate-limit pattern (D-RateLimit-pattern / MPD-7)

Canonical reference: `backend/src/main/java/com/worshiproom/post/engagement/ReactionsRateLimitService.java` (61 lines) + sibling `ReactionsRateLimitConfig.java` (35 lines, `@ConfigurationProperties(prefix = "worshiproom.reactions")`). Shape:
- **Config class:** `@Configuration @ConfigurationProperties(prefix = "worshiproom.{feature}")` exposing nested `RateLimit { int maxPerHour; int bucketCacheSize; }`.
- **Service class:** Caffeine bounded bucket map `maximumSize(config.bucketCacheSize) expireAfterAccess(2h)`; bucket4j `Bandwidth.classic(N, Refill.intervally(N, Duration.ofHours(1)))` (or other duration); throws domain-specific `*RateLimitException` carrying `retryAfterSec`.
- **Exception:** extends `PostException`, code `RATE_LIMITED`, status `429`, message generic; handled by `PostExceptionHandler.handleRateLimit` which sets `Retry-After` header.
- **Caller:** `service.checkAndConsume(userId)` at controller-or-service boundary.

6.1 ships two new rate-limit services in this pattern (W15 / D-RateLimit-pattern). The new `RateLimiter` interface from 5.6 is NOT consumed (strangler preserved).

## API Changes

| Method | Endpoint | Auth | Rate Limit | Request Body | Response |
|--------|----------|------|-----------|-------------|----------|
| GET | `/api/v1/posts/{id}/prayer-receipt` | Required (Bearer JWT); 403 unless `requester.user_id == post.author_id` | 60/hour/user | — | `200 OK` with `{ data: PrayerReceiptResponse, meta: { requestId } }`. 401 unauthenticated, 403 non-author, 404 post not found / soft-deleted, 429 rate-limited. |
| POST | `/api/v1/posts/{id}/prayer-receipt/share` | Required (Bearer JWT); 403 unless `requester.user_id == post.author_id` | 5/day/user/post (composite bucket key `{postId}:{userId}`) | — (no body) | `204 No Content` on success. 401, 403, 404, 429 same as above. No DB write — pure rate-limit check that records the share event in the bucket. |

**`PrayerReceiptResponse` wire format (Gate-32):**
```
{
  total_count: number,                          // posts.praying_count
  attributed_intercessors: Array<{
    user_id: string,                            // friend UUID
    display_name: string,                       // friend display name
    avatar_url: string | null
  }>,                                            // friends of author who reacted
  anonymous_count: number                       // non-friend reactors (count only)
}
// invariant: total_count == attributed_intercessors.length + anonymous_count
```

**Invariant (server-side enforced):** non-friend intercessor `user_id` / `display_name` / `avatar_url` never on the wire.

## Edge Cases & Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| `@CacheEvict` atomicity vs commit | Use default `@CacheEvict("prayer-receipt", key = "#postId")` annotation on `ReactionWriteService.toggle()` and `remove()`. Eviction fires post-method-body, pre-commit. Worst-case rollback scenario → cache miss (next reader queries fresh DB), not stale data. | Standard Spring pattern; matches brief D-Cache-invalidation; avoids `TransactionSynchronizationManager.registerSynchronization(...)` complexity. Gate-33's "no stale-count window beyond commit" is satisfied because the only failure mode is over-eviction. |
| Viewer plumbing in PrayerCard | Add `useAuth()` inside PrayerCard. Receipt component itself reads `useAuth()` + `useSettings()` and returns `null` at any gate failure (defense in depth W30). | Smaller diff than prop-threading viewer from 4 parents (PrayerWall, PrayerDetail, PrayerWallDashboard, PrayerWallProfile); confines gating logic to one place. |
| `usePrayerReceipt` fire condition | Hook accepts `enabled` arg; only fires fetch when `enabled === true`. Caller (PrayerReceipt component) computes `enabled = isAuthor && praying_count > 0 && prayerReceiptsVisible`. | Avoids needless 403/empty fetches; matches W30 defense in depth. |
| Settings toggle placement | `PrivacySection.tsx` with new "Prayer Receipts" subsection. NOT a new top-level Settings section. | Single click from `/settings`; doesn't break the existing 6-section tab structure; consistent with "privacy-related visibility control" semantics. |
| PNG aspect ratio default | **1080×1080 (Instagram square).** Single canonical aspect ratio in 6.1; deferring 1200×630 (Twitter/X card) to a future spec if demand emerges. | Most universal share surface; symmetrical design easier to typeset. Brief D-PNG-size left as "plan recon picks the default." |
| PII-stripping path | Defer to follow-up issue. Spec 6.7 (which owns the PII-strip path) ships after 6.1; html2canvas-generated PNGs don't carry EXIF (Canvas API renders fresh pixels, no camera metadata). Master plan AC explicitly permitted defer per brief § 11. | Functional outcome equivalent (PNG has no PII), without preempting 6.7's design. |
| Bundle budget test mechanism | `frontend/scripts/check-bundle-budget.mjs` — Node post-build script. Reads `frontend/dist/assets/*.js.gz` total gzipped size, compares to `frontend/scripts/bundle-budget-baseline.json` (committed BEFORE Step 12 adds html2canvas), asserts delta ≤ 30 KB, fails with non-zero exit on overrun. Wired as `pnpm check-bundle-budget` script. | Simplest mechanism that's CI-portable, deterministic, and doesn't require a Vite plugin. Baseline captured pre-html2canvas in Step 1. |
| Receipt for soft-deleted posts | 404 (matches `loadVisiblePost` in `ReactionWriteService` — soft-deleted posts not visible to anyone, even author). | Consistent with existing post-visibility semantics in `PostSpecifications.visibleTo`. |
| Test integration test Redis activation | `PrayerReceiptIntegrationTest` declares an inline `@TestConfiguration` that overrides the default `inMemoryCacheManager` bean with a `RedisCacheManager` for this test class only (calls `TestContainers.registerRedisProperties()` in `@DynamicPropertySource`). import com.worshiproom.friends.FriendRelationship;

```java
import com.worshiproom.post.dto.AttributedIntercessor;

import com.worshiproom.user.User;
import com.worshiproom.user.UserRepository;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.cache.annotation.Cacheable;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.util.*;

@Service
public class PrayerReceiptService {

    private static final Logger log = LoggerFactory.getLogger(PrayerReceiptService.class);

    private final PostRepository postRepository;
    private final ReactionRepository reactionRepository;
    private final FriendRelationshipRepository friendRepository;
    private final UserRepository userRepository;

    public PrayerReceiptService(/* constructor injection */) { ... }

    /**
     * Author-only — caller MUST verify requester.userId == post.authorId before invocation.
     * Service does NOT re-check (controller's responsibility, after the @AuthenticationPrincipal extraction).
     */
    @Cacheable(value = "prayer-receipt", key = "#postId")
    @Transactional(readOnly = true)
    public PrayerReceiptResponse buildReceipt(UUID postId) {
        Post post = postRepository.findById(postId)
            .orElseThrow(PostNotFoundException::new);

        long totalCount = post.getPrayingCount();

        // Enumerate praying reactors
        List<UUID> reactorIds = reactionRepository.findReactorIdsByPostIdAndReactionType(postId, "praying");

        // Author's friends (active relationships, A → B direction)
        Set<UUID> authorFriendIds = friendRepository
            .findAllByUserIdAndStatus(post.getAuthorId(), FriendRelationshipStatus.ACTIVE)
            .stream()
            .map(FriendRelationship::getFriendUserId)
            .collect(Collectors.toSet());

        // Classify
        List<UUID> friendReactorIds = reactorIds.stream()
            .filter(authorFriendIds::contains)
            .toList();
        long anonymousCount = reactorIds.size() - friendReactorIds.size();

        // Hydrate friend display data
        List<AttributedIntercessor> attributed = userRepository.findAllById(friendReactorIds).stream()
            .sorted(Comparator.comparing(User::getDisplayName, String.CASE_INSENSITIVE_ORDER))
            .map(u -> new AttributedIntercessor(u.getId(), u.getDisplayName(), u.getAvatarUrl()))
            .toList();

        log.info("prayerReceiptBuilt postId={} totalCount={} attributedCount={} anonymousCount={}",
            postId, totalCount, attributed.size(), anonymousCount);

        return new PrayerReceiptResponse(totalCount, attributed, anonymousCount);
    }
}
```

**Rationale and trade-off (Gate-33 analysis):**
Default `@CacheEvict` runs `beforeInvocation=false` (the default), meaning the cache is evicted AFTER the method body returns but BEFORE the surrounding transaction commits. Sequence:
1. `toggle()` invoked
2. DB INSERT into `post_reactions` + UPDATE `posts.praying_count`
3. Method body returns (success)
4. `@CacheEvict` fires (cache entry removed)
5. Transaction commits

Failure mode analysis:
- If the method body throws (DB error, validation error), `@CacheEvict` does NOT fire — cache stays whatever it was. Acceptable.
- If `@CacheEvict` itself fails (Redis unreachable mid-eviction), the circuit breaker (5.6) catches it and the eviction is silently no-op'd. Next reader gets stale data until TTL (30s) — acceptable degraded behavior.
- If `@CacheEvict` succeeds but the transaction THEN rolls back: cache now empty, DB unchanged. Next reader gets a cache miss, queries DB, repopulates cache from unchanged data. **Acceptable** — produces a cache miss, never stale data.

**Guardrails (DO NOT):**
- Do not set `beforeInvocation=true` — that evicts before the DB write, opening a stale-repopulation window
- Do not switch to programmatic `TransactionSynchronizationManager.registerSynchronization()` unless Eric explicitly requests the more rigorous pattern (current default is acceptable per Gate-33 analysis above)
- Do not also evict on `loadVisiblePost` errors — that's exception path, no state change to invalidate

```typescript
      setError(null)
      return
    }
    setLoading(true)
    setError(null)
    try {
      const response = await apiFetch<PrayerReceiptResponse>(
        `/api/v1/posts/${postId}/prayer-receipt`,
      )
      setData(response)
    } catch (e) {
      setError(e instanceof ApiError ? e : new ApiError('UNKNOWN', 0, 'Unknown error', null))
      setData(null)
    } finally {
      setLoading(false)
    }
  }, [postId, enabled])

  useEffect(() => {
    fetcher()
  }, [fetcher])

  return { data, loading, error, refetch: fetcher }
}
```

```tsx
  // Dynamic import keeps html2canvas out of the initial bundle (W34)
  const { default: html2canvas } = await import('html2canvas')

  setShareCardMounted(true)
```
