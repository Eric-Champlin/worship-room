### Exception strategy

| Case | Strategy | HTTP status | Code |
|---|---|---|---|
| `from == to` | Reuse `com.worshiproom.friends.SelfActionException` (caught by unscoped `FriendsExceptionHandler`) | 400 | `SELF_ACTION_FORBIDDEN` |
| Target user missing/deleted/banned | Reuse `com.worshiproom.friends.UserNotFoundException` | 404 | `USER_NOT_FOUND` |
| `weekStart` validation, message length, etc. | Reuse `com.worshiproom.friends.InvalidInputException` (service-thrown), and `SocialValidationExceptionHandler` for `@Valid` body failures | 400 | `INVALID_INPUT` |
| Not friends | NEW `com.worshiproom.social.NotFriendsException` (extends new `SocialException` base) | **403** | `NOT_FRIENDS` |
| Hourly or per-friend daily cap exceeded | NEW `com.worshiproom.social.RateLimitedException` | 429 | `RATE_LIMITED` |
| Nudge cooldown active | NEW `com.worshiproom.social.NudgeCooldownException` | 409 | `NUDGE_COOLDOWN` |

`SocialExceptionHandler` is package-scoped to `com.worshiproom.social` and handles ONLY `SocialException` subtypes. `FriendsExceptionHandler` (unscoped) catches the cross-package friends-domain exceptions thrown from `SocialController`/`SocialInteractionsService`. `SocialValidationExceptionHandler` is package-scoped to `com.worshiproom.social` and handles `@Valid` body failures, missing query params, and path-param type mismatches — same shape as `FriendsValidationExceptionHandler`.

### JSONB serialization pattern (canonical)

Per `ActivityLog.metadata`:

```java
@JdbcTypeCode(SqlTypes.JSON)
@Column(name = "payload", columnDefinition = "jsonb")
private Map<String, Object> payload;
```

Same for `MilestoneEvent.eventMetadata`. No third-party `hibernate-types` dependency. No custom converter. Hibernate 6 native pattern.

### Streak/Friend milestone thresholds

- **Streak thresholds:** `{7, 14, 30, 60, 90, 180, 365}` from `BadgeThresholds.STREAK`. Emit `STREAK_MILESTONE` when `newStreak.currentStreak() == threshold` AND `oldStreak.currentStreak() < threshold` for any threshold in the set (i.e., a clean crossing).
- **Friend thresholds:** `{1, 10}` from `BadgeThresholds.FRIENDS`. Emit `FRIEND_MILESTONE` when the user's post-accept friend count `== threshold` AND pre-accept count `< threshold`. Computed from `friend_relationships` rows after the two `INSERT`s in `acceptRequest`.
- **Prayer count milestones:** Enum value `PRAYER_COUNT_MILESTONE` declared (DB CHECK requires it) but **no emission logic in this spec** — defer per spec body. A future spec adds emission when prayer count thresholds become canonical.

## Spec-Category-Specific Guidance

This is a **dual-write spec** (Phase 2.5 migration pattern). Pattern applied:

1. Existing `wr_social_interactions` writes stay unchanged (localStorage source-of-truth).
2. Backend shadow writer added via fire-and-forget POSTs from the three `useSocialInteractions` mutations.
3. `VITE_USE_BACKEND_SOCIAL` flag introduced (default `false`). The flag-flip and read-switch are owned by Spec 2.5.5 (Phase 2.5 cutover).
4. Drift detection between local and backend state is NOT mandatory for this dual-write spec — Phase 2 used drift detection for the activity engine because the calculation logic was duplicated. Social interactions are insert-only writes with no recomputed-derived state, so drift detection adds no value and is omitted.
5. Step order: backend tables (already done) → backend services + endpoints → backend milestone emission integration → frontend env flag + API client → frontend hook wiring + tests.

## Edge Cases & Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Reuse `friends.SelfActionException` / `friends.UserNotFoundException` / `friends.InvalidInputException`? | Yes, import directly. | `FriendsExceptionHandler` is unscoped (per its class JavaDoc). HTTP code + error code already match. Duplicating would be churn. |
| Reuse `friends.NotFriendsException`? | **No**, create `social.NotFriendsException`. | Friends version returns 404; social spec requires 403. Same name, different package, different status. |
| `to_user_id` for recap dismissal? | Self-row: `to_user_id = from_user_id`. | Schema requires NOT NULL (Spec 2.5.1 changeset line 30). Migration to nullable is out of scope per spec body. Document in service Javadoc. |
| Hourly rate-limit window query? | `OffsetDateTime.now(ZoneOffset.UTC).minusHours(1)`. | Matches `ActivityService` pattern. Not `Instant`, not `LocalDateTime`. |
| JSONB serialization? | `@JdbcTypeCode(SqlTypes.JSON)` + `Map<String, Object>`. | Canonical pattern from `ActivityLog`. Hibernate 6 native — no `hibernate-types` dependency. |
| `MilestoneEventType.PRAYER_COUNT_MILESTONE` emission? | Declared in enum, NOT emitted in this spec. | DB CHECK requires the enum value; emission logic deferred per spec. |
| Frontend rate-limit cap for backend `application.properties`? | `daily-cap-per-friend=3`, `nudge.cooldown-days=7`. | Matches existing frontend constants — drift would cause inconsistent UX. |
| Hourly cap for nudges and encouragements? | `60/hr per user` for both, separate config keys. | Spec body. Each enforcement counts only its own interaction type. |
| Drift-detection test? | Skipped. | No recomputed-derived state; insert-only writes. Activity engine drift test was load-bearing because frontend and backend computed the same level/streak math; this spec has nothing computed both sides. |
| Streak crossing detection in `ActivityService`? | Compare pre-call streak (`streakData.currentStreak()`) to post-call streak (`streakResult.newState().currentStreak()`); emit `STREAK_MILESTONE` once per threshold the new streak crossed. | The streak calculation already runs in `recordActivity`. Reusing that result is cheaper than recalculating. |
| Friend count after `acceptRequest`? | Query `friend_relationships` after the two saves: `relationshipRepo.countByUserIdAndStatus(actingUserId, ACTIVE)`. | A new repo method; cheaper and more accurate than tracking deltas. |

**Objective:** Service with three public methods (`sendEncouragement`, `sendNudge`, `dismissRecap`) implementing all validation, rate-limit, and cooldown logic.

**Files to create:**
- `backend/src/main/java/com/worshiproom/social/SocialInteractionsService.java`.

**Details:**

```java
@Service
public class SocialInteractionsService {

    private static final ZoneOffset UTC = ZoneOffset.UTC;

    private final SocialInteractionRepository socialRepo;
    private final FriendRelationshipRepository relationshipRepo;
    private final UserRepository userRepository;
    private final SocialLimitsConfig limits;

    public SocialInteractionsService(SocialInteractionRepository socialRepo,
                                     FriendRelationshipRepository relationshipRepo,
                                     UserRepository userRepository,
                                     SocialLimitsConfig limits) {
        this.socialRepo = socialRepo;
        this.relationshipRepo = relationshipRepo;
        this.userRepository = userRepository;
        this.limits = limits;
    }

    @Transactional
    public SocialInteraction sendEncouragement(UUID fromUserId, UUID toUserId, String message) {
        if (fromUserId.equals(toUserId)) {
            throw new com.worshiproom.friends.SelfActionException(
                "Cannot send encouragement to yourself");
        }
        userRepository.findById(toUserId)
            .filter(u -> !u.isDeleted() && !u.isBanned())
            .orElseThrow(() -> new com.worshiproom.friends.UserNotFoundException(
                "Target user not found"));
        if (!relationshipRepo.existsByUserIdAndFriendUserIdAndStatus(
                fromUserId, toUserId,
                com.worshiproom.friends.FriendRelationshipStatus.ACTIVE)) {
            throw new NotFriendsException(
                "Cannot send encouragement to a non-friend");
        }
        OffsetDateTime hourAgo = OffsetDateTime.now(UTC).minusHours(1);
        OffsetDateTime dayAgo = OffsetDateTime.now(UTC).minusDays(1);
        long hourly = socialRepo.countByFromUserIdAndInteractionTypeAndCreatedAtGreaterThanEqual(
            fromUserId, SocialInteractionType.ENCOURAGEMENT, hourAgo);
        if (hourly >= limits.encouragement().hourlyCapPerUser()) {
            throw new RateLimitedException(
                "Hourly encouragement cap reached. Please try again later.");
        }
        long perFriend = socialRepo.countByFromUserIdAndToUserIdAndInteractionTypeAndCreatedAtGreaterThanEqual(
            fromUserId, toUserId, SocialInteractionType.ENCOURAGEMENT, dayAgo);
        if (perFriend >= limits.encouragement().dailyCapPerFriend()) {
            throw new RateLimitedException(
                "Daily encouragement cap for this friend reached.");
        }
        SocialInteraction row = new SocialInteraction(
            fromUserId, toUserId, SocialInteractionType.ENCOURAGEMENT,
            Map.of("message", message));
        return socialRepo.save(row);
    }

    @Transactional
    public SocialInteraction sendNudge(UUID fromUserId, UUID toUserId) {
        if (fromUserId.equals(toUserId)) {
            throw new com.worshiproom.friends.SelfActionException(
                "Cannot nudge yourself");
        }
        userRepository.findById(toUserId)
            .filter(u -> !u.isDeleted() && !u.isBanned())
            .orElseThrow(() -> new com.worshiproom.friends.UserNotFoundException(
                "Target user not found"));
        if (!relationshipRepo.existsByUserIdAndFriendUserIdAndStatus(
                fromUserId, toUserId,
                com.worshiproom.friends.FriendRelationshipStatus.ACTIVE)) {
            throw new NotFriendsException(
                "Cannot nudge a non-friend");
        }
        Optional<SocialInteraction> last = socialRepo
            .findFirstByFromUserIdAndToUserIdAndInteractionTypeOrderByCreatedAtDesc(
                fromUserId, toUserId, SocialInteractionType.NUDGE);
        if (last.isPresent()) {
            OffsetDateTime cutoff = OffsetDateTime.now(UTC)
                .minusDays(limits.nudge().cooldownDays());
            if (last.get().getCreatedAt().isAfter(cutoff)) {
                throw new NudgeCooldownException(
                    "Nudge cooldown active for this friend.");
            }
        }
        OffsetDateTime hourAgo = OffsetDateTime.now(UTC).minusHours(1);
        long hourly = socialRepo.countByFromUserIdAndInteractionTypeAndCreatedAtGreaterThanEqual(
            fromUserId, SocialInteractionType.NUDGE, hourAgo);
        if (hourly >= limits.nudge().hourlyCapPerUser()) {
            throw new RateLimitedException(
                "Hourly nudge cap reached. Please try again later.");
        }
        SocialInteraction row = new SocialInteraction(
            fromUserId, toUserId, SocialInteractionType.NUDGE, null);
        return socialRepo.save(row);
    }

    /**
     * Dismiss a weekly recap. Schema requires {@code to_user_id NOT NULL}
     * (Spec 2.5.1 changeset 011), so this self-action row sets
     * {@code to_user_id = from_user_id}. Future schema change to make
     * {@code to_user_id} nullable is out of scope.
     */
    @Transactional
    public SocialInteraction dismissRecap(UUID fromUserId, String weekStart) {
        // weekStart already validated by @Pattern; keep a defensive parse so a
        // future spec calling this method directly still gets an INVALID_INPUT.
        try {
            java.time.LocalDate.parse(weekStart);
        } catch (java.time.format.DateTimeParseException e) {
            throw new com.worshiproom.friends.InvalidInputException(
                "weekStart must be a valid YYYY-MM-DD date");
        }
        SocialInteraction row = new SocialInteraction(
            fromUserId, fromUserId, // self-row hack per Javadoc
            SocialInteractionType.RECAP_DISMISSAL,
            Map.of("weekStart", weekStart));
        return socialRepo.save(row);
    }
}
```

- DO NOT skip the `from == to` check in any of the three methods (well, recap-dismissal is self-action — it specifically does not check, by design).

private void emitFriendMilestoneIfThresholdCrossed(UUID userId) {
    long count = relationshipRepo.countByUserIdAndStatus(
        userId, FriendRelationshipStatus.ACTIVE);
    for (int threshold : com.worshiproom.activity.constants.BadgeThresholds.FRIENDS) {
        if (count == threshold) {
            milestoneEventsService.recordEvent(
                userId, MilestoneEventType.FRIEND_MILESTONE,
                Map.of("friendCount", threshold));
            break; // only one threshold per call (count == threshold can only match once)
        }
    }
}

Per the spec's phrasing this is ambiguous. **Decision: emit for both parties.** Both users equally cross a threshold; suppressing the sender's milestone would be unfair (and arbitrarily complex to detect later — you'd need a separate emission point on send). The sender's `friend_relationships` count grows by 1 in this call, identical to the recipient's. Both deserve the milestone emission.
- DO NOT skip the `break` — without it, an unbounded list of thresholds with the same value would emit duplicates (currently safe but defensive).
