### G) Indirect call sites (where `recordActivity` is threaded through a helper)

Three indirect surfaces. The helpers `useChallengeProgress.completeDay` and `useChallengeAutoDetect` accept a `recordActivityFn?: (type: ActivityType) => void` parameter and forward it. Updating the helpers' parameter types would force every caller of those helpers to also be aware of `sourceFeature`, which couples concerns. **Cleaner pattern:** at each call site that passes `recordActivity` through, wrap it once with the appropriate source feature:

```ts
// Before:

// After:
completeDay(challengeId, selectedDay, (type) => faithPoints.recordActivity(type, 'challenge'))
```

## Spec-Category-Specific Guidance

This spec is a **DUAL-WRITE spec** (Phase 2 migration pattern). Required step structure per the plan template applies:

1. **localStorage writer preservation** — the existing flow inside `recordActivity` (lines 138–253) is NOT removed or modified. It runs first and produces all UX (celebrations, sound effects, state updates).
2. **Backend shadow writer** — fire-and-forget `POST /api/v1/activity` after the localStorage path completes. Failure must NOT block the localStorage write or surface to the user. Logs at `console.warn` for observability (no Sentry per spec Decision 5).
3. **Feature flag wiring** — `VITE_USE_BACKEND_ACTIVITY` (default `false`). The dual-write branch is gated by `isBackendActivityEnabled()`. There is no read branch in this spec — reads stay on localStorage; the backend is shadow only. Spec 2.9 owns the cutover.
4. **Drift detection test** — owned by Spec 2.8, NOT this spec. Mentioned for completeness; do not implement here.
5. **Step order** — env-flag plumbing first, then helper modules, then `recordActivity` signature + body changes, then call site updates, then tests. Tests come last because TypeScript signature errors at test sites are caught after the call site updates settle.

## Edge Cases & Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Field naming on the wire | camelCase (`activityType`, `sourceFeature`) | Matches `backend/src/main/resources/openapi.yaml` § `ActivityRequest` and `frontend/src/types/api/activity.ts`. Spec body Decision 6 is wrong about snake_case. |
| `sourceFeature` typing | Required, not optional | Backend `@NotBlank` validation rejects empty strings with 400. TypeScript-required prevents the failure mode entirely. |
| Helpers (`useChallengeProgress.completeDay`, `useChallengeAutoDetect`) signature | Unchanged — wrap at call site | Avoids cascading sourceFeature awareness into challenge internals. Wrapper closure at the 3 indirect call sites is local and clear. |
| `console.warn` vs `console.error` for dual-write failures | `console.warn` | This is a non-critical, non-user-impacting failure. `console.error` is reserved for genuine errors that need attention. The codebase uses `console.error` for real failure paths in `useFaithPoints` (e.g., `localStorage.setItem` failures during `repairStreak`). |
| Env flag accessor location | `frontend/src/lib/env.ts` | Consistent with `getVapidPublicKey`. Avoids inline `import.meta.env.VITE_USE_BACKEND_ACTIVITY === 'true'` at the consumer. |
| `postActivityToBackend` location | Inside `frontend/src/hooks/useFaithPoints.ts` (private to this module) | Single consumer; no need for a separate file. Per spec deliverables: "private to this file OR exported from a new util — recon's call." Scope is tight enough for inline. |
| `metadata` field on the request | Omitted entirely | Spec Decision 7 — frontend has no metadata to send today. Omitted from JSON body. Backend `metadata` is `Record<string, unknown> \| null` (optional). |
| Where the wrapper closure goes for indirect call sites | Inline at each call site | A shared helper `recordActivityForChallenge(type)` would re-cross the boundary into the indirect helpers. Local closures are clearer at small scale (3 sites). |
| `useChallengeAutoDetect` and `useChallengeProgress` parameter types | Stay as `(type: ActivityType) => void` | The wrapped closure satisfies this type. Helpers don't need to know about `sourceFeature`. |
| Existing `useFaithPoints.test.ts` second-argument fill | Use `'test'` constant | Doesn't shadow real source-feature vocabulary; signals "test sentinel" to future readers. The dual-write tests use real source-feature strings. |
