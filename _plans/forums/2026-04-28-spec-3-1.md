## Architecture Context

**Existing Liquibase changeset directory:**
`backend/src/main/resources/db/changelog/` contains 13 changesets dated 2026-04-23-001 through 2026-04-27-013 plus the `master.xml` index and the `contexts/dev-seed.xml` context-scoped file. Most recent file is `2026-04-27-013-create-user-mutes-table.xml` (Phase 2.5.7). New files continue the date-sequence pattern starting at `2026-04-27-014`.

**`master.xml` is append-only.** The current file ends with `<include file="db/changelog/2026-04-27-013-create-user-mutes-table.xml"/>` followed by `<include file="db/changelog/contexts/dev-seed.xml"/>`. Six new `<include>` entries land BEFORE the `dev-seed.xml` line so dev-seed continues to run last (it depends on every preceding schema being present). Existing entries are NEVER reordered; existing changeset files are NEVER edited.

**Established XML conventions (verified across 2026-04-23-001, 2026-04-27-009, 2026-04-27-010, 2026-04-27-013):**
- `xmlns="http://www.liquibase.org/xml/ns/dbchangelog"` + `xsi:schemaLocation` with `dbchangelog-latest.xsd`
- One `<changeSet>` per file with `id="{filename-without-extension}"` and `author="worship-room"`
- Header comment naming the spec ("Forums Wave Spec 3.1 — …"), Decision references, and the canonical "Once this changeset runs, its MD5 is permanent in DATABASECHANGELOG; future schema changes go in new YYYY-MM-DD-NNN-*.xml changesets" warning
- UUID columns: `<column name="..." type="UUID" defaultValueComputed="gen_random_uuid()">` (Postgres 13+ built-in; verified in `2026-04-23-001` and `2026-04-27-010`). NOT `uuid_generate_v4()`.
- Timestamp columns: `type="TIMESTAMP WITH TIME ZONE" defaultValueComputed="NOW()"` (NOT `valueDate` with Z suffix — see `05-database.md` § "Timezoned timestamps")
- Boolean columns: `type="BOOLEAN" defaultValueBoolean="false"`
- CHECK constraints use raw `<sql>` blocks via `ALTER TABLE ... ADD CONSTRAINT name_check CHECK (...)` — Liquibase's structured `<addCheckConstraint>` is acceptable for trivial single-column checks but the established pattern (matching every prior CHECK in the repo) uses `<sql>`. Use `&lt;&gt;` for `<>` and `&lt;` for `<` inside `<sql>` (XML escaping).
- Partial indexes (with `WHERE` clauses) use raw `<sql>` blocks via `CREATE INDEX ... WHERE ...` — Liquibase's `<createIndex>` does not support `WHERE`. Verified in `2026-04-27-008-add-activity-log-backfill-idempotency-index.xml`.
- Plain (non-partial) indexes use `<createIndex tableName="..." indexName="...">` with `<column>` children, `descending="true"` for DESC ordering. Verified in `2026-04-27-013` (`idx_user_mutes_muter`).
- FK constraints use `<addForeignKeyConstraint>` with explicit `constraintName`, `onDelete="CASCADE"` (or `SET NULL` per spec), and `referencedTableName`/`referencedColumnNames`. Verified across all Phase 2.5 changesets.
- Composite PKs use `<addPrimaryKey constraintName="pk_{table}" columnNames="col_a, col_b"/>` after `<createTable>`. Verified in `2026-04-27-009`, `2026-04-27-013`.
- Rollback: `<rollback><dropTable tableName="..."/></rollback>`. Indexes and constraints attached to the dropped table vanish transitively — no need to drop them individually in the rollback.

## Edge Cases & Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| `author` attribute on changesets | `author="worship-room"` (NOT `author="claude"` as the spec body suggested in line 354) | `03-backend-standards.md` § Liquibase Standards is explicit: "Use `author=\"worship-room\"` for all changesets". Every existing changeset on disk (verified across 2026-04-23-001 through 2026-04-27-013) uses `worship-room`. Changeset author is part of the MD5 checksum input — diverging here would create an inconsistency that future schema-archaeology queries (`SELECT DISTINCT author FROM databasechangelog`) would surface as a one-off. Universal Rule 16 (respect existing patterns) governs. |
| Changeset date prefix | `2026-04-27-014` through `2026-04-27-019` (Spec Divergence 1) | Master plan body's `2026-04-17-*` dates are illustrative; the established convention is "execution date sequence." Using today's date (2026-04-28) would create a gap with the most recent file (2026-04-27-013); continuing the 2026-04-27 sequence keeps numbering tight. Liquibase MD5 is computed from id+author+content, NOT filename, so the date prefix is purely organizational. |
| UUID generation function | `gen_random_uuid()` | Verified in `2026-04-23-001-create-users-table.xml` line 17. Postgres 13+ built-in; no `CREATE EXTENSION` needed. Mixing with `uuid_generate_v4()` (uuid-ossp) would create dual-generator drift. |
| Partial-index implementation | Raw `<sql>` blocks inside `<changeSet>` | Liquibase's `<createIndex>` element does not support `WHERE` clauses. Verified pattern in `2026-04-27-008-add-activity-log-backfill-idempotency-index.xml`. The 12 partial indexes across `posts`, `post_comments`, `post_reports`, `qotd_questions` all use raw SQL. |
| CHECK constraint implementation | Raw `<sql>` blocks via `ALTER TABLE ... ADD CONSTRAINT name CHECK (...)` | Matches every existing Phase 1, 2, 2.5 changeset. Names every constraint explicitly so test assertions can target the constraint by name. XML escapes: `&lt;&gt;` for `<>`, `&lt;` for `<`. |
| `posts_category_check` allows NULL | `category IS NULL OR category IN (...)` | Spec § "Watch For" item #7. Non-prayer post types (e.g., `discussion`) may not have a category. CHECK constraint must explicitly allow NULL or PostgreSQL would reject all NULL inserts (which is opposite of intent). |
| `post_reactions.reaction_type` CHECK enum | `('praying', 'candle')` only — NOT `('praying', 'candle', 'praising', 'celebrate')` | Spec Divergence 3. Phase 6.6 Answered Wall ships a separate ALTER changeset to add `praising` and `celebrate`. Tightening the constraint now prevents Phase 3 read endpoints from ever encountering data values they don't know how to render. |
| `parent_comment_id` ships now | Yes, with `ON DELETE CASCADE` self-reference | Spec § "Watch For" item #8. Adding the column in a Phase 4 migration would touch a high-traffic table; including it now with `NULL` default is zero-impact during Phase 3. |
| `is_helpful` on `post_comments` ships now | Yes, with `DEFAULT FALSE` | Spec § "Watch For" item #9. Same rationale as `parent_comment_id`. |
| Denormalized counter defaults | `INTEGER NOT NULL DEFAULT 0` for all 5 counters on `posts` | Application-layer increment/decrement logic in Specs 3.5 + 3.7 maintains them. NOT NULL with default 0 prevents NULL-arithmetic bugs in counter updates. |
| Master.xml `<include>` insertion point | BEFORE the existing `<include file="db/changelog/contexts/dev-seed.xml"/>` line | The dev-seed file references seed users that depend on `users` being present; preserving "schema includes first, dev-seed last" maintains correct ordering. The 6 new schema includes go between the 2026-04-27-013 line and the dev-seed line. |
| Test changelog row-count assertion | Update from `hasSize(13)` to `hasSize(19)` and append 6 new `Map<String, Object>` blocks | The existing assertion explicitly enumerates each row by index. Adding a row without updating the count makes the test fail with "expected 13 but was 19." |
