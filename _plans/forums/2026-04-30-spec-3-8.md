## Edge Cases & Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| `post_id` vs `comment_id` mutual exclusion | XOR enforced by DB CHECK; service-layer `targetType` enum picks which column to set | Prevents impossible (both-set) state at the DB layer; service-layer enum branches to set exactly one (Watch-For #7) |
| Idempotency dedup scope | Only `status='pending'` reports dedup; closed reports allow re-reporting | Master plan body says "duplicate is idempotent" but doesn't specify pending-only; D4 refines: closed = moderator already processed, user re-flagging is new signal |
| Concurrency control | `SELECT FOR UPDATE` via `@Lock(LockModeType.PESSIMISTIC_WRITE)` on the existence-check query | Matches Phase 3 reaction-toggle pattern; lower overhead than SERIALIZABLE isolation; prevents two concurrent insert attempts from same user creating two pending rows (the schema has NO UNIQUE constraint per MPD-1) |
| Rate limit bucket scope | ONE bucket per user shared across post-reports AND comment-reports endpoints | Prevents target-alternation evasion (D6, Watch-For #3): user can't alternate `posts/.../reports` and `comments/.../reports` to get 2x the bucket |
| Reason enum casing | Java SCREAMING_SNAKE_CASE values, JSON wire format lowercase via `@JsonProperty` per value | Watch-For #14 Option A (preferred); idiomatic Java + idiomatic TypeScript. OpenAPI lists lowercase. JPA persists `name()` via `ReportReasonConverter`. |
| Visibility rule for target lookup | `postRepository.findById(postId)` directly — NOT `findVisibleTo(viewerId, postId)` | D9 deliberate departure from Phase 3 Addendum #7 visibility-filtering rule. Reports against crisis-flagged or soft-deleted posts ARE valid moderator-queue signal. |
| Self-report prevention | Service-layer 400 SELF_REPORT + UI hides Report button on own posts | D11 + Watch-For #1 + #18: defense in depth. UI is for UX; service is for security/data integrity. |
| Logging policy | Log only `reporterId, targetType, targetId, reason, status, requestId` — NEVER `details` content | D12 + Watch-For #2: PII risk. Users may include personal information about reported user, content quotes, etc. |
| Activity events | None | D7 — reports do NOT fire activity events; do NOT award faith points (mirrors bookmarks Spec 3.7 D3). Gamifying would create perverse incentives. |
| `last_activity_at` bump | None | D8 — a reported post should NOT jump to top of feed (mirrors bookmarks). |
| ExceptionHandler scope | `@RestControllerAdvice(basePackages = "com.worshiproom.post")` | R4 reasoning: Spring's `basePackages` filter applies to the CONTROLLER's package; mirrors EngagementExceptionHandler exactly. Type-based dispatch (ReportException is distinct base) keeps advices separate. |
| Idempotency-Key header | NOT used | Watch-For #16: dedup-on-pending policy provides idempotent semantics naturally; no header needed. |
| Frontend ReportDialog reason picker | Add `<fieldset>` with 6 radio buttons; "Other" default | D3 + D14 + MPD-2: existing dialog has only freeform textarea; add reason picker, repurpose textarea as `details`. |
| Successful-submission UX | Keep existing 1.5s inline "Report submitted. Thank you for keeping this safe." | Watch-For #21: more careful and intentional than a transient toast for a moderation action. Toasts reserved for FAILURE cases only. |
| Comment-report UI | Backend ships fully; frontend defers to a future spec | MPD-3 + D15: zero comment-reporting UI in `frontend/src/`. Captured in `_plans/post-1.10-followups.md`. |
| Test count | ~28 tests target (16 controller + 6 service + 4 rate-limit + 2 converter, plus ~6 frontend) | Spec argues 28+ for thorough coverage of idempotency × rate-limit × visibility combinations; Phase 3 medium-large specs typically ship 30-50. Master plan AC says ≥10 (clearly exceeded). |

### Step 2: ReportRepository with pessimistic-lock query

**Objective:** Define the JPA repository with the find-pending-for-update query that backs idempotent dedup.

**Files to create:**
- `backend/src/main/java/com/worshiproom/post/report/ReportRepository.java`

**Details:**

```java
package com.worshiproom.post.report;

import jakarta.persistence.LockModeType;
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Lock;
import org.springframework.data.jpa.repository.Query;
import org.springframework.data.repository.query.Param;
import org.springframework.stereotype.Repository;

import java.util.Optional;
import java.util.UUID;

@Repository
public interface ReportRepository extends JpaRepository<Report, UUID> {

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("""
            SELECT r FROM Report r
             WHERE r.reporterId = :reporterId
               AND r.postId = :postId
               AND r.status = com.worshiproom.post.report.ReportStatus.PENDING
            """)
    Optional<Report> findPendingByReporterAndPostForUpdate(
            @Param("reporterId") UUID reporterId,
            @Param("postId") UUID postId);

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("""
            SELECT r FROM Report r
             WHERE r.reporterId = :reporterId
               AND r.commentId = :commentId
               AND r.status = com.worshiproom.post.report.ReportStatus.PENDING
            """)
    Optional<Report> findPendingByReporterAndCommentForUpdate(
            @Param("reporterId") UUID reporterId,
            @Param("commentId") UUID commentId);
}
```

**Guardrails (DO NOT):**
- DO NOT use `@Modifying` — these are SELECTs, not bulk updates (addendum gate #3 doesn't apply)
- DO NOT remove the `PESSIMISTIC_WRITE` lock — it's the only thing preventing duplicate inserts under concurrent requests (no UNIQUE constraint per MPD-1)
- DO NOT add a generic `existsByReporterIdAndPostId` — service must lock the row, not just check existence

**Verification:**
- [ ] Code compiles: `./mvnw compile`

```java
package com.worshiproom.post.report;

import org.springframework.http.HttpStatus;

public abstract class ReportException extends RuntimeException {
    private final HttpStatus status;
    private final String code;
    protected ReportException(HttpStatus status, String code, String message) {
        super(message);
        this.status = status;
        this.code = code;
    }
    public HttpStatus getStatus() { return status; }
    public String getCode() { return code; }
}
```

`ReportTargetNotFoundException.java`:

```java
public class ReportTargetNotFoundException extends ReportException {
```

```java
public class SelfReportException extends ReportException {
```

```java
import com.worshiproom.post.report.dto.CreateReportRequest;
import com.worshiproom.post.report.dto.ReportData;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.util.Optional;
import java.util.UUID;

@Service
@Transactional(readOnly = true)
public class ReportService {

    private static final Logger log = LoggerFactory.getLogger(ReportService.class);

    public enum TargetType { POST, COMMENT }

    public record ReportResult(ReportData data, boolean created) {}

    private final ReportRepository reportRepository;
    private final PostRepository postRepository;
    private final PostCommentRepository commentRepository;
    private final ReportsRateLimitService rateLimitService;

    public ReportService(ReportRepository reportRepository,
                         PostRepository postRepository,
                         PostCommentRepository commentRepository,
                         ReportsRateLimitService rateLimitService) {
        this.reportRepository = reportRepository;
        this.postRepository = postRepository;
        this.commentRepository = commentRepository;
        this.rateLimitService = rateLimitService;
    }

    @Transactional
    public ReportResult report(TargetType targetType,
                               UUID targetId,
                               UUID reporterId,
                               CreateReportRequest request,
                               String requestId) {
        rateLimitService.checkAndConsume(reporterId);

        UUID authorId = loadTargetAuthorId(targetType, targetId);
        if (reporterId.equals(authorId)) {
            throw new SelfReportException();
        }

        Optional<Report> existing = (targetType == TargetType.POST)
                ? reportRepository.findPendingByReporterAndPostForUpdate(reporterId, targetId)
                : reportRepository.findPendingByReporterAndCommentForUpdate(reporterId, targetId);

        Report report;
        boolean created;
        if (existing.isPresent()) {
            report = existing.get();
            created = false;
        } else {
            report = new Report(
                    reporterId,
                    targetType == TargetType.POST ? targetId : null,
                    targetType == TargetType.COMMENT ? targetId : null,
                    request.reason(),
                    request.details());
            report = reportRepository.save(report);
            created = true;
        }

        log.info("Report submitted reporterId={} targetType={} targetId={} reason={} status={} requestId={}",
                reporterId, targetType.name().toLowerCase(), targetId,
                request.reason().name().toLowerCase(),
                created ? "created" : "idempotent",
                requestId);

        return new ReportResult(new ReportData(report.getId(), created), created);
    }

    private UUID loadTargetAuthorId(TargetType targetType, UUID targetId) {
        if (targetType == TargetType.POST) {
            return postRepository.findById(targetId)
                    .orElseThrow(ReportTargetNotFoundException::new)
                    .getAuthorId();
        } else {
            return commentRepository.findById(targetId)
                    .orElseThrow(ReportTargetNotFoundException::new)
                    .getAuthorId();
        }
    }
}
```

**Notes on the implementation:**

- Class-level `@Transactional(readOnly = true)`, method-level `@Transactional` override on `report()` (Watch-For #22; mirrors `BookmarkWriteService`)
- `loadTargetAuthorId` calls `findById()` directly — NOT a visibility-filtered query. Reports against crisis-flagged or soft-deleted posts ARE accepted (D9 deliberate Phase 3 Addendum #7 exception).
- Pessimistic lock window must be tight (Watch-For #6). No external API calls inside the `@Transactional` method. The save + log are the only actions after the lock acquires.
- **L1-cache trap**: We do NOT read the `createdAt` timestamp from `report` in the response (the response carries only `reportId` and `created`), so Phase 3 Addendum gate #2 does NOT require `entityManager.refresh()`. If a future spec extends the response to include `createdAt`, that follow-up work MUST add the refresh + non-null-timestamp regression test.
- Logging line per D12 — log only `reporterId, targetType, targetId, reason, status, requestId`; NEVER `details` content (PII risk per Watch-For #2)

- DO NOT log `request.details()` anywhere

```java
package com.worshiproom.post.report;

import com.worshiproom.proxy.common.ProxyError;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.slf4j.MDC;
import org.springframework.core.Ordered;
import org.springframework.core.annotation.Order;
import org.springframework.http.HttpHeaders;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

/**
 * Report-domain exception advice (Spec 3.8). Maps {@link ReportException} subclasses
 * to {@link ProxyError} HTTP responses.
 *
 * <p>Package-scoped to {@code com.worshiproom.post} (NOT
 * {@code com.worshiproom.post.report}) — Spring's {@code basePackages} filter
 * applies to the CONTROLLER's package. {@code ReportController} lives in
 * {@code com.worshiproom.post.report}, which is nested under
 * {@code com.worshiproom.post}, so the broader scope works. Mirrors
 * {@code EngagementExceptionHandler}'s scoping per R4.
 *
 * <p>Dispatch ambiguity is prevented by TYPE: {@link ReportException} is a
 * distinct base from {@code EngagementException}, {@code PostException}, and
 * {@code CommentException}. Each domain advice's {@code @ExceptionHandler}
 * methods route only to their own base type.
 *
 * <p>{@link ReportsRateLimitException} has a dedicated handler that adds the
 * {@code Retry-After} header. Java's most-specific-handler dispatch picks it
 * first when one is thrown.
 */
@RestControllerAdvice(basePackages = "com.worshiproom.post")
@Order(Ordered.HIGHEST_PRECEDENCE)
public class ReportExceptionHandler {

    private static final Logger log = LoggerFactory.getLogger(ReportExceptionHandler.class);

    @ExceptionHandler(ReportsRateLimitException.class)
    public ResponseEntity<ProxyError> handleRateLimit(ReportsRateLimitException ex) {
        var requestId = MDC.get("requestId");
        log.info("Reports rate limit hit retryAfter={}s", ex.getRetryAfterSeconds());
        return ResponseEntity
                .status(ex.getStatus())
                .header(HttpHeaders.RETRY_AFTER, String.valueOf(ex.getRetryAfterSeconds()))
                .body(ProxyError.of(ex.getCode(), ex.getMessage(), requestId));
    }

    @ExceptionHandler(ReportException.class)
    public ResponseEntity<ProxyError> handleReport(ReportException ex) {
        var requestId = MDC.get("requestId");
        log.info("Report-domain rejection: code={} message={}", ex.getCode(), ex.getMessage());
        return ResponseEntity
                .status(ex.getStatus())
                .body(ProxyError.of(ex.getCode(), ex.getMessage(), requestId));
    }
}
```

- DO NOT add a generic `@ExceptionHandler(Exception.class)` catch-all — let Spring's default handlers + the existing PostValidationExceptionHandler handle MethodArgumentNotValidException

```typescript
    },

}
```
