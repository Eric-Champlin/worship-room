## Edge Cases & Decisions

| Decision | Choice | Rationale |
|---|---|---|
| One changeset per table vs combined | Five separate XML files | Matches Spec 1.3 discipline (one logical unit per file); allows independent rollback; smaller code-review surface |
| Filename date prefix | `2026-04-25` (NOT today's `2026-04-26`) | Brief explicitly mandates `2026-04-25` per spec ¬ß "Master Plan Divergences" ‚Äî once shipped, the filename's MD5 is permanent in `DATABASECHANGELOG`; the +1-day stale date is harmless |
| Numeric sequence | `003` through `007` | Continues the existing `001`/`002` numeric sequence in `db/changelog/` |
| Timestamp type | `TIMESTAMP WITH TIME ZONE` everywhere | Matches the users-table convention; master plan's `TIMESTAMP` interpreted as `TIMESTAMP WITH TIME ZONE` per Architectural Decision #2 in the spec |
| `last_active_date` type | `DATE` (not `TIMESTAMP WITH TIME ZONE`) | Master plan Decision 5 explicitly says `DATE`; same for `grace_week_start`, `grief_pause_until` |
| FK ON DELETE behavior | `CASCADE` on every FK to `users(id)` | Decision 5 + `.claude/rules/05-database.md` Decision 10 rule 8 ‚Äî child rows disappear when parent user deletes |
| CHECK constraint declaration | Separate `<sql>` block with explicit name | Matches Spec 1.3's `users_display_name_preference_check` pattern; named so future ALTER/DROP can target without introspection |
| Index policy | Two on `activity_log` only; nothing on the other four (PKs cover) | Architectural Decision #5 ‚Äî minimal, justified; future tuning spec adds more when `pg_stat` reveals real query patterns |
| Liquibase context | NONE on these five changesets | Tables must materialize in dev/test/prod (same as users); only `dev-seed.xml` uses `context="dev"`. Per `.claude/rules/05-database.md` ¬ß Test-profile context override |
| CHECK rejection test approach | One JUnit `@ParameterizedTest` exercising all 7 constraints | Less ceremony than 7 individual methods; each row in the `@CsvSource` carries `(table, columnList, valueList, expectedConstraintName)`; failure messages still pinpoint the offending constraint |
| `LiquibaseSmokeTest` row-count bump | `hasSize(2)` ‚Üí `hasSize(7)` with five new explicit row assertions | Follows the pattern of the existing two row assertions; the explicit checks document exact filename + author for each new changeset |
| `metadata` column type on `activity_log` | `JSONB` (not `JSON`) | PostgreSQL convention; supports indexing if a future tuning spec needs it |
| `points_earned` default | `INTEGER NOT NULL DEFAULT 0` | Per master plan Decision 5; allows zero-point activity rows (e.g., shadow-write of an event that didn't qualify for points) |
| Should rollback be tested? | No | Spec ¬ß "Testing Notes" explicitly says rollback is a manual operator action via `liquibase rollback`; XML parser validation at startup is sufficient |

### Step 9: Add CHECK constraint rejection tests for all 7 named constraints

**Objective:** Verify every named CHECK constraint rejects an out-of-domain value with a `DataIntegrityViolationException` carrying the constraint name.

**Files to modify:**
- `backend/src/test/java/com/worshiproom/db/LiquibaseSmokeTest.java`

**Details:**

Add ONE `@ParameterizedTest` method that exercises all seven CHECK constraints. Use `@CsvSource` to declare the rows; each row carries enough info to construct a violating INSERT and the expected constraint name in the exception message.

**Method signature:**
```java
@ParameterizedTest(name = "{0} rejects invalid value")
@CsvSource({
    "faith_points_total_points_nonneg_check, ...",
    "faith_points_current_level_positive_check, ...",
    "streak_state_current_streak_nonneg_check, ...",
    "streak_state_longest_streak_nonneg_check, ...",
    "streak_state_grace_days_used_nonneg_check, ...",
    "user_badges_display_count_positive_check, ...",
    "activity_counts_count_value_nonneg_check, ..."
})
void checkConstraintRejectsInvalidValue(String constraintName, ...)
```

**Approach:** Because the seven constraints span four tables with different INSERT shapes, the cleanest approach is NOT to parameterize the SQL but to dispatch on the constraint name inside the test body. Each branch:

1. Inserts a fresh `users` row with `gen_random_uuid()` to capture a valid `user_id` (FK requirement).
2. Attempts the violating child-table INSERT with a single negative or zero value targeting the constraint under test.
3. Asserts `DataIntegrityViolationException` is thrown with a message containing the constraint name.
4. Cleans up the parent users row in `@AfterEach` if needed (or use `@Transactional` to auto-rollback ‚Äî but `LiquibaseSmokeTest` does not currently mark anything `@Transactional`; either approach works as long as it's consistent).

**Cleanup approach:** Use a simple `@AfterEach` that runs `DELETE FROM <child_table>; DELETE FROM users WHERE email LIKE 'check-test-%';` before each test (or after) ‚Äî keeps tests independent. Cleaner than `@Transactional` here because each parameter case handles a different table and rollback semantics get noisy.

**Per-constraint violation INSERT shape:**

| Constraint | Table | Violating value |
|---|---|---|
| `faith_points_total_points_nonneg_check` | `faith_points` | `total_points = -1` |
| `faith_points_current_level_positive_check` | `faith_points` | `current_level = 0` |
| `streak_state_current_streak_nonneg_check` | `streak_state` | `current_streak = -1` |
| `streak_state_longest_streak_nonneg_check` | `streak_state` | `longest_streak = -1` |
| `streak_state_grace_days_used_nonneg_check` | `streak_state` | `grace_days_used = -1` |
| `user_badges_display_count_positive_check` | `user_badges` | `display_count = 0` |
| `activity_counts_count_value_nonneg_check` | `activity_counts` | `count_value = -1` |

**Assertion shape (modeled on the existing `displayNamePreferenceCheckConstraintRejectsInvalidValues`):**

```java
assertThatThrownBy(() ->
    jdbcTemplate.update(<violating INSERT SQL>, <bind args>)
).isInstanceOf(DataIntegrityViolationException.class)
 .hasMessageContaining(constraintName);
```

**Helper for the parent users insert** (since each branch needs a valid `user_id`):

```java
private UUID insertTestUser(String emailSuffix) {
    UUID id = UUID.randomUUID();
    jdbcTemplate.update(
        "INSERT INTO users (id, email, password_hash, first_name, last_name) " +
        "VALUES (?, ?, ?, ?, ?)",
        id, "check-test-" + emailSuffix + "@example.com",
        "bcrypt-placeholder", "Test", "User"
    );
    return id;
}
```

(Method added private to the test class. Not autowired, no Spring annotations.)

**Why parameterize and not split into 7 methods:** Spec ¬ß "Files to Modify" item 3 explicitly allows either approach ("CC's call"). Parameterized is more compact, easier to extend in future specs (adding an 8th constraint = adding one CSV row), and keeps the 7 tests green-or-red as a unit. Spec ¬ß "Testing Notes" anticipates ~445 total tests post-spec; this approach gives us +6 method-instance test counts (5 new shape tests in Step 8 + 1 parameterized method that itself runs 7 cases). JUnit's report shows each parameterized case as a separate row, so the 7 individual cases are visible in the test summary.

**Guardrails (DO NOT):**
- Do NOT skip any of the seven constraints.
- Do NOT use `hasMessageContaining("CHECK")` ‚Äî must match the exact constraint name (Spec acceptance criterion).
- Do NOT leak users rows into subsequent tests ‚Äî clean up via `@AfterEach` or transactional rollback.
- Do NOT depend on the order of parameterized rows ‚Äî each row must be independently runnable.
