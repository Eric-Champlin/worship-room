## Spec-Category-Specific Guidance

This is a **BACKEND-ONLY infrastructure spec**. The "Affected Frontend Routes" section is `N/A — backend-only spec`. `/verify-with-playwright` is skipped. Focus on:

1. **Liquibase changeset with rollback block** (Rule 3) — mandatory
2. **Testcontainers integration test** (`06-testing.md` — never H2) — establishes the pattern future schema specs follow
3. **No OpenAPI extension** — no API change in this spec
4. **No JPA entity / repository / service / controller** — arrive in Spec 1.4/1.5
5. **No rate limiting additions** — no new endpoints

**Primary risks:**

- **Accidentally enabling Liquibase in dev-only profile.** If `spring.liquibase.*` goes into `application-dev.properties` by mistake, prod deploys will skip migrations and fail at startup. Verification: `grep -r 'spring.liquibase' backend/src/main/resources/` — exactly two lines, both in `application.properties` (base file). Zero lines in `application-{dev,test,prod}.properties`.
- **Overwriting Spec 4 Deviation #1 log suppressions.** The `application-dev.properties` file has four critical `INFO`-level overrides under `org.springframework.web.*` that prevent API key / PII leaks in dev-profile logs. This spec does NOT modify `application-dev.properties` at all. Any diff there is a regression.
- **Liquibase `$` character gotcha in XML.** BCrypt hashes contain `$`, which Liquibase treats as `${}` template reference. This spec does NOT seed data (no BCrypt hashes) — the Phase 1.8 dev-seed spec owns that risk. Noted here only for future reference.
- **Testcontainers Docker dependency.** The smoke test fails without Docker running. This is expected behavior (matches Spec 1.2); document it in the test's class-level JavaDoc.
- **Changeset checksum conflict on re-edit.** Once a changeset runs, Liquibase stores its MD5 checksum in `DATABASECHANGELOG`. Editing the changeset file in place causes boot to fail with a checksum mismatch. Future edits MUST create a new changeset. Note this in the changeset file's inline comment.

| CHECK constraint syntax in Liquibase XML | **Use inline `<sql>` block after the `<createTable>`.** Specifically: `ALTER TABLE users ADD CONSTRAINT users_display_name_preference_check CHECK (display_name_preference IN ('first_only','first_last_initial','first_last','custom'));` | Liquibase 4.x supports both `<addCheckConstraint>` and `<sql>`. `<sql>` is explicit about the exact DDL emitted and matches the `03-backend-standards.md` § Changeset Template convention of keeping constraint DDL human-readable. Name the constraint `users_display_name_preference_check` so future ALTER/DROP operations are trivially identifiable. Include the inverse `ALTER TABLE users DROP CONSTRAINT ...` in the `<rollback>` — but actually `<dropTable>` covers it transitively (the constraint dies with the table), so the rollback stays `<dropTable tableName="users"/>`. |

| Testcontainers image for smoke test | **`postgres:16-alpine`** (matches Spec 1.2 docker-compose image) | Dev and test use the same image → eliminates a class of "works on dev, fails in CI" bugs. |
| Testcontainers JUnit extension | **`@Testcontainers` + `@Container` static field + `@DynamicPropertySource`.** | Matches `06-testing.md` § Testcontainers Setup Pattern example. The static `@Container` ensures one container per JVM (Testcontainers reuses it across test methods in the class). `@DynamicPropertySource` injects the random-port JDBC URL before Spring boots. |
| Test method count | **3 test methods** (matches spec minimum: schema-exists, changelog-row-recorded, CHECK-constraint-rejects). | Spec says "3 minimum, more acceptable but not mandatory." Expect `DataIntegrityViolationException` (Spring's wrapper around the PostgreSQL `23514 check_violation`). | Spec provides the literal INSERT. The exception is Spring's generic `DataIntegrityViolationException` (a `DataAccessException`), not a raw `PSQLException` — Spring's `SQLExceptionTranslator` wraps it. AssertJ's `hasMessageContaining("check constraint")` catches the nested PSQL message text. |
| Package for smoke test | **`com.worshiproom.db`** (creates the new `db/` test subpackage). | Spec Notes #12 explicitly permits `db` as an infrastructure subpackage. Fits the existing `proxy/common`, `proxy/ai`, `proxy/maps`, `proxy/bible` convention. |
