## Edge Cases & Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Changeset filename date + sequence | `2026-04-23-002-add-users-timezone-column.xml` | Master plan suggested `2026-04-14-003` — stale. Today is 2026-04-23; Spec 1.3's changeset was `2026-04-23-001` (same day). Sequence increments within a date per `03-backend-standards.md` Liquibase naming rules. Confirmed no collision via `ls` in the pre-exec checklist. |
| `defaultValue="UTC"` vs `defaultValueComputed="'UTC'"` | `defaultValue="UTC"` | Liquibase's `defaultValue` attribute is a string literal and is correctly quoted by Liquibase when emitting SQL. `defaultValueComputed` is for SQL expressions like `gen_random_uuid()` — wrong tool here. Spec § Notes-for-plan-phase-recon #8 confirms. |
| Column length | `VARCHAR(50)` | Longest IANA name (`America/Argentina/ComodRivadavia`) is 31 chars. 50 exceeds that with headroom but signals expected shape. Matches master plan. Not expanding to 100 per spec Out-of-band note. |
| `master.xml` include ordering | APPEND — new `<include>` goes AFTER Spec 1.3's `<include>` | Liquibase applies in include-order. Out-of-order includes cause subtle bugs when later changesets depend on earlier ones. Spec § Notes-for-plan-phase-recon #2. |
| Atomicity of changeset creation + `master.xml` update | Treat as ONE atomic step (Step 1) | Per spec § Notes-for-plan-phase-recon #11 — running `./mvnw test` between "create changeset file" and "update master.xml include" would either skip the changeset or apply it against stale include order, masking the real test result. Spec 1.1 regression lesson applied. |
| `usersTableExistsAfterMigration()` method name | Update the actual method `usersTableExistsWithAllDecision3Columns()` in-place (keep name OR rename — plan chooses keep-name to minimize churn) | The spec refers to this method by its suggested name from master plan recon. Actual on-disk name differs. Keeping `usersTableExistsWithAllDecision3Columns` preserves Spec 1.3's naming convention; the method body updates to assert 21 columns including `timezone`. No method-name churn. |
| `liquibaseChangelogRecordsExactlyOneChangeset()` method name | RENAME to `liquibaseChangelogRecordsAllChangesets()` | The "ExactlyOne" qualifier is a lie once a second changeset exists. Renaming to a count-neutral name prevents future churn each time a changeset is added. Body updates to assert `hasSize(2)` + the specific IDs in order. |
| New test method name | `timezoneColumnHasCorrectDefaultAndConstraints()` | Matches spec § Testing notes verbatim. Clear single-purpose name. |
| Final test count | 284 total (283 + 1) | Per spec § Notes-for-plan-phase-recon #9 — one new method, two existing methods updated-in-place. Acceptance Criterion 7 allows 283 or 284. |
| `column_default` PostgreSQL format | Substring assertion — `contains("UTC")` | PostgreSQL commonly surfaces the default as `'UTC'::character varying`. Exact-match `.isEqualTo("'UTC'")` would be brittle across PG versions / Liquibase quoting. `contains("UTC")` is spec-approved (§ Acceptance criteria test-assertions). |
