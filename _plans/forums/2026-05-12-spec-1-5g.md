## Edge Cases & Decisions

| # | Decision | Choice | Rationale |
|---|----------|--------|-----------|
| 1 | AuthenticatedUser refactor strategy (RO-1/R11 — Pattern A `forTest(...)` vs Pattern B record secondary constructor) | **Pattern B — record secondary constructor.** Add `public AuthenticatedUser(UUID userId, boolean isAdmin) { this(userId, isAdmin, null); }` to the record. Production code (`JwtAuthenticationFilter` line 112) refactors to 3-arg. The 32 test call sites compile unchanged. | Smaller diff. Records explicitly support secondary constructors. Two ways to construct `AuthenticatedUser` is acceptable when production code consistently uses the 3-arg form (only the filter constructs it in prod). Brief's preferred direction (`forTest` factory) is acceptable, but the master-plan addendum's Pattern A nomenclature is reserved for reactive-store subscription (per `11b-local-storage-keys-bible.md`) — using "Pattern A factory" terminology here would conflict. Secondary constructor is the lower-cognitive-load path. |
| 2 | Atomic `incrementSessionGeneration` query — native vs JPQL | **Native SQL `RETURNING` clause** per R13: `UPDATE users SET session_generation = session_generation + 1 WHERE id = :userId RETURNING session_generation`. `@Modifying(clearAutomatically=true, flushAutomatically=true) @Query(nativeQuery=true)`, returns `Integer`. | JPQL `UPDATE` cannot return scalar values; native `RETURNING` is the canonical PostgreSQL pattern (per Decision 4 in master plan). Single round-trip. Atomicity by SQL row-lock. |
| 3 | JwtBlocklistService write order — Redis first vs Postgres first | **Postgres first, then Redis.** If Postgres write succeeds and Redis write fails, the blocklist entry still exists in the durable layer (read path's fallback handles this). If Redis-first and Postgres-second, a Redis success + Postgres failure leaves a Redis-only entry that disappears on TTL expiry — and on Redis restart in between, the entry vanishes silently. Postgres-first is the safer order. | Per MPD-2: durable layer is the source of truth. Postgres-first ensures any successful blocklist entry survives a Redis outage. Read path consults Redis first (fast path) and falls back to Postgres (durable path). |
| 4 | Postgres-only fallback when Redis fails — fail-closed vs fail-open | **Fail-closed at the BLOCKLIST CHECK boundary only.** If Redis is unreachable, `isRevoked(jti)` consults Postgres. If BOTH Redis AND Postgres are unreachable, the filter MUST reject (Gate-G-FAIL-CLOSED) with 401 TOKEN_INVALID — never authenticate. Revoke writes during this state still go to Postgres (Redis dual-write silently fails); reads consult Postgres. | MPD-2 mandates no in-memory fallback. Security-critical: silent failure of blocklist would un-revoke previously revoked tokens. |
| 5 | TTL semantics — Redis blocklist entry TTL = remaining JWT lifetime | Per D-TTL: `ttl_seconds = jwt.exp - now` clamped to `[1, jwtConfig.expirationSeconds]`. Edge case: a token already past `exp` but not yet pruned by JJWT's parser (clock skew) → set TTL to 60s minimum. | Caps blocklist size to "currently-valid tokens that were revoked." Postgres entry uses `expires_at = jwt.exp` (no clamp). Cleanup job sweeps Postgres rows where `expires_at < now()`. |
| 6 | active_sessions write throttling — application-level vs SQL-level | **SQL-level.** Per D-WriteThrottling: `UPDATE active_sessions SET last_seen_at = NOW() WHERE jti = :jti AND last_seen_at < NOW() - INTERVAL '60 seconds'`. No application-level state needed. If no row matches (throttle gate), zero rows updated — silent no-op. | Stateless; no per-process cache to drift. SQL `WHERE` enforces the throttle atomically. |
| 7 | Concurrent revoke + auth race | **Accept the in-flight authentication; next request fails.** The filter's parse → blocklist-check → sessionGen-check → principal-construction is NOT atomic with the revoke endpoint. A request that has passed all three checks before the revoke commits proceeds successfully. The NEXT request from the same token fails 401. | Per W22 / test spec `.concurrentRevokeAndAuth`. The race window is bounded by filter execution time (sub-millisecond after Redis hit, sub-10ms on Postgres fallback). Per Gate-G-SESSION-VISIBILITY, this is acceptable — revocation actually disables the token; what changes is "when" not "if." |
| 8 | "Sign out everywhere" — frontend redirect strategy | **Query-param flash.** After successful `DELETE /api/v1/sessions/all`, frontend manually `clearStoredToken()` + `navigate('/?auth=login&reason=signed_out_everywhere', { replace: true })`. AuthModal opens (via existing AuthQueryParamHandler) and reads `reason` to render flash message "You've been signed out everywhere." | Per W27. Consistent with the canonical query-param-driven AuthModal pattern from Spec 7 (02-security.md). |
| 9 | Pre-1.5g JWT tolerance — filter ordering | Filter inserts checks AFTER existing claim extraction (line 110) but BEFORE principal construction (line 112). Order: (a) parse `jti` from claims → if null, accept (no blocklist check) → if non-null, query blocklist. (b) parse `gen` from claims → if null, treat as `gen=0` → query `userRepository.findByIdAndSessionGeneration` or cached lookup → if mismatch, throw `tokenRevoked()`. | Per R2/Gate-G-MIGRATION. Pre-1.5g tokens have neither `jti` nor `gen`; both must be treated as absent-but-acceptable. |
| 10 | Session-generation lookup — DB on every request vs cached | **Cached via Spring's `@Cacheable` on `UserRepository.getSessionGenerationByUserId(UUID)`**, TTL 30s, max 10K entries (Caffeine). Cache eviction on `incrementSessionGeneration` writes — implemented via `@CacheEvict(value = "user-session-gen", key = "#userId")` on the increment method. | Per W15. Filter runs on every authenticated request — `O(1)` lookups are mandatory. 30s TTL trades a brief window of stale gen against DB hammering. Cache-evict on increment ensures correctness within ~30s for all OTHER concurrent requests; the current request (which triggered the increment) uses the returned value directly. |
