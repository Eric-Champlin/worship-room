## Edge Cases & Decisions

| Decision | Choice | Rationale |
|---|---|---|
| **URL format reconciliation (R1, load-bearing)** | Option (a) — match BibleReader's existing `?scroll-to=<n>&verse=<n>` contract | The stub claimed `#verse-{n}` hash anchors; live code uses query params. Mirroring `BiblePlanDay.tsx:188-200` keeps the change bounded to ScriptureChip and preserves the BB-38 deep-link contract. Options (b) and (c) expand scope. |
| **Parameter order (`scroll-to` first vs `verse` first)** | `?scroll-to=<n>&verse=<n>` (scroll-to first) | Mirrors `BiblePlanDay.tsx:196-197` insertion order; matches `BibleReader.deeplink.test.tsx:490` asserted URL `'/bible/john/3?scroll-to=16&verse=3'`. Order is functionally irrelevant (both params parse independently) but consistency is worth it. |
| **Setting both params vs scroll-to only (R2)** | Set BOTH `?scroll-to=` and `?verse=` to the same value | Matches BiblePlanDay precedent — chip click is equivalent to a plan-day passage open. Persistent selection (`?verse=`) lets the user re-find the verse if they navigate away inside the reader; one-shot scroll (`?scroll-to=`) handles arrival behavior. |
| **Range references (R3)** | Use `parsed.verse` (start verse) for both `verse` and `scroll-to` params; do NOT use `verseEnd` | `parseReference` already returns the start verse on `.verse` for ranges. Range-highlight (`?scroll-to=28-30`) is explicitly out of scope per spec Section 5. Single-verse format is what the existing chip does, just with one more param added. |
| **Mobile touch target (R4)** | No change — existing `min-h-[44px]` on the linked variant preserved | Already WCAG-compliant per recon. R4 is verified-by-inspection. |
| **Crisis-flagged cards (R5)** | Chip rendering unchanged — no crisis check in the chip mount condition | `crisisFlag` is server-side-only (line 93 of `prayer-wall.ts` types). The frontend `PrayerRequest` shape has no crisis fields. Crisis-flagged content is filtered upstream — the chip cannot introspect it even if the design wanted to. The R5 default (chip stays visible) is structurally enforced. |
| **Gate-G-CRISIS-CARD-CHIP-VISIBLE reframing** | Test asserts the chip render condition has NO post-type / category / `isAnswered` / other special-case branching — only `prayer.scriptureReference && (...)`. | Since `crisisFlag` is not on the wire shape, we cannot test "with crisis flag set" directly. Instead, we lock in the absence of crisis-conditional logic by exercising chip render across post types + categories without finding a branch that suppresses it. |
| **URL builder implementation** | `URLSearchParams` (matches BiblePlanDay) | Cleaner than template-string concatenation. Forward-compatible if a future spec adds more reader query params. Output URL has insertion-order parameters: `?scroll-to=X&verse=X`. |

### Step 1: Update `ScriptureChip.tsx` URL builder

**Objective:** Chip's `<Link to={path}>` builds `/bible/<book>/<chapter>?scroll-to=<n>&verse=<n>` when the parsed reference has a verse number; chapter-only references continue to build `/bible/<book>/<chapter>` with no query string; unparseable references continue to render as an unlinked span.

**Files to create/modify:**
- `frontend/src/components/prayer-wall/ScriptureChip.tsx` — Replace lines 29–30 with a `URLSearchParams`-based builder mirroring `BiblePlanDay.tsx:193-200`.

**Details:**

Replace the existing lines 29–30:

```tsx
const verseSuffix = parsed.verse !== undefined ? `?verse=${parsed.verse}` : ''
const path = `/bible/${parsed.book}/${parsed.chapter}${verseSuffix}`
```

With:

```tsx
const path = (() => {
  const base = `/bible/${parsed.book}/${parsed.chapter}`
  if (parsed.verse === undefined) return base
  // BB-38: both ?scroll-to= (one-shot arrival glow) and ?verse= (persistent
  // selection in the reader). Mirrors BiblePlanDay.tsx:193-200.
  const params = new URLSearchParams()
  params.set('scroll-to', String(parsed.verse))
  params.set('verse', String(parsed.verse))
  return `${base}?${params.toString()}`
})()
```

Everything else in the file (imports, props interface, unlinked-span branch, JSX class strings, `aria-label`, icon) stays verbatim. This is the entire material change.

**Guardrails (DO NOT):**
- Do NOT modify `BibleReader.tsx` — Gate-G-NO-BIBLEREADER-CHANGES
- Do NOT modify `PrayerCard.tsx` — the chip mount condition is correct
- Do NOT modify `parseReference` / `reference-parser.ts` — range handling already works (R3)
- Do NOT use the range format (`?scroll-to=28-30`) — explicitly out of scope per spec Section 5
- Do NOT add `?verse=28-30` for ranges — `?verse=` accepts a single integer per BB-38 contract
- Do NOT add a chapter-only `?scroll-to=` — chapter-only refs have no verse to scroll to (unlinked span branch covers it)
- Do NOT change parameter insertion order — `scroll-to` first, `verse` second (matches BiblePlanDay)
- Do NOT change the `aria-label` (Gate-G-EXISTING-PRAYERCARD-TEST-UNCHANGED depends on it)
