## Spec-Category-Specific Guidance

**Spec 6.3 is FRONTEND-ONLY.** Phase 3 Execution Reality Addendum gates are mostly N/A (no Liquibase, no OpenAPI, no rate limiting). The applicable gates are:

- **Gate-G-NO-FOUC (HARD):** Inline `<script>` in `index.html` runs synchronously before React, reads localStorage + browser hour, applies a route-aware `data-prayer-wall-night-mode-pending` attribute to `<html>`. <5ms execution.
- **Gate-G-COPY-PAIRS (HARD):** 8 day/night copy pairs Eric-approved before execute; CC may light-edit (within 2 words per variant) for fit; no wholesale replacement.
- **Gate-G-A11Y (HARD):** WCAG AA contrast verified; keyboard nav (Tab/Enter/Esc) on chip; focus trap on popover; reduced-motion disables breathing-glow + palette transition; axe-core 0 violations.
- **Gate-G-SCOPE-PRAYER-WALL-ONLY (HARD):** `data-night-mode` attribute lives only on Prayer Wall containers. `data-prayer-wall-night-mode-pending` (used by the inline script) is allowed on `<html>` ONLY as a transient no-FOUC route-detection signal — the gate forbids `:root[data-night-mode]` and `html[data-night-mode]` specifically (literal attribute name); the pending attribute is a separate name and satisfies the gate by construction.
- **Gate-G-LIVE-TRANSITION (SOFT):** 60s polling tick re-evaluates while mounted; cleared on unmount.

**Details:**

```ts
// frontend/src/lib/night-mode-resolver.ts
import type { NightModePreference } from '@/types/settings'

/**
 * Returns true when the hour falls within the Night Mode window
 * (21:00 inclusive to 06:00 exclusive).
 *
 * MIRROR — the same logic is inlined in frontend/index.html for no-FOUC.
 * If you change this function, update the inline script and re-run
 * `night-mode-resolver-parity.test.ts`.
 */
export function isNightHour(hour: number): boolean {
  return hour >= 21 || hour < 6
}

/**
 * Resolves a 3-state preference + browser hour into a boolean active state.
 *
```

```ts
// frontend/src/hooks/useNightMode.ts
import { useEffect, useState, useCallback } from 'react'
import type { NightModePreference } from '@/types/settings'
import { useSettings } from '@/hooks/useSettings'
import { resolveNightModeActive } from '@/lib/night-mode-resolver'

export const NIGHT_MODE_HINT_KEY = 'wr_night_mode_hint'

export interface UseNightModeReturn {
  active: boolean
  source: 'auto' | 'manual'
  userPreference: NightModePreference
}

function readCurrentHour(): number {
  return new Date().getHours()
}

function writeHint(active: boolean) {
  try {
    localStorage.setItem(NIGHT_MODE_HINT_KEY, active ? 'on' : 'off')
  } catch (_) {
    // private mode / quota — graceful no-op
  }
}

export function useNightMode(): UseNightModeReturn {
  const { settings } = useSettings()
  const userPreference = settings.prayerWall.nightMode
  const [hour, setHour] = useState<number>(() => readCurrentHour())

  // Polling tick — re-evaluate every 60s while mounted.
  useEffect(() => {
    const id = setInterval(() => {
      setHour(readCurrentHour())
    }, 60_000)
    return () => clearInterval(id)
  }, [])

  const active = resolveNightModeActive(userPreference, hour)
  const source: 'auto' | 'manual' = userPreference === 'auto' ? 'auto' : 'manual'

  // Reconciliation: persist resolved state to hint key for next-load no-FOUC.
  useEffect(() => {
    writeHint(active)
  }, [active])

  return { active, source, userPreference }
}
```

### Fix #1 — Hero `<h1>` violet gradient at night (HIGH)

**Symptom:** `[data-night-mode='on'] [data-night-hero-heading]` CSS rule did not paint amber; H1 stayed violet because the inline `style={GRADIENT_TEXT_STYLE}` on `PrayerWallHero.tsx`'s `<h1>` won the cascade (specificity 1,0,0,0 inline vs 0,0,2,0 attribute selector).

**Fix:** Added `!important` to all five declarations in the Step 7 hero-heading rule (`frontend/src/index.css` lines 158-172). Per CSS Cascade spec, `!important` author rules DO beat inline `style=""` without `!important`. Pattern matches the existing precedents in the same file: BackgroundCanvas multi-bloom override (lines 174-182, same problem) and marker-class surgical overrides (lines 187-200).

**Verification:** `getComputedStyle(h1).backgroundImage` now reports `linear-gradient(rgb(255, 255, 255) 0%, rgb(212, 163, 115) 100%)` — white → amber (#d4a373), no longer white → violet (#8b5cf6).

### Fix #2 — SPA pending-attr leak (HIGH)

**Symptom:** `data-prayer-wall-night-mode-pending="on"` persisted on `<html>` after SPA navigation away from `/prayer-wall`. The inline `index.html` script does not re-run on SPA route changes, so visiting `/prayer-wall` (with pref='on' or auto+night-hour) and then SPA-navigating to `/daily` (or any other route) leaked the warm body backdrop (`html[data-prayer-wall-night-mode-pending='on'] body { background-color: var(--pw-night-bg-base) }`) onto every non-Prayer-Wall route until the next full page reload. Gate-G-SCOPE-PRAYER-WALL-ONLY scope violation in spirit.

**Fix:** Added a top-level `NightModePendingCleanup` effect component in `frontend/src/App.tsx` (mounted as a sibling of `NotificationSchedulerEffect`, inside the BrowserRouter). The component subscribes to `location.pathname` via `useLocation` and removes the pending attribute when the path does not match `/prayer-wall(/.*)?$`. Removing on every non-PW route is safe because the pending attribute exists only to paint the body backdrop BEFORE React mounts on a Prayer Wall surface; after mount, the canonical `[data-night-mode='on']` rule on the BackgroundCanvas / PageShell wrapper takes over for the visible night palette.

**Verification:** Visited `/prayer-wall` (pref='on') → pending attr was `'on'` and body bg was warm dim. SPA-nav to `/daily` → pending attr now correctly `null` and body bg returns to `rgb(8, 5, 26)` (#08051a standard hero-bg).
