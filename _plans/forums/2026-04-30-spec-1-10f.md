## Spec-Category-Specific Guidance

This is a **BACKEND-HEAVY full-stack spec, NOT a dual-write migration**. There is no localStorage source-of-truth to migrate — legal acceptance is born on the backend. There is no feature flag (`VITE_USE_BACKEND_LEGAL`) — both reads and writes go directly to the backend.

Apply Phase 3 Execution Reality Addendum items where applicable (per spec § "Phase 3 Execution Reality Addendum gates — applicability"):

- **Item 2 (L1-cache trap):** `POST /me/legal/accept` updates `user.terms_version` and `user.privacy_version` then either returns 204 (no body) OR a fresh `UserResponse`. Plan choice: return **204 No Content** (simpler, matches Spec D5). Frontend explicitly re-fetches `GET /users/me` after a successful accept to refresh the cached `AuthUser` (per spec D15's "frontend explicitly refreshes user state after a successful accept"). No L1-cache test guard needed because the response carries no DB-default-populated columns.
- **Item 3 (`@Modifying` flags):** N/A — the update is implemented as entity-based `userRepository.save(user)`, NOT a JPQL `UPDATE` statement (per spec § "Phase 3 Execution Reality Addendum gates — applicability" item 3 recommendation).
- **Item 4 (SecurityConfig rule ordering):** Applies. `POST /api/v1/users/me/legal/accept` is `.authenticated()`, placed BEFORE `OPTIONAL_AUTH_PATTERNS.permitAll()`. `GET /api/v1/legal/versions` is added to `PublicPaths.PATTERNS` (fully public, JWT filter skips entirely).
- **Item 5 (Caffeine-bounded bucket):** Applies. `LegalAcceptRateLimitConfig` + `LegalAcceptRateLimitService` mirror `PostsRateLimitConfig` + `PostsRateLimitService` exactly. `@ConfigurationProperties(prefix = "worshiproom.legal.accept")`. Cache: `Caffeine.newBuilder().maximumSize(10_000).expireAfterAccess(Duration.ofMinutes(70))` (slightly longer than 1h refill window so an active user doesn't get a fresh bucket from eviction).
- **Item 6 (domain-scoped advice):** Applies. `@RestControllerAdvice(basePackages = "com.worshiproom.legal")` for `LegalExceptionHandler`. The new exceptions are NOT thrown from filters, so no unscoped companion advice needed. Validation-failure exceptions thrown from inside `@Valid` on the registration request body bubble through the existing `MethodArgumentNotValidException` handlers (already in `UserValidationExceptionHandler` or similar).
- **Item 7 (CrisisAlertService):** N/A.
- **Item 8 (do NOT recreate schema):** Applies — confirmed via recon that `users.terms_version` and `users.privacy_version` do NOT exist. The `<addColumn>` is the correct operation (per spec § Watch-for #6).

## Edge Cases & Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Spec D1 — column nullability | Both columns `nullable=true` initially | Existing dev-seed users get NULL → modal fires → users accept → row updated. Tightening to NOT NULL is a future spec post-launch. |
| Spec D2 — version format | ISO-8601 date string `2026-04-29` | Human-readable, sortable, communicates change-time. Documented in `LegalVersionService` Javadoc. |
| Spec D3 — version constants location | `LegalVersionService.java` as `public static final String` | Single source of truth; bumping a doc = bumping a constant. Rejected config-properties (operator-changeable would be wrong for legal docs). |
| Spec D4 — GET /legal/versions auth | `permitAll` (added to `PublicPaths.PATTERNS`) | No PII, no leakage; needed by registration form before the user has a token. JWT filter skips entirely. |
| Spec D5 — POST /me/legal/accept response | `204 No Content` | Simpler than returning UserResponse. Frontend fetches `GET /users/me` after a successful accept (per D15) to refresh cached AuthUser. |
| Spec D6 — registration version fields | Required (`@NotBlank`) | Re-using existing `@NotBlank` validation produces 400 INVALID_INPUT on missing fields without a custom validator. Mismatched versions return 400 VERSION_MISMATCH via service-layer check. |
| Anti-enumeration during registration | Validate version match BEFORE the existing-email branch | If the version is wrong, return 400 BEFORE burning the timing-equalization BCrypt encode. This means version-error and existing-email branches have different timing — but that's acceptable because version mismatch is a protocol error, not a content-leak channel. |
| Spec D7 — useLegalVersions | Single-fetch on mount, cached in module-level state, refetch on `wr:auth-invalidated` | Versions are static per release; refetching every render or every route change is wasteful. |
| Spec D8 — modal dismissal | X button + Esc + "Later" → session-scoped React state flag (NOT persisted) | Reset on next page load if still stale, per anti-pressure design. |
| Spec D9 — useGatedAction wrapping | Wrap inside `useFriends` (call site of `sendRequest`/`acceptRequest`); future specs adopt for new mutations | Eric's "wire only what exists today" directive; future specs add their own gates. |
| Spec D9 — queued action replay on accept | Pending action stored in a single ref; on successful 204 from accept, replay; if user dismisses without accepting, ref clears and action drops | Per spec Watch-for #15. |
| Spec D10 — modal copy treatment | Boot-stale modal → "Review and accept" / "Later"; click "Review and accept" → inline accept form with 3 doc links + checkbox + Accept/Cancel | Per spec D10. The two-state pattern (notice + inline accept) is implemented as React state inside the same modal component, NOT two separate modals. |
| Spec D11 — registration consent label | Plain language with 3 inline links to ToS / Privacy / Community Guidelines | Per spec D11. Submit disabled via React state until checkbox checked. |
| Spec D12 — page routes | `/terms-of-service` and `/privacy-policy` (NOT `/terms` / `/privacy`) | Per spec D12; matches markdown filenames; SEO-friendly. |
| Spec D13 — footer column case | "1.10m has shipped" — add 2 entries to existing column | Verified via recon (SiteFooter.tsx:31–33 has 1 entry). |
| Spec D14 — rate-limit config | 5/hour per user, Caffeine 10K cache, mirror `PostsRateLimitConfig` shape | Per spec D14. Rejected per-IP fallback (the endpoint is auth-gated, no anonymous abuse vector). |

| Watch-for #14 — link clicks ≠ acceptance | Doc links are `<a target="_blank">` only | The explicit checkbox + Submit is the legal action. |
| Watch-for #16 — no email/in-app banner notifications | Out of scope this spec | Spec 15.x territory. |

```java
/**
 * Canonical source of truth for the three current legal-document versions.

 * <p>Versions are encoded as ISO-8601 dates. To bump a version, you update
 * BOTH the markdown source at {@code content/{terms-of-service,privacy-policy,
 * community-guidelines}.md} AND the corresponding constant here AND the JSX

 * lexicographic comparison. Submitting a version older than current OR newer
 * than current both reject with VERSION_MISMATCH.

@Service
public class LegalVersionService {

    public static final String TERMS_VERSION = "2026-04-29";
    public static final String PRIVACY_VERSION = "2026-04-29";

        return TERMS_VERSION.equals(submitted);
    }

    public boolean isPrivacyVersionCurrent(String submitted) {
        return PRIVACY_VERSION.equals(submitted);
    }

    public String currentPrivacyVersion() { return PRIVACY_VERSION; }
```

```java
// LegalAcceptRateLimitService.java
package com.worshiproom.legal;

import com.github.benmanes.caffeine.cache.Cache;
import com.github.benmanes.caffeine.cache.Caffeine;
import io.github.bucket4j.Bandwidth;
import io.github.bucket4j.Bucket;
import io.github.bucket4j.ConsumptionProbe;
import io.github.bucket4j.Refill;
import org.springframework.stereotype.Service;

import java.time.Duration;
import java.util.UUID;
import java.util.concurrent.TimeUnit;

/**
 * Per-user legal-accept rate limit. 5 tokens per hour with intervally refill.
 *
 * <p>Bucket map is bounded per the BOUNDED EXTERNAL-INPUT CACHES rule —
 * Caffeine maximumSize(10_000), expireAfterAccess(70 minutes) so the eviction
 * window is slightly longer than the refill window (no risk of evicting a
 * still-active bucket and giving a user free retries).
 *
 * <p>Mirrors {@link com.worshiproom.post.PostsRateLimitService} pattern.
 */
@Service
public class LegalAcceptRateLimitService {

    private final Cache<UUID, Bucket> userBuckets;
    private final LegalAcceptRateLimitConfig config;

    public LegalAcceptRateLimitService(LegalAcceptRateLimitConfig config) {
        this.config = config;
        this.userBuckets = Caffeine.newBuilder()
                .maximumSize(config.getRateLimit().getBucketCacheSize())
                .expireAfterAccess(Duration.ofMinutes(70))
                .build();
    }

    public void checkAndConsume(UUID userId) {
        Bucket bucket = userBuckets.get(userId, k -> newBucket());
        ConsumptionProbe probe = bucket.tryConsumeAndReturnRemaining(1);
        if (!probe.isConsumed()) {
            long retryAfterSec = Math.max(1L, TimeUnit.NANOSECONDS.toSeconds(probe.getNanosToWaitForRefill()));
            throw new LegalAcceptanceRateLimitException(retryAfterSec);
        }
    }

    private Bucket newBucket() {
        int max = config.getRateLimit().getMaxPerHour();
        Bandwidth limit = Bandwidth.classic(max, Refill.intervally(max, Duration.ofHours(1)));
        return Bucket.builder().addLimit(limit).build();
    }
}
```

```java
    public ResponseEntity<Void> acceptVersions(
            @AuthenticationPrincipal AuthenticatedUser principal,
            @Valid @RequestBody AcceptLegalVersionsRequest request) {

        // Per spec § Watch-for #1: caller == subject is enforced via the JWT principal.
        // The /me/ path segment is for URL clarity; the actual authorization is the
        // principal.userId() — never trust a userId from request body or path.
        rateLimitService.checkAndConsume(principal.userId());

        if (!versionService.isTermsVersionCurrent(request.termsVersion())) {
            throw new VersionMismatchException();
        }
        if (!versionService.isPrivacyVersionCurrent(request.privacyVersion())) {
            throw new VersionMismatchException();
        }

        User user = userRepository.findById(principal.userId())
            .orElseThrow(UserException::userNotFound);

        user.setTermsVersion(request.termsVersion());
        user.setPrivacyVersion(request.privacyVersion());
        user.setUpdatedAt(OffsetDateTime.now(ZoneOffset.UTC));
        userRepository.save(user);

        log.info("legalAcceptRecorded userId={} termsVersion={} privacyVersion={}",
            principal.userId(), request.termsVersion(), request.privacyVersion());

        return ResponseEntity.noContent().build();
    }
}
```

### Step 17: `useLegalVersions` hook — single-fetch + cache + refetch on auth-invalidated

**Objective:** Hook that fetches current versions from `GET /legal/versions` once on mount, caches in module state, and refetches on the `wr:auth-invalidated` event.

**Files to create:**

- `frontend/src/hooks/useLegalVersions.ts`

**Details:**

```ts
import { useEffect, useState } from 'react'
import { getLegalVersionsApi, type LegalVersions } from '@/services/api/legal-api'
import { AUTH_INVALIDATED_EVENT } from '@/lib/api-client'

interface UseLegalVersionsResult {
  versions: LegalVersions | null
  isLoading: boolean
  error: Error | null
}

// Module-level cache so multiple components don't re-fire the fetch.
let cachedVersions: LegalVersions | null = null
let inflightFetch: Promise<LegalVersions> | null = null

function fetchOnce(): Promise<LegalVersions> {
  if (inflightFetch) return inflightFetch
  if (cachedVersions) return Promise.resolve(cachedVersions)
  inflightFetch = getLegalVersionsApi()
    .then((v) => {
      cachedVersions = v
      inflightFetch = null
      return v
    })
    .catch((err) => {
      inflightFetch = null
      throw err
    })
  return inflightFetch
}

function invalidateCache() {
  cachedVersions = null
  inflightFetch = null
}

export function useLegalVersions(): UseLegalVersionsResult {
  const [versions, setVersions] = useState<LegalVersions | null>(cachedVersions)
  const [isLoading, setIsLoading] = useState<boolean>(!cachedVersions)
  const [error, setError] = useState<Error | null>(null)

  useEffect(() => {
    let cancelled = false

    function load() {
      setIsLoading(true)
      setError(null)
      fetchOnce()
        .then((v) => {
          if (cancelled) return
          setVersions(v)
          setIsLoading(false)
        })
        .catch((err) => {
          if (cancelled) return
          setError(err instanceof Error ? err : new Error(String(err)))
          setIsLoading(false)
        })
    }

    load()

    function handleAuthInvalidated() {
      invalidateCache()
      load()
    }
    window.addEventListener(AUTH_INVALIDATED_EVENT, handleAuthInvalidated)
    return () => {
      cancelled = true
      window.removeEventListener(AUTH_INVALIDATED_EVENT, handleAuthInvalidated)
    }
  }, [])

  return { versions, isLoading, error }
}

// Test helper — exported but only consumed by tests
export function __resetLegalVersionsCache() {
  cachedVersions = null
  inflightFetch = null
}
```

### Step 18: `useGatedAction` hook — soft-enforcement wrapper

**Objective:** Hook that wraps an action callback. Before invoking, checks if the user has stale legal acceptance (per `LegalVersionGate` context). If stale: shows the modal, queues the action; on successful accept, replays the queued action.

**Files to create:**

- `frontend/src/hooks/useGatedAction.ts`

**Details:**

```ts
import { useCallback } from 'react'
import { useLegalVersionGate } from '@/components/legal/LegalVersionGate'

/**
 * Wraps an action so it only fires when the current user's legal acceptance
 * is current. If stale, shows the TermsUpdateModal and queues the action;
 * on successful accept, the queued action replays. If user dismisses without
 * accepting, the queued action drops.
 *
 * Per Spec 1.10f D9.
 */
export function useGatedAction<TArgs extends unknown[]>(
  action: (...args: TArgs) => void,
): (...args: TArgs) => void {
  const gate = useLegalVersionGate()

  return useCallback(
    (...args: TArgs) => {
      if (gate.isStaleAcceptance) {
        gate.queueAndShow(() => action(...args))
        return
      }
      action(...args)
    },
    [gate, action],
  )
}
```

### Step 20: `LegalVersionGate` provider

**Objective:** Provider that wraps the app, exposes `isStaleAcceptance` + `queueAndShow(action)` via context, mounts `TermsUpdateModal` overlay when stale, manages session-scoped dismissal flag, queued-action ref + replay on successful accept.

**Files to create:**

- `frontend/src/components/legal/LegalVersionGate.tsx`

**Details:**

```tsx
import { createContext, useCallback, useContext, useMemo, useRef, useState, type ReactNode } from 'react'
import { useAuth } from '@/hooks/useAuth'
import { useLegalVersions } from '@/hooks/useLegalVersions'
import { TermsUpdateModal } from './TermsUpdateModal'

interface LegalVersionGateContextValue {
  isStaleAcceptance: boolean
  queueAndShow: (action: () => void) => void
}

const LegalVersionGateContext = createContext<LegalVersionGateContextValue | null>(null)

export function LegalVersionGate({ children }: { children: ReactNode }) {
  const { isAuthenticated, user } = useAuth()
  const { versions, isLoading } = useLegalVersions()
  const [sessionDismissed, setSessionDismissed] = useState(false)
  const queuedActionRef = useRef<(() => void) | null>(null)

  const isStaleAcceptance = useMemo(() => {
    if (isLoading || !versions) return false
    if (!isAuthenticated || !user) return false
    return (
      user.termsVersion !== versions.termsVersion ||
      user.privacyVersion !== versions.privacyVersion
    )
  }, [isLoading, versions, isAuthenticated, user])

  const showModal = isStaleAcceptance && !sessionDismissed

  const queueAndShow = useCallback((action: () => void) => {
    queuedActionRef.current = action
    setSessionDismissed(false) // re-show even if previously dismissed this session
  }, [])

  const handleAccepted = useCallback(() => {
    // Re-fetch /users/me to refresh the cached user state with new versions.
    // This mirrors the boot-time hydration pattern; the simplest path is to
    // dispatch the existing `wr:auth-invalidated` event NO — that would log
    // the user out. Instead, we directly re-call getCurrentUser() and update
    // AuthContext via a new method, OR we hard-reload. Simplest correct path:
    // use a dedicated AuthContext method `refreshUser()` to be added.
    //
    // For this plan: extend AuthContext with `refreshUser()` (calls
    // getCurrentUser + setState). Wired in Step 21.

    // Replay queued action AFTER user state has updated.
    const action = queuedActionRef.current
    queuedActionRef.current = null
    if (action) {
      // Defer to next tick so the user state setter from refreshUser() has
      // completed before the action fires.
      setTimeout(action, 0)
    }
    setSessionDismissed(true) // hide modal until next stale state
  }, [])

  const handleDismiss = useCallback(() => {
    queuedActionRef.current = null // dismissed without accepting → drop queued action
    setSessionDismissed(true)
  }, [])

  const value: LegalVersionGateContextValue = useMemo(
    () => ({ isStaleAcceptance, queueAndShow }),
    [isStaleAcceptance, queueAndShow],
  )

  return (
    <LegalVersionGateContext.Provider value={value}>
      {children}
      {showModal && versions && (
        <TermsUpdateModal
          isOpen={showModal}
          versions={versions}
          onAccepted={handleAccepted}
          onDismiss={handleDismiss}
        />
      )}
    </LegalVersionGateContext.Provider>
  )
}

// eslint-disable-next-line react-refresh/only-export-components
export function useLegalVersionGate(): LegalVersionGateContextValue {
  const ctx = useContext(LegalVersionGateContext)
  if (!ctx) throw new Error('useLegalVersionGate must be used within LegalVersionGate')
  return ctx
}
```
