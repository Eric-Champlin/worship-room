## Architecture Context

### Existing patterns confirmed by recon

- **Controller shape (canonical):** `com.worshiproom.user.UserController` is the closest precedent.
  - `@RestController` + `@RequestMapping("/api/v1/users")` (note path prefix; subroutes via `@GetMapping("/me")`, `@PatchMapping("/me")`).
  - Constructor injection only (per `03-backend-standards.md`).
  - Principal injected via `@AuthenticationPrincipal AuthenticatedUser principal` (Spring Security's built-in resolver — no custom resolver).
  - All methods return `ResponseEntity<ProxyResponse<T>>` with `MDC.get("requestId")`.
  - `@Valid` on `@RequestBody` parameters.
- **AuthenticatedUser principal:** record `(UUID userId, boolean isAdmin)` — fixed contract from Spec 1.4. Does NOT carry timezone. The week-start helper in this spec must fetch the User entity (or just the timezone string) via `UserRepository.findById(principal.userId())`.
- **Exception advice pattern:** Two flavors documented in `03-backend-standards.md` § "@RestControllerAdvice Scoping":
  1. **Package-scoped** (e.g., `UserExceptionHandler`, `UserValidationExceptionHandler`): `@RestControllerAdvice(basePackages = "com.worshiproom.user")`. Catches exceptions thrown from controllers in that package only. Spring's `isApplicableToBeanType()` resolves by **controller's** package, NOT by the exception's package.
  2. **Unscoped, single-exception-class** (e.g., `RateLimitExceptionHandler`): no `basePackages`. Safe because the exception type is owned by one feature and never thrown elsewhere. Class-level JavaDoc explains the deliberate non-scoping.
- **`UserController.search()` cross-package case (Watch-for #4 in spec):** Per Divergence 2, `GET /api/v1/users/search?q=...` lives on `UserController`, not `FriendsController`. It delegates to `FriendsService.searchUsers(...)` which throws `InvalidInputException` (a `FriendsException` subclass) when `q` < 2 chars. Spring's `@RestControllerAdvice(basePackages = "com.worshiproom.friends")` does NOT catch exceptions raised from a controller in `com.worshiproom.user` — the advice resolves on the controller's package. **Decision (D-Advice below): make `FriendsExceptionHandler` UNSCOPED single-class** (handles only `FriendsException` and subclasses; safe because no other code throws those). `FriendsValidationExceptionHandler` stays package-scoped (validation exceptions are generic and could be over-caught if globally scoped — and search endpoint validates at service layer, so MethodArgumentNotValidException only arises from FriendsController's own `@Valid` request bodies).
- **Validation error code (`INVALID_INPUT` vs `VALIDATION_FAILED`):** `UserValidationExceptionHandler` emits `VALIDATION_FAILED` with a `fieldErrors` map. The 2.5.3 spec acceptance criteria explicitly say `INVALID_INPUT` for missing/invalid request fields. **Decision (D-ValidationCode below): `FriendsValidationExceptionHandler` emits `INVALID_INPUT` with `fieldErrors`** (deliberate divergence from User's `VALIDATION_FAILED` to satisfy spec acceptance criteria). The friends domain consolidates all 400 errors under `INVALID_INPUT` (matching the `InvalidInputException` thrown from the service layer), giving the frontend a single error code to branch on for friends 400s.
- **Test pattern (canonical):** `UserControllerIntegrationTest` is the closest precedent.
  - Extends `AbstractIntegrationTest` (full Spring context, singleton Testcontainers PostgreSQL).
  - `@AutoConfigureMockMvc` to inject `MockMvc`.
  - Subclass `@DynamicPropertySource` registers `jwt.secret` (≥32 bytes) AND `auth.rate-limit.per-email.capacity`/`.per-ip.capacity` (inflated to never fire — `LoginRateLimitFilter` is in the bean graph regardless).
  - `@BeforeEach` calls `userRepository.deleteAll()` then re-seeds users; CASCADE cleans transitive child rows (friend_relationships, friend_requests, etc.).
  - JWT acquired via `jwtService.generateToken(seedUser.getId(), seedUser.isAdmin())` — there is NO `/auth/login` round-trip in tests. This is faster and avoids password-hash setup.
- **Service-level validation surfaces through controller as friends-domain exception:** `FriendsService.searchUsers` throws `InvalidInputException` for `q` < 2 chars. Service-thrown `InvalidInputException` is caught by `FriendsExceptionHandler` (unscoped) → 400 INVALID_INPUT. Same path as all other friends-domain rejections (`AlreadyFriendsException`, `BlockedUserException`, etc.). Bean-Validation @Valid failures (missing toUserId, etc.) surface as `MethodArgumentNotValidException` and are caught by `FriendsValidationExceptionHandler` → 400 INVALID_INPUT with `fieldErrors`.
- **JsonNullable / OpenAPI / DTO conventions:** `UserController.PATCH /me` uses `JsonNullable<T>` for three-state PATCH semantics. **Friends DTOs do NOT need JsonNullable** — request bodies are creates and toggles, no PATCH-with-clear-to-null semantics. Use plain `@NotNull UUID toUserId` etc.
- **OpenAPI insertion points:** `backend/src/main/resources/openapi.yaml` already exists (1604 lines). Existing structure:
  - `paths:` section (line 40) ends at line 823. Insert new friends paths AFTER `/api/v1/activity/backfill` (line 774) so chronological order matches phase order.
  - `components.schemas:` (line 826) ends ~1537. Insert new friends schemas at the end of the schemas block, after `ActivityBackfillResponse`.
  - `components.responses:` (line 1538) — already has `BadRequest`, `Unauthorized`, `RateLimited`, `UpstreamError`, `UpstreamTimeout`, `NotFound`, `SafetyBlocked`, `InternalError`. **Need to ADD: `Conflict` (409) and `Forbidden` (403)** — heavily used by friends endpoints, missing today.
  - `components.securitySchemes.bearerAuth` already defined (line 1597).
- **`RateLimitFilter` scope:** Spec 1's filter explicitly excludes everything except `/api/v1/proxy/**` via `shouldNotFilter`. New friends paths under `/api/v1/users/me/friends/**`, `/api/v1/friend-requests/**`, `/api/v1/users/me/blocks/**`, `/api/v1/users/search` are NOT subject to the existing rate limit. No filter changes needed (per Divergence 1).

## Edge Cases & Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **D-Advice: Advice scoping for `FriendsExceptionHandler`** | UNSCOPED single-class advice (handles only `FriendsException`). Class-level JavaDoc mirrors `RateLimitExceptionHandler`'s deliberate-non-scoping comment. | Per Watch-for #4 + recon. The `UserController.search()` endpoint (Divergence 2) lives in `com.worshiproom.user` and throws friends-domain exceptions (`InvalidInputException` from `FriendsService.searchUsers`). Spring's `@RestControllerAdvice(basePackages = "...")` filters by **controller's** package, so a friends-package-scoped advice would NOT catch the exception from `UserController`. Unscoping is safe because `FriendsException` is owned by `com.worshiproom.friends` and not thrown anywhere else. Same architectural shape as `RateLimitExceptionHandler`. |
| **D-ValidationAdvice: Advice scoping for `FriendsValidationExceptionHandler`** | Package-scoped to `com.worshiproom.friends` (matches `UserValidationExceptionHandler` shape). | Bean Validation exceptions (`MethodArgumentNotValidException`, etc.) are generic — an unscoped advice would compete with the existing scoped advices for User and Auth. The friends-package-scoped advice catches @Valid failures from FriendsController's request DTOs. The `UserController.search()` endpoint does NOT use @Valid (no request body) — its `q` validation runs in `FriendsService.searchUsers` (service layer) and surfaces as `InvalidInputException`, caught by `FriendsExceptionHandler` (unscoped, see D-Advice). |
| **D-ValidationCode: Code emitted by `FriendsValidationExceptionHandler`** | `INVALID_INPUT` (NOT `VALIDATION_FAILED`). | Spec acceptance criteria explicitly say `INVALID_INPUT` for missing/invalid request fields (e.g., "POST friend-request with missing toUserId returns 400 INVALID_INPUT"). `UserValidationExceptionHandler` uses `VALIDATION_FAILED` — that's a deliberate cross-domain divergence: friends consolidates all 400s under `INVALID_INPUT`. Frontend gets a single error code to branch on for friends 400s. Field errors still emitted under a `fieldErrors` map (matches User's structure for parser compatibility). |
| **D-PatchDispatch: PATCH /friend-requests/{id} action handling** | Single endpoint, dispatched server-side by `action` enum in body. `RespondToFriendRequestRequest` has one field: `@NotNull @Pattern(regexp="^(accept\|decline\|cancel)$") String action`. Controller uses a switch on action string. | Per Divergence 3 + master plan line 3327. Spec watch-for #2 calls out the temptation to triplicate endpoints — push back. Three actions share validation + auth + exception shape, so dispatching server-side is simpler. |
| **D-PatchResponse: PATCH /friend-requests/{id} response shape** | 200 with body `{"status": "accepted" \| "declined" \| "cancelled"}`. | Spec line 116 says CC picks between 204 and 200-with-status. The 200 form is friendlier for frontend optimistic UI updates (frontend gets confirmation of new state without re-fetching). Aligns with Spec Z's anti-pressure framing — the frontend sees its own action reflected back, not a guessed result. |
| **D-LocationHeader: Location header on POST endpoints** | POST friend-request: YES, `Location: /api/v1/friend-requests/{id}` (resource is separately addressable). POST blocks: NO (block has no GET-by-id endpoint, only DELETE). | Watch-for #6. Don't fabricate `/api/v1/blocks/{userId}` — that endpoint doesn't exist. Block POST returns body `{"blockedUserId": "...", "blockedAt": "..."}` instead. |
| **D-WeekStart: Week-start computation matches frontend** | New utility class `WeekStartCalculator` with `monday00InZone(ZoneId zone, Clock clock)` returning `OffsetDateTime`. Mirrors `frontend/src/utils/date.ts` `getCurrentWeekStart()` logic: take **NOW** in the user's local timezone, find the **most recent Monday** (Monday=start, Sunday=6 days back), set time to 00:00 local, return as `OffsetDateTime` (with the zone's offset for that wall-time). | Per Divergence 4. Frontend uses `Date.getDay()` (Sunday=0, Monday=1) and rolls back to Monday in LOCAL time — not UTC. Java equivalent: `LocalDate.now(zone)` → `with(TemporalAdjusters.previousOrSame(DayOfWeek.MONDAY))` → `atStartOfDay(zone)` → `.toOffsetDateTime()`. The DST gotcha: `atStartOfDay(zone)` correctly handles spring-forward (returns 01:00 local when 00:00 doesn't exist) and fall-back (returns 00:00 with the earlier offset). Service receives `OffsetDateTime` — already supports zone-aware comparisons. |
| **D-WeekStartFetch: Where to fetch user's timezone** | `UserRepository.findById(principal.userId())` in `FriendsController.listFriends`. NO Caching, NO injection of timezone into AuthenticatedUser. | Watch-for #10. AuthenticatedUser is a fixed contract from Spec 1.4 — only `userId` and `isAdmin`. The list-friends endpoint already fetches the User's friends list (denormalized join in 2.5.2's native query), so adding one more `findById` for the calling user is acceptable cost. Future optimization (cache timezone in JWT or short-lived in-memory map) is its own spec. |
| **D-WeekStartUnit: Unit-testable extraction** | Extract `WeekStartCalculator` as a top-level class with a static method, NOT as a private method on FriendsController. Inject `Clock` for testability (default to `Clock.systemDefaultZone()` via Spring `@Bean` if a Clock isn't already in the bean graph — recon shows NO `Clock` bean exists; **decision: pass `Clock` as a method parameter** so the controller stays Clock-free and the helper is purely functional). | Aligned with `WeekStartCalculatorTest` requirements: spring-forward DST week, fall-back DST week, non-zero-offset zone (Asia/Tokyo). A static `monday00InZone(ZoneId zone, Clock clock)` is the minimal testable surface. Controller calls `WeekStartCalculator.monday00InZone(ZoneId.of(user.getTimezone()), Clock.systemDefaultZone())`. |
| **D-DirectionParam: Missing/invalid `direction` query param** | Missing → 400 INVALID_INPUT (caught by `FriendsValidationExceptionHandler` via `MissingServletRequestParameterException`). Invalid value (e.g., `direction=foo`) → 400 INVALID_INPUT (caught by FriendsController explicitly checking the value and throwing `InvalidInputException` from the friends domain). | Spec § 2 acceptance criteria explicitly call for both. `MissingServletRequestParameterException` is a Spring Bean Validation cousin handled by `FriendsValidationExceptionHandler`. Invalid value goes through the service-thrown InvalidInputException path so the user gets a consistent 400 INVALID_INPUT either way. |
| **D-EmptyList: Empty list response** | Always 200 with `{ "data": [], "meta": { ... } }`. NEVER 404. | Watch-for #5. "No friends" / "no pending requests" is not an error condition. Tests assert this explicitly. |
| **D-PublicPaths: PublicPaths.java unchanged** | NO friends path added to `PublicPaths.PATTERNS`. | Watch-for #1. All 8 endpoints require auth. JWT filter rejects unauthenticated requests with 401. |
| **D-FriendsTypeShape: Frontend hand-typed types in `frontend/src/types/api/friends.ts`** | Hand-typed TypeScript interfaces matching the OpenAPI schemas (FriendDto, FriendRequestDto, UserSearchResultDto, SendFriendRequestRequest, RespondToFriendRequestRequest, BlockUserRequest, plus a small `Direction = 'incoming' \| 'outgoing'` union). No openapi-typescript codegen run (pipeline not wired). | Watch-for #12. Recon confirmed `frontend/package.json` has no `types:generate` script; existing API type modules are hand-typed. Document the hand-typed path in this file (already done in `03-backend-standards.md`). |
| **D-OpenApiSharedResponses: Add `Conflict` and `Forbidden` shared responses to openapi.yaml** | Add at the end of `components.responses:` (after `Unauthorized`). Each `$ref`s `ProxyError` schema. Friends endpoints `$ref` them (e.g., `'409': { $ref: '#/components/responses/Conflict' }`). | Recon confirmed only `BadRequest`, `Unauthorized`, `RateLimited`, `NotFound`, `UpstreamError`, `UpstreamTimeout`, `SafetyBlocked`, `InternalError` exist today. Friends adds 3 endpoints that emit 409 (POST friend-request/AlreadyFriends/Duplicate, PATCH/InvalidRequestState) and several 403 paths (BlockedUser, UnauthorizedAction). package com.worshiproom.friends.dto;

import jakarta.validation.constraints.NotNull;
import jakarta.validation.constraints.Size;

import java.util.UUID;

/**
 * POST /api/v1/users/me/friend-requests body. Plain-text message per
 * Universal Rule 14 — no HTML, no Markdown rendering. The 280-char cap

public record SendFriendRequestRequest(

```

`RespondToFriendRequestRequest.java`:

package com.worshiproom.friends.dto;

import jakarta.validation.constraints.NotNull;
import jakarta.validation.constraints.Pattern;

/**

 * dispatched server-side per Divergence 3. Pattern regex is the source
 * of truth for valid values; no shadow Java enum.
 */
public record RespondToFriendRequestRequest(
    @NotNull(message = "action is required")
    @Pattern(regexp = "^(accept|decline|cancel)$",

) {}
```

`BlockUserRequest.java`:

`WeekStartCalculator.java`:

package com.worshiproom.friends;

import java.time.Clock;
import java.time.DayOfWeek;
import java.time.LocalDate;
import java.time.OffsetDateTime;
import java.time.ZoneId;
import java.time.temporal.TemporalAdjusters;

/**
 * Computes the start of the current week (Monday 00:00 in the given timezone)
 * to match the frontend convention from {@code frontend/src/utils/date.ts}.
 *
 * <p>Frontend logic (verbatim equivalent):
 * <pre>
 * const now = new Date();
 * const day = now.getDay(); // 0 = Sunday, 1 = Monday
 * const diff = now.getDate() - day + (day === 0 ? -6 : 1); // most-recent Monday
 * const monday = new Date(year, month, diff); // 00:00 LOCAL time
 * </pre>
 *
 * <p>Java equivalent: take "now" in the user's zone, adjust to the previous-or-
 * same Monday, set time to 00:00 in that zone, return as an OffsetDateTime
 * (with the zone's offset for that wall-time, which handles DST correctly).
 *
 * <p>DST behavior:
 * <ul>
 *   <li>Spring-forward week: {@code atStartOfDay(zone)} returns the actual
 *       earliest valid time of day in the zone (typically 00:00, but 01:00 on
 *       gap days). Friends list weekly window starts at that point — correct.</li>
 *   <li>Fall-back week: {@code atStartOfDay(zone)} returns 00:00 with the
 *       PRE-fall-back offset (the earlier of the two possible offsets), which
 *       matches the frontend's "Monday 00:00 local" framing.</li>
 * </ul>
 *
 * <p>Clock parameter is for testability — production callers pass
 * {@link Clock#systemDefaultZone()}.
 */
public final class WeekStartCalculator {

    private WeekStartCalculator() {}

    /**
     * @param zone user's IANA timezone (e.g., "America/Chicago", "UTC")
     * @param clock injected clock — production: Clock.systemDefaultZone()
     * @return Monday 00:00 in the user's timezone as an OffsetDateTime
     */
    public static OffsetDateTime monday00InZone(ZoneId zone, Clock clock) {
        LocalDate todayInZone = LocalDate.now(clock.withZone(zone));
        LocalDate monday = todayInZone.with(TemporalAdjusters.previousOrSame(DayOfWeek.MONDAY));
        return monday.atStartOfDay(zone).toOffsetDateTime();
    }
}

package com.worshiproom.friends;

import com.worshiproom.proxy.common.ProxyError;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.slf4j.MDC;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

/**
 * Maps {@link FriendsException} to {@link ProxyError} HTTP responses.
 *
 * <p><b>Deliberately unscoped</b> (no {@code basePackages} filter). Per Spec 2.5.3
 * Divergence 2, the user-search endpoint lives in {@link com.worshiproom.user.UserController}
 * but throws friends-domain exceptions (e.g., {@link InvalidInputException} from
 * {@link FriendsService#searchUsers}). A package-scoped advice keyed to
 * {@code com.worshiproom.friends} would NOT catch exceptions raised from a controller
 * in {@code com.worshiproom.user} — Spring's advice resolution checks the controller's
 * package, not the exception's package.
 *
 * <p>Unscoping is safe here because {@link FriendsException} is owned by the friends
 * domain and is never thrown by any other code. Same architectural shape as
 * {@link com.worshiproom.proxy.common.RateLimitExceptionHandler}, which is also
 * unscoped for the same single-exception-class reason.
 */
@RestControllerAdvice
public class FriendsExceptionHandler {

    private static final Logger log = LoggerFactory.getLogger(FriendsExceptionHandler.class);

    @ExceptionHandler(FriendsException.class)
    public ResponseEntity<ProxyError> handleFriends(FriendsException ex) {
        var requestId = MDC.get("requestId");
        log.info("Friends-domain rejection: code={} message={}", ex.getCode(), ex.getMessage());
        return ResponseEntity
            .status(ex.getStatus())
            .body(ProxyError.of(ex.getCode(), ex.getMessage(), requestId));
    }
}

package com.worshiproom.friends;

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.slf4j.MDC;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.MissingServletRequestParameterException;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;
import org.springframework.web.method.annotation.MethodArgumentTypeMismatchException;

import java.time.Instant;
import java.util.LinkedHashMap;
import java.util.Map;
import java.util.stream.Collectors;

/**
 * Validation-error advice scoped to friends controllers.
 *
 * <p><b>Emits {@code INVALID_INPUT}</b> (NOT {@code VALIDATION_FAILED}) to match
 * Spec 2.5.3 acceptance criteria. The friends domain consolidates all 400 errors
 * under {@code INVALID_INPUT}, giving the frontend a single error code to branch
 * on for friends 400s. Field-level errors are still emitted under a {@code fieldErrors}
 * map for parser compatibility with the user-domain response shape.
 *
 * <p>Package-scoped to {@code com.worshiproom.friends} so this advice does NOT
 * compete with sibling validation advices (User, Auth) for {@code MethodArgumentNotValidException}
 * raised from non-friends controllers.
 *
 * <p>Note: {@link com.worshiproom.user.UserController#search} (Divergence 2)
 * does NOT use {@code @Valid} request bodies — it validates {@code q} length at
 * the service layer, which throws {@link InvalidInputException} (handled by
 * {@link FriendsExceptionHandler}). So this advice is correctly scoped to the
 * friends package even though the search endpoint is cross-package.
 */
@RestControllerAdvice(basePackages = "com.worshiproom.friends")
public class FriendsValidationExceptionHandler {

    private static final Logger log = LoggerFactory.getLogger(FriendsValidationExceptionHandler.class);

    /** @Valid request-body failures (e.g., missing toUserId, invalid action regex). */
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String, Object>> handleValidation(MethodArgumentNotValidException ex) {
        Map<String, String> fieldErrors = ex.getBindingResult().getFieldErrors().stream()
            .collect(Collectors.toMap(
                fe -> fe.getField(),
                fe -> fe.getDefaultMessage() == null ? "invalid" : fe.getDefaultMessage(),
                (a, b) -> a,
                LinkedHashMap::new
            ));
        return invalidInput("Request validation failed.", fieldErrors);
    }

    /** Missing required query param (e.g., direction= unspecified). */
    @ExceptionHandler(MissingServletRequestParameterException.class)
    public ResponseEntity<Map<String, Object>> handleMissingParam(
            MissingServletRequestParameterException ex) {
        return invalidInput(ex.getParameterName() + " query parameter required",
            Map.of(ex.getParameterName(), "is required"));
    }

    /** Path-param type mismatch (e.g., {id} not parseable as UUID). */
    @ExceptionHandler(MethodArgumentTypeMismatchException.class)
    public ResponseEntity<Map<String, Object>> handleTypeMismatch(
            MethodArgumentTypeMismatchException ex) {
        return invalidInput(
            "Invalid value for parameter '" + ex.getName() + "'",
            Map.of(ex.getName(), "must be a valid " +
                (ex.getRequiredType() != null ? ex.getRequiredType().getSimpleName() : "value")));
    }

    private ResponseEntity<Map<String, Object>> invalidInput(
            String message, Map<String, String> fieldErrors) {
        String requestId = MDC.get("requestId");
        log.info("friendsValidationFailed fields={}", fieldErrors.keySet());
        Map<String, Object> body = new LinkedHashMap<>();
        body.put("code", "INVALID_INPUT");
        body.put("message", message);
        body.put("requestId", requestId);
        body.put("timestamp", Instant.now().toString());
        body.put("fieldErrors", fieldErrors);
        return ResponseEntity.status(HttpStatus.BAD_REQUEST).body(body);
    }
}

            .orElseThrow(() -> new UserNotFoundException("Caller not found"));

    @GetMapping("/api/v1/users/me/friend-requests")
    public ResponseEntity<ProxyResponse<List<FriendRequestDto>>> listFriendRequests(
            @AuthenticationPrincipal AuthenticatedUser principal,
            @RequestParam("direction") String direction) {
        List<FriendRequestDto> requests;
        if ("incoming".equals(direction)) {
            requests = friendsService.listIncomingPendingRequests(principal.userId());
        } else if ("outgoing".equals(direction)) {
            requests = friendsService.listOutgoingPendingRequests(principal.userId());
        } else {
            throw new InvalidInputException(
                "direction must be 'incoming' or 'outgoing'");
        }
        return ResponseEntity.ok(ProxyResponse.of(requests, MDC.get("requestId")));
    }

    @PostMapping("/api/v1/users/me/friend-requests")

            @AuthenticationPrincipal AuthenticatedUser principal,
            @Valid @RequestBody SendFriendRequestRequest body) {
        FriendRequest req = friendsService.sendRequest(
            principal.userId(), body.toUserId(), body.message());
        FriendRequestDto dto = new FriendRequestDto(
            req.getId(),
            req.getFromUserId(),
            req.getToUserId(),
            null, // otherPartyDisplayName: not denormalized at create-time; frontend re-fetches if needed
            null,
            req.getMessage(),
            req.getStatus().value(),
            req.getCreatedAt(),
            req.getRespondedAt()
        );
        URI location = URI.create("/api/v1/friend-requests/" + req.getId());
        return ResponseEntity.created(location)
            .body(ProxyResponse.of(dto, MDC.get("requestId")));
    }

    // 4. PATCH /api/v1/friend-requests/{id}
    @PatchMapping("/api/v1/friend-requests/{id}")

            @AuthenticationPrincipal AuthenticatedUser principal,
            @PathVariable("id") UUID id,
            @Valid @RequestBody RespondToFriendRequestRequest body) {
        String newStatus = switch (body.action()) {
            case "accept" -> { friendsService.acceptRequest(id, principal.userId()); yield "accepted"; }
            case "decline" -> { friendsService.declineRequest(id, principal.userId()); yield "declined"; }
            case "cancel" -> { friendsService.cancelRequest(id, principal.userId()); yield "cancelled"; }
            default -> throw new InvalidInputException(
                "action must be 'accept', 'decline', or 'cancel'");
            // Defensive default — @Pattern on the DTO catches it first, but a
            // belt-and-suspenders fallback prevents an unreachable IllegalStateException.
        };
        return ResponseEntity.ok(ProxyResponse.of(
            Map.of("status", newStatus), MDC.get("requestId")));
    }

     *

     * FriendsController) for URL-namespace cohesion: {@code /api/v1/users/me/...}
     * and {@code /api/v1/users/search} both belong to the user namespace. The

     * caller-self exclusion.
