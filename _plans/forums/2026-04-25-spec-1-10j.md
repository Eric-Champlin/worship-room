- **Spring Boot version:** 3.5.11. Liveness/readiness probes built-in since 2.3.

### Public-path matching (verified by recon)
- `JwtAuthenticationFilter` uses **Spring's `AntPathMatcher`** to match `request.getRequestURI()` against `PublicPaths.PATTERNS`. Source: `backend/src/main/java/com/worshiproom/auth/JwtAuthenticationFilter.java` lines 60, 83-89.
- Spring's `AntPathMatcher.match("/actuator/health", "/actuator/health/liveness")` returns **`false`**. Ant-style matching requires an explicit `/**` wildcard to cover sub-paths.
- **Therefore the spec's "if needed" question is decided:** `PublicPaths.PATTERNS` MUST be extended. Two options:
  - **Option A (chosen):** Add the two specific paths `/actuator/health/liveness` and `/actuator/health/readiness`. Tightest scope, prevents accidental future leakage of a new actuator/health/* sub-path becoming unauthenticated by default.
  - **Option B (rejected):** Replace `/actuator/health` with `/actuator/health/**`. Would auto-include any future probe sub-path Spring adds (e.g., custom-named probe groups). Rejected because Spring exposes more under `/actuator/health/**` than just probes (e.g., per-component endpoints like `/actuator/health/db` when `show-details=always`); Option B would expose those without authentication too. Stays narrow and intentional.

### Filter chain order (verified from `SecurityHeadersConfig` JavaDoc and `JwtAuthenticationFilter` JavaDoc)
1. `RequestIdFilter` (`HIGHEST_PRECEDENCE`) — threads `X-Request-Id` into MDC.
2. `CorsFilter` (`HIGHEST_PRECEDENCE + 5`) — Spring framework default.
3. `SecurityHeadersFilter` (`HIGHEST_PRECEDENCE + 6`) — sets the six security headers (1.10g).
4. `RateLimitFilter` (`HIGHEST_PRECEDENCE + 10`) — scoped to `/api/v1/proxy/**` only via `shouldNotFilter`.
5. `LoginRateLimitFilter` (`HIGHEST_PRECEDENCE + 20`) — scoped to `/api/v1/auth/login` only.
6. `DelegatingFilterProxy` → `JwtAuthenticationFilter` (Spring Security chain).

Probe responses traverse this entire chain. `SecurityHeadersFilter` decorates them at +6 BEFORE Spring Security's authorization check runs, so the security headers are guaranteed to land on the probe response. `RateLimitFilter` and `LoginRateLimitFilter` `shouldNotFilter` already exclude `/actuator/**`; no rate limit on probes.

### Test pattern reference (verified)
- **Base class:** `com.worshiproom.support.AbstractIntegrationTest`. Wires the singleton Testcontainers PostgreSQL and pins `spring.liquibase.contexts=test`.
- **Reference test class:** `com.worshiproom.config.SecurityHeadersConfigTest.IntegrationTests` exercises `GET /actuator/health` end-to-end, matches against `SecurityHeadersConfig.HSTS_VALUE` etc., and registers `jwt.secret` + `proxy.trust-forwarded-headers` in `@DynamicPropertySource`. The new `LivenessReadinessProbeTest` follows the same shape.
- **`auth.rate-limit.*` properties** must be supplied via test annotation when `extends AbstractIntegrationTest` + `@AutoConfigureMockMvc` is used. Reference: `SecurityHeadersConfigTest.IntegrationTests` annotation block.
- **Liquibase context override** is handled by the base class — no per-test override needed.

### railway.toml (verified by recon)
- **No `railway.toml` exists at repo root.** Railway is using platform defaults (TCP-port-listening check on 8080), which routes user traffic to a Spring process that has bound the port BEFORE Liquibase finishes. This spec creates the file.

## Edge Cases & Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Should `PublicPaths` use a wildcard `/actuator/health/**` or list the two probe paths explicitly? | **List the two paths explicitly.** | Tightest scope. With `show-details=always`, Spring Boot exposes per-component endpoints under `/actuator/health/{component}` (e.g., `/actuator/health/db` showing DataSource state) — a wildcard would unauthenticate those too. The probes are the only sub-paths intentionally public. |
| Should the test class extend `AbstractIntegrationTest` or use `@WebMvcTest`? | **Extend `AbstractIntegrationTest`.** | The probes' readiness state depends on `ApplicationContext` being fully refreshed (which transitively requires Liquibase having completed and the DataSource being ready). `@WebMvcTest` skips that and would assert "readiness UP" against a mock context, which doesn't prove anything. The 5-test class is small enough that the singleton-container reuse keeps the cost negligible (<2s per spec's estimate). |
| Should the probe-enable config go in `application.properties` (all profiles) or per-profile? | **All profiles, in `application.properties`.** | Liveness and readiness semantics are correct in dev, prod, and tests. Tests benefit from the same probe behavior they'd see in prod. Per-profile activation would create dev/prod drift for no benefit. |
| Should the test assert the probe response body has NO extra fields beyond `status`? | **Assert only `$.status == "UP"`.** | Spring Boot probes default to a minimal body (`{"status":"UP"}`) regardless of `show-details=always`, but framework versions can change which fields appear. Asserting just the contract field (`status`) is robust to framework upgrades. The "no extra fields" wording in the spec is intent-level, not a strict structural assertion. |
| Should `probesEmitSecurityHeaders` test live in `LivenessReadinessProbeTest` or be added to `SecurityHeadersConfigTest`? | **In `LivenessReadinessProbeTest`.** | Test colocation: the cross-spec verification belongs alongside the new endpoints it covers. `SecurityHeadersConfigTest` already has a `headersPresentOnControllerSuccess` test exercising `/actuator/health`; that test stays unchanged. The new probe-specific assertion is the new test class's job. |
| Should `railway.toml` include `restartPolicyType` and `restartPolicyMaxRetries`? | **Yes — keep both.** | Spec body lists them. They're orthogonal to the healthcheck change but are reasonable Railway baselines. If a future spec wants to tune restart behavior, it edits the same file. Keeping the file minimal-but-complete now beats a follow-up spec to add `restartPolicy`. |
| Should the `.claude/rules/08-deployment.md` update describe the probes at length, or just one line about Railway? | **Brief — ~5-10 lines.** | The rule files are reference docs, not runbooks. The spec brief in `_specs/forums/spec-1-10j.md` is the runbook (rationale, semantics, verification commands). The rule file just needs to anchor: "Railway uses readiness probe; liveness is sibling; probes carry security headers; probes are public by design." |
| Should the test simulate a `readiness=DOWN` scenario? | **No.** | Out of scope per spec § 7 ("Marking the app as readiness=DOWN during graceful shutdown — Spring Boot 2.3+ does this automatically; no extra config needed"). Forcing readiness=DOWN in a test would require either reflection into `ApplicationAvailability` or a custom HealthIndicator — both are wider than the spec's intent. Test the happy path; trust Spring Boot's built-in behavior for the unhappy path. |

## Implementation Steps

### Step 1: Enable liveness and readiness probes in `application.properties`

**Objective:** Tell Spring Boot Actuator to expose `/actuator/health/liveness` and `/actuator/health/readiness` on this non-Kubernetes deployment. Without these three lines, Spring Boot detects the absence of K8s and silently leaves the probe endpoints disabled — the URLs would return 404.

**Files to modify:**

- `backend/src/main/resources/application.properties` — add three lines under the existing `# ─── Actuator ──────────────────` block.

**Details:**

Locate the existing actuator block (lines 10–13 of `application.properties`):

```properties
# ─── Actuator ──────────────────────────────────────────────────────────────
management.endpoints.web.exposure.include=health,info
management.endpoint.health.show-details=always
management.health.defaults.enabled=true
```

Append three lines so the block reads:

```properties
# ─── Actuator ──────────────────────────────────────────────────────────────
management.endpoints.web.exposure.include=health,info
management.endpoint.health.show-details=always
management.health.defaults.enabled=true
# Liveness/readiness probes (Spec 1.10j). Auto-configured on Kubernetes-detected
# platforms; explicit on non-K8s platforms (Railway). The three properties
# together expose /actuator/health/liveness and /actuator/health/readiness with
# the right indicator sets — liveness=ApplicationAvailability LIVE state only;
# readiness=ApplicationAvailability READY state, which flips to UP after
# ApplicationContext refresh (after Liquibase completes).
management.endpoint.health.probes.enabled=true
management.health.livenessstate.enabled=true
management.health.readinessstate.enabled=true
```

Do NOT modify `application-prod.properties` or `application-dev.properties`. The probes apply uniformly across profiles — there is no scenario where dev/test wants probes disabled.

**Guardrails (DO NOT):**

- Do NOT change `management.endpoints.web.exposure.include` to add `metrics`, `prometheus`, etc. Those belong to Spec 1.10d (Production Monitoring Foundation).
- Do NOT add `management.endpoint.health.group.*` config to define custom probe groups. The default groups (`liveness`, `readiness`) have correct semantics; custom groups are out of scope.
- Do NOT touch the `show-details=always` line. It controls the legacy `/actuator/health` aggregation endpoint, which stays as-is.
- Do NOT add custom `HealthIndicator` beans. Out of scope per spec § 7.

**Verification:**

  - `curl -i http://localhost:8080/actuator/health/liveness` (no Authorization header) → HTTP 200 (NOT 401).

**Test specifications:** None added in this step. Test class lands in Step 3.

### Step 3: Author `LivenessReadinessProbeTest` with five test methods

**Objective:** Lock in probe contract — endpoints exist, return correct shape, are publicly accessible, and carry the 1.10g security headers. This test is the spec's primary verification.

**Files to create:**

- `backend/src/test/java/com/worshiproom/health/LivenessReadinessProbeTest.java` (new file; new package `com.worshiproom.health` follows existing `auth/`, `config/`, `controller/` layout convention).

**Details:**

Class skeleton — extends `AbstractIntegrationTest`, uses `@AutoConfigureMockMvc`, registers `jwt.secret` and `auth.rate-limit.*` per the `SecurityHeadersConfigTest.IntegrationTests` precedent so the Spring context boots cleanly:

```java
package com.worshiproom.health;

import com.worshiproom.config.SecurityHeadersConfig;
import com.worshiproom.support.AbstractIntegrationTest;
import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.autoconfigure.web.servlet.AutoConfigureMockMvc;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.test.context.DynamicPropertyRegistry;
import org.springframework.test.context.DynamicPropertySource;
import org.springframework.test.web.servlet.MockMvc;

import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.get;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.header;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.jsonPath;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.status;

/**
 * Verifies the Spring Boot Actuator liveness and readiness probes (Spec 1.10j).
 *
 * <p>Purpose: prove that
 * <ol>
 *   <li>{@code /actuator/health/liveness} and {@code /actuator/health/readiness}
 *       exist (probe enablement in {@code application.properties} took effect),</li>
 *   <li>both endpoints return {@code 200 {"status":"UP"}} once the
 *       ApplicationContext is fully refreshed (which is the case in any
 *       {@code @SpringBootTest} run),</li>
 *   <li>both endpoints are publicly accessible without an Authorization header
 *       ({@link com.worshiproom.auth.PublicPaths} extension took effect),</li>
 *   <li>both endpoints carry the six security headers from Spec 1.10g
 *       (cross-spec integration check — proves
 *       {@link SecurityHeadersConfig.SecurityHeadersFilter} decorates probe
 *       responses too).</li>
 * </ol>
 *
 * Extends {@link AbstractIntegrationTest} (full Spring context + singleton
 * Testcontainers PostgreSQL) because readiness state depends on
 * {@code ApplicationContext} refresh completing — which requires Liquibase
 * having run against a real database. {@code @WebMvcTest} would short-circuit
 * that and prove nothing.
 */
@AutoConfigureMockMvc
@SpringBootTest(properties = {
    "auth.rate-limit.per-email.capacity=5",
    "auth.rate-limit.per-email.window-minutes=15",
    "auth.rate-limit.per-ip.capacity=20",
    "auth.rate-limit.per-ip.window-minutes=15"
})
class LivenessReadinessProbeTest extends AbstractIntegrationTest {

    private static final String HSTS = "Strict-Transport-Security";
    private static final String XCTO = "X-Content-Type-Options";
    private static final String XFO = "X-Frame-Options";
    private static final String RP = "Referrer-Policy";
    private static final String CSP = "Content-Security-Policy";
    private static final String PP = "Permissions-Policy";

    @DynamicPropertySource
    static void probeProperties(DynamicPropertyRegistry registry) {
        registry.add("jwt.secret",
            () -> "test-jwt-secret-at-least-32-bytes-long-xxxxxxxxxxxxxx");
    }

    @Autowired private MockMvc mvc;

    @Test
    @DisplayName("livenessProbe_returnsUpWithStatusOnly — GET /actuator/health/liveness → 200, $.status=UP")
    void livenessProbe_returnsUpWithStatusOnly() throws Exception {
        mvc.perform(get("/actuator/health/liveness"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.status").value("UP"));
    }

    @Test
    @DisplayName("readinessProbe_returnsUpWhenAppFullyBooted — GET /actuator/health/readiness → 200, $.status=UP")
    void readinessProbe_returnsUpWhenAppFullyBooted() throws Exception {
        mvc.perform(get("/actuator/health/readiness"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.status").value("UP"));
    }

    @Test
    @DisplayName("livenessProbe_isPubliclyAccessible_noJwtRequired — GET with no Authorization → 200, NOT 401")
    void livenessProbe_isPubliclyAccessible_noJwtRequired() throws Exception {
        // Asserts PublicPaths.PATTERNS extension took effect — without
        // /actuator/health/liveness in the list, JwtAuthenticationFilter would
        // not skip the path and the missing Authorization header would route
        // through SecurityConfig's authenticated() rule to RestAuthenticationEntryPoint
        // → 401.
        mvc.perform(get("/actuator/health/liveness"))
            .andExpect(status().isOk());
    }

    @Test
    @DisplayName("readinessProbe_isPubliclyAccessible_noJwtRequired — GET with no Authorization → 200, NOT 401")
    void readinessProbe_isPubliclyAccessible_noJwtRequired() throws Exception {
        mvc.perform(get("/actuator/health/readiness"))
            .andExpect(status().isOk());
    }

    @Test
    @DisplayName("probesEmitSecurityHeaders — both probes carry all six 1.10g security headers")
    void probesEmitSecurityHeaders() throws Exception {
        // Cross-spec integration check: 1.10g's SecurityHeadersFilter at
        // HIGHEST_PRECEDENCE+6 must decorate probe responses. If a future
        // refactor moves SecurityHeadersFilter behind the actuator chain, or
        // if Spring Boot's actuator response writer bypasses the filter chain
        // for some sub-paths, this test fails loudly.
        mvc.perform(get("/actuator/health/liveness"))
            .andExpect(status().isOk())
            .andExpect(header().string(HSTS, SecurityHeadersConfig.HSTS_VALUE))
            .andExpect(header().string(XCTO, SecurityHeadersConfig.X_CONTENT_TYPE_OPTIONS_VALUE))
            .andExpect(header().string(XFO, SecurityHeadersConfig.X_FRAME_OPTIONS_VALUE))
            .andExpect(header().string(RP, SecurityHeadersConfig.REFERRER_POLICY_VALUE))
            .andExpect(header().string(CSP, SecurityHeadersConfig.CSP_VALUE))
            .andExpect(header().string(PP, SecurityHeadersConfig.PERMISSIONS_POLICY_VALUE));

        mvc.perform(get("/actuator/health/readiness"))
            .andExpect(status().isOk())
            .andExpect(header().string(HSTS, SecurityHeadersConfig.HSTS_VALUE))
            .andExpect(header().string(XCTO, SecurityHeadersConfig.X_CONTENT_TYPE_OPTIONS_VALUE))
            .andExpect(header().string(XFO, SecurityHeadersConfig.X_FRAME_OPTIONS_VALUE))
            .andExpect(header().string(RP, SecurityHeadersConfig.REFERRER_POLICY_VALUE))
            .andExpect(header().string(CSP, SecurityHeadersConfig.CSP_VALUE))
            .andExpect(header().string(PP, SecurityHeadersConfig.PERMISSIONS_POLICY_VALUE));
    }
}
```

**Notes on test design:**

- **Why `extends AbstractIntegrationTest` rather than `@WebMvcTest`:** Readiness state requires the full `ApplicationContext` to have refreshed. Mock-only contexts skip the ApplicationAvailability state machine wiring and would short-circuit the assertion.
- **Why `auth.rate-limit.*` properties on the class annotation:** `LoginRateLimitFilter` requires these to construct its bucket builder; without them, context refresh fails. Pattern lifted verbatim from `SecurityHeadersConfigTest.IntegrationTests`.
- **Why no body cleanup (`userRepository.deleteAll()`):** This test never writes to the database. The `AbstractIntegrationTest` JavaDoc note about per-class cleanup applies only to tests that mutate state.
