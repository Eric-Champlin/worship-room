## Architecture Context

### FrostedCard (verified on disk at `frontend/src/components/homepage/FrostedCard.tsx`)

- **Variants & base classes:**
  - `accent`: `bg-violet-500/[0.08] backdrop-blur-md md:backdrop-blur-[12px] border border-violet-400/70 rounded-3xl p-6 shadow-frosted-accent` + `before:` gradient stripe + hover variants when `onClick` is provided.
  - `default`: `bg-white/[0.07] backdrop-blur-sm md:backdrop-blur-md border border-white/[0.12] rounded-3xl p-6 shadow-frosted-base` + hover variants when `onClick` is provided.
  - `subdued`: `bg-white/[0.05] backdrop-blur-sm md:backdrop-blur-md border border-white/[0.10] rounded-3xl p-5`.
- **Props supported (FrostedCardProps):** `children`, `onClick?`, `className?`, `as?: 'div'|'button'|'article'|'section'` (defaults to `'div'`), `tabIndex?`, `role?`, `onKeyDown?`, `variant?` (defaults to `'default'`), `eyebrow?`, `eyebrowColor?: 'violet'|'white'`, `type?` (only used when `as='button'`, defaults to `'button'` inside the buttonProps spread — irrelevant when `as!=='button'`), `aria-label?`, `aria-labelledby?`, `style?`.
- **Does NOT forwardRef.** This is a planning-critical constraint — see Decision 1 below.
- **Eyebrow rendering:** When `eyebrow` is set, FrostedCard prepends a violet leading dot (`bg-violet-400`) + uppercase tracked label (`text-violet-300` on `variant='accent'`, `text-white/50` otherwise) inside the card, above children. No styling configuration beyond `eyebrowColor`.

### `cn()` utility (verified at `frontend/src/lib/utils.ts:4`)

`clsx` + `tailwind-merge` wrapper. Used for className composition. Do not reimplement.

The `<article>` outer wrapper carries ONLY:
- `ref={articleRef}` (for the pulse animation triggered via `usePrayerCardPulse()` context)

The brief's "preserve tabIndex / role / onKeyDown on PrayerCard" claim does not match on-disk reality; the only props to preserve are `aria-label` (via FrostedCard prop) and the `ref` attachment (for pulse animation — see Decision 5).

### Reconciliation-report alignment (key reframe of the brief's "visual parity" requirement)

Per the report (line 129), Phase 5.1's job is explicitly to lift Prayer Wall chrome from drift (`rounded-xl`, `bg-white/[0.06]`, `border-white/10`) to canonical (`rounded-3xl`, `bg-white/[0.07]`, `border-white/[0.12]`). **The visual output WILL change in measurable, intentional ways** for surfaces whose current chrome doesn't match FrostedCard's variants — the migration is bringing them TO canonical, not preserving their current state. The brief's "visual parity within 1-2 pixel tolerance" framing applies to OPACITIES OF THE PER-TYPE ACCENT LAYER (W8 — do not normalize the per-type opacity values), not to the FrostedCard base chrome. See Decision 2.

## Implementation Steps

### Step 1: Create `post-type-chrome.ts` constants file + tests

**Objective:** Establish the per-type accent class lookup as the single source of truth before any component migration consumes it.

**Files to create:**
- `frontend/src/constants/post-type-chrome.ts` — exports `PER_TYPE_CHROME: Record<PostType, string>` (typed map) and `getPerTypeChromeClass(postType: PostType): string` (helper).
- `frontend/src/constants/__tests__/post-type-chrome.test.ts` — 3 unit tests.

**Details:**

```typescript
// frontend/src/constants/post-type-chrome.ts
import type { PostType } from '@/constants/post-types'

/**
 * Per-type accent layer for Prayer Wall cards. Layered on top of
 * `<FrostedCard variant="default">` via the `className` prop. Provides ONLY
 * the border tint and background tint; the frosted glass base chrome
 * (rounded-3xl, backdrop-blur, default surface opacity) is provided by
 * FrostedCard itself.
 *
 * Lifted verbatim from PrayerCard.tsx's per-type switch (Spec 5.1, 2026-05-11).
 * Opacities are NOT to be normalized — per Spec 5.1 W8, the migration preserves
 * the per-type accent opacity values exactly as they were in the inline switch.
 */
export const PER_TYPE_CHROME: Record<PostType, string> = {
  prayer_request: '',
  testimony: 'border-amber-200/10 bg-amber-500/[0.04]',
  question: 'border-cyan-200/10 bg-cyan-500/[0.04]',
  discussion: 'border-violet-200/10 bg-violet-500/[0.04]',
  encouragement: 'border-rose-200/10 bg-rose-500/[0.04]',
}

export function getPerTypeChromeClass(postType: PostType): string {
  return PER_TYPE_CHROME[postType]
}
```

```typescript
// frontend/src/constants/__tests__/post-type-chrome.test.ts
import { describe, expect, it } from 'vitest'
import { POST_TYPES } from '@/constants/post-types'
import { PER_TYPE_CHROME, getPerTypeChromeClass } from '../post-type-chrome'

describe('post-type-chrome', () => {
  it('has an entry for every PostType in POST_TYPES', () => {
    for (const entry of POST_TYPES) {

    expect(getPerTypeChromeClass('prayer_request')).toBe('')
  })
})
```
