## Architecture Context

### Backend (post-recon)

**Visibility predicate is the canonical centralization point per master plan line 6521.** The implementation lives at `backend/src/main/java/com/worshiproom/post/PostSpecifications.java:57–97` and matches the canonical SQL at master plan lines 6504–6519 exactly:

- Anonymous viewer disjunction (lines 69–72): friends/private/own branches all evaluate to `cb.disjunction()` so only the PUBLIC clause can match.
- FRIENDS branch (lines 73–84): EXISTS subquery over `friend_relationships` with the load-bearing direction `fr.user_id = post.user_id AND fr.friend_user_id = :viewerId AND fr.status = ACTIVE`. **Reversing the direction leaks private content** — `PostSpecificationsTest.authenticated_doesNotSeeFriendsPostsWhenRelationshipReversed` (lines 172–181) is the regression guard.
- PRIVATE branch (lines 86–89): gated by `post.userId = viewerId`.
- ownPost branch (line 91): always matches when `post.userId = viewerId`, so the author sees every tier of their own posts.

The Java `PostVisibility` enum (`backend/src/main/java/com/worshiproom/post/PostVisibility.java`) defines `PUBLIC` / `FRIENDS` / `PRIVATE` with wire values `public` / `friends` / `private`, plus a `PostVisibilityConverter` for JPA. `Post.visibility` is a non-null column with default `'public'` and CHECK constraint enforcement.

### Step 1: Backend enforcement audit — integration tests demonstrating predicate composition at the endpoint layer

**Objective:** Lock in the R4 / MPD-6 "every endpoint uses the predicate" AC with end-to-end integration tests that exercise the friends/private visibility flow through the principal read endpoints. The predicate-level coverage already exists in `PostSpecificationsTest.java`; this step adds endpoint-level coverage so a future regression (e.g., a service method dropping `.and(visibleTo(...))`) fails a test rather than shipping silently.

**Files to create:**
- `backend/src/test/java/com/worshiproom/post/PostVisibilityEnforcementIntegrationTest.java` (NEW)

**Files to modify:** none

**Details:**

Create a new integration test class extending `AbstractIntegrationTest`. Tests should:

1. Seed three users via `userRepository.saveAndFlush(...)`: `author`, `friend`, `stranger`.
2. Seed a friend relationship: `friendRepository.saveAndFlush(new FriendRelationship(author.getId(), friend.getId(), FriendRelationshipStatus.ACTIVE))`.
3. Author creates THREE posts via `PostService.createPost(...)` — one with `visibility='public'`, one with `visibility='friends'`, one with `visibility='private'`.
4. For each principal read endpoint listed below, exercise the read as `friend` (should see PUBLIC + FRIENDS, not PRIVATE) and as `stranger` (should see PUBLIC only) and as `author` (should see all three) and as anonymous (should see PUBLIC only).

Endpoints to exercise (one nested test class per endpoint):

- `PostService.listFeed(viewerId, ...)` — main feed
- `PostService.getById(postId, viewerId)` — single post detail
- `PostService.listAuthorPosts(username, viewerId, ...)` — author profile feed
- `FriendPrayersService.listFriendPrayersToday(viewerId)` — Daily Hub friend prayers (friends-tier specific — friend should see, stranger should not even when post is `visibility='public'` because `byActiveFriendsOf` gates the surface; author should not see their own post here because `byActiveFriendsOf(author)` requires author to be in their own friends list)

Use `@BeforeEach` to clean `friend_relationships`, `posts`, and `users` (per the canonical pattern at `FriendPrayersServiceTest.java:46`). Use `jdbc.update("DELETE FROM ...")` for cleanup.

Each test asserts the visible post set by `extracting(PostDto::id).contains(...)` and `.doesNotContain(...)`.

**Guardrails (DO NOT):**

- DO NOT call `PostSpecifications.visibleTo()` directly in these tests — the point is to verify endpoint composition, not predicate behavior. `PostSpecificationsTest` already covers predicate behavior at the unit level.
- DO NOT mock the database or use H2. Testcontainers PostgreSQL only.
- DO NOT skip the friend-relationship cleanup in `@BeforeEach` — Spring Data test fixtures persist across tests within a class.
- DO NOT use the singleton `AbstractIntegrationTest` container per-test (already JVM-singleton via `support/TestContainers.java`).

**Verification:**

- [ ] Code compiles: `./mvnw -pl . compile -DskipTests` (or `./mvnw compile`)

**`frontend/src/components/prayer-wall/VisibilitySelector.tsx`** — controlled component:

```typescript
import { useRovingTabindex } from '@/hooks/useRovingTabindex'
import { cn } from '@/lib/utils'
import { VISIBILITY_OPTIONS, type PostVisibility } from '@/constants/visibility-options'

interface VisibilitySelectorProps {
  value: PostVisibility
  onChange: (next: PostVisibility) => void
  disabled?: boolean
}

export function VisibilitySelector({ value, onChange, disabled }: VisibilitySelectorProps) {
  const initialIndex = VISIBILITY_OPTIONS.findIndex(opt => opt.value === value)
  const { getItemProps } = useRovingTabindex({
    itemCount: VISIBILITY_OPTIONS.length,
    onSelect: (index) => onChange(VISIBILITY_OPTIONS[index].value),
    orientation: 'horizontal',
    initialIndex: initialIndex >= 0 ? initialIndex : 0,
  })

  return (
    <fieldset className="mt-3" disabled={disabled}>
      <legend className="mb-2 text-sm font-medium text-white/70">Visibility</legend>
      <div
        role="radiogroup"
        aria-label="Post visibility"
        className="flex flex-nowrap gap-2"
      >
        {VISIBILITY_OPTIONS.map((opt, index) => {
          const itemProps = getItemProps(index)
          const Icon = opt.icon
          const isSelected = value === opt.value
          return (
            <button
              key={opt.value}
              type="button"
              role="radio"
              aria-checked={isSelected}
              title={opt.tooltip}
              onClick={() => onChange(opt.value)}
              className={cn(
                'inline-flex min-h-[44px] shrink-0 items-center gap-1.5 whitespace-nowrap rounded-full border px-4 py-2 text-sm font-medium transition-colors',
                isSelected
                  ? 'border-violet-400/45 bg-violet-500/[0.13] text-white'
                  : 'border-white/[0.12] bg-white/[0.07] text-white/70 hover:bg-white/[0.12]',
              )}
              tabIndex={itemProps.tabIndex}
              onKeyDown={itemProps.onKeyDown}
              ref={itemProps.ref}
            >
              <Icon className="h-3.5 w-3.5" aria-hidden="true" />
              {opt.label}
            </button>
          )
        })}
      </div>
    </fieldset>
  )
}
```

  // relationship userA <-> userC via API or seed fixture. userB is a stranger.
