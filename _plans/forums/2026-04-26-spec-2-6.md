### Auth pattern (verified)

- `JwtAuthenticationFilter` validates the Bearer token, sets `AuthenticatedUser` (record: `UUID userId`, `boolean isAdmin`) as the Spring Security principal.
- `PublicPaths.PATTERNS` lists routes that bypass auth. `/api/v1/activity` is NOT added there → endpoint requires auth.
- Controllers extract via `@AuthenticationPrincipal AuthenticatedUser principal`. Canonical example: `UserController.getMe(@AuthenticationPrincipal AuthenticatedUser principal)`.
- Missing-user case: `UserService` does `userRepository.findById(userId).orElseThrow(UserException::userNotFound)` → 401 USER_NOT_FOUND. Spec 2.6 follows the same pattern: `userRepository.findById(principal.userId()).orElseThrow(UserException::userNotFound)`.

### Activity-type → count-type mapping (verbatim from frontend)

7 of 12 `ActivityType` values map to a `CountType`. Other 5 (`MOOD`, `REFLECTION`, `CHALLENGE`, `LOCAL_VISIT`, `DEVOTIONAL`) trigger no count increment via this endpoint. Wire values are identical between the two enums for the 7 mapped pairs (`pray`, `journal`, `meditate`, `listen`, `prayerWall`, `readingPlan`, `gratitude`).

### Standard response envelope (verified from `UserController` and proxy controllers)

```java
ProxyResponse.of(payload, MDC.get("requestId"))
// → { "data": <payload>, "meta": { "requestId": "..." } }
```

`ProxyResponse<T>` is the project's canonical success envelope. Defined in `com.worshiproom.proxy.common`. Reused — DO NOT introduce a new envelope.

### Standard error envelope and existing handlers

- `ProxyError` is the canonical error body (fields: `code`, `message`, `requestId`, `timestamp`). In `com.worshiproom.proxy.common`.
- Validation handlers are **package-scoped** per project convention. `UserValidationExceptionHandler` lives in `com.worshiproom.user`; `AuthValidationExceptionHandler` in `com.worshiproom.auth`. Each emits 400 VALIDATION_FAILED with a `fieldErrors` map.
- This spec adds `ActivityValidationExceptionHandler` in `com.worshiproom.activity` matching the same shape (see Architectural Decision #11). The spec text saying "Don't add new exception handlers" is incorrect for the codebase's package-scoping convention; per Rule 16 (respect existing patterns), the package-scoped handler is mandatory.

### Jackson naming convention (verified)

- No global snake_case strategy in `application.properties`. Existing DTOs use camelCase field names (`firstName`, `displayName`, `verseText`, `requestId`).
- Per Architectural Decision #12 below, the entire activity API uses **camelCase wire format** to maintain consistency with the rest of the codebase. This is a deliberate divergence from Decision 5's snake_case shape — codebase convention beats master-plan example.

### JSONB pattern (no precedent)

No existing entity uses JSONB columns. The `activity_log.metadata` column is the first. Hibernate 6 (Spring Boot 3.x) supports the canonical pattern natively without `hibernate-types` dependency:

```java
@JdbcTypeCode(SqlTypes.JSON)
@Column(name = "metadata", columnDefinition = "jsonb")
private Map<String, Object> metadata;
```

### Frontend type generation pipeline (does NOT exist yet)

- `frontend/package.json` has NO `types:generate` or `openapi-typescript` script.
- `frontend/src/types/api/` directory does NOT exist.
- `03-backend-standards.md` § "OpenAPI Spec and Type Generation" confirms: *"Frontend types: Generated to `frontend/src/types/api/generated.ts` via openapi-typescript (pipeline wired up by Forums Wave Phase 1; **hand-typed in the interim**)."*

- **Write**: `activity_log` (always), `activity_counts` (always when mapping exists), `faith_points` / `streak_state` / `user_badges` (only on first-time-today).

## Edge Cases & Decisions

| Decision | Choice | Rationale |
|---|---|---|
| **#1 Controller + orchestrator service split** | `ActivityController` (HTTP boundary) + `ActivityService` (transactional orchestration) | Per master plan files-to-create list; keeps controller thin and orchestration testable. |
| **#2 Single transaction** | `@Transactional` on `ActivityService.recordActivity` (default isolation, default propagation) | All four service calls + 5 table writes inside one transaction; any exception → full rollback. |
| **#3 Activity-type → count-type mapping** | Private `static final EnumMap<ActivityType, CountType>` inside `ActivityService` | Lives at the call site per Spec 2.5 brief out-of-scope; 7 entries only. |
| **#4 Points-delta computation** | Two `FaithPointsService.calculate` calls — one with `oldSet`, one with `newSet`; delta = `newResult.pointsEarned() - oldResult.pointsEarned()` | Multiplier tier crosses retroactively bump previous activities' contribution; the delta captures this. NOT "this single activity earned X points." |
| **#5 faith_points / streak_state UPSERT** | JPA `repository.save()` on the entity loaded via `findById(userId).orElse(...fresh constructor...)` | Race conditions on the same user's row within the same second are exceptionally rare; transaction isolation handles them. Native UPSERT is overkill here. |
| **#6 user_badges persistence** | Non-repeatable badges → `repository.save(new UserBadge(userId, badgeId))`; repeatable badges (only `full_worship_day` today) → new `BadgeRepository.incrementDisplayCount(userId, badgeId)` native UPSERT | `BadgeService` already filters non-repeatable badges out via `alreadyEarned`; non-repeatable INSERTs cannot collide. Repeatable needs UPSERT for the `display_count` increment. |
| **#7 activity_log JSONB metadata** | `Map<String, Object> metadata` field annotated with `@JdbcTypeCode(SqlTypes.JSON)` and `@Column(name = "metadata", columnDefinition = "jsonb")` | Hibernate 6 native; no extra dependency. First JSONB usage in the codebase — establishes the canonical pattern. |
| **#8 Bean Validation on request** | `@NotNull ActivityType activityType`, `@NotBlank @Size(max = 50) String sourceFeature`, `Map<String, Object> metadata` (no validation; optional) | Matches `UpdateUserRequest` and `ExplainRequest` patterns. |
| **#9 OpenAPI tag + schemas** | New tag `Activity`; new path `/api/v1/activity`; 6 new schemas; reuse `BadRequest`, `Unauthorized`, `RateLimited`, `InternalError` responses; reuse `bearerAuth` security scheme. Path uses `security: [{ bearerAuth: [] }]` matching `/api/v1/users/me`. | Mirrors existing OpenAPI structure. |
| **#10 No endpoint-specific rate limit** | Document the gap; rely on existing `RateLimitFilter` (which does NOT cover this path) until JWT-aware per-user rate limiting lands | Master plan acceptance criterion explicitly defers it. |
| **#11 Add `ActivityValidationExceptionHandler`** | Create new package-scoped advice (`@RestControllerAdvice(basePackages = "com.worshiproom.activity")`) handling `MethodArgumentNotValidException` (→ 400 VALIDATION_FAILED) AND `HttpMessageNotReadableException` (→ 400 INVALID_INPUT) | **DIVERGES from spec text** which says "Don't add new exception handlers." But validation advices are package-scoped per project convention (Rule 16); user/auth handlers don't catch activity exceptions. The "INVALID_INPUT for unknown enum value" requirement also requires intercepting Jackson's `HttpMessageNotReadableException` to produce the standard `ProxyError` shape (Spring's default response would not have `code`/`message`/`requestId`/`timestamp`). |
| **#12 camelCase wire format + hand-authored frontend types** | Use camelCase JSON field names throughout (`pointsEarned`, `totalPoints`, `currentLevel`, `levelUp`, `multiplierTier`, `newBadges`, `newToday`, `graceUsed`, `graceRemaining`, etc.); hand-author `frontend/src/types/api/activity.ts` since no codegen pipeline exists | **DIVERGES from Decision 5's snake_case shape** but matches the codebase convention (`firstName`, `displayName`, `verseText`, `requestId`, `joinedAt` are all camelCase in existing OpenAPI schemas and Java DTOs). Spec 2.6 § Recon H explicitly says "match the existing convention"; the existing convention is camelCase. Frontend codegen pipeline doesn't exist yet (`03-backend-standards.md` § "hand-typed in the interim") — hand-authoring is the only option. Future codegen spec wires `openapi-typescript` and the hand file becomes the migration target. |
| **#13 first-time-today gating window** | Compute window in user's timezone: `today.atStartOfDay(tz).toInstant()` to `today.plusDays(1).atStartOfDay(tz).toInstant()` (exclusive upper bound). Query `activity_log` `countByUserIdAndActivityTypeAndOccurredAtGreaterThanEqualAndOccurredAtLessThan` BEFORE the new row is saved. | Using `LocalTime.MAX` (the spec's example) gives `23:59:59.999999999` and excludes the very last microsecond; `today.plusDays(1).atStartOfDay()` is cleaner and exact. Saving the new `activity_log` row LAST in the orchestration ensures the count query sees the pre-insert state. |
| **#14 today's activity-type set query** | `findDistinctActivityTypesByUserIdAndDateRange` returning `List<String>` — converted to `Set<ActivityType>` in service | The `oldSet` for delta computation is "distinct activity types in `activity_log` rows for this user, today, in user's tz" — uses the same window as #13. Returns wire strings; service converts via `ActivityType.fromWireValue`. |
| **#15 metadata is opaque** | Backend does NOT inspect `metadata` for crisis content, PII, or any other content (Rule 13 not applicable) | `metadata` is structured event data (e.g., `{ devotionalDay: 5 }`), not user-generated text. Crisis detection lives at post/comment write boundaries (Phase 3 specs). |
| **#16 grace_used / grace_remaining always 0** | Always serialize as `0` in `StreakSnapshot` | Per Spec 2.3 Divergence 1, frontend has no grace-day concept; backend has no grace logic either. Fields exist for API stability; future grace spec populates them. |
| **#17 newToday derivation** | `newToday = streakResult.transition() != StreakTransition.SAME_DAY` (true for FIRST_EVER, INCREMENT, RESET) | Mirrors the spec's "true unless transition == SAME_DAY" rule. |

### Step 2: Author `ActivityLog` JPA entity

**Objective:** Create the JPA entity mapping to the existing `activity_log` table.

**Files to create:**
- `backend/src/main/java/com/worshiproom/activity/ActivityLog.java` — JPA entity. Manual getters/setters, no Lombok, equals/hashCode by primary key (UUID `id`), matching `FaithPoints` and `StreakState` style.

**Details:**

```java
@Entity
@Table(name = "activity_log")
public class ActivityLog {

    @Id
    @Column(name = "id", nullable = false, updatable = false)
    private UUID id;

    @Column(name = "user_id", nullable = false, updatable = false)
    private UUID userId;

    @Column(name = "activity_type", nullable = false, length = 50, updatable = false)
    private String activityType;  // wire-format string (e.g., "pray", "prayerWall")

    @Column(name = "source_feature", nullable = false, length = 50, updatable = false)
    private String sourceFeature;

    @Column(name = "occurred_at", nullable = false, updatable = false)
    private OffsetDateTime occurredAt;

    @Min(0)
    @Column(name = "points_earned", nullable = false)
    private int pointsEarned;

    @JdbcTypeCode(SqlTypes.JSON)
    @Column(name = "metadata", columnDefinition = "jsonb")
    private Map<String, Object> metadata;

    protected ActivityLog() {}

    public ActivityLog(UUID userId, String activityType, String sourceFeature,
                       OffsetDateTime occurredAt, int pointsEarned,
                       Map<String, Object> metadata) {
        this.id = UUID.randomUUID();
        this.userId = userId;
        this.activityType = activityType;
        this.sourceFeature = sourceFeature;
        this.occurredAt = occurredAt;
        this.pointsEarned = pointsEarned;
        this.metadata = metadata;
    }

    // getters: id, userId, activityType, sourceFeature, occurredAt, pointsEarned, metadata
    // setters: pointsEarned ONLY (rest are immutable post-construction)
    // equals/hashCode by id only
    // toString without metadata (avoid logging arbitrary content)
}
```

**Guardrails (DO NOT):**
- DO NOT add a `@PrePersist` hook that generates a UUID — assign `UUID.randomUUID()` in the all-args constructor (matches `FaithPoints` style; `id` is `nullable=false` and `updatable=false`).
- DO NOT add `@CreatedDate` / `@LastModifiedDate` — `occurred_at` is set explicitly by the orchestrator (one consistent value per request), not by JPA auditing.
- DO NOT log the `metadata` field in `toString()` — it's arbitrary client-supplied JSON.
- DO NOT mark `pointsEarned` as `updatable = false` — the orchestrator updates it after first-time-today determination (see Step 6).
- DO NOT change the `activity_type` column to a Java enum mapping — the column stores the wire-format string per Spec 2.1's schema; conversion to/from `ActivityType` happens in the service.

### Step 3: Author `ActivityLogRepository`

**Objective:** Spring Data repository with the two custom queries needed by the orchestrator.

**Files to create:**
- `backend/src/main/java/com/worshiproom/activity/ActivityLogRepository.java` — extends `JpaRepository<ActivityLog, UUID>`. Two custom methods.

**Details:**

```java
@Repository
public interface ActivityLogRepository extends JpaRepository<ActivityLog, UUID> {

    /**
     * Count rows matching (userId, activityType wire string) where
     * occurredAt is in [start, end). Used to detect first-time-today
     * before saving a new activity_log row.
     */
    @Query("""
        SELECT COUNT(a) FROM ActivityLog a
        WHERE a.userId = :userId
          AND a.activityType = :activityType
          AND a.occurredAt >= :start
          AND a.occurredAt < :end
        """)
    long countTodaysOccurrences(@Param("userId") UUID userId,
                                 @Param("activityType") String activityType,
                                 @Param("start") OffsetDateTime start,
                                 @Param("end") OffsetDateTime end);

    /**
     * Distinct activity_type wire strings the user logged today.
     * Used to assemble the oldSet for the points-delta computation.
     */
    @Query("""
        SELECT DISTINCT a.activityType FROM ActivityLog a
        WHERE a.userId = :userId
          AND a.occurredAt >= :start
          AND a.occurredAt < :end
        """)
    List<String> findDistinctActivityTypesToday(@Param("userId") UUID userId,
                                                 @Param("start") OffsetDateTime start,
                                                 @Param("end") OffsetDateTime end);
}
```

**Guardrails (DO NOT):**

### Step 6: Author `ActivityService` orchestrator

**Objective:** The transactional orchestration that composes all four services.

**Files to create:**
- `backend/src/main/java/com/worshiproom/activity/ActivityService.java`

**Details:**

The service has ONE public method, `recordActivity`, which executes the algorithm laid out in Spec 2.6 § Architectural Decision #4. Pseudo-structure:

```java
@Service
public class ActivityService {

    private static final Logger log = LoggerFactory.getLogger(ActivityService.class);

    /** 7 mappings; the other 5 ActivityType values trigger no count increment. */
    private static final Map<ActivityType, CountType> ACTIVITY_TO_COUNT;
    static {
        Map<ActivityType, CountType> m = new EnumMap<>(ActivityType.class);
        m.put(ActivityType.PRAY,         CountType.PRAY);
        m.put(ActivityType.JOURNAL,      CountType.JOURNAL);
        m.put(ActivityType.MEDITATE,     CountType.MEDITATE);
        m.put(ActivityType.LISTEN,       CountType.LISTEN);
        m.put(ActivityType.PRAYER_WALL,  CountType.PRAYER_WALL);
        m.put(ActivityType.READING_PLAN, CountType.READING_PLAN);
        m.put(ActivityType.GRATITUDE,    CountType.GRATITUDE);
        ACTIVITY_TO_COUNT = Collections.unmodifiableMap(m);
    }

    private final UserRepository userRepository;
    private final ActivityLogRepository activityLogRepository;
    private final FaithPointsRepository faithPointsRepository;
    private final StreakRepository streakRepository;
    private final BadgeRepository badgeRepository;
    private final FaithPointsService faithPointsService;
    private final StreakService streakService;
    private final BadgeService badgeService;
    private final ActivityCountsService activityCountsService;

    // constructor injection of all 9 dependencies

    @Transactional
    public ActivityResponseData recordActivity(UUID userId, ActivityRequest request) {
        // 1. Load user, compute today/currentWeekStart in user's tz
        User user = userRepository.findById(userId)
            .orElseThrow(UserException::userNotFound);
        ZoneId tz = ZoneId.of(user.getTimezone());
        LocalDate today = LocalDate.now(tz);
        OffsetDateTime windowStart = today.atStartOfDay(tz).toOffsetDateTime();
        OffsetDateTime windowEnd = today.plusDays(1).atStartOfDay(tz).toOffsetDateTime();
        OffsetDateTime now = OffsetDateTime.now(ZoneOffset.UTC);

        ActivityType type = request.activityType();
        String typeWire = type.wireValue();

        // 2. Determine first-time-today (BEFORE any writes)
        boolean firstTimeToday = activityLogRepository.countTodaysOccurrences(
            userId, typeWire, windowStart, windowEnd) == 0;

        // 3. Always: increment count if mapping exists
        CountType countType = ACTIVITY_TO_COUNT.get(type);
        if (countType != null) {
            activityCountsService.incrementCount(userId, countType);
        }

        // 4. Load existing state (creates fresh entities if missing)
        FaithPoints fp = faithPointsRepository.findById(userId)
            .orElseGet(() -> new FaithPoints(userId));
        StreakState streakEntity = streakRepository.findById(userId)
            .orElseGet(() -> new StreakState(userId));
        StreakStateData streakData = new StreakStateData(
            streakEntity.getCurrentStreak(),
            streakEntity.getLongestStreak(),
            streakEntity.getLastActiveDate());

        // 5. Branch: full path (firstTimeToday) vs record-only path
        int pointsDelta = 0;
        boolean levelUp = false;
        int totalPoints;
        int currentLevel;
        StreakResult streakResult;
        List<BadgeDefinition> newBadgeDefs;
        MultiplierTier multiplierTier;

        if (firstTimeToday) {
            // 5a. Compute oldSet from activity_log (pre-insert state)
            Set<ActivityType> oldSet = readTodaysActivityTypes(
                userId, windowStart, windowEnd);
            Set<ActivityType> newSet = EnumSet.copyOf(oldSet);
            newSet.add(type);

            // 5b. Two FaithPointsService.calculate calls for the delta
            FaithPointsResult oldResult = faithPointsService.calculate(oldSet, 0);
            FaithPointsResult newResult = faithPointsService.calculate(newSet, 0);
            pointsDelta = newResult.pointsEarned() - oldResult.pointsEarned();
            multiplierTier = newResult.multiplierTier();

            // 5c. Update lifetime points + level
            int newLifetime = fp.getTotalPoints() + pointsDelta;
            LevelInfo levelInfo = LevelThresholds.levelForPoints(newLifetime);
            int previousLevel = fp.getCurrentLevel();
            levelUp = levelInfo.level() > previousLevel;
            fp.setTotalPoints(newLifetime);
            fp.setCurrentLevel(levelInfo.level());
            fp.setLastUpdated(now);
            faithPointsRepository.save(fp);

            // 5d. Update streak
            streakResult = streakService.updateStreak(streakData, today);
            StreakStateData newStreakState = streakResult.newState();
            streakEntity.setCurrentStreak(newStreakState.currentStreak());
            streakEntity.setLongestStreak(newStreakState.longestStreak());
            streakEntity.setLastActiveDate(newStreakState.lastActiveDate());
            streakRepository.save(streakEntity);

            // 5e. Check badges
            BadgeCheckContext context = buildBadgeContext(
                userId, newStreakState, levelInfo.level(), previousLevel,
                newSet);
            Set<String> alreadyEarned = badgeRepository.findAllByUserId(userId).stream()
                .map(UserBadge::getBadgeId)
                .collect(Collectors.toSet());
            BadgeResult badgeResult = badgeService.checkBadges(context, alreadyEarned);
            newBadgeDefs = badgeResult.newlyEarnedDefinitions();
            persistNewBadges(userId, newBadgeDefs, alreadyEarned);

            totalPoints = newLifetime;
            currentLevel = levelInfo.level();
        } else {
            // Record-only path: all derived values come from current state
            streakResult = new StreakResult(streakData, StreakTransition.SAME_DAY,
                streakData.currentStreak(), false);
            // multiplierTier reflects today's existing set (NOT including a duplicate
            // of the current activityType — the boolean activity model treats
            // duplicates as no-op for tier computation)
            Set<ActivityType> existingSet = readTodaysActivityTypes(
                userId, windowStart, windowEnd);
            multiplierTier = MultiplierTiers.forActivityCount(existingSet.size());
            totalPoints = fp.getTotalPoints();
            currentLevel = fp.getCurrentLevel();
            newBadgeDefs = List.of();
        }

        // 6. Always: insert activity_log row (even on record-only path)
        ActivityLog logRow = new ActivityLog(
            userId, typeWire, request.sourceFeature(), now,
            firstTimeToday ? pointsDelta : 0,
            request.metadata());
        activityLogRepository.save(logRow);

        // 7. Build response
        return buildResponse(
            firstTimeToday ? pointsDelta : 0,
            totalPoints, currentLevel, levelUp,
            streakResult, newBadgeDefs, multiplierTier);
    }

    // ─── Helpers ──────────────────────────────────────────────────────

    private Set<ActivityType> readTodaysActivityTypes(
            UUID userId, OffsetDateTime start, OffsetDateTime end) {
        List<String> wireValues = activityLogRepository.findDistinctActivityTypesToday(
            userId, start, end);
        Set<ActivityType> result = EnumSet.noneOf(ActivityType.class);
        for (String wire : wireValues) {
            try {
                result.add(ActivityType.fromWireValue(wire));
            } catch (IllegalArgumentException e) {
                // Defensive: rogue wire string in DB. Log + skip (matches
                // ActivityCountsService.getAllCounts pattern).
                log.warn("Skipping unknown activity_type='{}' in activity_log for userId={}",
                    wire, userId);
            }
        }
        return result;
    }

    private BadgeCheckContext buildBadgeContext(
            UUID userId, StreakStateData newStreak, int currentLevel, int previousLevel,
            Set<ActivityType> todayActivities) {
        Map<CountType, Integer> counts = activityCountsService.getAllCounts(userId);
        ActivityCountsSnapshot snapshot = new ActivityCountsSnapshot(
            counts.getOrDefault(CountType.PRAY, 0),
            counts.getOrDefault(CountType.JOURNAL, 0),
            counts.getOrDefault(CountType.MEDITATE, 0),
            counts.getOrDefault(CountType.LISTEN, 0),
            counts.getOrDefault(CountType.PRAYER_WALL, 0),
            counts.getOrDefault(CountType.READING_PLAN, 0),
            counts.getOrDefault(CountType.GRATITUDE, 0),
            counts.getOrDefault(CountType.REFLECTION, 0),
            counts.getOrDefault(CountType.ENCOURAGEMENTS_SENT, 0),
            counts.getOrDefault(CountType.FULL_WORSHIP_DAYS, 0),
            counts.getOrDefault(CountType.CHALLENGES_COMPLETED, 0),
            counts.getOrDefault(CountType.INTERCESSION_COUNT, 0),
            counts.getOrDefault(CountType.BIBLE_CHAPTERS_READ, 0),
            counts.getOrDefault(CountType.PRAYER_WALL_POSTS, 0)
        );
        // Per Spec 2.4 Divergence 1: backend has empty data for friendCount,
        // reading plans, bible progress, meditation history, gratitude entries,
        // local visits, listening history during the dual-write phase.
        // Pass the safe-empty defaults; BadgeCheckContext's compact constructor
        // replaces nulls with safe empties, but explicit is clearer.
        return new BadgeCheckContext(
            newStreak.currentStreak(),
            newStreak.longestStreak(),
            currentLevel,
            previousLevel,
            todayActivities,
            snapshot,
            0,                                  // friendCount — not in backend yet
            false,                              // allActivitiesWereTrueBefore — not tracked yet
            List.of(),                          // readingPlanProgress — empty
            BibleProgressSnapshot.empty(),      // bibleProgress — empty
            List.of(),                          // meditationHistory — empty
            GratitudeEntriesSnapshot.empty(),   // gratitudeEntries — empty
            LocalVisitsSnapshot.zero(),         // localVisits — empty
            List.of()                           // listeningHistory — empty
        );
    }

    private void persistNewBadges(UUID userId, List<BadgeDefinition> defs,
                                   Set<String> alreadyEarned) {
        for (BadgeDefinition def : defs) {
            if (def.repeatable()) {
                badgeRepository.incrementDisplayCount(userId, def.id());
            } else if (!alreadyEarned.contains(def.id())) {
                badgeRepository.save(new UserBadge(userId, def.id()));
            }
            // Else: non-repeatable AND already earned. BadgeService should
            // have filtered these out, but guard defensively.
        }
    }

    private ActivityResponseData buildResponse(
            int pointsDelta, int totalPoints, int currentLevel, boolean levelUp,
            StreakResult streakResult, List<BadgeDefinition> newBadgeDefs,
            MultiplierTier tier) {
        StreakStateData s = streakResult.newState();
        boolean newToday = streakResult.transition() != StreakTransition.SAME_DAY;
        StreakSnapshot streak = new StreakSnapshot(
            s.currentStreak(), s.longestStreak(), newToday, 0, 0);

        OffsetDateTime now = OffsetDateTime.now(ZoneOffset.UTC);
        List<NewBadge> newBadges = newBadgeDefs.stream()
            .map(d -> new NewBadge(d.id(), d.name(), d.celebrationTier(), now))
            .toList();

        MultiplierTierSnapshot multiplierSnap = new MultiplierTierSnapshot(
            tier.label(), tier.multiplier());

        return new ActivityResponseData(
            pointsDelta, totalPoints, currentLevel, levelUp,
            streak, newBadges, multiplierSnap);
    }
}
```

**Decisions inside the orchestrator:**
- **Window calculation** uses `today.atStartOfDay(tz).toOffsetDateTime()` and `today.plusDays(1).atStartOfDay(tz).toOffsetDateTime()` — half-open `[start, end)`, exact, no microsecond truncation issues with `LocalTime.MAX`.
- **Time of insertion (`now`)** is captured ONCE at the start of the transaction and reused for `activity_log.occurred_at`, `faith_points.last_updated`, and `NewBadge.earnedAt`. Single value per request → consistent timeline.
- **`oldSet` for delta** is read fresh from `activity_log` (after the count increment but before the new row is inserted) — matches Spec 2.6 § Architectural Decision #4 algorithm.
- **`fp.getCurrentLevel()` baseline** is captured BEFORE the level update, so `levelUp` compares post-update vs pre-update level (not pre-update vs pre-update).
- **`previousLevel` argument to `BadgeCheckContext`** is the pre-update level from `fp`, NOT the level computed from `oldSet` (the BadgeService's `levelBadges` only checks `currentLevel`, but we faithfully populate `previousLevel` for future use).
- **`activityCountsService.incrementCount` is called BEFORE the badge check** so that `getAllCounts` inside `buildBadgeContext` sees the post-increment value — required for milestone-badge eligibility (e.g., 100th prayer triggers `prayer_100`).
- **Record-only path's `multiplierTier`** is computed from the pre-existing today's set size (which does NOT include the current activity since it was already in the set on the FIRST occurrence today). Same value as the most recent first-time-today call's `newResult.multiplierTier()`. No second `FaithPointsService.calculate` call needed.
- **Record-only path returns `levelUp=false`, `pointsEarned=0`, `newBadges=[]`**. Streak `newToday=false` (transition SAME_DAY).
- **Rogue `activity_type` strings in DB** are skipped with WARN log (matches `ActivityCountsService.getAllCounts` resilience pattern).

**Guardrails (DO NOT):**
- DO NOT skip `activity_log.save(logRow)` on the record-only path — Spec 2.6 § Acceptance criteria mandate "row created on every successful call."
- DO NOT skip `activityCountsService.incrementCount(...)` on the record-only path when a mapping exists — the pray-count is a lifetime counter.
- DO NOT compute `pointsEarned` by passing a single-element set to `FaithPointsService.calculate` — use the delta algorithm (Architectural Decision #4).
- DO NOT add `Sentry.captureException` calls — the global handler covers it.
- DO NOT add caching — reads come from DB on every request.
- DO NOT add a per-method `@Transactional` to helper methods (`buildBadgeContext`, etc.) — they inherit the outer transaction; nested annotations cause noise.
- DO NOT compute `today` / `windowStart` / `windowEnd` in the helper methods — pass them in or compute once at the top of `recordActivity` (per spec guardrail).
- DO NOT increment `activity_log.points_earned` after the row is saved — set it correctly in the constructor (immutability is preferred for audit-style logs).
- DO NOT use `Instant.now()` and convert later — use `OffsetDateTime.now(ZoneOffset.UTC)` directly to match `OffsetDateTime` fields throughout the codebase.
- DO NOT write `BibleProgressSnapshot.empty()` etc. inline — these factories already exist in the DTO classes per Spec 2.4. Verify each factory exists in recon Step 1; if missing, surface as a deviation rather than inventing one.

```java
 *  - MethodArgumentNotValidException (Bean Validation @NotNull / @NotBlank /
 *    @Size failures on @Valid @RequestBody) → 400 VALIDATION_FAILED with a
 *    fieldErrors map matching the existing user/auth shape.
 *
 *  - HttpMessageNotReadableException (Jackson deserialization failure —
 *    most importantly, an unknown ActivityType wire string raises
 *    IllegalArgumentException from ActivityType.fromWireValue, which Jackson
 *    wraps in JsonMappingException) → 400 INVALID_INPUT with the standard
 *    ProxyError shape so the response shape matches every other 400 in the
 *    project.
 */
@RestControllerAdvice(basePackages = "com.worshiproom.activity")
public class ActivityValidationExceptionHandler {

    private static final Logger log =
        LoggerFactory.getLogger(ActivityValidationExceptionHandler.class);

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String, Object>> handleValidation(
            MethodArgumentNotValidException ex) {
        String requestId = MDC.get("requestId");
        Map<String, String> fieldErrors = ex.getBindingResult().getFieldErrors().stream()
            .collect(Collectors.toMap(
                FieldError::getField,
                fe -> fe.getDefaultMessage() == null ? "invalid" : fe.getDefaultMessage(),
                (a, b) -> a,
                LinkedHashMap::new));

        log.info("activityValidationFailed fields={}", fieldErrors.keySet());

        Map<String, Object> body = new LinkedHashMap<>();
        body.put("code", "VALIDATION_FAILED");
        body.put("message", "Request validation failed.");
        body.put("requestId", requestId);
        body.put("timestamp", Instant.now().toString());
        body.put("fieldErrors", fieldErrors);

        return ResponseEntity.status(HttpStatus.BAD_REQUEST).body(body);
    }

    @ExceptionHandler(HttpMessageNotReadableException.class)
    public ResponseEntity<ProxyError> handleNotReadable(
            HttpMessageNotReadableException ex) {
        String requestId = MDC.get("requestId");
        // Generic safe message — do NOT echo the upstream parser detail
        // (could include user-submitted strings).
        log.info("activityRequestNotReadable cause={}", ex.getMostSpecificCause().getClass().getSimpleName());
        return ResponseEntity.status(HttpStatus.BAD_REQUEST)
            .body(ProxyError.of("INVALID_INPUT",
                "Request body is malformed or contains invalid values.",
                requestId));
    }
}
```

**Decisions:**
- **Package-scoped** to `com.worshiproom.activity` — does NOT catch user/auth/proxy exceptions.
- **`MethodArgumentNotValidException` shape** matches `UserValidationExceptionHandler` exactly (LinkedHashMap, fieldErrors, INSTANT timestamp) so frontend error handling stays uniform.
- **`HttpMessageNotReadableException` shape** uses `ProxyError.of(...)` (the canonical 4-field error) for consistency.
- **Generic error message** — does NOT include `ex.getMessage()` (could contain user content). Per `02-security.md` § "Never Leak Upstream Error Text" applied defensively to deserialization too.

**Guardrails (DO NOT):**
- DO NOT make this advice unscoped (no `basePackages`) — that would catch the same exceptions thrown by user/auth/proxy controllers and shadow their handlers.
- DO NOT include `ex.getMessage()` in the response body — could include user strings.
- DO NOT add a generic `@ExceptionHandler(Exception.class)` catch-all — global error handling is reserved for a future spec; unhandled exceptions should propagate to Spring's default 500.

Test asserts `is4xxClientError()` instead. The rollback verification (5 table-state assertions) is unchanged and still proves the transactional contract that this test exists to verify. Fixing the catch-all is out of scope for Spec 2.6 — recommended as a follow-up.
