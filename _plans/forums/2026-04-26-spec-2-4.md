## Architecture Context

### Existing patterns to mirror (verified during recon)

**Spec 2.2 / 2.3 service pattern** — `com.worshiproom.activity.FaithPointsService` and `com.worshiproom.activity.StreakService`:
- Annotated `@Service`, no field injection
- Single public method, no `@Autowired` deps, no `@Transactional`
- Throws `IllegalArgumentException` on null/invalid input with explicit messages
- Constants in `com.worshiproom.activity.constants.{PointValues,LevelThresholds,MultiplierTiers}` — `final class` with private constructor, public static immutable maps/lists
- DTOs as Java records in `com.worshiproom.activity.dto.{FaithPointsResult,StreakResult}` with `record (...)` syntax and Javadoc on every field

**Existing `ActivityType` enum** at `com.worshiproom.activity.ActivityType` — 12 values: `MOOD, PRAY, LISTEN, PRAYER_WALL, READING_PLAN, MEDITATE, JOURNAL, GRATITUDE, REFLECTION, CHALLENGE, LOCAL_VISIT, DEVOTIONAL` — uses `@JsonValue` + `@JsonCreator` for camelCase wire format. `CelebrationTier` follows the same `@JsonValue` pattern with kebab-case wire strings.

**Existing JPA entity pattern** — `com.worshiproom.activity.FaithPoints` and `StreakState`:
- `@Entity` + `@Table(name = "...")`
- Manual getters/setters (no Lombok)
- `protected` no-arg constructor for JPA, public arg constructor for application use
- `equals()` / `hashCode()` based on primary key only (uses `instanceof` pattern matching: `if (!(o instanceof FaithPoints other)) return false`)
- `@Id @Column(name = "...", nullable = false, updatable = false)` for primary key
- Bean Validation: `@Min(0)` on numeric columns, mirroring DB CHECK constraints

**Composite-key precedent** — Forums Wave uses `@IdClass` for composite keys (per master plan Decision 4 patterns + JPA convention). No existing composite-key entity in `com.worshiproom.activity` to mirror; `UserBadge` is the first. Plan picks `@IdClass(UserBadgeId.class)` because:
1. It keeps the `@Entity` field shape flat (matches `StreakState`)
2. It avoids an extra `@Embeddable` class with separate field declarations
3. JPA equality semantics are identical between `@IdClass` and `@EmbeddedId` for `findById()` calls

**Repository stub pattern** — `StreakRepository extends JpaRepository<StreakState, UUID>` is a one-line interface annotated `@Repository` with no custom methods (Spec 2.6 will add what it needs). `BadgeRepository` follows the same shape with `JpaRepository<UserBadge, UserBadgeId>`.

**Test pattern** — `StreakServiceTest` and `FaithPointsServiceTest` are plain JUnit 5 (no `@SpringBootTest`, no Testcontainers), instantiate the service directly (`private final StreakService service = new StreakService()`), and use AssertJ assertions (`assertThat`, `assertThatThrownBy`). Group tests with `@Nested @DisplayName(...)` classes for readability. `BadgeServiceTest` mirrors this layout with one `@Nested` class per eligibility category.

## Edge Cases & Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Composite key strategy | `@IdClass(UserBadgeId.class)` on `UserBadge` | Keeps `@Entity` field shape flat (matches `StreakState`); JPA equality is identical to `@EmbeddedId`. `UserBadgeId` is a separate class with `Serializable`, equals/hashCode, and a public no-arg + all-args constructor. |
| `BadgeCatalog.lookup` return type | `Optional<BadgeDefinition>` | Cleaner Java idiom; `BadgeService` uses `.map(BadgeDefinition::repeatable).orElse(false)` for the repeatability guard. Fast-fails on unknown IDs without the caller writing `try/catch`. |
| `BadgeCheckContext.todayActivities` shape | `Set<ActivityType>` (not a 12-boolean record) | Matches existing `ActivityType` enum; simpler to construct and reason about. The 7 frontend boolean reads (`mood, pray, listen, prayerWall, meditate, journal, readingPlan`) become `set.contains(ActivityType.MOOD)` etc. |
| `BibleProgressSnapshot` internal type | `Map<String, Set<Integer>>` (book slug → completed chapter numbers) | Frontend persists `Record<string, number[]>`; backend uses `Set<Integer>` for semantic accuracy (chapters are unique). The caller in Spec 2.6 dedupes on the way in. |
| `GratitudeEntriesSnapshot` internal type | `List<LocalDate>` (dates as `LocalDate`, not `String`) | Java type system advantage. Service reduces to `Set<LocalDate>` internally for unique-day counts. |
| Gratitude consecutive-streak algorithm | `ChronoUnit.DAYS.between(prev, curr) == 1` over sorted `List<LocalDate>` | Per Architectural Decision 9. `LocalDate` has no time component — DST is not a concern, so the frontend's noon-anchor JS trick is irrelevant. Cleaner and byte-identical for any input where the dates parse correctly. |
| `BadgeService` repeatability path | After per-category eligibility computation, filter step: `alreadyEarned ∧ ¬repeatable → drop` | Mirrors frontend `addEarnedBadge` semantics (non-repeatable already-earned skipped; repeatable always added/incremented). The CALLER (Spec 2.6) reads the result and increments `display_count` for repeatable badges. |
| Welcome badge in catalog | INCLUDED in `BadgeCatalog` (not in `checkBadges` evaluation) | Per Divergence 2: catalog returns metadata for any `badge_id` even when the eligibility service doesn't fire it. Future registration spec calls a separate "grant welcome" path that uses `BadgeCatalog.lookup("welcome")` for metadata. |
| Challenge badges in catalog | INCLUDED (same rationale as welcome) | Same Divergence 2 reasoning. Future challenge-completion spec grants them via separate code path. |
| Pure-function service (no DB) | `BadgeService` has no `@Autowired`, no repository injection, no DB access | Mirrors `FaithPointsService` and `StreakService`. Spec 2.6 composes the service with persistence in the controller layer. |
| `BadgeResult.newlyEarnedDefinitions` ordering | Same order as `newlyEarnedBadgeIds` | The two fields are parallel arrays so the API consumer can iterate in lockstep. The order reflects the 15-category iteration order from the frontend. |
| Defensive null/empty inputs | Empty collections produce zero badges from those categories without throwing | Per Divergence 1: most context fields will be empty during dual-write phase. Throwing on empty would defeat the purpose. Service treats `null` collections same as empty (defensive — caller in Spec 2.6 should pass empties, but a `null` is a safe fallback rather than a 500). |
| `Set<String> alreadyEarnedBadgeIds` null handling | Throw `IllegalArgumentException` | Mirrors `FaithPointsService.calculate` and `StreakService.updateStreak` posture: required arguments are explicit; `null` is a programming error. The empty set is the legitimate "user has no badges yet" input. |

```java
package com.worshiproom.activity;

import com.fasterxml.jackson.annotation.JsonCreator;
import com.fasterxml.jackson.annotation.JsonValue;

/**
 * Frontend-mirrored celebration tier (4 values, kebab-case wire strings).
 *
 * <p>Java identifiers use SCREAMING_SNAKE_CASE per Java convention; the
 * kebab-case wire-format string used in JSON request/response bodies is
 * mapped via {@link JsonValue} (serialization) and {@link JsonCreator}
 * (deserialization). The wire strings MUST match the frontend's
 * {@code CelebrationTier} union exactly (verbatim port of
 * {@code frontend/src/types/dashboard.ts CelebrationTier}).
 */
public enum CelebrationTier {
    TOAST("toast"),
    TOAST_CONFETTI("toast-confetti"),
    SPECIAL_TOAST("special-toast"),
    FULL_SCREEN("full-screen");

    private final String wireValue;

    CelebrationTier(String wireValue) {
        this.wireValue = wireValue;
    }

    @JsonValue
    public String wireValue() {
        return wireValue;
    }

    @JsonCreator
    public static CelebrationTier fromWireValue(String value) {
        if (value == null) {
            throw new IllegalArgumentException("CelebrationTier wire value must not be null");
        }
        for (CelebrationTier t : values()) {
            if (t.wireValue.equals(value)) {
                return t;
            }
        }
        throw new IllegalArgumentException("Unknown CelebrationTier wire value: " + value);
    }
}
```

`BibleProgressSnapshot` — wraps `Map<String, Set<Integer>>`:

```java
package com.worshiproom.activity.dto;

import java.util.Collections;
import java.util.Map;
import java.util.Set;

/**
 * Bible reading progress: book slug -> set of completed chapter numbers.
 * Mirrors frontend {@code wr_bible_progress} ({@code Record<string, number[]>}).
 *
 * <p>The snapshot uses {@code Set<Integer>} for chapter uniqueness;
 * Spec 2.6's caller is responsible for de-duplicating on assembly.
 *
 * @param chaptersByBook unmodifiable map; empty map represents "no progress yet"
 */
public record BibleProgressSnapshot(Map<String, Set<Integer>> chaptersByBook) {

    public BibleProgressSnapshot {
        chaptersByBook = chaptersByBook == null ? Map.of() : Collections.unmodifiableMap(chaptersByBook);
    }

    public static BibleProgressSnapshot empty() {
```

```java
 * pattern.
 *
 * <p>All collection fields are unmodifiable after construction. Callers
 * pass empty collections (via the static factories on each snapshot DTO)
 * when data is unavailable.

    int currentStreak,
    int longestStreak,
    int currentLevel,
    int previousLevel,
    Set<ActivityType> todayActivities,
    ActivityCountsSnapshot activityCounts,
    int friendCount,
    boolean allActivitiesWereTrueBefore,
    List<ReadingPlanProgress> readingPlanProgress,
    BibleProgressSnapshot bibleProgress,
    List<MeditationSession> meditationHistory,
    GratitudeEntriesSnapshot gratitudeEntries,
    LocalVisitsSnapshot localVisits,
    List<ListeningSession> listeningHistory

    public BadgeCheckContext {
        // Defensive: replace nulls with safe empties so the service body
        // can iterate freely without null guards on every field.
        todayActivities = todayActivities == null
            ? EnumSet.noneOf(ActivityType.class)
            : Collections.unmodifiableSet(EnumSet.copyOf(todayActivities.isEmpty()
                ? EnumSet.noneOf(ActivityType.class)
                : todayActivities));
        activityCounts = activityCounts == null ? ActivityCountsSnapshot.zeros() : activityCounts;
        readingPlanProgress = readingPlanProgress == null
            ? List.of()
            : Collections.unmodifiableList(readingPlanProgress);
        bibleProgress = bibleProgress == null ? BibleProgressSnapshot.empty() : bibleProgress;
        meditationHistory = meditationHistory == null
            ? List.of()
            : Collections.unmodifiableList(meditationHistory);
        gratitudeEntries = gratitudeEntries == null ? GratitudeEntriesSnapshot.empty() : gratitudeEntries;
        localVisits = localVisits == null ? LocalVisitsSnapshot.zero() : localVisits;
```

### Step 12: Author `BadgeService` (15 categories + idempotency + repeatability)

**Objective:** Pure-function `@Service` that runs the 15 eligibility categories in frontend declaration order, applies the idempotency-vs-repeatability filter, and returns the result.

**Files to create:**
- `backend/src/main/java/com/worshiproom/activity/BadgeService.java`

**Details:**

The service has one public method, `checkBadges(BadgeCheckContext, Set<String>) → BadgeResult`. Its body:

1. **Validate inputs.** `context == null` or `alreadyEarnedBadgeIds == null` → `IllegalArgumentException`.
2. **Build a candidate list** by running 15 private helpers in order (each returns `List<String>`):
   - `streakBadges(context)`
   - `levelBadges(context)`
   - `activityMilestoneBadges(context)`
   - `fullWorshipDay(context)`
   - `communityBadges(context)`
   - `readingPlanBadges(context)`
   - `localSupportSeeker(context)`
   - `bibleBookBadges(context)`
   - `meditationSessionBadges(context)`
   - `prayerWallPostBadges(context)`
   - `intercessor(context)`
   - `bibleChapterBadges(context)`
   - `gratitudeBadges(context)`
   - `localFirstVisit(context)`
   - `worshipListener(context)`
3. **Deduplicate while preserving order** (a candidate list could in principle contain duplicates if categories overlap — they don't today, but the dedup is a cheap correctness guard).
4. **Apply the idempotency-vs-repeatability filter:** for each candidate ID, drop it if it's already in `alreadyEarnedBadgeIds` AND its catalog entry's `repeatable` flag is `false`. Repeatable badges (`full_worship_day`) pass through.
5. **Hydrate definitions** by calling `BadgeCatalog.lookup(id)` per surviving ID. Skip any ID whose catalog entry is missing (paranoid guard — should never happen).
6. **Return** `new BadgeResult(ids, definitions)`.

**Per-category logic** — each helper mirrors the frontend's iteration shape:

- **streakBadges:** for each `t` in `BadgeThresholds.STREAK`, push `"streak_" + t` if `context.currentStreak() >= t`. (The eligibility filter at step 4 handles already-earned exclusion; the helper itself does NOT consult `alreadyEarnedBadgeIds`.) Note: the frontend embeds the `!earned[badgeId]` check in the loop; we hoist that to the central filter.

  > **Precedent in port:** the frontend always pushes a candidate when condition holds, then the candidate-vs-earned filter happens at end. This Java port unifies the filter for symmetry — same behavior, cleaner control flow.

- **levelBadges:** push `"level_" + context.currentLevel()` (single push per call). The filter at step 4 handles repeated-level idempotency.

- **activityMilestoneBadges:** for each entry in `BadgeThresholds.ACTIVITY_MILESTONES`, look up the count in `context.activityCounts()` (use a private `getCount(ActivityType, ActivityCountsSnapshot)` helper that switches on `ActivityType` and returns the right field; covers PRAY, JOURNAL, MEDITATE, LISTEN, PRAYER_WALL). private List<String> applyEarnedFilter(List<String> candidates, Set<String> alreadyEarned) {
```java
    List<String> survivors = new ArrayList<>();
    LinkedHashSet<String> seen = new LinkedHashSet<>();
    for (String id : candidates) {
        if (!seen.add(id)) continue; // dedup, preserve order
        boolean repeatable = BadgeCatalog.lookup(id).map(BadgeDefinition::repeatable).orElse(false);
        if (alreadyEarned.contains(id) && !repeatable) continue;
        survivors.add(id);
    }
    return survivors;
}
```

```java
        if (context == null) {
            throw new IllegalArgumentException("context must not be null");

            throw new IllegalArgumentException("alreadyEarnedBadgeIds must not be null");

        candidates.addAll(streakBadges(context));
        candidates.addAll(levelBadges(context));
        candidates.addAll(activityMilestoneBadges(context));
        candidates.addAll(fullWorshipDay(context));
        candidates.addAll(communityBadges(context));
        candidates.addAll(readingPlanBadges(context));

        candidates.addAll(intercessor(context));
        candidates.addAll(bibleChapterBadges(context));
        candidates.addAll(gratitudeBadges(context));
        candidates.addAll(localFirstVisit(context));
        candidates.addAll(worshipListener(context));

        List<String> survivors = applyEarnedFilter(candidates, alreadyEarnedBadgeIds);
        List<BadgeDefinition> defs = hydrate(survivors);
        return new BadgeResult(survivors, defs);
```
