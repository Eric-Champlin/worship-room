### Frontend mount pattern

The bridge lives inside `InlineComposer.tsx`, conditionally rendered. Render position: **immediately after the closing `</fieldset>` of the category fieldset (after line 667) and BEFORE the `WaysToHelpPicker` block (line 670).**

Rationale for placement:
- The category fieldset is below the textarea per spec MPD-2 ("below textarea, above submit row"). Submit row is at line 698-725.
- Placing the bridge directly below the category fieldset creates a clear visual cause-and-effect: user selects Mental Health → bridge appears just below.
- This is functionally "below textarea + below scripture input + below challenge checkbox + below category" — still meets MPD-2 ("below textarea, above submit row").
- Alternative placements considered: above the category fieldset (rejected — bridge would push fieldset down, feels like interrogation); just before the submit row (rejected — too far from the trigger pill).

The parent-level gate uses both conditions explicitly for defense-in-depth (even though `selectedCategory === 'mental-health'` is impossible without the category fieldset being rendered, and the fieldset only renders for `prayer_request`):

```tsx
{postType === 'prayer_request' && selectedCategory === 'mental-health' && (
  <CounselorBridge />
)}
```

The `CounselorBridge` component internally checks `shouldShowCounselorBridge()` on mount and returns `null` if dismissed. This isolates dismissal logic inside the component (cleaner than threading state through InlineComposer).

### Reply-mode exclusion (R3 — architecturally enforced)

`PrayerDetail.tsx:432` mounts `CommentInput`, NOT `InlineComposer`, for the reply composer. `CommentInput.tsx` has no `postType` concept and no category fieldset. The bridge therefore CANNOT fire in reply mode without explicit code changes to import CounselorBridge into CommentInput — and this spec does NOT do that. Gate-G-REPLY-MODE-EXCLUSION holds via architecture, not via a runtime check.

A defensive code comment will live in CounselorBridge.tsx documenting this so a future spec author doesn't add it to CommentInput without re-evaluating Gate-G-MH-OMISSION-RESPECTED + brand voice.

### Crisis-UI coexistence (R5 default — both render)

InlineComposer's existing inline crisis banner (lines 738-771) renders when `crisisDetected === true`, which is set in `handleSubmit` after the client-side `containsCrisisKeyword()` check. The bridge renders based on category selection alone. The two render independently:

- User types crisis keywords + picks Mental Health + clicks Submit → bridge already visible (from category pick) + crisis banner appears after Submit
- The bridge does not change positioning or visibility based on crisisDetected
- The crisis banner sits at the bottom of the FrostedCard (lines 738-771); the bridge sits between the category fieldset and the ways-to-help picker

Tests verify both render simultaneously when both conditions are met (Gate-G-NO-CRISIS-UI-CONFLICT).

### Step 1: Create `counselor-bridge-storage.ts` service module

**Objective:** Provide the canonical session-scoped dismissal API for the bridge. Mirrors `welcome-back-storage.ts` exactly except for the key and the function names.

**Files to create:**
- `frontend/src/services/counselor-bridge-storage.ts` — new file, ~30 lines

**Details:**

```ts
// frontend/src/services/counselor-bridge-storage.ts

/**
 * Spec 7.5 — Counselor bridge dismissal storage.
 *
 * Mirrors the session-scoped dismissal pattern from welcome-back-storage.ts
 * and surprise-storage.ts. Once dismissed, the bridge stays hidden for the
 * remainder of the browser session (sessionStorage clears on tab close).
 *
 * The key is wr_counselor_bridge_dismissed and intentionally describes the
 * bridge (the surfaced offering), NOT the category that triggers it (Mental
 * Health), per Gate-G-MH-OMISSION-RESPECTED in the spec.
 */

const SESSION_KEY = 'wr_counselor_bridge_dismissed'

/**
 * Returns true if the bridge should be shown — i.e., it has NOT been
 * dismissed in the current browser session.
 */
export function shouldShowCounselorBridge(): boolean {
  try {
    return sessionStorage.getItem(SESSION_KEY) !== 'true'
  } catch {
    // sessionStorage unavailable (private mode, quota, security) — fail open
    // (show the bridge). Same posture as welcome-back-storage.ts.
    return true
  }
}

/**
 * Mark the bridge as dismissed for this session. Idempotent.
 */
export function markCounselorBridgeDismissed(): void {
  try {
    sessionStorage.setItem(SESSION_KEY, 'true')
  } catch {
    // Silently fail — acceptable per existing pattern. Worst case: bridge
    // re-appears on next category toggle within the same session.
  }
}
```

- Do NOT switch to localStorage — dismissal is intentionally session-scoped per MPD-4 / AC-4 / R7
- Do NOT introduce a reactive store — single consumer per render, plain CRUD is appropriate per Spec 6.9 precedent (`composer-drafts-storage.ts`)
- Do NOT add analytics hooks — Section 4 "Anti-Pressure + Privacy Decisions" explicitly prohibits engagement telemetry on this bridge
- Do NOT export the `SESSION_KEY` constant — internal implementation detail; tests can hardcode the literal string (matches welcome-back-storage / surprise-storage convention)

```tsx
// frontend/src/components/prayer-wall/CounselorBridge.tsx

import { useState, useCallback } from 'react'
import { Link } from 'react-router-dom'
import { Heart, X } from 'lucide-react'
import { cn } from '@/lib/utils'
import {
  shouldShowCounselorBridge,
  markCounselorBridgeDismissed,
} from '@/services/counselor-bridge-storage'

/**
 * Spec 7.5 — Quiet bridge offering /local-support/counselors when the user
 * composes a Mental Health prayer request.
 *
 * Render condition is enforced by the parent (InlineComposer), which only
 * mounts this component when postType === 'prayer_request' AND
 * selectedCategory === 'mental-health'. This component additionally checks
 * shouldShowCounselorBridge() on mount and returns null if the user has
 * dismissed it in the current session.
 *
 * Architectural note: This component is intentionally NOT imported by
 * CommentInput.tsx (the reply composer used in PrayerDetail). Comment
 * replies do not have a postType / category and would not satisfy the
 * bridge's trigger conditions. Do not import this into CommentInput
 * without revisiting Gate-G-MH-OMISSION-RESPECTED and the brand voice
 * review in the spec.
 *
 * Visual treatment per Spec 7.5 MPD-3 (with Recon Addendum item 1 / Spec-10A
 * correction): no surrounding card / panel / border / background; only an
 * inline icon + link + dismiss button. Link color is
 * text-violet-300 hover:text-violet-200 (post-Spec-10A canonical), NOT
 * text-primary (which CrisisBanner uses but is on the deprecated list).
 */
export function CounselorBridge() {
  const [visible, setVisible] = useState<boolean>(() =>
    shouldShowCounselorBridge(),
  )

  const handleDismiss = useCallback(() => {
    markCounselorBridgeDismissed()
    setVisible(false)
  }, [])

  if (!visible) return null

  return (
    <div
      className="mt-3 flex items-center gap-2 text-sm text-white/70"
      data-testid="counselor-bridge"
    >
      <Heart
        className="h-4 w-4 flex-shrink-0 text-white/50"
        aria-hidden="true"
      />
      <Link
        to="/local-support/counselors"
        className={cn(
          'underline underline-offset-2',
          'text-violet-300 hover:text-violet-200',
          'focus-visible:outline-none focus-visible:ring-2',
          'focus-visible:ring-violet-300/60 focus-visible:ring-offset-2',
          'focus-visible:ring-offset-hero-bg rounded-sm',
        )}
      >
        Find a counselor near you &rarr;
      </Link>
      <button
        type="button"
        onClick={handleDismiss}
        className={cn(
          'ml-auto inline-flex h-11 w-11 items-center justify-center',
          'rounded-full text-white/40 hover:text-white/70',
          'focus-visible:outline-none focus-visible:ring-2',
          'focus-visible:ring-white/50',
        )}
        aria-label="Dismiss counselor suggestion"
      >
        <X className="h-4 w-4" aria-hidden="true" />
      </button>
    </div>
  )
}
```

**Guardrails (DO NOT):**
- Do NOT use `text-primary` for the link color — that's on the post-Spec-10A deprecated list (`09-design-system.md` § "Deprecated Patterns"). The Recon Addendum item 1 and Gate-G-LINK-COLOR-SPEC-10A make this explicit.
- Do NOT wrap the bridge in a `<FrostedCard>` — MPD-3 says NO surrounding card / panel / border / background. The bridge is a sibling block, not a callout.
- Do NOT use `bg-warning/10`, `bg-rose-500/10`, or any other tinted background — same MPD-3 prohibition. This is NOT a crisis intervention; it's a quiet offering.
- Do NOT add a `role="alert"` or `aria-live` region — the bridge is informational, not alerting. Crisis content has its own `role="alert"` in InlineComposer.tsx:739.
- Do NOT use a `UserCircle` or `MessageCircle` icon — picked `Heart` per the Edge Cases & Decisions table.
- Do NOT add an animation on enter / exit — the bridge appears/disappears synchronously with the category toggle, matching the existing fieldset's no-animation behavior. Adding animations introduces motion-sensitivity concerns near a Mental Health surface; default to none.
- Do NOT render a `(Mental Health prayer detected)` subtitle or any text that names the category — the bridge is a quiet offering, not an explanation of why we're showing it. Naming the category back to the user would feel surveillance-like and risks violating R6 brand voice.
- Do NOT add a `data-category` or any other DOM attribute that exposes the trigger category to scripts or tests beyond the necessary `data-testid` — keeps the user's category choice out of inspectable surfaces.
- Do NOT add analytics / telemetry hooks — Section 4 explicitly prohibits this.
- Do NOT use the deprecated `text-success` / `text-danger` color tokens — use `text-white/*` for muted greys and `text-violet-300` for the link.
