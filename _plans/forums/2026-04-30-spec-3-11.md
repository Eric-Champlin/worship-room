**Existing patterns to match:**

- **Pattern A subscription store** (BB-45 / Phase 0.5): `lib/prayer-wall/reactionsStore.ts` exposes `subscribe(listener: () => void): () => void`, `getSnapshot(): T`, and writes that call `notify()` to invalidate a snapshot cache and fire listeners. The hook (`hooks/usePrayerReactions.ts`) calls `useSyncExternalStore(subscribe, getSnapshot, getServerSnapshot)`. **No changes to the subscription mechanism in this spec** — only to what fires `notify()`.
- **API error taxonomy** (Spec 3.10): `mapApiErrorToToast(error, retryAfterSeconds?)` returns `{message, severity, retryAfterSeconds?}`. `AnonymousWriteAttemptError` is the throw-before-network sentinel; the canonical handler is "catch this class first, then fall through to ApiError generic handling."
- **Read-swap, NOT dual-write** (Spec 3.10 D7): when flag is on, reads come from backend with module-lifetime in-memory cache; localStorage is offline-cache-on-hydration-only, never written during a toggle. Spec 3.11 extends the same model to the store layer.
- **`wr:auth-invalidated` event** (Spec 1.9 / api-client): dispatched on 401 by `apiFetch` and `apiFetchWithMeta`. AuthContext listens; on receipt it clears auth state. Spec 3.11 also listens so the store can clear its in-memory backend cache.
- **DI for non-React-to-React communication** (Spec 3.10 in `services/api/prayer-wall-api.ts` — uses `getStoredToken()` from auth-storage, not a React hook): non-React modules access app singletons via module-scoped functions. Spec 3.11 applies the same shape via `configure({showToast, openAuthModal})` exposed by the store and called from inside the hook's `useEffect`.

**Recon-confirmed consumer reality:**

`PrayerWall.tsx:264`, `PrayerWallDashboard.tsx:72`, `PrayerDetail.tsx:43`, and `PrayerWallProfile.tsx:49` all read the synchronous `togglePraying` return value:

```ts
const wasPraying = togglePraying(prayerId)
if (!wasPraying) { recordActivity('prayerWall', 'prayer_wall'); /* ... */ }
setPrayers(prev => prev.map(p =>
  p.id === prayerId ? { ...p, prayingCount: p.prayingCount + (wasPraying ? -1 : 1) } : p
))
```

If `togglePraying` returned `Promise<boolean>`, `wasPraying ? -1 : 1` would always evaluate to `-1` (a Promise is always truthy) — a silent correctness bug across all four consumer files. This forces a divergence from the spec author's D4 proposal: keep toggles **synchronous-returning** and fire-and-forget the API call. See Plan-Time Divergence #1.

**Reactions hydration shape:**

`reactionsResponseToReactionsMap` (`postMappers.ts:148`) returns `Record<string, PrayerReaction>` with all four fields populated from the backend's `/users/me/reactions` response (`isPraying`, `isBookmarked`, `isCandle`). `listMyBookmarks` returns `PostListResult` with `posts: PrayerRequest[]`. The hydration merger sets the reactions map first, then overlays bookmarked post IDs (forcing `isBookmarked: true`) — matches Watch-for #12.

## Edge Cases & Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Toggle return type when flag is on | **Synchronous boolean (fire-and-forget the API call)** — diverges from spec D4 | Four consumer files use `wasPraying` synchronously (`prayingCount + (wasPraying ? -1 : 1)`); `Promise<boolean>` would silently coerce to truthy and break the count math. See Plan-Time Divergence #1. |
| Hydration trigger | `useEffect` in `usePrayerReactions` keyed on `user?.id` | Recon-verified: `useAuth()` exposes `{isAuthenticated, user, ...}` with `user.id` as the stable identity. `useEffect` deps `[user?.id]` produces hydrate-on-login, clear-on-logout, re-hydrate-on-user-switch. |
| User-switch detection | Module-scoped `lastHydratedUserId: string \| null` | Cache from User A must not leak to User B on the same browser session. `init(userId)` checks against `lastHydratedUserId` — different non-null userId → clear cache fully before re-hydrating. |
| Bookmark hydration pagination | First page only (`page: 1, limit: 100`) per spec D3 | Adding bookmarks is idempotent (Spec 3.7 D5); a >100-bookmark user clicking re-bookmark on cosmetically-uncached post 101+ produces correct backend state. Documented for >100-bookmark followup. |
| `init(null)` localStorage behavior | Clear in-memory cache, do NOT clear localStorage (per spec MPD-3 + Watch-for #3) | Logout shouldn't wipe offline cache; user might log back in offline and the prior session's cache is the best fallback. |
| Toast / AuthModal access from non-React store | DI via `configure({showToast, openAuthModal})` called from hook `useEffect` (per spec D5) | Store is non-React; events bus would diffuse the signal across the codebase. Direct DI keeps the wiring explicit; null-check on every call site keeps the store safe before the hook mounts. |
| Pending mutation guard key shape | `${prayerId}:${concern}` where concern ∈ {`praying`, `bookmark`, `candle`} | Per-prayer + per-concern locking — a user can simultaneously double-tap the bookmark icon and the praying icon on the same prayer; only same-concern double-taps coalesce. |
| Server-state-disagreement handling | `forceState(prayerId, partial)` bypasses optimistic logic, calls `notify()` (per spec D8) | Rare cross-device race; trust server. Listeners re-render with corrected state. |
| `toggleBookmark` route decision | Cached `isBookmarked` boolean determines `addBookmark` vs `removeBookmark` (per spec D9 + Watch-for #16) | Both endpoints are idempotent (Spec 3.7 D5); cache-stale-vs-server window is forgiving. |
| `wr:auth-invalidated` listener registration | Once at module load (per Watch-for #11) | Multiple subscriptions cause duplicate `init(null)` calls; module-load registration matches the existing api-client pattern. |
| Hydration-vs-mutation race | Block hydration writes while any mutation is pending (per Watch-for #14 simpler alternative) | Stale data for ~hundreds of ms is better than overwriting a user's just-clicked Pray. Implementation: hydration result held in a local variable; if `pendingMutations.size > 0`, defer the cache replacement until all mutations settle. |
| Hydration error toast scope | Network/5xx fires "couldn't refresh" toast; 401 fires nothing (apiFetch already triggered AuthModal); partial failure (one of two API calls fails) toasts only for the failed concern (per spec D11) | Matches existing apiErrors taxonomy; new HYDRATION_FAILED copy entry distinguishes from action-failure NETWORK_ERROR. |
| Hook mount cleanup behavior | `useEffect` cleanup does NOT call `init(null)` (per Watch-for #15) | Prayer Wall navigation mounts/unmounts the hook on every page transition; calling `init(null)` on unmount would clear cache + re-hydrate on every nav. |

### Step 1: Extend `apiErrors.ts` with hydration-failed copy

**Objective:** Add a distinct toast string for hydration failures so the user sees "couldn't refresh" instead of the action-oriented "couldn't reach the server" copy.

**Files to create/modify:**
- `frontend/src/lib/prayer-wall/apiErrors.ts` — add `HYDRATION_FAILED` to `PRAYER_WALL_API_ERROR_COPY`; add `mapHydrationErrorToToast(error, concern?)` helper that returns the new copy for network/5xx and falls through to existing `mapApiErrorToToast` for other codes.

**Details:**

```typescript
// In PRAYER_WALL_API_ERROR_COPY (extend existing const):
HYDRATION_FAILED: "We couldn't refresh your reactions. Try again later.",
HYDRATION_FAILED_PARTIAL_REACTIONS: "We couldn't refresh your reactions. Try again later.",
HYDRATION_FAILED_PARTIAL_BOOKMARKS: "We couldn't refresh your bookmarks. Try again later.",

/**
 * Maps a hydration error to a toast descriptor. Distinct from
 * mapApiErrorToToast because hydration failures use "couldn't refresh"
 * copy (passive — failed background sync) rather than "couldn't reach
 * the server" (active — user-initiated action failed).
 *
 * Concern: 'reactions' | 'bookmarks' | 'both' — controls which copy variant.
 */
export function mapHydrationErrorToToast(
  error: ApiError,
  concern: 'reactions' | 'bookmarks' | 'both' = 'both',
): ToastDescriptor {
  // 401 — apiFetch already cleared token; no toast (AuthModal-only flow)
  if (error.status === 401) return { message: '', severity: 'error' }

  // Network / 5xx → hydration-specific copy
  if (error.code === 'NETWORK_ERROR' || error.status === 0 || error.status >= 500) {
    const message =
      concern === 'reactions'
        ? PRAYER_WALL_API_ERROR_COPY.HYDRATION_FAILED_PARTIAL_REACTIONS
        : concern === 'bookmarks'
        ? PRAYER_WALL_API_ERROR_COPY.HYDRATION_FAILED_PARTIAL_BOOKMARKS
        : PRAYER_WALL_API_ERROR_COPY.HYDRATION_FAILED
    return { message, severity: 'warning' }
  }

  // Anything else falls through to the standard taxonomy
  return mapApiErrorToToast(error)
}
```

- [x] No exclamation

- [x] No comparison

### Step 3: Add `init(userId)` hydration logic + `wr:auth-invalidated` listener

**Objective:** Wire the hydration entry point that fires `getMyReactions()` and `listMyBookmarks()` in parallel, merges results, and replaces the in-memory cache. Wire the auth-invalidated listener so a stale-token 401 elsewhere triggers `init(null)`.

**Files to create/modify:**
- `frontend/src/lib/prayer-wall/reactionsStore.ts` — add `init(userId: string | null)`, register window event listener at module load

**Details:**

```typescript
import * as prayerWallApi from '@/services/api/prayer-wall-api'
import { isBackendPrayerWallEnabled } from '@/lib/env'
import { ApiError } from '@/types/auth'
import { mapHydrationErrorToToast } from './apiErrors'

const AUTH_INVALIDATED_EVENT = 'wr:auth-invalidated'

/**
 * Hydration entry point. Called from usePrayerReactions hook on auth state
 * change. Merges getMyReactions + listMyBookmarks into a unified
 * Record<prayerId, PrayerReaction>, replaces cache, writes to localStorage.
 *
 * - userId === null → clear in-memory cache, do NOT touch localStorage
 *   (offline cache preservation per MPD-3)
 * - userId !== null AND flag off → no-op
 * - userId !== null AND flag on → fire both API calls in parallel; merge
 *   on partial failure
 */
export async function init(userId: string | null): Promise<void> {
  // Logout / no-user
  if (userId === null) {
    cache = {}
    snapshotCache = null
    lastHydratedUserId = null
    notify()
    // Note: localStorage is NOT cleared — preserve offline cache per MPD-3
    return
  }

  if (!isBackendPrayerWallEnabled()) {
    // Flag-off mode — leave existing localStorage seed behavior intact
    return
  }

  // User-switch detection — clear cache fully before re-hydrating
  if (lastHydratedUserId !== null && lastHydratedUserId !== userId) {
    cache = {}
    snapshotCache = null
    notify()
  }

  // Fire both API calls in parallel; allow partial failure
  const [reactionsResult, bookmarksResult] = await Promise.allSettled([
    prayerWallApi.getMyReactions(),
    prayerWallApi.listMyBookmarks({ page: 1, limit: 100 }),
  ])

  // Hydration-vs-mutation race guard (Watch-for #14):
  // Defer cache replacement if any mutation is in flight. The simpler
  // alternative — overwrite anyway — would clobber an optimistic flip
  // the user just made. Stale data for hundreds of ms is preferable.
  if (pendingMutations.size > 0) {
    // Re-call ourselves once mutations drain. We use a microtask poll
    // because mutations settle inside the Promise chain; setTimeout(0)
    // is sufficient. Keep the poll bounded — give up after 5 seconds.
    deferHydrationCommit(reactionsResult, bookmarksResult, userId, 0)
    return
  }

  commitHydration(reactionsResult, bookmarksResult, userId)
}

function commitHydration(
  reactionsResult: PromiseSettledResult<Record<string, PrayerReaction>>,
  bookmarksResult: PromiseSettledResult<{ posts: PrayerRequest[]; pagination: PaginationInfo }>,
  userId: string,
): void {
  // Build the merged shape. Reactions FIRST (sets isPraying / isCandle / isBookmarked),
  // then bookmarks LAST (overrides isBookmarked: true) per Watch-for #12.
  const merged: Record<string, PrayerReaction> = {}

  if (reactionsResult.status === 'fulfilled') {
    Object.assign(merged, reactionsResult.value)
  } else {
    showHydrationToast(reactionsResult.reason, 'reactions')
  }

  if (bookmarksResult.status === 'fulfilled') {
    for (const post of bookmarksResult.value.posts) {
      const existing = merged[post.id]
      merged[post.id] = {
        prayerId: post.id,
        isPraying: existing?.isPraying ?? false,
        isBookmarked: true,
        isCandle: existing?.isCandle ?? false,
      }
    }
  } else {
    showHydrationToast(bookmarksResult.reason, 'bookmarks')
  }

  // Both failed — keep existing cache, don't replace
  if (reactionsResult.status === 'rejected' && bookmarksResult.status === 'rejected') {
    return
  }

  cache = merged
  snapshotCache = null
  lastHydratedUserId = userId
  writeToStorage(merged)
  notify()
}

function deferHydrationCommit(
  reactions: PromiseSettledResult<Record<string, PrayerReaction>>,
  bookmarks: PromiseSettledResult<{ posts: PrayerRequest[]; pagination: PaginationInfo }>,
  userId: string,
  attempt: number,
): void {
  if (attempt > 50) return // 50 * 100ms = 5s cap
  setTimeout(() => {
    if (pendingMutations.size === 0) {
      commitHydration(reactions, bookmarks, userId)
    } else {
      deferHydrationCommit(reactions, bookmarks, userId, attempt + 1)
    }
  }, 100)
}

function showHydrationToast(reason: unknown, concern: 'reactions' | 'bookmarks'): void {
  if (!toastFn) return
  if (!(reason instanceof ApiError)) return
  const descriptor = mapHydrationErrorToToast(reason, concern)
  if (descriptor.message.length > 0) {
    toastFn(descriptor.message)
  }
}

// --- Module-load: register wr:auth-invalidated listener exactly once ---
if (typeof window !== 'undefined') {
  window.addEventListener(AUTH_INVALIDATED_EVENT, () => {
    void init(null)
  })
}
```

**Details:**

```typescript
export function togglePraying(prayerId: string): boolean {
  // Synchronous optimistic flip — listeners notified immediately
  const wasPraying = applyPrayingFlip(prayerId)

  if (!isBackendPrayerWallEnabled()) {
    // Flag-off mode: write to localStorage, return previous value
    writeToStorage(getCache())
    return wasPraying
  }

  // Flag-on mode: read-swap, NOT dual-write — do NOT writeToStorage here.
  // Pending-mutation guard prevents double-tap races (D7).
  const key = `${prayerId}:praying`
  if (pendingMutations.has(key)) {
    // Roll back the optimistic flip — second tap is a no-op
    applyPrayingFlip(prayerId)
    return wasPraying
  }
  pendingMutations.add(key)

  // Fire-and-forget the API call. Note we do NOT await — the function
  // remains synchronous for consumers per Plan-Time Divergence #1.
  void prayerWallApi.toggleReaction(prayerId, 'praying')
    .then((result) => {
      // D8: server-state-disagreement case → forceState
      const serverIsPraying = result.state === 'added'
      const cachedIsPraying = getCache()[prayerId]?.isPraying ?? false
      if (serverIsPraying !== cachedIsPraying) {
        forceState(prayerId, { isPraying: serverIsPraying })
      }
    })
    .catch((err) => {
      // Rollback the optimistic flip
      applyPrayingFlip(prayerId)
      handleToggleError(err, 'togglePraying')
    })
    .finally(() => {
      pendingMutations.delete(key)
    })

  return wasPraying
}
```

## Plan-Time Divergences from Brief

| # | Decision | Reason | Reversible? |
|---|---|---|---|
| 1 | **Toggle functions remain synchronous-returning** (`togglePraying: (id) => boolean`, `toggleBookmark: (id) => void`, `toggleCandle: (id) => boolean`). API call is fire-and-forget. Brief D4 proposed `Promise<boolean>`. | Recon (`PrayerWall.tsx:264`, `PrayerWallDashboard.tsx:72`, `PrayerDetail.tsx:43`, `PrayerWallProfile.tsx:49`) shows all four consumer files use `wasPraying ? -1 : 1` synchronously to update `prayingCount`. A `Promise<boolean>` return would coerce to truthy in those expressions, silently inverting the count math (always `-1`). Brief MPD-2 ("Hook return shape genuinely unchanged") survives correctly only with synchronous returns. | Yes — could be reverted by changing return types, but the same consumer-fix would be required. |
| 2 | **`HYDRATION_FAILED` is a separate copy entry** (Step 1) rather than reusing `NETWORK_ERROR`. | Brief D11 + AC #14 calls for distinct copy ("couldn't refresh" vs "couldn't reach the server"). Adding a discrete copy entry preserves the existing `NETWORK_ERROR` semantics for action failures and keeps the taxonomy clean. | Yes |
| 3 | **Hydration-vs-mutation defer uses `setTimeout` polling (5s cap)** rather than a dedicated subscription mechanism. | Mutations settle inside Promise chains; a `setTimeout(0)` retry is the simplest available primitive. The 5s cap avoids unbounded retry if a pending mutation never settles (e.g., a network call that never errors). | Yes — could be reworked into a subscription on `pendingMutations` Set changes if this turns out to be flaky; flagged as a followup if observed in practice. |
| 4 | **`init(null)` notifies listeners with `notify()` after clearing cache** even though spec D2 doesn't explicitly require it. | If the user is on `/prayer-wall` when their session is invalidated, listeners (the hook → consumer components) should re-render with empty state immediately instead of showing the previous user's reactions until the next interaction. Sympathetic to the spec's intent. | Yes |
