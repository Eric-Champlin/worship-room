**Canonical changeset pattern** (read from `2026-04-25-003-create-activity-log-table.xml`, `2026-04-25-005-create-streak-state-table.xml`, `2026-04-25-006-create-user-badges-table.xml`):

- `<databaseChangeLog>` root with `xmlns` + `xsi`
- One `<changeSet id="<filename-without-xml>" author="worship-room">` per file
- Leading XML comment block: spec scope + permanence reminder
- `<createTable>` with explicit column types via `<column>` elements
- UUID primary keys: `type="UUID" defaultValueComputed="gen_random_uuid()"` with `<constraints primaryKey="true" nullable="false"/>` (single-column PK only)
- Composite primary keys: declared via separate `<addPrimaryKey>` element AFTER `<createTable>` (NOT inline) — pattern from `user_badges` and `activity_counts`
- `<addForeignKeyConstraint>` declared OUTSIDE `<createTable>`, never inline. `onDelete="CASCADE"` mandatory for FKs to `users(id)` per `05-database.md`
- Named CHECK constraints declared via `<sql>ALTER TABLE ... ADD CONSTRAINT ... CHECK (...)</sql>` blocks (NOT inline `checkConstraint=` attribute) — pattern from `streak_state` and `users`
- `<rollback>` block with `<dropTable tableName="..."/>` — drops cascade transitively to indexes, FKs, CHECKs, UNIQUEs
- TIMESTAMP columns: `TIMESTAMP WITH TIME ZONE` with `defaultValueComputed="NOW()"` (project standard, overrides Decision 8's plain `TIMESTAMP`)
- JSONB: `type="JSONB"` with no default
- Comparison operators in `<sql>` use `&gt;` / `&lt;` (XML-encoded)

**Master.xml pattern** (read from current file): explicit `<include file="..."/>` per changeset in chronological order. The `contexts/dev-seed.xml` entry stays last. Append-only — never edit existing entries (Liquibase checksum failure on deploy).

**LiquibaseSmokeTest pattern** (read from `backend/src/test/java/com/worshiproom/db/LiquibaseSmokeTest.java`):

- Extends `AbstractIntegrationTest` (full Spring context, singleton Postgres container via `TestContainers`)
- `@Autowired JdbcTemplate jdbcTemplate` for raw queries
- Table-existence tests: query `information_schema.columns` ordered by `ordinal_position`, assert `hasSize(N)`, assert `extracting("column_name").containsExactly(...)`, spot-check 2–3 specific column types via `data_type` lookup
- Composite PK tests: query `information_schema.table_constraints` joined to `key_column_usage` filtered to `constraint_type='PRIMARY KEY'`, assert `containsExactly(...)` on column names
- UNIQUE constraint tests: same pattern, filter `constraint_type='UNIQUE'`
- Changelog row test: query `databasechangelog ORDER BY orderexecuted`, assert `hasSize(N)`, then sequential row-by-row assertions on `id`/`author`/`filename` suffix
- CHECK rejection tests (parameterized): `@ParameterizedTest @CsvSource({...})` enumerating constraint names; switch statement maps each name to its INSERT
- `@AfterEach cleanupCheckTestUsers()` deletes users with `email LIKE 'check-test-%'` — child rows clean up via `ON DELETE CASCADE`. New tests that insert users SHOULD use the `check-test-` email prefix to participate in this cleanup
- Existing parameterized CHECK rejection helper `insertTestUser(String emailSuffix)` returns a `UUID`; reuse it

**No JPA entities, services, controllers, or DTOs in this spec.** Phase 2.5.2 owns those. The package `com.worshiproom.friends/` does NOT yet exist and will not be created here.

## Edge Cases & Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Plain `TIMESTAMP` (Decision 8) vs `TIMESTAMP WITH TIME ZONE` | `TIMESTAMP WITH TIME ZONE` everywhere | Project standard per `05-database.md` § "Liquibase Seed Data & Value Patterns"; matches users + activity_log + streak_state. Prevents JVM-timezone drift. |
| `friend_requests.message` type | `VARCHAR(280)`, NULL allowed | Matches mood-text limit elsewhere in the app; bounded payload prevents abuse. NULL allowed because sender may opt out of message. Decision 8 says `TEXT NULL`; tightening per spec divergence #3. |
| CHECK constraints inline vs named via `<sql>` | Named via `<sql>` blocks | Project convention per Specs 1.3 + 2.1. Names enable targeted ALTER/DROP and produce readable test failure messages (`hasMessageContaining("friend_relationships_status_check")`). |
| `friend_relationships` composite PK declaration | Separate `<addPrimaryKey>` element AFTER `<createTable>` | Pattern from `user_badges` (Spec 2.1) and `activity_counts` (Spec 2.1). Avoids ambiguity of inline PK on multiple columns. |
| `friend_requests` UNIQUE constraint | FULL UNIQUE on `(from_user_id, to_user_id)`, NOT partial | Decision 8 line 1180 specifies full UNIQUE. Anti-harassment policy: a user who declined once should not have to keep declining the same sender. The other party can send a fresh request — their pair is distinct. |
| `friend_relationships.status` enum values | `'active'` and `'blocked'` ONLY (NO `'accepted'`) | Decision 8 line 1146. The two tables use different vocabularies intentionally: `friend_requests` has a lifecycle (`pending`/`accepted`/`declined`/`cancelled`); `friend_relationships` has a state (`active`/`blocked`). |
| `milestone_events.occurred_at` vs `created_at` | `occurred_at` | Decision 8 line 3263. Events HAPPEN at a moment; interactions are CREATED. `social_interactions` uses `created_at` per line 3247. Asymmetry is intentional. |
| `friend_requests.responded_at` vs `resolved_at` | `responded_at` | Decision 8 line 1178 names it `responded_at`. NULL while status is `pending`. |
| Mutuality enforcement (A↔B both rows on accept) | Service-layer (Spec 2.5.2), NOT schema | Decision 8 specifies `FriendsService.acceptRequest` does the dual INSERT inside a `@Transactional`. No CHECK across rows, no trigger, no FK cycle. |
| Cross-user weekly-recap aggregator index `(occurred_at DESC)` on `milestone_events` | Deferred to Phase 13 | Master plan line 3266 mentions it; Phase 13 spec that queries it owns it. Adding it speculatively wastes write bandwidth. |
| Liquibase changeset granularity | Four files, one per table | Discipline pattern from Spec 2.1 (5 separate files for 5 tables). Future spec can roll back ONE table independently if a flaw surfaces. Code review easier. |
| Liquibase contexts on new changesets | None (no `context="dev"`, no `context="test"`) | Tables must materialize in dev, test, AND prod. Contexts reserved for seed-data per `05-database.md`. |
| Seed data | None | Phase 2.5 has no friend or social-interaction seed data. The 5 dev-seed users from Spec 1.8 exist; no friendships seeded between them. Spec 2.5.2 may decide to add fixtures — its call. |
| Smoke test base class | `AbstractIntegrationTest` (existing pattern, unchanged) | The current `LiquibaseSmokeTest` extends `AbstractIntegrationTest`. Splitting tests across `AbstractDataJpaTest` would fragment. Keep the existing base. |
| CHECK rejection test pattern | Single new `@ParameterizedTest` method modeled on existing `checkConstraintRejectsInvalidValue` | Reduces test method count, matches established pattern. UNIQUE blocking tests stay as standalone `@Test` methods because they need multi-statement setup (insert pending → update declined → re-insert). |

**Objective:** Create the `friend_relationships` table — symmetric mutual friendship + unidirectional block storage.

**Files to create:**

**Details:**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<databaseChangeLog xmlns="http://www.liquibase.org/xml/ns/dbchangelog"
  xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
  xsi:schemaLocation="http://www.liquibase.org/xml/ns/dbchangelog
    http://www.liquibase.org/xml/ns/dbchangelog/dbchangelog-latest.xsd">

  <!-- Forums Wave Spec 2.5.1 — `friend_relationships` per master plan Decision 8.
       First of four Phase 2.5 schema changesets. Mutual friendships are stored as
       two rows (A→B and B→A inserted by Spec 2.5.2's FriendsService.acceptRequest
       inside one @Transactional). Block is unidirectional: blocker → blocked,
       single row, status='blocked'. The CHECK on status forbids 'accepted' here
       — that value belongs to friend_requests.status (different vocabulary). The
       reverse-direction index covers "who friended user Y?" queries; the forward
       direction is covered by the composite PK.
       Once this changeset runs, its MD5 is permanent in DATABASECHANGELOG;
       future schema changes go in new YYYY-MM-DD-NNN-*.xml changesets. -->
  <changeSet id="2026-04-27-009-create-friend-relationships-table" author="worship-room">

    <createTable tableName="friend_relationships">
      <column name="user_id" type="UUID">
        <constraints nullable="false"/>
      </column>
      <column name="friend_user_id" type="UUID">
        <constraints nullable="false"/>
      </column>
      <column name="status" type="VARCHAR(20)">
        <constraints nullable="false"/>
      </column>
      <column name="created_at" type="TIMESTAMP WITH TIME ZONE" defaultValueComputed="NOW()">
        <constraints nullable="false"/>
      </column>
    </createTable>

    <addPrimaryKey
      tableName="friend_relationships"
      constraintName="pk_friend_relationships"
      columnNames="user_id, friend_user_id"/>

    <addForeignKeyConstraint
      baseTableName="friend_relationships"
      baseColumnNames="user_id"
      constraintName="fk_friend_relationships_user"
      referencedTableName="users"
      referencedColumnNames="id"
      onDelete="CASCADE"/>

    <addForeignKeyConstraint
      baseTableName="friend_relationships"
      baseColumnNames="friend_user_id"
      constraintName="fk_friend_relationships_friend_user"
      referencedTableName="users"
      referencedColumnNames="id"
      onDelete="CASCADE"/>

    <!-- Named CHECK: status must be 'active' or 'blocked' (NOT 'accepted' —
         that value belongs to friend_requests.status). -->
    <sql>
      ALTER TABLE friend_relationships ADD CONSTRAINT friend_relationships_status_check
      CHECK (status IN ('active', 'blocked'));
    </sql>

    <!-- Named CHECK: a user cannot friend or block themselves. -->
    <sql>
      ALTER TABLE friend_relationships ADD CONSTRAINT friend_relationships_no_self_reference
      CHECK (user_id &lt;&gt; friend_user_id);
    </sql>

    <!-- Reverse-direction index supports "who friended user Y?" / "who blocked
         user Y?" queries. The composite PK covers the forward direction. -->
    <createIndex tableName="friend_relationships" indexName="idx_friend_relationships_friend_user">
      <column name="friend_user_id"/>
    </createIndex>

    <rollback>
      <dropTable tableName="friend_relationships"/>
    </rollback>

  </changeSet>
</databaseChangeLog>
```

### Step 2: Create `friend_requests` changeset

**Objective:** Create the `friend_requests` table — request lifecycle storage with full UNIQUE anti-harassment constraint.

**Files to create:**
- `backend/src/main/resources/db/changelog/2026-04-27-010-create-friend-requests-table.xml`

**Details:**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<databaseChangeLog xmlns="http://www.liquibase.org/xml/ns/dbchangelog"
  xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
  xsi:schemaLocation="http://www.liquibase.org/xml/ns/dbchangelog
    http://www.liquibase.org/xml/ns/dbchangelog/dbchangelog-latest.xsd">

  <!-- Forums Wave Spec 2.5.1 — `friend_requests` per master plan Decision 8.
       Second of four Phase 2.5 schema changesets. Tracks the lifecycle of a
       friend request: pending → accepted/declined/cancelled. responded_at is
       NULL while pending; set when status transitions out of pending.
       FULL UNIQUE on (from_user_id, to_user_id) — not partial. Anti-harassment
       policy: once A sends a request to B, A cannot send another regardless of
       outcome. B can still send a fresh request to A (different pair). The
       UNIQUE constraint creates its own index, so no separate <createIndex>
       for (from_user_id, to_user_id) lookups. The two indexes below cover
       per-user list-by-status queries.
       Once this changeset runs, its MD5 is permanent in DATABASECHANGELOG;
       future schema changes go in new YYYY-MM-DD-NNN-*.xml changesets. -->
  <changeSet id="2026-04-27-010-create-friend-requests-table" author="worship-room">

    <createTable tableName="friend_requests">
      <column name="id" type="UUID" defaultValueComputed="gen_random_uuid()">
        <constraints primaryKey="true" nullable="false"/>
      </column>
      <column name="from_user_id" type="UUID">
        <constraints nullable="false"/>
      </column>
      <column name="to_user_id" type="UUID">
        <constraints nullable="false"/>
      </column>
      <column name="status" type="VARCHAR(20)" defaultValue="pending">
        <constraints nullable="false"/>
      </column>
      <column name="message" type="VARCHAR(280)"/>
      <column name="created_at" type="TIMESTAMP WITH TIME ZONE" defaultValueComputed="NOW()">
        <constraints nullable="false"/>
      </column>
      <column name="responded_at" type="TIMESTAMP WITH TIME ZONE"/>
    </createTable>

    <addForeignKeyConstraint
      baseTableName="friend_requests"
      baseColumnNames="from_user_id"
      constraintName="fk_friend_requests_from_user"
      referencedTableName="users"
      referencedColumnNames="id"
      onDelete="CASCADE"/>

    <addForeignKeyConstraint
      baseTableName="friend_requests"
      baseColumnNames="to_user_id"
      constraintName="fk_friend_requests_to_user"
      referencedTableName="users"
      referencedColumnNames="id"
      onDelete="CASCADE"/>

    <!-- FULL UNIQUE (Decision 8 line 1180). Once A → B request exists in any
         state, A cannot send another to B. -->
    <addUniqueConstraint
      tableName="friend_requests"
      columnNames="from_user_id, to_user_id"
      constraintName="friend_requests_unique_sender_recipient"/>

    <!-- Named CHECK: status enum. -->
    <sql>
      ALTER TABLE friend_requests ADD CONSTRAINT friend_requests_status_check
      CHECK (status IN ('pending', 'accepted', 'declined', 'cancelled'));
    </sql>

    <!-- Named CHECK: a user cannot send a request to themselves. -->
    <sql>
      ALTER TABLE friend_requests ADD CONSTRAINT friend_requests_no_self_reference
      CHECK (from_user_id &lt;&gt; to_user_id);
    </sql>

    <!-- Recipient's pending-incoming-requests query. -->
    <createIndex tableName="friend_requests" indexName="idx_friend_requests_to_user_status">
      <column name="to_user_id"/>
      <column name="status"/>
    </createIndex>

    <!-- Sender's pending-outgoing-requests query. -->
    <createIndex tableName="friend_requests" indexName="idx_friend_requests_from_user_status">
      <column name="from_user_id"/>
      <column name="status"/>
    </createIndex>

    <rollback>
      <dropTable tableName="friend_requests"/>
    </rollback>

  </changeSet>
</databaseChangeLog>
```

**Guardrails (DO NOT):**
- DO NOT make the UNIQUE partial (no `WHERE status = 'pending'`). Decision 8 line 1180 mandates FULL UNIQUE.
- DO NOT name the column `resolved_at` — Decision 8 line 1178 names it `responded_at`. Easy to typo.
- DO NOT use `TEXT` for `message` — spec divergence #3 tightens to `VARCHAR(280)`.
- DO NOT add a separate `<createIndex>` on `(from_user_id, to_user_id)` — the UNIQUE constraint creates its own backing index automatically.

**Test 8 — Parameterized cascade-on-user-delete (one method, 4 cases):**

```java
@ParameterizedTest(name = "deleting user cascades rows from {0}")
@CsvSource({
    "friend_relationships",
    "friend_requests",
    "social_interactions",
    "milestone_events"
})
void phase25TableCascadesOnUserDelete(String tableName) {
    UUID userA = insertTestUser("cascade-" + tableName + "-a");
    UUID userB = insertTestUser("cascade-" + tableName + "-b");

    switch (tableName) {
        case "friend_relationships" ->
            jdbcTemplate.update(
                "INSERT INTO friend_relationships (user_id, friend_user_id, status) " +
                "VALUES (?, ?, ?)",
                userA, userB, "active");
        case "friend_requests" ->
            jdbcTemplate.update(
                "INSERT INTO friend_requests (from_user_id, to_user_id, status) " +
                "VALUES (?, ?, ?)",
                userA, userB, "pending");
        case "social_interactions" ->
            jdbcTemplate.update(
                "INSERT INTO social_interactions (from_user_id, to_user_id, interaction_type) " +
                "VALUES (?, ?, ?)",
                userA, userB, "encouragement");
        case "milestone_events" ->
            jdbcTemplate.update(
                "INSERT INTO milestone_events (user_id, event_type) VALUES (?, ?)",
                userA, "level_up");
        default ->
            throw new IllegalStateException("Unhandled table: " + tableName);
    }

    // Sanity: row exists before delete.
    String userColumn = tableName.equals("milestone_events") ? "user_id" : "from_user_id";
    if (tableName.equals("friend_relationships")) {
        userColumn = "user_id";
    }
    Integer beforeCount = jdbcTemplate.queryForObject(
        "SELECT COUNT(*) FROM " + tableName + " WHERE " + userColumn + " = ?",
        Integer.class, userA
    );
    assertThat(beforeCount).isEqualTo(1);

    // Delete userA — cascade should remove all rows referencing userA.
    jdbcTemplate.update("DELETE FROM users WHERE id = ?", userA);

    Integer afterCount = jdbcTemplate.queryForObject(
        "SELECT COUNT(*) FROM " + tableName + " WHERE " + userColumn + " = ?",
        Integer.class, userA
    );
    assertThat(afterCount).isEqualTo(0);
}
```

**Note on the cascade test's `userColumn` resolution:** `friend_relationships` uses `user_id` for the "owner" side; `friend_requests` and `social_interactions` use `from_user_id`; `milestone_events` uses `user_id`. The conditional logic captures this asymmetry. The cascade verifies the FK targeting `userA` removes the row; the row referencing `userB` is untouched (separate row, not in the COUNT for this test).
