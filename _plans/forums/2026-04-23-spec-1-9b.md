## Affected Frontend Routes

Cross-cutting design-system spec. The audit inspects components that render across most of the app; the `/verify-with-playwright` pass captures new-or-modified primitives in their rendered states at 375×667, 768×1024, and 1440×900 viewports. Primary consumer routes used for capture:

- `/register` — `FormField` field-error + `aria-describedby` wiring; Button disabled/loading; form-level error surface
- `/daily` — `Toast` / `WhisperToast` callers; `DailyHubSkeleton`; tab-content empty states
- `/prayer-wall` — `FeatureEmptyState`, `Toast`, `PrayerWallSkeleton`
- `/friends` — `FriendsSkeleton`, `FeatureEmptyState` (no-friends state)
- `/insights` — `ChartFallback`, `InsightsSkeleton`, chart empty states
- `/bible` — `BibleBrowserSkeleton`, route-level `ChunkErrorBoundary` fallback
- `/bible/:book/:chapter` — `BibleReaderSkeleton`, inline AI Explain/Reflect error states

**Component-in-isolation capture strategy:** LoadingSpinner and FormError (the likely new primitives) don't have dedicated demo routes. Strategy per spec line 21: capture them in their chosen consumer contexts by forcing the loading/error state via Playwright — e.g., click Register submit with empty fields to force field errors, intercept the registration network call and reject it to force a form-level error. No temporary demo harness is added; the spec is explicit about avoiding Storybook.

### Confirmed gaps (to address in this spec)

1. **Button lacks `isLoading` prop** → add prop that renders an inline spinner, sets `aria-busy="true"` + `aria-disabled="true"`, preserves button width so layout doesn't jump. (Modifies `Button.tsx`.)
2. **No form-level error banner** → create `FormError.tsx` primitive for "your submission failed" banners above a form (distinct from per-field `FormField` errors). `role="alert"` + `aria-live="assertive"` by default; configurable via prop.
3. **No standalone loading spinner** → create `LoadingSpinner.tsx` for contexts where skeleton is infeasible (button interior, tiny inline indicator, modal content while fetching). `role="status"` + sr-only default label `"Loading"`.
4. **`ChartFallback` lacks `role="status"`** → minor a11y fix in Step 3.
5. **`ErrorBoundary.tsx` default fallback copy is corporate-apologetic** → Step 4 rewrites the body to warmer, blameless language. No `role="alert"` on the fallback div → Step 5 adds it.

**`LoadingSpinner.tsx` public API:**
```tsx
interface LoadingSpinnerProps {
  size?: number           // px, default 18
  className?: string      // merges with defaults
  label?: string          // sr-only label, default "Loading"
}
```

Implementation: SVG circle with `stroke-dasharray` rotation, `motion-safe:animate-spin`, `role="status"`, sr-only `<span>{label}</span>`. Reduced-motion fallback: static circle at `opacity-60`. Uses `currentColor` stroke so it inherits from parent text color. Imports NO animation ms value — relies on Tailwind's `animate-spin` keyframe (1000ms default, documented-tokens-adjacent since it's not in BB-33 tokens but is a platform-standard CSS-level spinner that matches `skeletons/*`'s `motion-safe:animate-shimmer` precedent).

**`Button.tsx` `isLoading` prop:**

```tsx
interface ButtonProps extends ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: 'primary' | 'secondary' | 'outline' | 'ghost' | 'light'
  size?: 'sm' | 'md' | 'lg'
  asChild?: boolean
  isLoading?: boolean   // NEW
}
```

When `isLoading` is true:
- `disabled` is forced `true`
- `aria-busy="true"` set on the button element
- `aria-disabled="true"` set (redundant with `disabled` but screen-reader-friendly per some older ATs)
- Internal layout: visible children get `opacity-0` (preserves width); a centered `<LoadingSpinner size={sizeForButton}>` overlays. When `asChild`, isLoading is ignored + a dev-mode console warning fires (because `asChild` passes props to a non-button consumer where layout assumptions break).

Size mapping: `sm`→16px, `md`→18px, `lg`→20px spinner.

- DO NOT allow a click to fire while `isLoading`; enforce via both the disabled attribute AND an early return in a merged onClick if consumers supply one (defensive — React disabled attribute should suffice).
- DO NOT use hardcoded `ms` strings in LoadingSpinner's rotation — rely on `animate-spin` Tailwind utility.
- DO NOT couple LoadingSpinner to Button — it is reusable standalone.

## Anti-Pressure Design Decisions (preserved from spec)

1. **Skeleton over spinner by default.** Standalone `LoadingSpinner` exists only for contexts where a skeleton is infeasible. Documented in Step 8's Decision Tree.
2. **`prefers-reduced-motion` static opacity-60 fallback.** Applied to LoadingSpinner and verified on skeletons. No new motion tokens introduced.
3. **Error severity without emergency-red.** FormError's `error` severity uses muted `red-950/30 bg` + `red-100 text` + `red-400/30 border` — NOT pure red.
4. **`aria-live="polite"` by default, `assertive` a sharp edge.** FormError's `error` default is `assertive` (submission failure IS urgent); `warning` and `info` are `polite`. Documented in Step 8's rule file.
5. **Empty states affirm absence as valid.** Good/bad paired examples included in Step 8's Copy Guidelines.
6. **Focus-on-error for forms.** Documented pattern in Step 8's a11y checklist. Consuming spec (1.9 for AuthModal) implements.
