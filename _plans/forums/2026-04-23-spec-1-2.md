## Edge Cases & Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Where to place `spring.datasource.*` config | `application-dev.properties` (dev only) | The base `application.properties` has zero datasource config today. Prod datasource comes from `DATABASE_URL` env var via Spec 1.10b. Test profile gets its own datasource via Testcontainers in Spec 1.7. Dev-only placement keeps the base file database-agnostic and avoids the need for a prod override. |
| Spring Boot JDBC driver property | Omit `spring.datasource.driver-class-name` | Spring Boot auto-resolves the driver from the JDBC URL (`jdbc:postgresql://...` ‚Üí `org.postgresql.Driver`). Explicit setting adds noise with no benefit. |
| Maven dependency scope for `org.postgresql:postgresql` | Default (compile) scope | Matches what Spring Initializr generates. `runtime` scope is marginally more correct but complicates IDE indexing for a non-benefit. The Spring Boot BOM manages the version ‚Äî no `<version>` tag needed. |
| PostgreSQL version pin | `postgres:16-alpine` | Matches `03-backend-standards.md` tech stack. `16` is the current major LTS. Alpine is standard for size. The master plan warns against `latest` explicitly. |
| Dev database credentials | `worshiproom` / `worshiproom` / `worshiproom_dev` (user/password/db) | Local-only database, never exposed outside `localhost`. Password in plaintext in `application-dev.properties` is acceptable because the database never accepts external connections. Production creds come from env vars per Spec 1.10b. README will state this explicitly. |
| Named volume name | `worshiproom_postgres_data` | Project-scoped to avoid collision with other Docker projects on Eric's machine. Compose auto-prefixes with project name (`worship-room_worshiproom_postgres_data`), but the internal name staying project-meaningful aids manual inspection. |
| Healthcheck binary | `pg_isready -U worshiproom -d worshiproom_dev` | `pg_isready` IS bundled in official `postgres` images (unlike `curl` in Alpine JRE, which caused Spec 1.1's healthcheck failure). Cannot repeat the Alpine-missing-curl bug here. |
| Healthcheck timing | `interval: 5s, timeout: 3s, retries: 5, start_period: 10s` | Matches spec Note #4 example. `start_period` of 10s gives postgres cold-boot time before healthcheck failures count toward retries. Typical first-boot readiness is 5‚Äì10s. |
| `depends_on` wiring | `backend.depends_on.postgres.condition: service_healthy` | Gates backend boot on postgres health, preventing HikariCP connection-refused race at cold start. Frontend's existing `depends_on: backend: service_healthy` stays unchanged ‚Äî chain is now frontend ‚Üí backend ‚Üí postgres. |
| Update `backend/.env.example`? | **No.** | This spec introduces no new env vars. Dev creds are hard-coded in `application-dev.properties`. If a future spec adds `DATABASE_URL` for prod, that spec updates `.env.example`. |
| Update `.gitignore`? | No, unless a `data/` or `pgdata/` directory appears | Named Docker volumes live in Docker's storage area, not the project tree. If one appears in the project directory, that indicates a bind-mount misconfiguration ‚Äî fix the compose file, don't gitignore the leak. |

### Step 1: Add PostgreSQL service to `docker-compose.yml`

**Objective:** Stand up a local PostgreSQL 16 container with a healthcheck, named volume for persistence, and explicit port mapping. Gate `backend` boot on postgres health.

**Files to modify:**
- `docker-compose.yml` ‚Äî add `postgres` service block; add `depends_on` block under `backend`; add top-level `volumes:` section

**Details:**

Add the `postgres` service (insert before `backend` to make the service graph read top-down):

```yaml
  postgres:
    image: postgres:16-alpine
    ports:
      - "5432:5432"
    environment:
      POSTGRES_DB: worshiproom_dev
      POSTGRES_USER: worshiproom
      POSTGRES_PASSWORD: worshiproom
    volumes:
      - worshiproom_postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U worshiproom -d worshiproom_dev"]
      interval: 5s
      timeout: 3s
      retries: 5
      start_period: 10s
```

Add a `depends_on` block under the existing `backend` service (before `healthcheck:`):

```yaml
    depends_on:
      postgres:
        condition: service_healthy
```

Add a top-level volumes declaration at the end of the file:

```yaml
volumes:
  worshiproom_postgres_data:
```

Preserve every other line of `docker-compose.yml` exactly as-is ‚Äî do NOT touch the existing `backend.healthcheck` (the broken `curl`-in-Alpine bug is explicitly out of scope per spec Out-of-scope item 5) and do NOT touch the `frontend` service's existing `depends_on: backend: service_healthy`.
