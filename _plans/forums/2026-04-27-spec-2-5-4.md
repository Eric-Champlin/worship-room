### B) The existing `apiFetch` client

`frontend/src/lib/api-client.ts` ships `apiFetch<T>(path, options): Promise<T>`:
- Auto-prepends `VITE_API_BASE_URL` (line 44)
- Auto-attaches `Authorization: Bearer <token>` from `auth-storage.getStoredToken()` unless `skipAuth: true` (lines 61–64)
- 30s default timeout via `AbortController`
- Parses `{ data, meta }` envelope; returns `data` typed (line 136)
- Throws `ApiError` on non-2xx (line 131)
- 401 on authenticated request → clears stored token, dispatches `AUTH_INVALIDATED_EVENT` (lines 119–129)
- Network error → throws `ApiError('NETWORK_ERROR', 0, ...)` (lines 81–87)

The success envelope strip at line 136 means `friends-api.ts`'s `sendFriendRequestApi` callers see the `FriendRequestDto` fields directly (including the backend's UUID `id`), not a wrapped `{data, meta}` shape. This validates the spec's Watch-For #4.

### C) The AuthContext AUTH_INVALIDATED_EVENT handler — Watch-For #1 confirmed

`frontend/src/contexts/AuthContext.tsx:283–285`:

```ts
function handleInvalidated() {
  clearStoredLegacyAuth()
  setState({ isAuthenticated: false, isAuthResolving: false, user: null })
}
```

This handler **does NOT spare legacy simulated-auth users.** It calls `clearStoredLegacyAuth()` (which removes `wr_auth_simulated` and `wr_user_name`) AND sets `isAuthenticated: false`. So if a simulated-auth user's dual-write fires and apiFetch returns 401 (because no `Authorization` header was attached — `getStoredToken()` returned null), the AUTH_INVALIDATED dispatch will silently log them out.

**Conclusion:** `shouldDualWrite()` MUST gate on BOTH `isBackendFriendsEnabled()` AND `getStoredToken() !== null`. The simpler `isAuthenticated && isBackendFriendsEnabled()` pattern would unintentionally log out simulated-auth users in dev when the flag is flipped on. Watch-For #1 is mandatory, not defensive.

### D) Existing TypeScript types from Spec 2.5.3

`frontend/src/types/api/friends.ts` (already shipped) exports the canonical types we'll consume:

- `FriendRequestDto` — backend's response shape for POST `/users/me/friend-requests` and GET listings. **`id` field is the backend UUID we capture for the local-to-backend mapping.**
- `SendFriendRequestRequest` — `{ toUserId: string, message?: string | null }`

### F) Existing `friends-storage.ts` pattern

`frontend/src/services/friends-storage.ts` is a flat collection of pure operation functions (lines 60–135): `sendFriendRequest`, `acceptFriendRequest`, `declineFriendRequest`, `cancelOutgoingRequest`, `removeFriend`, `blockUser`, plus query helpers `isBlocked`/`isFriend`/`hasPendingRequest`. All take `data: FriendsData` and return a new `FriendsData`. No side effects, no localStorage in the operation functions themselves — `saveFriendsData(newData)` is the explicit persist step.

The new `attachBackendId(data, toUserId, backendId)` function follows this pattern exactly: take data + scalars, return new data, no I/O.

The match-criterion is **the most-recently-sent outgoing request to `toUserId` that has no `backendId` yet**. Since `sendFriendRequest` already prevents duplicate outgoing requests to the same user (line 75–80 in `friends-storage.ts`), the "most recent matching" criterion collapses to "the unique outgoing request to `toUserId`." Implementation uses `.map` to update only matching entries with `!req.backendId`, leaving already-mapped entries untouched (idempotency).

### J) Spec 2.7's canonical pattern in `useFaithPoints.ts:287-305`

The shipped Spec 2.7 dual-write block:

```ts
if (isBackendActivityEnabled()) {
  // ... backfill trigger ...
  postActivityToBackend(type, sourceFeature).catch((err) => {
    console.warn('[useFaithPoints] backend dual-write failed:', err);
  });

```

NOTE: Spec 2.7 uses ONLY `isBackendActivityEnabled()` — NO `getStoredToken() !== null` guard. Why is this safe for activity but not for friends? Because activity dual-write fires from `useFaithPoints` which is itself auth-gated (`if (!isAuthenticated) return` at line 171); when `wr_auth_simulated` is true, the legacy fallback in AuthContext (`mirrorToLegacyKeys` keeps `wr_auth_simulated='true'` AND `wr_jwt_token` set in the same login flow) means simulated-auth in dev is paired with a JWT. **However**, that pairing is a Phase 1.9 transitional pattern; if a future scenario produces simulated-auth without a JWT (legacy localStorage state from before Spec 1.9 + flag flipped on), the activity dual-write would have the same bug. Watch-For #1 protects against this scenario for friends; activity has the same vulnerability documented but inherits the existing pattern. **Do not "fix" Spec 2.7 in this spec** — that's its own followup spec.

The error log prefix is `[useFaithPoints] backend dual-write failed:`. The friends prefix is `[useFriends] backend {operation} dual-write failed:` per the spec's Watch-For #12.

## Spec-Category-Specific Guidance

This is a **DUAL-WRITE spec** (Phase 2.5 migration pattern). Required step structure (mirror Spec 2.7):

1. **localStorage writer preservation** — every existing `storage*` operation function in `friends-storage.ts` continues to be called unchanged.
2. **Backend shadow writer** — `friends-api.ts` exports four functions wrapping `apiFetch`. Each `useFriends` mutation calls the relevant function fire-and-forget AFTER `persist(...)`. Failures are caught via `.catch(console.warn)`.
3. **Feature flag wiring** — `VITE_USE_BACKEND_FRIENDS` (default `false`) gates the backend dispatch via `isBackendFriendsEnabled()`. Cutover (flag default flip to `true` in `.env.example`) is Spec 2.5.5, NOT this spec.
4. **No drift detection test** — Phase 2.5 doesn't have a dual-write parity test like Phase 2's `activity-drift.test.ts` (master plan Decision 12 mandates drift detection only for activity engine; friends data has no drift-relevant client-side calculations because the backend re-derives `FriendDto.weeklyPoints` from `activity_log`). Skip without a followup.
5. **Step order** — env flag first → storage operation function + types → API client → useFriends wiring. Each step independently verifiable.

| Decision | Choice | Rationale |
|----------|--------|-----------|
| `shouldDualWrite()` guard | Both `isBackendFriendsEnabled()` AND `getStoredToken() !== null` | AuthContext's `handleInvalidated` clobbers simulated-auth state; gating on flag alone would log out simulated users when flag flipped on. Spec 2.7 uses flag-only because Phase 1.9's `mirrorToLegacyKeys` pairs simulated-auth with a JWT; same vulnerability exists but is a Spec 2.7 followup, not a Spec 2.5.4 fix. |

The `!req.backendId` filter ensures idempotency under double-callback firing. |
| Number of API client functions | 4, not 6 | accept/decline/cancel collapse into `respondToFriendRequestApi(id, action)` because the backend's PATCH dispatches by `action` field. No thin-wrapper helpers — `useFriends` callers pass the action directly. |

```ts
/**
 * Attaches the backend's UUID to the outgoing friend request matching `toUserId`,
 * if and only if that request does not already have a `backendId`. Idempotent
 * against double-callback firing — a second call with the same `toUserId` is
 * a no-op once `backendId` is set. Spec 2.5.4 (Frontend Friends Dual-Write).
 */
export function attachBackendId(
  data: FriendsData,
  toUserId: string,
  backendId: string,
): FriendsData {
  return {
    ...data,
    pendingOutgoing: data.pendingOutgoing.map((req) =>
      req.to.id === toUserId && !req.backendId ? { ...req, backendId } : req,
    ),
  }
}
```

```ts
/**
 * Friends API client — Spec 2.5.4 (Frontend Friends Dual-Write).
 *
 * Wraps the Spec 2.5.3 friend-mutation endpoints as fire-and-forget calls
 * from useFriends. localStorage stays the source of truth for reads; the
 * backend receives a shadow copy. Failures are logged at the call site
 * (console.warn) and never block UX.
 *
 * Authorization header is attached automatically by apiFetch from auth-storage.
 *
 * Shapes: imports from frontend/src/types/api/friends.ts (hand-typed, shipped
 * with Spec 2.5.3). When the openapi-typescript pipeline ships, the types
 * file regenerates and this module's imports remain stable.
 */

import { apiFetch } from '@/lib/api-client'
import type {
  FriendRequestDto,
  FriendRequestAction,
  SendFriendRequestRequest,
  RespondToFriendRequestRequest,
  BlockUserRequest,
} from '@/types/api/friends'

/**
 * POST /api/v1/users/me/friend-requests — send a friend request.
 * Returns the FriendRequestDto including the backend's UUID `id`, which the
 * caller (useFriends) attaches to the local FriendRequest via attachBackendId.
 */
export async function sendFriendRequestApi(
  toUserId: string,
  message: string | null,
): Promise<FriendRequestDto> {
  const body: SendFriendRequestRequest = { toUserId, message }
  return apiFetch<FriendRequestDto>('/api/v1/users/me/friend-requests', {
    method: 'POST',
    body: JSON.stringify(body),
  })
}

/**
 * PATCH /api/v1/friend-requests/{backendRequestId} — accept, decline, or cancel.
 * The backend dispatches on the `action` field; response body is discarded
 * (caller doesn't need confirmation — localStorage is the contract).
 */
export async function respondToFriendRequestApi(
  backendRequestId: string,
  action: FriendRequestAction,
): Promise<void> {
  const body: RespondToFriendRequestRequest = { action }
  await apiFetch<unknown>(`/api/v1/friend-requests/${backendRequestId}`, {
    method: 'PATCH',
    body: JSON.stringify(body),
  })
}

/**
 * DELETE /api/v1/users/me/friends/{friendUserId} — remove a friend.
 * No body. apiFetch's default Content-Type=application/json header is harmless
 * on a body-less DELETE; the backend ignores it.
 */
export async function removeFriendApi(friendUserId: string): Promise<void> {
  await apiFetch<void>(`/api/v1/users/me/friends/${friendUserId}`, {
    method: 'DELETE',
  })
}

/**
 * POST /api/v1/users/me/blocks — block a user. Idempotent on re-block.
 */
export async function blockUserApi(userId: string): Promise<void> {
  const body: BlockUserRequest = { userId }
  await apiFetch<void>('/api/v1/users/me/blocks', {
    method: 'POST',
    body: JSON.stringify(body),
  })
}
```

**Shape A — sendRequest (95–112)**: AFTER `persist(storageSendRequest(...))`, dual-write with `.then` callback that captures `backendId` via `attachBackendId`:

```ts
    if (!isAuthenticated || !user) return
```

**NOTE on the `.then` re-persist:** Reading `data` from the closure inside `.then` would use stale state. We pick (a) because `persist(storageAttachBackendId(...))` is the same shape as the rest of the hook, and `getOrInitFriendsData` is already imported. The race is acceptable per Watch-For #2 — duplicate sends are already storage-layer rejected.

**Shape B — respond mutations (acceptRequest, declineRequest, cancelRequest)**: Capture `backendId` BEFORE `persist(...)` (because `persist` removes the request from the array). Two early-skip cases — neither shouldDualWrite, or no backendId:

```ts

  (requestId: string) => {

    const request = data.pendingIncoming.find((r) => r.id === requestId)

        console.warn('[useFriends] backend acceptRequest dual-write failed:', err)

      )
    }
  },
  [isAuthenticated, data, persist],
)
```
