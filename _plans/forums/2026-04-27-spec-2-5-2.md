## Architecture Context

### Existing patterns confirmed by recon

- **Composite-PK entity pattern**: `com.worshiproom.activity.UserBadge` + `UserBadgeId` is the canonical reference. `@IdClass(...)`; both `@Id`-tagged fields named identically across both classes; ID class implements `Serializable` with value-based equals/hashCode; entity equals/hashCode is by primary-key fields (the composite). Manual getters/setters, no Lombok.
- **Single-PK entity pattern**: `com.worshiproom.user.User` and `com.worshiproom.activity.ActivityLog` use `@Entity` + `@Table` + manual getters/setters. ActivityLog uses `@Id` + caller-set `UUID.randomUUID()` in constructor (no `@GeneratedValue` because the column has `gen_random_uuid()` DB default but the entity sets it client-side). `User` has `@PrePersist` to set `id` if null. **Decision for our entities below: caller-set UUID in constructor matching `ActivityLog` (simpler, matches the most recent pattern).**
- **Wire-string enum pattern**: `ActivityType` uses `@JsonValue`/`@JsonCreator` + private `wireValue` field + static `fromWireValue()`. Used here because the JSON contract uses camelCase strings while Java enums are SCREAMING_SNAKE. **For our internal-only status enums (no wire-format requirement; the controller in 2.5.3 will accept enum-mode JSON if needed) the simpler `dbValue()`/`fromDbValue()` shape from `DisplayNamePreference` is sufficient.** Use a hybrid: lowercase string column value via `value()` + `fromValue()` static factory.
- **Service shape**: `ActivityService` uses constructor injection of repositories + sub-services, `@Transactional` per write method, throws domain exceptions caught by package-scoped `@RestControllerAdvice`. Direct `OffsetDateTime.now(ZoneOffset.UTC)` for timestamps — **NO `Clock` bean exists anywhere in the codebase** (the spec mentions Clock injection as a "Phase 2 pattern" but `grep -rn "Clock" backend/src/main/java/com/worshiproom/` returns zero hits — see Edge Case D1 below).
- **Exception pattern**: `UserException` is a single class with static factories + `HttpStatus` field + `code` field. `UserExceptionHandler` is `@RestControllerAdvice(basePackages = "com.worshiproom.user")` (package-scoped) and emits `ProxyError` from `com.worshiproom.proxy.common`. Controller-raised exceptions use package-scoped advice (filter-raised use unscoped advice — not relevant here). **The 2.5.2 spec asks for a per-error-code class hierarchy under `FriendsException`. Decision: follow the spec (multiple classes), not `UserException`'s single-class pattern, because there are 11 distinct error states and the per-class shape is more discoverable. The `FriendsException` base will carry the `HttpStatus` + `code` fields like `UserException` does.**
- **Native query pattern**: `BadgeRepository.incrementDisplayCount` uses `@Query(value = """...""", nativeQuery = true)` with text block. `ActivityCountsRepository` does the same. Spring Data projection interfaces (returning `List<FriendsListProjection>`) are not yet used elsewhere — **this spec introduces the pattern**.
- **Repository pattern**: `extends JpaRepository<Entity, IdType>`, `@Repository` annotation, JPQL `@Query` for joins, `@Modifying` + `@Query` for bulk delete.
- **Test pattern**: extends `AbstractIntegrationTest` (singleton Testcontainers), `@AutoConfigureMockMvc` for HTTP layer (NOT needed here — service-level tests), `@BeforeEach` calls `userRepository.deleteAll()` then re-seeds users (CASCADE cleans transitive children: friend_relationships, friend_requests, etc.). Tests use `@Nested` + `@DisplayName` for grouping. `JdbcTemplate` is autowired for direct DB row counts.
- **Liquibase already shipped**: `friend_relationships` (changeset 009), `friend_requests` (010), `social_interactions` (011), `milestone_events` (012). Tables, FKs (`ON DELETE CASCADE` on both `user_id` and `friend_user_id`), CHECK constraints (status enum, no-self-reference), indexes, and the FULL UNIQUE on `(from_user_id, to_user_id)` named `friend_requests_unique_sender_recipient` are all in place. **No changeset work in 2.5.2.**
- **`LevelThresholds` already provides the canonical level→name map**: `LevelThresholds.levelForPoints(points).name()` returns the 6 canonical names (Seedling/Sprout/Blooming/Flourishing/Oak/Lighthouse). **The spec's recommendation to create a separate `LevelNameLookup.java` is unnecessary — reuse `LevelThresholds`.**

## Edge Cases & Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **D1: Timestamp source for `respondedAt`** | Direct `OffsetDateTime.now(ZoneOffset.UTC)` (no `Clock` injection) | Spec mentions Clock as a "Phase 2 pattern" but `grep -rn "Clock"` in backend code returns zero hits. The activity engine (`ActivityService`, `ActivityBackfillService`, `FaithPoints`, `UserBadge`, `ActivityCount`) all call `OffsetDateTime.now(ZoneOffset.UTC)` directly. Introducing Clock here breaks the established pattern. If timestamp testability becomes a problem, a future infrastructure spec can introduce Clock across all services in one consistent sweep. |
| **D2: Level name source** | Reuse `LevelThresholds.levelForPoints(points).name()` from `com.worshiproom.activity.constants` | Already verified verbatim port of `frontend/src/constants/dashboard/levels.ts`. The drift parity test mentioned in spec § "Testing Notes" reduces to "the constant is already canonical; assert all 6 levels return expected names." No new `LevelNameLookup.java`. |
| **D3: `UserNotFoundException` and `InvalidInputException` reuse** | Create friends-specific copies under `com.worshiproom.friends`. Recon confirmed there are NO standalone classes by these names in `com.worshiproom.user` (UserException is a single class with static factories). | The spec instructs reuse "if it exists." It does not. Friends-specific copies live under `com.worshiproom.friends` extending `FriendsException`. A future spec can extract a shared `WorshipRoomNotFoundException` if more domains accumulate similar errors. |
| **D4: Composite PK pattern** | `@IdClass(FriendRelationshipId.class)` matching `UserBadge` + `UserBadgeId` shape | Per master plan v2.9 backend standards and existing precedent. Spec § "What to Watch For" item #1 makes this explicit. `@EmbeddedId` is the alternative — rejected for consistency. |
| **D5: Status enum string mapping** | Lowercase `value()` + static `fromValue(String)` mirroring `DisplayNamePreference.dbValue()`/`fromDbValue()` shape (NOT the `@JsonValue`/`@JsonCreator` pattern from `ActivityType`) | Status enums are stored as lowercase strings in DB CHECK constraints (`'active'`, `'blocked'`, `'pending'`, ...). No JSON wire format is needed in 2.5.2 (no controller). 2.5.3 can add `@JsonValue` later without breaking changes if it chooses to expose status as a string in JSON. |
| **D6: `searchUsers` query language** | JPQL with `@Query`, NOT native SQL | Search is a simple `LIKE` + NOT IN; JPQL handles it cleanly with `:q` parameter binding. Native SQL is reserved for the friends-list query where JPQL can't express the SUM windowed subquery cleanly. JPQL parameter binding is parameterized → no injection risk. |
| **D7: Block side-effect order** | (a) Delete friend_relationships rows in either direction, (b) delete pending friend_requests rows in either direction, (c) insert single block row | Per spec § "What to Watch For" item #6. Conceptually clean: undo prior state, then add the block. Order does not affect correctness but matches the spec recommendation. |
| **D8: `isEitherDirectionBlocked` re-check on accept** | Yes — checked again in `acceptRequest` even though `sendRequest` already checked | Per spec § "What to Watch For" item #7. Time gap between send and accept can be hours/days; a block placed in between must be honored. Test specifically exercises this rollback case. |
| **D9: Block idempotency** | `blockUser` returns silently (no-op) when target is already blocked | Per spec § Operation 9 validation step 3. Avoids spurious state changes and matches typical idempotent block-button UX. |
| **D10: `removeFriend` idempotency** | `removeFriend` throws `NotFriendsException` when not currently friends | Per spec § Operation 8 — caller "should know whether they were friends." Distinct from block, where idempotency is desirable. |
| **D11: Test class split** | `FriendsServiceTest` (send/accept/decline/cancel/remove + happy paths), `FriendsServiceBlockTest` (block/unblock/side-effects/search), `FriendsServiceListTest` (denormalized list-friends, list-incoming, list-outgoing, FK CASCADE) | Per spec § Files to Create. Three classes keep each ≤12 tests, matching existing test-class size norms. |
| **D12: Test base class** | All three test classes extend `AbstractIntegrationTest` (full Spring context) | The accept-flow's two-table transactional rollback semantics need the full context. `AbstractDataJpaTest` (slice) cannot verify rollback across multiple repositories cleanly. Per spec § "Testing Notes." |
| **D13: Mid-flight block test mechanism** | Use `@Transactional`-aware test helper that issues a direct `JdbcTemplate.update` to insert the block row between sendRequest and acceptRequest calls | Cleaner than a Mockito spy (which would require partial mocking the repository under test). The block row inserted via JdbcTemplate commits immediately because JdbcTemplate is outside the test's `@Transactional` scope. |
| **D14: `findFriendsListForUser` weekStart parameter** | Service accepts `OffsetDateTime weekStart` as a method parameter; controller (2.5.3) computes it from user's timezone | Service stays pure — no implicit timezone math. 2.5.3's controller is the natural owner of "week start in user TZ" computation since it has access to the request context. |
| **D15: `UserNotFoundException` semantics** | Throws when target user doesn't exist OR is `is_deleted=true` OR `is_banned=true` | Per spec § Operation 4 validation step 2. The exception is information-leak-equivalent to "user not found" — banned and deleted users return the same error so attackers can't enumerate ban status. |

### Step 1: Create `FriendRelationshipStatus` and `FriendRequestStatus` enums

**Objective:** Provide application-layer enum representations for the two status columns. Both DB columns are `VARCHAR(20)` with CHECK constraints; the enums map Java-side identifiers to the lowercase string values.

**Files to create:**
- `backend/src/main/java/com/worshiproom/friends/FriendRelationshipStatus.java` — values `ACTIVE("active")`, `BLOCKED("blocked")`
- `backend/src/main/java/com/worshiproom/friends/FriendRequestStatus.java` — values `PENDING("pending")`, `ACCEPTED("accepted")`, `DECLINED("declined")`, `CANCELLED("cancelled")`

**Details:**
Mirror `DisplayNamePreference`'s shape but include the explicit lowercase value via a private field (matching DB CHECK strings). Each enum exposes:
- `public String value()` — returns the lowercase DB string
- `public static <Type> fromValue(String)` — static factory parsing DB string → enum, throws `IllegalArgumentException("Unknown <type> value: " + value)` on miss; null input returns `IllegalArgumentException` (not silent default — these enums must round-trip cleanly)

```java
public enum FriendRelationshipStatus {
    ACTIVE("active"),
    BLOCKED("blocked");

    private final String value;

    FriendRelationshipStatus(String value) {
        this.value = value;
    }

    public String value() { return value; }

    public static FriendRelationshipStatus fromValue(String value) {
        if (value == null) {
            throw new IllegalArgumentException("FriendRelationshipStatus value must not be null");
        }
        for (FriendRelationshipStatus s : values()) {
            if (s.value.equals(value)) return s;
        }
        throw new IllegalArgumentException("Unknown FriendRelationshipStatus value: " + value);
    }
}
```

Same shape for `FriendRequestStatus` with its 4 values.

**Guardrails (DO NOT):**
- Do NOT use `@JsonValue` / `@JsonCreator` annotations — JSON wire-format is 2.5.3's concern. Keep enums clean of Jackson today.
- Do NOT reuse `ActivityType`'s `wireValue` naming — these are stored as DB strings, not wire strings (no JSON exposure today). Use `value()` to mirror `DisplayNamePreference.dbValue()`.
- Do NOT add `JPA @Convert` — the JPA `@Column` will receive a String; the entity field type is the enum and we use a Hibernate-supported `EnumType.STRING` mapping by default (or store as `String` field internally and convert in getter/setter — see Step 2 details).

**Verification:**
- [ ] Code compiles: `./mvnw compile`
- [ ] No new test class — enum behavior is exercised by Step 7+ tests through entity round-trip.

**Test specifications:**
None at this step (covered transitively by entity tests in later steps).

**Expected state after completion:**

### Step 2: Create `FriendRelationship` entity + `FriendRelationshipId` composite key

**Objective:** Map the `friend_relationships` table to a JPA entity with composite PK `(user_id, friend_user_id)`.

**Files to create:**
- `backend/src/main/java/com/worshiproom/friends/FriendRelationshipId.java` — composite PK class implementing `Serializable`, with `userId: UUID` and `friendUserId: UUID` fields, value-based equals/hashCode
- `backend/src/main/java/com/worshiproom/friends/FriendRelationship.java` — `@Entity @Table(name = "friend_relationships") @IdClass(FriendRelationshipId.class)`

**Details:**

`FriendRelationshipId.java` mirrors `UserBadgeId.java` exactly (same field-pair shape, same Serializable, same equals/hashCode-on-both-fields):

```java
public class FriendRelationshipId implements Serializable {
    private UUID userId;
    private UUID friendUserId;

    public FriendRelationshipId() {}

    public FriendRelationshipId(UUID userId, UUID friendUserId) {
        this.userId = userId;
        this.friendUserId = friendUserId;
    }

    public UUID getUserId() { return userId; }
    public UUID getFriendUserId() { return friendUserId; }

    public void setUserId(UUID userId) { this.userId = userId; }
    public void setFriendUserId(UUID friendUserId) { this.friendUserId = friendUserId; }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof FriendRelationshipId other)) return false;
        return Objects.equals(userId, other.userId) && Objects.equals(friendUserId, other.friendUserId);
    }

    @Override
    public int hashCode() {
        return Objects.hash(userId, friendUserId);
    }

    @Override
    public String toString() {
        return "FriendRelationshipId{userId=" + userId + ", friendUserId=" + friendUserId + "}";
    }
}
```

`FriendRelationship.java` fields (all manual getters/setters, no Lombok):
- `@Id @Column(name = "user_id", nullable = false, updatable = false) private UUID userId;`
- `@Id @Column(name = "friend_user_id", nullable = false, updatable = false) private UUID friendUserId;`
- `@Enumerated(EnumType.STRING) @Column(name = "status", nullable = false, length = 20) private FriendRelationshipStatus status;` — Hibernate stores enum names by default but we want the lowercase value. **Important: use `@Convert` or store as `String`-backed field with manual conversion in getters/setters.** Decision: use a `@Convert(converter = FriendRelationshipStatusConverter.class)` attribute converter (similar to `DisplayNamePreferenceConverter`) — see "Subfile" below.
- `@Column(name = "created_at", nullable = false, insertable = false, updatable = false) private OffsetDateTime createdAt;` — DB default `NOW()` populates; entity reads only.

Constructors:
- `protected FriendRelationship()` — JPA default
- `public FriendRelationship(UUID userId, UUID friendUserId, FriendRelationshipStatus status)` — application-side constructor; `createdAt` is set by DB on insert (entity refreshes it via `@PostPersist` if needed by callers; otherwise re-fetch)

Equals/hashCode by composite key (both fields):
```java
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof FriendRelationship other)) return false;
    return Objects.equals(userId, other.userId) && Objects.equals(friendUserId, other.friendUserId);
}

@Override
public int hashCode() { return Objects.hash(userId, friendUserId); }
```

**Subfile required**: `backend/src/main/java/com/worshiproom/friends/FriendRelationshipStatusConverter.java` — `@Converter(autoApply = false)` implementing `AttributeConverter<FriendRelationshipStatus, String>`:

```java
@Converter(autoApply = false)
public class FriendRelationshipStatusConverter
        implements AttributeConverter<FriendRelationshipStatus, String> {

    @Override
    public String convertToDatabaseColumn(FriendRelationshipStatus attribute) {
        return attribute == null ? null : attribute.value();
    }

    @Override
    public FriendRelationshipStatus convertToEntityAttribute(String dbData) {
        return dbData == null ? null : FriendRelationshipStatus.fromValue(dbData);
    }
}
```

The entity field declaration becomes:
```java
@Convert(converter = FriendRelationshipStatusConverter.class)
@Column(name = "status", nullable = false, length = 20)
private FriendRelationshipStatus status;
```

**Guardrails (DO NOT):**
- Do NOT use `cascade = CascadeType.ALL` on any relationship — DB has `ON DELETE CASCADE`; double-cascade confuses Hibernate (per spec § "What to Watch For" #9).
- Do NOT use `@EmbeddedId` — codebase convention is `@IdClass`.
- Do NOT make `createdAt` writable — DB default `NOW()` is the source of truth.
- Do NOT add `@PrePersist` to set `userId`/`friendUserId` — caller-supplied composite PK must come from the service.

- [ ] Code compiles: `./mvnw compile`

**Test specifications:**
None directly; entity is exercised by repository + service tests later.

- [ ] `FriendRelationship.java` and `FriendRelationshipId.java` and `FriendRelationshipStatusConverter.java` exist

```java
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Modifying;
import org.springframework.data.jpa.repository.Query;
import org.springframework.data.repository.query.Param;
import org.springframework.stereotype.Repository;

import java.time.OffsetDateTime;
import java.util.List;
import java.util.Optional;
import java.util.UUID;

@Repository
public interface FriendRelationshipRepository
        extends JpaRepository<FriendRelationship, FriendRelationshipId> {

    List<FriendRelationship> findAllByUserIdAndStatus(UUID userId, FriendRelationshipStatus status);

    Optional<FriendRelationship> findByUserIdAndFriendUserId(UUID userId, UUID friendUserId);

    boolean existsByUserIdAndFriendUserIdAndStatus(
        UUID userId, UUID friendUserId, FriendRelationshipStatus status);

    /**
     * True when EITHER (a → b) OR (b → a) has status = BLOCKED. Used to gate
     * sendRequest and acceptRequest. JPQL because the comparison fits.
     */
    @Query("""
        SELECT COUNT(fr) > 0 FROM FriendRelationship fr
        WHERE ((fr.userId = :a AND fr.friendUserId = :b)
            OR (fr.userId = :b AND fr.friendUserId = :a))
          AND fr.status = com.worshiproom.friends.FriendRelationshipStatus.BLOCKED
        """)
    boolean isEitherDirectionBlocked(@Param("a") UUID a, @Param("b") UUID b);

    /**
     * Bulk delete both directions of a friendship in one statement. Used by
     * removeFriend and as the cleanup step in blockUser.
     * Returns the number of rows deleted (0, 1, or 2).
     */
    @Modifying
    @Query("""
        DELETE FROM FriendRelationship fr
        WHERE (fr.userId = :a AND fr.friendUserId = :b)
           OR (fr.userId = :b AND fr.friendUserId = :a)
        """)
    int deleteBothDirections(@Param("a") UUID a, @Param("b") UUID b);

    /**
     * Denormalized friends-list query (Decision 8 mandate).
     *
     * <p>Single SQL execution returns one row per ACTIVE friend with all
     * fields the FriendDto needs. The COALESCE clauses provide defaults
     * for users with no faith_points row, no streak_state row, or no
     * activity_log rows in the current week.
     *
     * <p>Filters: ACTIVE status only, friend not deleted, friend not banned.
     * Order: most-recently-active first (NULLS LAST), then alphabetical
     * by first name as a tiebreak.
     *
     * <p>Native query because the windowed SUM subquery on activity_log
     * is awkward to express in JPQL. Parameter binding is parameterized.
     */
    @Query(value = """
        SELECT
            u.id                       AS friend_user_id,
            u.first_name               AS first_name,
            u.last_name                AS last_name,
            u.display_name_preference  AS display_name_preference,
            u.custom_display_name      AS custom_display_name,
            u.avatar_url               AS avatar_url,
            u.last_active_at           AS last_active_at,
            COALESCE(fp.current_level, 1)  AS level,
            COALESCE(fp.total_points, 0)   AS faith_points,
            COALESCE(ss.current_streak, 0) AS current_streak,
            COALESCE((
                SELECT SUM(al.points_earned)
                FROM activity_log al
                WHERE al.user_id = u.id
                  AND al.occurred_at >= :weekStart
            ), 0) AS weekly_points
        FROM friend_relationships fr
            JOIN users u ON u.id = fr.friend_user_id
            LEFT JOIN faith_points fp ON fp.user_id = u.id
            LEFT JOIN streak_state ss ON ss.user_id = u.id
        WHERE fr.user_id = :userId
          AND fr.status = 'active'
          AND u.is_deleted = FALSE
          AND u.is_banned = FALSE
        ORDER BY u.last_active_at DESC NULLS LAST, u.first_name ASC
        """,
        nativeQuery = true)
    List<FriendsListProjection> findFriendsListForUser(
        @Param("userId") UUID userId,
        @Param("weekStart") OffsetDateTime weekStart);

    /**
     * Search users by name prefix, excluding the caller, blocked-by-caller,
     * blocked-the-caller, deleted, and banned users.
     *
     * <p>JPQL with LIKE concatenation via parameter binding (no SQL injection
     * vector). Limit applied at the query level via Pageable in the caller, OR
     * the service can wrap this in a Pageable query. Decision: service uses
     * `org.springframework.data.domain.PageRequest.of(0, limit)` and calls
     * the variant that takes Pageable for cleaner result-cap semantics.
     *
     * <p>Subqueries for blocked-relationship exclusion: the service computes
     * "users I blocked" and "users who blocked me" in two NOT EXISTS subqueries.
     */
    @Query("""
        SELECT u FROM com.worshiproom.user.User u
        WHERE u.id <> :actingUserId
          AND u.isDeleted = FALSE
          AND u.isBanned = FALSE
          AND (LOWER(u.firstName) LIKE LOWER(CONCAT(:q, '%'))
            OR LOWER(u.lastName) LIKE LOWER(CONCAT(:q, '%')))
          AND NOT EXISTS (
              SELECT 1 FROM FriendRelationship fr
              WHERE fr.userId = :actingUserId
                AND fr.friendUserId = u.id
                AND fr.status = com.worshiproom.friends.FriendRelationshipStatus.BLOCKED)
          AND NOT EXISTS (
              SELECT 1 FROM FriendRelationship fr
              WHERE fr.userId = u.id
                AND fr.friendUserId = :actingUserId
                AND fr.status = com.worshiproom.friends.FriendRelationshipStatus.BLOCKED)
        ORDER BY u.firstName ASC
        """)
    List<com.worshiproom.user.User> searchUsersExcludingBlocks(
        @Param("actingUserId") UUID actingUserId,
        @Param("q") String q,

}
```

```java
@Transactional
public FriendRequest sendRequest(UUID fromUserId, UUID toUserId, String message) {
    // Validation order per spec § Operation 4
    if (fromUserId.equals(toUserId)) {
        throw new SelfActionException("Cannot send friend request to yourself");
    }
    User target = userRepository.findById(toUserId)
        .filter(u -> !u.isDeleted() && !u.isBanned())
        .orElseThrow(() -> new UserNotFoundException("Target user not found"));
    if (relationshipRepo.isEitherDirectionBlocked(fromUserId, toUserId)) {
        throw new BlockedUserException("Cannot send friend request to a blocked user");
    }
    if (relationshipRepo.existsByUserIdAndFriendUserIdAndStatus(
            fromUserId, toUserId, FriendRelationshipStatus.ACTIVE)) {
        throw new AlreadyFriendsException("Already friends with this user");
    }
    FriendRequest req = new FriendRequest(fromUserId, toUserId, message);
    try {
        return requestRepo.saveAndFlush(req);  // saveAndFlush forces immediate UNIQUE check
    } catch (DataIntegrityViolationException e) {
        if (isUniqueConstraintViolation(e)) {
            throw new DuplicateFriendRequestException(
                "A friend request from this sender to this recipient already exists");
        }
        throw e;  // Unrelated integrity violation — let it propagate
    }
}

private boolean isUniqueConstraintViolation(DataIntegrityViolationException e) {
    Throwable cur = e;
    while (cur != null) {
        String msg = cur.getMessage();
        if (msg != null && msg.contains(UNIQUE_CONSTRAINT_NAME)) {
            return true;
        }
        if (cur.getCause() == cur) break;
        cur = cur.getCause();
    }
    return false;
}
```

**Why `saveAndFlush`**: the default `save` defers INSERT until transaction commit, which means `DataIntegrityViolationException` is thrown OUT of the service method (after the `try`/`catch`). `saveAndFlush` forces the INSERT immediately so the catch block can translate the constraint violation into `DuplicateFriendRequestException`. This is the canonical pattern for catch-and-translate of constraint violations in Spring Data JPA.

#### Operation 5: `acceptRequest(UUID requestId, UUID actingUserId) : void`

```java
@Transactional
public void acceptRequest(UUID requestId, UUID actingUserId) {
    FriendRequest req = requestRepo.findById(requestId)
        .orElseThrow(() -> new FriendRequestNotFoundException("Friend request not found"));
    if (!req.getToUserId().equals(actingUserId)) {
        throw new UnauthorizedActionException("Only the recipient can accept this request");
    }
    if (req.getStatus() != FriendRequestStatus.PENDING) {
        throw new InvalidRequestStateException(
            "Request is not pending; cannot accept (current status: " + req.getStatus().value() + ")");
    }
    // Re-check block AFTER the original send — guards against a block placed
    // between send and accept (D8 — see spec § What to Watch For #7)
    if (relationshipRepo.isEitherDirectionBlocked(req.getFromUserId(), req.getToUserId())) {
        throw new BlockedUserException("Block in place; cannot accept friend request");
    }
    OffsetDateTime now = OffsetDateTime.now(ZoneOffset.UTC);
    req.setStatus(FriendRequestStatus.ACCEPTED);
    req.setRespondedAt(now);
    requestRepo.save(req);
    // Insert mutual friendship: A→B and B→A
    relationshipRepo.save(new FriendRelationship(
        req.getFromUserId(), req.getToUserId(), FriendRelationshipStatus.ACTIVE));
    relationshipRepo.save(new FriendRelationship(
        req.getToUserId(), req.getFromUserId(), FriendRelationshipStatus.ACTIVE));
}
```

**Transactional rollback**: any exception thrown during the two `save` calls (e.g., FK violation if a user was deleted mid-flight) rolls back the request status update too — `@Transactional` is the boundary.

```java
    if (relationshipRepo.existsByUserIdAndFriendUserIdAndStatus(
            blockerId, blockedId, FriendRelationshipStatus.BLOCKED)) {
        return;
    }
    // (a) Delete any existing relationships in either direction
    relationshipRepo.deleteBothDirections(blockerId, blockedId);
    // (b) Delete pending requests in either direction (Divergence 3)
    requestRepo.deletePendingBetween(blockerId, blockedId);
    // (c) Insert single block row from blocker → blocked
    relationshipRepo.save(new FriendRelationship(
        blockerId, blockedId, FriendRelationshipStatus.BLOCKED));
}
```

#### Operation 10: `unblockUser(UUID blockerId, UUID blockedId) : void`

```java
@Transactional
public void unblockUser(UUID blockerId, UUID blockedId) {
    if (blockerId.equals(blockedId)) {
        throw new SelfActionException("Cannot unblock yourself");
    }
    boolean wasBlocked = relationshipRepo.existsByUserIdAndFriendUserIdAndStatus(
        blockerId, blockedId, FriendRelationshipStatus.BLOCKED);
    if (!wasBlocked) {
        throw new NotBlockedException("This user is not blocked");
    }
    relationshipRepo.deleteBothDirections(blockerId, blockedId);
    // Note: prior friendship is NOT restored (per spec § Operation 10).
    // FULL UNIQUE on friend_requests means prior request rows still block re-send.
}
```
