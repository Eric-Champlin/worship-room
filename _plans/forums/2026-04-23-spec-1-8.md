**Test profile gap (spec-uncovered, resolved in Step 6):**

The spec's Current-state item #2 says "`application-test.properties` exists". Reconnaissance shows this is false — no such file exists. Tests run under the `dev` profile by default (`spring.profiles.default=dev` in `application.properties`; only `RateLimitIntegrationTest` has `@ActiveProfiles("dev")`, the rest rely on the default). Consequences:

- If `spring.liquibase.contexts=dev` is added ONLY to `application-dev.properties` (Step 4), then tests inherit it too, which means `dev-seed.xml` WOULD apply during test runs. That breaks isolation.
- Liquibase's context filter semantics: empty/unset filter runs ALL changesets (including `context="dev"`); a non-matching filter (e.g., `contexts=test`) skips changesets whose context doesn't match and applies only the no-context ones. Creating a new `application-test.properties` + wiring `@ActiveProfiles("test")` onto every base class is invasive.
- Chosen solution: override `spring.liquibase.contexts` to `test` in both `AbstractIntegrationTest` and `AbstractDataJpaTest` via `@DynamicPropertySource`. This keeps the dev profile semantics intact for tests (DB URL, JWT fallback) but explicitly pins the Liquibase context filter to `test`, excluding `dev-seed.xml` from test runs. The method is named `registerLiquibaseTestContext` to avoid clashing with `registerBaseDatasourceProperties` (Spring aggregates across the inheritance hierarchy — both run).

## Database Changes

| Change | Table | Liquibase Changeset | Notes |
|--------|-------|---------------------|-------|
| INSERT × 5 | `users` | `contexts/dev-seed.xml` (changeset id `2026-04-23-003-dev-seed-users`, context `dev`) | 5 deterministic-UUID users; all share BCrypt hash of `WorshipRoomDev2026!` at strength 10; CDATA-wrapped hash column; fixed `joined_at` `2026-01-15T10:00:00Z`. No schema changes. |

**Migration safety:** Zero-downtime. Seed runs only when `spring.liquibase.contexts=dev` (local dev profile). Prod runs with no context filter — wait, this needs clarification: prod currently has NO `spring.liquibase.contexts` setting, which means ALL changesets including `context="dev"` would run in prod. **Step 5 closes this gap** by explicitly setting `spring.liquibase.contexts=` (empty) in `application-prod.properties` — but an empty value STILL runs all changesets per Liquibase semantics. The actual defense: set `spring.liquibase.contexts=prod` in prod (a value that doesn't match `dev`) OR rely on platform env var override. To avoid introducing a "prod" context convention mid-phase, the defensive approach is `spring.liquibase.contexts=none` (a sentinel value that no changeset uses) — this causes Liquibase to run ONLY no-context changesets in prod. Final choice: **`spring.liquibase.contexts=production`** — a descriptive sentinel that no seed changeset will ever use, making the intent self-documenting. This is belt-and-suspenders; the ACTUAL protection is that prod deploys set `SPRING_PROFILES_ACTIVE=prod` via platform env, activating `application-prod.properties`.

**Rollback strategy:** Seed is a data-only change. The canonical "rollback" is `docker compose down -v` (wipe volume) followed by a restart. No Liquibase `<rollback>` block — matches Decision 10 rule 3 ("Rollback blocks only for non-destructive changes"): deleting populated users rows isn't cleanly reversible mid-development.

## Edge Cases & Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| BCrypt hash storage in changeset XML | CDATA inline, single-line per row | Per master plan Decision 2 recommendation. Single-line CDATA (`<column name="password_hash"><![CDATA[$2a$10$...]]></column>`) avoids whitespace preservation issues that multi-line CDATA would introduce. Liquibase does NOT template-expand content inside CDATA, so `$` characters survive verbatim. |
| Same hash for all 5 users or 5 distinct hashes | Same hash reused 5 times | All 5 users share the same password `WorshipRoomDev2026!`. BCrypt is non-deterministic (random salt), so 5 `encode()` calls produce 5 different hashes — but all 5 hashes would validate the same plaintext via `matches()`. Reusing one hash across all 5 rows produces a smaller diff, easier visual verification, and identical functional behavior. |
| Deterministic UUID pattern | `00000000-0000-0000-0000-00000000000N` for N=1..5 | Per spec Key Design Decision 3. Visually obvious as seed UUIDs in DB queries and logs; reproducible across volume wipes; frontend dev against `/u/:username` (Phase 8.1) can bookmark specific users. |
| `joined_at` value | Fixed `2026-01-15T10:00:00Z` | Reproducibility across volume wipes; deterministic for any future test that asserts on seed user's `joined_at`. 3 months before the seed changeset runs (2026-04-23), plausible. |
| `created_at` / `updated_at` | Omitted from INSERT; DB defaults to `NOW()` | The `users` table has `defaultValueComputed="NOW()"` on both columns. These represent row creation in infrastructure terms, not business terms — intentionally drift from `joined_at`. Matches spec Gotcha #6. |
| `last_active_at` | NULL (not set) | Nullable column. Seed users have never "logged in" until Eric manually does so in dev; no meaningful value to set. |
| `bio` / `avatar_url` / `favorite_verse_reference` / `favorite_verse_text` | Populated on users 2 (Sarah) and 3 (Bob); NULL on users 1 (Admin), 4 (Mikey), 5 (Sakura) | Per spec roster table. Exercises profile display fields on a subset. |
| Test isolation mechanism | Override `spring.liquibase.contexts=test` in `AbstractIntegrationTest` + `AbstractDataJpaTest` via `@DynamicPropertySource` | See "Test profile gap" note in Architecture Context. Alternative of creating `application-test.properties` + `@ActiveProfiles("test")` on every base class is more invasive. The override pattern is already used by `AuthControllerIntegrationTest` for `jwt.secret` and `auth.rate-limit.*` — same discipline, new property. |
| Prod context defense | Set `spring.liquibase.contexts=production` in `application-prod.properties` | `production` is a sentinel value that matches no existing changeset context. Per Liquibase semantics, changesets with no context filter pass; changesets with `context="dev"` are skipped. Belt-and-suspenders with Decision 10's expected prod-profile activation. |
| Rollback block on seed changeset | None | Decision 10 rule 3: "Rollback blocks only for non-destructive changes." Seed data is destructive to reverse cleanly (DELETE WHERE id IN (...) works in principle but adds maintenance burden for zero benefit — `docker compose down -v` is the canonical reset path in dev). Matches the spec's idempotency approach. |
| Pre-condition on seed changeset | Add `<preConditions onFail="HALT">` checking users table exists | Per spec Key Design Decision 8. Belt-and-suspenders with the `<include>` ordering in `master.xml` (seed comes after users table creation). |
| BCrypt hash generation timing | Execution-time via throwaway `@Test` method in Step 1 | Planning cannot safely embed a real hash (BCrypt's random salt means no stable hash is available without running Java). Step 1 runs a one-off test, captures hash from stdout, embeds it into the changeset, then deletes the throwaway test. |

### Step 1: Generate a real BCrypt hash for `WorshipRoomDev2026!` at strength 10

**Objective:** Produce a real, validated BCrypt hash to embed in the dev-seed changeset. Spec Gotcha #1 is explicit: a placeholder hash causes every seed login to return 401 with no obvious cause.

**Files to create/modify:**
- `backend/src/test/java/com/worshiproom/auth/DevSeedBcryptHashGenerator.java` — THROWAWAY test class. Single `@Test` method prints the hash and asserts `matches()` round-trip. Deleted at end of Step 2.

**Details:**

Create a one-off test class (NOT extending any base — no container needed, no Spring context):

```java
package com.worshiproom.auth;

import org.junit.jupiter.api.Test;
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
import static org.assertj.core.api.Assertions.assertThat;

class DevSeedBcryptHashGenerator {

    private static final String DEV_PASSWORD = "WorshipRoomDev2026!";

    @Test
    void generateAndPrintHash() {
        BCryptPasswordEncoder encoder = new BCryptPasswordEncoder();
        String hash = encoder.encode(DEV_PASSWORD);

        System.out.println("========================================");
        System.out.println("DEV SEED BCRYPT HASH");
        System.out.println(hash);
        System.out.println("========================================");

        assertThat(hash).startsWith("$2a$10$");
        assertThat(encoder.matches(DEV_PASSWORD, hash)).isTrue();
    }
}
```

Run: `./mvnw test -Dtest=DevSeedBcryptHashGenerator`. Capture the printed hash from stdout (format `$2a$10$<53 chars>`). Record this hash verbatim — it gets embedded in 5 places in Step 2.

**Guardrails (DO NOT):**
- Do NOT embed a placeholder hash like `$2a$10$placeholderplaceholderplaceholder` — spec Gotcha #1 is explicit.
- Do NOT use `BCryptPasswordEncoder(12)` or any non-default strength. `PasswordConfig.java` uses the no-arg constructor (strength 10). A mismatch would make `matches()` succeed (BCrypt stores strength in the hash) but is confusing and drifts from Spec 1.4.
- Do NOT commit this file — delete at the start of Step 2.
- Do NOT regenerate the hash between test runs unnecessarily — capture ONCE in Step 1, reuse the same hash across all 5 rows in Step 2.

- [ ] Stdout shows a hash starting with `$2a$10$`.
- [ ] Round-trip assertion (`encoder.matches(...)`) returns true.

### Step 5: Add defensive `spring.liquibase.contexts=production` to `application-prod.properties`

### Step 6: Override `spring.liquibase.contexts=test` in `AbstractIntegrationTest` and `AbstractDataJpaTest`

**Objective:** Prevent the dev-seed from applying during Testcontainers-backed tests. Closes the test-profile gap identified during recon.

**Files to create/modify:**
- `backend/src/test/java/com/worshiproom/support/AbstractIntegrationTest.java` — add one new `@DynamicPropertySource` method.
- `backend/src/test/java/com/worshiproom/support/AbstractDataJpaTest.java` — add one new `@DynamicPropertySource` method.

**Details:**

Both base classes already have a `registerBaseDatasourceProperties` method for JDBC properties. Per the class-level JavaDoc ("Subclasses that need additional dynamic properties may declare their own static `@DynamicPropertySource` method with a distinct name — Spring aggregates across the inheritance hierarchy"), we add a second static method with a new distinct name. Spring runs both.

**`AbstractIntegrationTest.java` target state** (additions only, existing code unchanged):

```java
@SpringBootTest
public abstract class AbstractIntegrationTest {

    @DynamicPropertySource
    static void registerBaseDatasourceProperties(DynamicPropertyRegistry registry) {
        TestContainers.registerJdbcProperties(registry);
    }

    /**
     * Override spring.liquibase.contexts to "test" so the dev-only seed changeset
     * (Spec 1.8, contexts/dev-seed.xml, context="dev") is SKIPPED during test runs.
     *
     * <p>Tests inherit the dev profile by default via {@code spring.profiles.default=dev}
     * in {@code application.properties}, which would otherwise apply the dev-seed to the
     * shared singleton container and leak 5 user rows across every integration test class.
     *
     * <p>Liquibase context semantics: when the filter is set to "test" (a value no changeset
     * uses), only no-context changesets run; every changeset with an explicit context — dev-seed
     * included — is skipped. Future `contexts/test-seed.xml` would use {@code context="test"} if
     * ever added; this override would then include it automatically.
     */
    @DynamicPropertySource
    static void registerLiquibaseTestContext(DynamicPropertyRegistry registry) {
        registry.add("spring.liquibase.contexts", () -> "test");
    }
}
```

**`AbstractDataJpaTest.java` target state** (same additions):

```java
@DataJpaTest
@AutoConfigureTestDatabase(replace = Replace.NONE)
public abstract class AbstractDataJpaTest {

    @DynamicPropertySource
    static void registerBaseDatasourceProperties(DynamicPropertyRegistry registry) {
        TestContainers.registerJdbcProperties(registry);
    }

    /**
     * See {@link AbstractIntegrationTest#registerLiquibaseTestContext} — same rationale;
     * @DataJpaTest boots a Liquibase-integrated Spring context and must also skip the
     * dev-seed to keep the singleton container clean across test classes.
     */
    @DynamicPropertySource
    static void registerLiquibaseTestContextForJpaSlice(DynamicPropertyRegistry registry) {
        registry.add("spring.liquibase.contexts", () -> "test");
    }
}
```

Note the distinct method names (`registerLiquibaseTestContext` in `AbstractIntegrationTest`, `registerLiquibaseTestContextForJpaSlice` in `AbstractDataJpaTest`) — per the existing JavaDoc, method-name collision with the base's `registerBaseDatasourceProperties` would be static-method-hiding. Different names for different base classes is purely cosmetic here, but keeps the intent clear.

**Guardrails (DO NOT):**
- Do NOT reuse the method name `registerBaseDatasourceProperties` — static method hiding makes the intent ambiguous and Spring's aggregation behavior breaks.
- Do NOT set the property to an empty string (`registry.add("spring.liquibase.contexts", () -> "")`) — empty means "no filter, run everything", which would RE-ENABLE the dev-seed in tests. Defeats the entire purpose.
- Do NOT add the override only to `AbstractIntegrationTest` — `AbstractDataJpaTest` also boots Liquibase and mutates the shared singleton container; it needs the same guard.
- Do NOT set `@ActiveProfiles("test")` on the base classes — that would require also creating `application-test.properties` and wiring it with JDBC properties, a much larger change with no additional isolation benefit.

**Verification:**

Crucially: the dev-seed changeset does NOT appear in `DATABASECHANGELOG` rows when inspecting the test container (Step 7's isolation test will assert this).

Existing test suite verification (no new tests at this step). `DevSeedContextIsolationTest` in Step 7 covers the new behavior.

### Step 7: Create `DevSeedContextIsolationTest`

**Objective:** Add a regression test that fails if any of the following happens: (a) the `context="dev"` attribute is accidentally removed from `contexts/dev-seed.xml`, (b) the test-profile override breaks and seeds leak into tests, (c) `contexts/dev-seed.xml` is unwired from `master.xml`.

**Files to create/modify:**
- `backend/src/test/java/com/worshiproom/db/DevSeedContextIsolationTest.java` — NEW test file.

**Details:**

```java
package com.worshiproom.db;

import com.worshiproom.support.AbstractIntegrationTest;
import com.worshiproom.user.UserRepository;
import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.core.io.ClassPathResource;
import org.w3c.dom.Document;
import org.w3c.dom.Element;
import org.w3c.dom.NodeList;

import javax.xml.parsers.DocumentBuilder;
import javax.xml.parsers.DocumentBuilderFactory;
import java.util.UUID;

import static org.assertj.core.api.Assertions.assertThat;

/**
 * Drift-detection test for Spec 1.8 dev-seed isolation.
 *
 * <p>Asserts three invariants:
 * <ol>
 *   <li>The dev-seed XML file exists at the expected classpath location.</li>
 *   <li>Every changeSet inside dev-seed.xml declares {@code context="dev"}. Without this
 *       attribute the seed would run in ALL environments, including production.</li>
 *   <li>With {@code spring.liquibase.contexts=test} (pinned by {@link AbstractIntegrationTest}),
 *       the 5 deterministic-UUID seed users do NOT appear in the test database. This catches
 *       both test-profile-override regressions and context-attribute regressions in one assert.</li>
 * </ol>
 *
 * <p>Not a smoke test of the seed itself — that's exercised manually by Eric via
 * {@code ./mvnw spring-boot:run -Dspring-boot.run.profiles=dev} per Spec 1.8 success criteria.
 */
class DevSeedContextIsolationTest extends AbstractIntegrationTest {

    private static final UUID[] SEED_UUIDS = {
        UUID.fromString("00000000-0000-0000-0000-000000000001"),
        UUID.fromString("00000000-0000-0000-0000-000000000002"),
        UUID.fromString("00000000-0000-0000-0000-000000000003"),
        UUID.fromString("00000000-0000-0000-0000-000000000004"),
        UUID.fromString("00000000-0000-0000-0000-000000000005"),
    };

    @Autowired
    private UserRepository userRepository;

    @Test
    void seedUsersAreAbsentInTestContext() {
        // With spring.liquibase.contexts=test pinned by the base class, the dev-seed
        // changeset (context="dev") must be skipped. Consequently none of the 5 seed
        // UUIDs should exist in the users table.
        for (UUID seedId : SEED_UUIDS) {
            assertThat(userRepository.findById(seedId))
                .as("Seed UUID %s must NOT be present in test context — dev-seed leaked.", seedId)
                .isEmpty();
        }
    }

    @Test
    void devSeedXmlDeclaresDevContext() throws Exception {
        ClassPathResource resource = new ClassPathResource("db/changelog/contexts/dev-seed.xml");
        assertThat(resource.exists())
            .as("contexts/dev-seed.xml must exist at the expected classpath location")
            .isTrue();

        DocumentBuilder builder = DocumentBuilderFactory.newInstance().newDocumentBuilder();
        Document doc = builder.parse(resource.getInputStream());

        NodeList changesets = doc.getElementsByTagNameNS("*", "changeSet");
        assertThat(changesets.getLength())
            .as("contexts/dev-seed.xml should contain at least one <changeSet>")
            .isGreaterThanOrEqualTo(1);

        for (int i = 0; i < changesets.getLength(); i++) {
            Element cs = (Element) changesets.item(i);
            assertThat(cs.getAttribute("context"))
                .as("Every changeSet in contexts/dev-seed.xml MUST declare context='dev'")
                .isEqualTo("dev");
        }
    }

    @Test
    void masterXmlIncludesDevSeed() throws Exception {
        ClassPathResource master = new ClassPathResource("db/changelog/master.xml");
        assertThat(master.exists()).isTrue();

        DocumentBuilder builder = DocumentBuilderFactory.newInstance().newDocumentBuilder();
        Document doc = builder.parse(master.getInputStream());

        NodeList includes = doc.getElementsByTagNameNS("*", "include");
        boolean devSeedWired = false;
        for (int i = 0; i < includes.getLength(); i++) {
            Element inc = (Element) includes.item(i);
            if ("db/changelog/contexts/dev-seed.xml".equals(inc.getAttribute("file"))) {
                devSeedWired = true;
                break;
            }
        }
        assertThat(devSeedWired)
            .as("master.xml must include db/changelog/contexts/dev-seed.xml")
            .isTrue();
    }
}
```

**Guardrails (DO NOT):**
- Do NOT put a `@BeforeEach userRepository.deleteAll()` — Step 6's context override means seed rows never arrive in the first place; a cleanup step would mask a test-profile-override regression.
- Do NOT use raw JDBC to check for the seed UUIDs — `UserRepository.findById(UUID)` is more expressive and the repository already exists.
- Do NOT assert on count (`userRepository.count() == 0`) — adjacent test classes in the same JVM run may leave rows in the shared singleton container; count-based assertions are brittle. UUID-specific lookups are immune.
- Do NOT use `ClassLoader.getSystemResource("...")` or raw file paths — `ClassPathResource` is the Spring-idiomatic mechanism and aligns with how Liquibase itself resolves files.
- Do NOT use a namespaced XPath expression that hardcodes the Liquibase namespace URI — `getElementsByTagNameNS("*", "changeSet")` matches regardless of namespace prefix, which is more resilient to namespace changes.

**Verification:**
- [ ] `./mvnw test -Dtest=DevSeedContextIsolationTest` passes.
- [ ] All three test methods pass individually.
- [ ] `./mvnw test` full suite passes with the new test included.
- [ ] Negative-path sanity check (ONLY if time permits, and revert immediately): temporarily remove `context="dev"` from `contexts/dev-seed.xml` and re-run `./mvnw test -Dtest=DevSeedContextIsolationTest#devSeedXmlDeclaresDevContext` — expect a failure. Re-add the attribute.
